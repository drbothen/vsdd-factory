## 5. Cross-hook observations (shared envelope schema, exit-code patterns)

### 5.1 Shared `_emit` helper pattern

Every hook except `session-learning.sh`, `validate-index-self-reference.sh`, `warn-pending-wave-gate.sh` (3 of 44) defines an `_emit` function that:

1. Checks `${CLAUDE_PLUGIN_ROOT:-}` non-empty AND `${CLAUDE_PLUGIN_ROOT}/bin/emit-event` is executable.
2. Calls `${CLAUDE_PLUGIN_ROOT}/bin/emit-event "$@" 2>/dev/null || true`.
3. Always returns 0.

This is a **fail-tolerant shared sink** abstraction enforced by convention, not by a sourced library.

### 5.2 Standard event-payload key set (hook.block / hook.action / agent.start / agent.stop / commit.made / pr.opened / pr.merged)

Every block emission carries: `type=hook.block hook=<name> matcher=<event-or-tool> reason=<code> [file_path|command] [story_id|subagent|...]`.

- `severity=warn` is set on warn-only blocks (regression-gate, handoff-validator, purity-check, convergence-tracker warnings, validate-state-index-status-coherence, warn-pending-wave-gate, validate-input-hash, validate-anchor-capabilities-union missing-BC notes).
- Block reasons (block codes) are unique per script and stable identifiers — total of **49 distinct reason codes** observed across all hooks (e.g., `ai_attribution_coauthored`, `git_reset_hard`, `policy7_bc_title_mismatch`, `wave_gate_incomplete`).

### 5.3 stdin contract & jq parsing pattern

- Every hook reads JSON from stdin via `INPUT=$(cat)`.
- Every hook (except `session-learning.sh`, `warn-pending-wave-gate.sh`) extracts via `jq -r '.tool_input.X // empty'` or `jq -r '.tool_response.Y // ""'`.
- Most hooks: missing jq → silent exit 0.
- Hard-fail hooks (jq missing → exit 1): `brownfield-discipline.sh`, `protect-bc.sh`, `protect-vp.sh`, `red-gate.sh`. These four register on_error=block, so jq-absent is a hard block.
- `validate-vp-consistency.sh` prints jq-missing message but exits 0 — soft-fail.

### 5.4 Exit-code semantics

- **0** = pass / advisory / non-applicable. Used by every hook on the happy path. Used as the "warn-only" exit by convergence-tracker (warn case), regression-gate, purity-check, handoff-validator, validate-input-hash on advisory paths, validate-index-self-reference, capture-*, track-*, session-learning, warn-pending-wave-gate, validate-anchor-capabilities-union (missing BCs warn), update-wave-state-on-merge.
- **1** = special case used by 2 distinct paths:
  - jq-missing-fail-closed: brownfield-discipline, protect-bc, red-gate (state file parse error path).
  - explicitly-warn: validate-state-index-status-coherence (only hook to use exit 1 as a documented "warning, not block" semantic — script header line 13 says so explicitly).
- **2** = block / deny / violation. The dispatcher's standard block exit code.

> Legacy finding-ID examples in section 5 commentary are deliberately omitted from this round's prose; see BC-AUDIT-1109/1110 for the canonical format rules and the hook source for the regex.

### 5.5 Duplicate `protect-secrets` registration

Only multi-registration in the registry: `hooks/protect-secrets.sh` is registered as both:
- `[[hooks]] name="protect-secrets" event="PreToolUse" tool="Bash" priority=80`
- `[[hooks]] name="protect-secrets" event="PreToolUse" tool="Read" priority=80`

Same script handles both via tool dispatch in body (BC-AUDIT-1052). All other 43 scripts are registered exactly once.

### 5.6 PostToolUse `additionalContext` JSON pattern

Two scripts emit `{"additionalContext": "..."}` JSON to stdout instead of using stderr + exit code:
- `check-factory-commit.sh` (registered as PreToolUse — DRIFT noted in section 4)
- `verify-git-push.sh` on the success path (BC-AUDIT-1089)

This Claude Code envelope feature lets a hook attach context that the next assistant turn sees as part of the input. Used sparingly here because most hooks need to actually block.

### 5.7 PreToolUse permissionDecision JSON envelope

Two scripts use the structured PreToolUse envelope (richer than exit-code blocking):
- `protect-bc.sh` — emits `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"allow|deny","permissionDecisionReason":"..."}}`.
- `protect-vp.sh` — same pattern.

Both ALWAYS exit 0; the deny is encoded in the JSON. This is the only way to surface a structured `reason` to the user that Claude Code echoes back.

### 5.8 Common "set" prologue patterns

- **`set -euo pipefail`** — strictest; used by 32 hooks (most validators + all PreToolUse blockers).
- **`set +e`** — explicitly tolerant; used by `capture-commit-activity.sh`, `capture-pr-activity.sh`, `track-agent-start.sh`, `track-agent-stop.sh` (telemetry hooks that must never block on a parse error).

Telemetry hooks vs validators: distinct error-handling philosophies inscribed in the prologue.

### 5.9 Path normalization conventions

- `.factory/` path detection is universally `*.factory/*` glob or regex `(^|/)\.factory/`.
- Story ID extraction: `S-[0-9]+\.[0-9]+` regex first; `STORY-[0-9]+` fallback. Used by 5 hooks (validate-pr-merge-prerequisites, validate-wave-gate-prerequisite, update-wave-state-on-merge, track-agent-start, validate-anchor-capabilities-union via different scope).

### 5.10 Bash 3.2 compatibility

`validate-anchor-capabilities-union.sh`, `validate-state-index-status-coherence.sh`, `validate-state-pin-freshness.sh`, and `validate-subsystem-names.sh` explicitly state "no associative arrays" / "Bash 3.2 compatible" in their headers — they target macOS default bash. Other hooks use bash 4+ idioms (`[[ "$VAR" =~ regex ]]` is fine; `declare -A` is the line). Implementation discipline visible.
