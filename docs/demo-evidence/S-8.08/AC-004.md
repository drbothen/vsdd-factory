# AC-004 Evidence — story_id Two-Pattern Extraction Cascade

## Acceptance Criterion

Story ID extraction follows the two-pattern cascade from the bash source:
1. Search prompt for pattern `S-[0-9]+\.[0-9]+` (e.g., `S-8.03`); use first match.
2. If no match, search for `STORY-[0-9]+` (e.g., `STORY-042`); use first match.
3. If neither matches, omit `story_id` field from the event entirely.

Bats test verifies:
- prompt containing "S-6.07" produces `story_id=S-6.07`
- prompt containing "STORY-042" (no S-N.NN) produces `story_id=STORY-042`
- prompt with no story pattern produces event without story_id field

## Evidence

### Rust Extraction Implementation (src/lib.rs lines 48–62)

```rust
pub fn extract_story_id(prompt: &str) -> Option<String> {
    // Pattern 1: S-N.NN (e.g., S-8.03)
    let re1 = regex::Regex::new(r"S-[0-9]+\.[0-9]+").expect("static regex is valid");
    if let Some(m) = re1.find(prompt) {
        return Some(m.as_str().to_string());
    }

    // Pattern 2 (fallback): STORY-NNN (e.g., STORY-042)
    let re2 = regex::Regex::new(r"STORY-[0-9]+").expect("static regex is valid");
    if let Some(m) = re2.find(prompt) {
        return Some(m.as_str().to_string());
    }

    None
}
```

This exactly replicates the bash source (`grep -oE 'S-[0-9]+\.[0-9]+' | head -1` then
`grep -oE 'STORY-[0-9]+' | head -1`). Pattern 1 is tried first; only if it finds nothing
does pattern 2 run. Returns `None` when both fail — the caller omits `story_id` from the
field slice.

### Unit Tests (All Pass)

### EC-003: Pattern 1 Priority

When a prompt contains both `S-N.NN` and `STORY-NNN` patterns, pattern 1 wins.
The cascade returns at the first match of pattern 1, never reaching pattern 2.
