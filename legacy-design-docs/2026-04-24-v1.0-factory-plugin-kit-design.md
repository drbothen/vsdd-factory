# vsdd-factory v1.0 — Factory Plugin Kit

**Date:** 2026-04-24
**Status:** Design ready; open questions resolved; pending decomposition into stories.
**Scope:** Full rearchitecture of the hook system. New compiled Rust dispatcher,
WASM plugin ABI, multi-sink observability layer, cross-platform support
(macOS/Linux/Windows). Version bump from 0.79.x to 1.0.0.
**Prerequisite:** v0.79.4 shipped (last configuration-only workaround for the
`(PostToolUse, Bash)` matcher dispatch bug; still failing per live tests in prism).

## Problem

Three pressures justify a full rearchitecture rather than continued incremental
patching:

**1. Claude Code harness bugs we can't route around in config.**
v0.79.0 → v0.79.4 burned four releases trying to restore `PostToolUse` dispatch
for Bash-matched hooks: reordering the matcher array (v0.79.2), replacing
`"Bash"` with `"*"` (v0.79.3), and dropping the `matcher` field entirely
(v0.79.4 — matching the proven-in-production
[shanraisshan/claude-code-hooks](https://github.com/shanraisshan/claude-code-hooks)
pattern). Upstream
[claude-code#52715](https://github.com/anthropics/claude-code/issues/52715)
reports the mirror-image failure (their `PreToolUse:Bash` is silent while
`PostToolUse:Bash` fires — we're the opposite). A dispatcher that owns its own
routing can ignore the harness's matcher field entirely and filter internally
from the stdin JSON payload.

**2. Cross-platform support is aspirational, not real.**
All 30+ existing hooks are bash. Windows users need git-bash; failure modes are
undiscovered. `find`/`date`/path separators are known Linux-vs-BSD-vs-Windows
divergences scattered through the scripts. Observability scripts assume POSIX
shell. No CI job actually validates Windows.

**3. Observability is hardcoded to local Grafana.**
`emit-event` always writes to `.factory/logs/events-*.jsonl`. Operators who want
to ship logs directly to Datadog/Honeycomb/custom HTTP without the local OTel
collector have no path. Operators who want zero local disk footprint have no
path. The file path is the contract; everything else layers on top of tailing
that file.

## Decisions

- **Compiled Rust dispatcher binary**, shipped per-platform, committed directly
  to the repo by CI on release. Single entry point in `hooks.json`, no matcher
  field, routing internal.
- **WASM plugin ABI (WASI preview 1)** for all hook logic going forward. Each
  hook is a `.wasm` module. Language-agnostic at the plugin boundary; Rust is
  the reference SDK. The SDK is published to crates.io as `vsdd-hook-sdk`.
- **Hook registry in TOML** (`hooks-registry.toml`). Declares which `.wasm`
  fires on which event/tool with what priority/timeout/error-policy/capabilities.
  Adding a hook requires no dispatcher recompile — just drop a `.wasm` and
  update the TOML.
- **Multi-instance, multi-backend observability sinks** (`observability-config.toml`).
  File, HTTP, OTel gRPC, Datadog, Honeycomb as first-class sink types. Operators
  select any combination: local-only, remote-only, hybrid, or multiple instances
  of the same type (e.g., two Datadog orgs). Per-sink filters, tags, batching,
  retry/circuit-breaker resilience, and dead-letter queue on drop.
- **Legacy-bash-adapter plugin during migration.** A single WASM plugin that
  shells out to existing bash scripts (via host-granted `exec_subprocess`
  capability with explicit `shell_bypass_acknowledged`) keeps the 30 current
  hooks working on macOS/Linux while they're ported individually. Windows
  support for a given hook arrives when that hook is ported to pure WASM.
- **Plugins run parallel within priority tiers, sequential between tiers.**
  Priority tiers are the primitive for expressing ordering dependencies;
  within a tier, plugins are independent. Bounded wall-clock per event.
- **Activation-skill-driven platform selection.** Claude Code exposes no
  OS/arch variable expansion in `hooks.json` and no plugin install lifecycle
  hooks; `hooks.json` must be a literal path to the right-platform dispatcher
  binary. The existing activation skill detects OS+arch at activation time and
  copies the matching `hooks.json.<platform>` (CI-generated from a template)
  into place. `hooks.json` is gitignored; source of truth is the template.
- **Dispatcher-internal self-telemetry** at `.factory/logs/dispatcher-internal.jsonl`
  is always-on, regardless of sink configuration. Ensures the dispatcher stays
  debuggable when all configured sinks are misconfigured or down.
- **Staged 1.0 release:** `1.0.0-beta.N` → `1.0.0-rc.N` → `1.0.0`. Semver
  stability promise engages at `1.0.0`, not earlier; breaking changes during
  prereleases are allowed. Dogfood vsdd-factory's own repo on each prerelease
  before promoting to marketplace.
- **Semver promise at 1.0.** Plugin ABI, host function interface, registry
  schema, and observability config schema are stable contracts after 1.0.0
  ships. Breaking changes require a major bump. Separate `HOST_ABI_VERSION`
  tracked by the dispatcher; breaking host ABI = major bump on dispatcher
  and SDK.

### Control flow per tool invocation

```
Claude Code                  dispatcher (Rust)                          WASM plugins                    sinks
-----------                  -----------------                          ------------                    -----
Bash/Edit/Write tool call →
                             receive stdin JSON
                             assign dispatcher_trace_id
                             load hooks-registry.toml (cached, mtime-invalidated)
                             filter plugins by event + tool match
                             group plugins by priority tier

                             for each priority tier (sequential):
                                 spawn tokio tasks in parallel:
                                     instantiate wasmtime store →       on_hook(payload)
                                     apply epoch + fuel limits           emit_event() → queue → fan out to sinks
                                     invoke plugin                                                      (file, http, otel, datadog, honeycomb)
                                     capture result + timing
                                 wait for all tasks in tier to complete or timeout
                                 apply on_error policies
                                 if any 'block' policy triggered → record block intent

                             aggregate tier results
                             emit internal.* lifecycle events
                             exit with combined status (0 or 2)

                             (all emit_event calls also write through to
                              .factory/logs/dispatcher-internal.jsonl regardless
                              of configured sinks)
```

### Plugin ABI (Rust-facing SDK)

```rust
use vsdd_hook_sdk::{hook, HookPayload, HookResult};

#[hook]
pub fn on_hook(payload: HookPayload) -> HookResult {
    // payload.event_name: "PostToolUse"
    // payload.tool_name:  "Bash"
    // payload.tool_input / tool_response: serde_json::Value

    if payload.tool_name != "Bash" {
        return HookResult::continue_();
    }

    let command = payload.tool_input.get("command")?.as_str()?;
    if !regex::match_git_commit(command) {
        return HookResult::continue_();
    }

    let (sha, branch) = parse_commit_preamble(payload.tool_response_stdout())?;

    vsdd_hook_sdk::emit_event("commit.made", &[
        ("commit_sha", sha),
        ("branch", branch),
    ]);

    HookResult::continue_()
}
```

Plugin authors never see the raw WASI ABI — the `#[hook]` macro in `hook-sdk`
generates the `extern "C"` entry point, the JSON deserialization, and the panic
boundary. Plugins always build as `cdylib` targeting `wasm32-wasip1`.

### Host function surface (dispatcher-exposed)

Plugins call these via imports provided by the dispatcher. Capability-limited
by design; **every host function with blocking or resource-unbounded potential
takes explicit size and time bounds at the API level** — no unbounded
overloads.

| Host function | Purpose | Capability policy |
|---|---|---|
| `emit_event(type, [(k, v), …])` | Record observability event | Always allowed |
| `log(level, message)` | Structured log to dispatcher-internal | Always allowed |
| `read_file(path, max_bytes, timeout_ms)` → `Option<Bytes>` | Read from host FS | Only paths under preopened roots; bounded size + time required |
| `exec_subprocess(cmd, args, timeout_ms, max_output_bytes)` → `ExecResult` | Spawn subprocess, argv-only (no shell) | Disabled by default; per-plugin binary allow-list; bounded time + output required |
| `session_id()` | Current Claude session | Always allowed |
| `dispatcher_trace_id()` | Correlation ID for this tool invocation's dispatch | Always allowed |
| `plugin_root()` | Path to plugin's WASM file's dir | Always allowed |
| `plugin_version()` | The plugin's own declared version string | Always allowed |
| `cwd()` | Current working directory | Always allowed |
| `env(var)` → `Option<String>` | Read env var | Allow-list in registry (defaults to `PATH`, `HOME`, `TMPDIR`, `CLAUDE_PROJECT_DIR`, `VSDD_*`) |

**ABI version:** dispatcher declares `HOST_ABI_VERSION: u32`; plugins declare
compatibility in their SDK. Mismatch = plugin refused to load with a loud
error event. Breaking ABI changes require major bump on both dispatcher and
SDK; dispatcher MAY support multiple ABI versions concurrently if we commit
to back-compat.

### `hooks-registry.toml` — routing declaration

```toml
# Format version for future schema migrations
schema_version = 1

# Global defaults (overridable per plugin)
[defaults]
timeout_ms = 5000
on_error = "continue"

[[hooks]]
name = "capture-commit-activity"
event = "PostToolUse"
tool = "Bash"              # regex; omit to match all tools
plugin = "hook-plugins/capture-commit-activity.wasm"
priority = 100             # lower runs first
enabled = true

[[hooks]]
name = "validate-template-compliance"
event = "PostToolUse"
tool = "Edit|Write"
plugin = "hook-plugins/validate-template-compliance.wasm"
timeout_ms = 10000         # override default
on_error = "block"         # this hook's failure should block
priority = 50

[[hooks]]
name = "legacy-bash-adapter"
event = "PostToolUse"
plugin = "hook-plugins/legacy-bash-adapter.wasm"
priority = 900             # run last
enabled = true

# Capability escalation for the legacy adapter — multi-layer defense
[hooks.capabilities.exec_subprocess]
# Binaries this plugin may exec. Resolved to full paths at load time
# to prevent PATH hijacking. Setuid/setgid binaries refused categorically.
binaries = ["git", "jq", "gh"]

# Env vars the subprocess may inherit. Everything else stripped.
env_allowlist = ["PATH", "HOME", "TMPDIR", "CLAUDE_PROJECT_DIR", "VSDD_SESSION_ID"]

# Working directories the subprocess may be spawned in.
cwd_allowlist = ["${CLAUDE_PROJECT_DIR}", "${CLAUDE_PROJECT_DIR}/.factory"]

# Defaults for calls that omit these.
default_timeout_ms = 3000
max_output_bytes = 1048576  # 1 MiB

# Shell interpreter use is explicit and must be justified.
shell_bypass_acknowledged = ["bash"]
shell_bypass_reason = "legacy-bash-adapter executes unported bash hook scripts during v1.0 migration"

[[hooks]]
name = "block-ai-attribution"
event = "PreToolUse"
tool = "Bash"
plugin = "hook-plugins/block-ai-attribution.wasm"
on_error = "block"
```

Sink drivers live in the dispatcher as Rust modules. Supported types at v1.0:

- `file` — append JSONL to a path (current behavior)
- `http` — batched POST to a generic endpoint
- `otel-grpc` — OTel gRPC exporter (for existing local Grafana/Loki/Prometheus stack)
- `datadog` — Datadog Logs Intake API
- `honeycomb` — Honeycomb Events API

Adding a new sink type requires a dispatcher release. Adding a new sink
*instance* (another Datadog, another file) is config-only.

### Failure isolation

- **Sink failure never blocks plugin.** Plugin calls `emit_event()`; dispatcher
  adds the event to each enabled sink's outbound queue; plugin returns
  immediately. Sink failures (network down, auth rejected) are isolated to that
  sink's own worker, surfaced via an `internal.sink_error` event routed to
  all healthy sinks, and always written to `dispatcher-internal.jsonl`.
- **Plugin failure never blocks other plugins.** Each plugin runs in an
  isolated wasmtime store with its own epoch + fuel budgets. Plugins within
  the same priority tier run in parallel tokio tasks; one tokio task panic
  doesn't affect siblings.
- **Plugin failure policy (`on_error`)** governs how the dispatcher reports to
  Claude Code: `continue` = record it, route through sinks, exit 0;
  `block` = record it, route through sinks, exit 2 (blocks the tool call,
  only meaningful for PreToolUse and PermissionRequest).
- **Circuit breakers** isolate persistently-failing sinks (see
  `observability-config.toml` resilience blocks above).
- **Dead letter queues** capture dropped events so silent data loss is
  recoverable.
- **Dispatcher-internal log** (`dispatcher-internal.jsonl`) is always-on and
  captures lifecycle + self-diagnostics even when all configured sinks are
  down.

## ADRs

### ADR-001: Rust for the dispatcher

Options considered: Rust, Go, Python, Node, Bash.

- **Rust:** compiled binary, no runtime dep, fast startup (~1-5ms), type
  safety, memory safety, aligns with project direction (AI-generated Rust
  already in use elsewhere). Cons: cross-compile matrix adds CI complexity.
- **Go:** same startup and binary story, slightly simpler cross-compile. Cons:
  no existing presence in project; contributor base is Rust.
- **Python:** proven in shanraisshan's plugin. Cons: Python is not guaranteed
  on operator machines (native Claude binary doesn't bundle it; Windows users
  often don't have it).
- **Node:** would be free if Claude Code required Node — but confirmed it
  doesn't (Claude binary is standalone, not npm-installed). Rejected.
- **Bash:** works on Linux/macOS only. Fails the Windows requirement.

**Decision: Rust.** AI-generated-Rust concerns that matter for humans don't
apply here.

### ADR-002: WASM for plugin ABI

Options considered: monolithic Rust workspace, external native binary per
plugin, WASM plugins, embedded scripting (Rhai/Lua).

- **Monolithic:** Simplest. All hooks compiled into the dispatcher.
  Cons: every new hook = dispatcher recompile and re-release. Inhibits
  third-party and experimentation.
- **External binaries:** Each hook cross-compiled per OS. Cons: ~2MB × 30
  hooks × 5 platforms = ~300MB shipped. Rejected on size.
- **WASM:** One `.wasm` runs on every OS. ~100-200KB per plugin, ~6MB total.
  Sandbox by default. Language-agnostic (Rust is our reference, others can
  join). Mature runtime (wasmtime). Fast instantiation (~1ms).
- **Embedded scripting:** Rhai or Lua in the dispatcher. Cons: new language
  for plugin authors; Rust ecosystem isn't at home here.

**Decision: WASM plugins via wasmtime.**

### ADR-003: WASI preview 1 for v1.0

WASI p1 is stable and has enough capability: stdin/stdout, env vars, file I/O
via preopens. WASI p2 (component model) is the future direction and adds
wasi-http, wasi-sockets, wasi-command — but is still maturing.

**Decision: WASI p1 for v1.0.** Migrate to p2 in v2.0 when ecosystem matures.
Plugins that need network will shell out via `exec_subprocess` in the
legacy-bash-adapter pattern until then.

### ADR-004: TOML for configuration

Options: TOML, YAML, JSON.

- **TOML:** Rust-native (`toml` crate), comment-friendly, explicit about types,
  array-of-tables syntax is clean for multi-instance sinks.
- **YAML:** Verbose indentation, whitespace traps, slower to parse, weaker
  type system. Popular but error-prone at human-editing time.
- **JSON:** No comments, verbose for this use case, harder to diff.

**Decision: TOML.** Matches Cargo ecosystem, readable, strongly typed.

### ADR-005: Multi-sink observability architecture

Options: (a) leave OTel collector as-is, users configure collector for Datadog;
(b) dispatcher writes to file only, collector handles everything; (c)
dispatcher owns multi-sink fan-out natively.

- **(a) OTel-only:** Requires keeping the local OTel collector running even
  for remote-only setups. Doesn't support zero-disk mode (local file is always
  written). Collector config is YAML, duplicates routing logic.
- **(b) File-only:** Same zero-disk limitation.
- **(c) Dispatcher-owned:** First-class sink types in Rust. Plugins emit via
  host function; dispatcher routes. OTel-gRPC is one sink among many.
  Supports zero-disk mode (file sink disabled, datadog sink enabled).

**Decision: (c).** Existing OTel collector stack continues to work (file sink
+ existing filelog receiver + existing Grafana). New operators can bypass it.

### Decisions captured in Open Questions

Four additional architectural decisions were made during spec review but
live in the Open Questions resolutions rather than as standalone ADRs:

- **Parallel-within-tier execution** (Q3) — plugins in the same priority
  tier run in parallel tokio tasks; tiers run sequentially. Alternatives
  considered: sequential-only, parallel-all-regardless-of-tier,
  sequential-with-global-wall-clock-cap.
- **Activation-skill-driven platform selection** (Q5) — neither
  `hooks.json` variable expansion nor plugin install lifecycle hooks exist;
  the activation skill writes a platform-specific `hooks.json` from CI-
  generated variants of a template.
- **Always-on dispatcher self-telemetry** (Q6) — `dispatcher-internal.jsonl`
  is written regardless of configured sinks, ensuring debuggability when
  all sinks are misconfigured.
- **Staged release** (Q7) — `1.0.0-beta.N` → `1.0.0-rc.N` → `1.0.0` rather
  than single `1.0.0`; semver stability engages only at final `1.0.0`.

See the Open Questions section for full rationale and alternatives on each.

## Deliverables (phased)

Release tags map to phases:
- `1.0.0-beta.1` = Phases 1+2 complete
- `1.0.0-rc.1` = Phases 3+4 complete (plus ≥2 weeks of beta shakedown)
- `1.0.0` = Phase 5 complete (plus ≥1 week of rc shakedown)

Each tag requires all CI matrix platforms green and the dogfood factory
(vsdd-factory's own repo) running on the prerelease successfully.

### Phase 0 — Infrastructure prep

1. Verify `scripts/bump-version.sh` accepts semver prerelease format
   (`1.0.0-beta.1`, `1.0.0-rc.1`). Extend if not.
2. Verify Release workflow handles prerelease tags (artifacts, GitHub Release
   draft vs full, etc.). Extend if not.
3. Add `plugins/vsdd-factory/hooks/hooks.json` to `.gitignore`.
4. Document the `/vsdd-factory:activate` flow's new platform-selection
   behavior (updates to existing skill).

### Phase 1 — Dispatcher foundation (target: `1.0.0-beta.1` prep)

1. Cargo workspace at repo root: `crates/factory-dispatcher`,
   `crates/hook-sdk`, `crates/hook-plugins/*`
2. `factory-dispatcher` Rust crate:
   - Reads stdin JSON (Claude Code hook payload)
   - Assigns `dispatcher_trace_id`
   - Loads `hooks-registry.toml` (cached + mtime-invalidated)
   - Routes to matching WASM plugins via wasmtime
   - Parallel-within-tier execution via tokio, sequential between tiers
   - Epoch + fuel enforcement with per-plugin budgets
   - Implements host function surface (see Architecture section)
   - Always writes `dispatcher-internal.jsonl`
3. `hook-sdk` crate with `#[hook]` proc-macro, `HookPayload`, `HookResult`,
   host function bindings, `VERSION_STRING` export requirement.
4. `hook-sdk` published to crates.io as `vsdd-hook-sdk` at `0.1.0`.
5. Cross-platform CI matrix: darwin-arm64, darwin-x64, linux-x64, linux-arm64,
   windows-x64. All must build; all must pass smoke tests.
6. Binary distribution: CI commits per-platform binaries to
   `plugins/vsdd-factory/hooks/dispatcher/bin/<platform>/` as part of the
   release chore commit. Never on feature branches.
7. `hooks.json.template` source of truth + CI generation of all 5
   `hooks.json.<platform>` variants. CI verifies no drift against committed
   versions.
8. Extend activation skill: detect OS+arch, copy `hooks.json.<platform>` →
   `hooks.json`, verify dispatcher binary, record activated-for-platform in
   `.claude/settings.local.json`. Deactivation removes generated
   `hooks.json`.

### Phase 2 — Legacy bash adapter (target: `1.0.0-beta.1`)

1. `legacy-bash-adapter` WASM plugin in `crates/hook-plugins/legacy-bash-adapter/`.
2. Takes a bash script path as configuration, invokes `exec_subprocess`
   with full Q4 safety envelope (allow-list, bounded output, bounded
   timeout, shell bypass acknowledged).
3. Generate `hooks-registry.toml` mapping every existing bash hook through
   the adapter. Preserves current behavior on Linux/macOS during migration.
4. Regression test: full bats suite (1177+ tests) passes against adapter-
   routed hooks.
5. **Release `1.0.0-beta.1`** once all CI green + dogfood passes. Collect
   feedback. Iterate `1.0.0-beta.2..N` as bugs surface.

### Phase 3 — High-value port to native WASM (target: `1.0.0-rc.1`)

Port these hooks from bash to native WASM. Drops subprocess overhead; adds
Windows support for the ported hooks:

1. `capture-commit-activity` (simplest; already self-filters)
2. `capture-pr-activity`
3. `block-ai-attribution` (PreToolUse/Bash — validates both halves of the
   original matcher bug are now routed correctly)
4. `emit-event` — becomes a host function rather than a plugin

Remaining 26 hooks stay on the legacy adapter for v1.0. Post-1.0 port
backlog driven by observed latency/usage telemetry.

### Phase 4 — Observability sinks (target: `1.0.0-rc.1`)

1. `observability-config.toml` schema + loader
2. Sink driver crates:
   - `sink-file` — JSONL append (default behavior, preserved from 0.79.x)
   - `sink-http` — batched POST with auth header support
   - `sink-otel-grpc` — OTel exporter (feeds existing local Grafana stack)
   - `sink-datadog` — Datadog Logs Intake
   - `sink-honeycomb` — Honeycomb Events
3. Per-sink resilience: retry with exponential backoff + jitter, circuit
   breaker, bounded queue with drop-oldest, dead-letter queue, per-sink
   health metrics.
4. Per-sink routing filters (event_type) and tag enrichment.
5. Dispatcher-internal log always-on at fixed path.
6. **Release `1.0.0-rc.1`** once Phases 3 + 4 green + 2+ weeks of beta
   shakedown with no critical bugs. Iterate `1.0.0-rc.2..N`.

### Phase 5 — New Claude Code hook events (target: `1.0.0`)

Wire the following event types through the dispatcher, populate
`hooks-registry.toml` examples, add to authoring docs:

- `SessionStart` — emit `session.started` with session_id + dep checks +
  factory-health brief
- `SessionEnd` — emit `session.ended` with duration + cost
- `WorktreeCreate` — auto-register new worktree with observability stack
  (obviates the v0.78.0 multi-factory escapade for new worktrees)
- `WorktreeRemove` — emit `worktree.removed`; cleanup observability
  registration
- `PostToolUseFailure` — emit `tool.error` for the Factory ROI error-rate
  panel (new post-1.0 dashboard work)

Deferred past v1.0: `PreCompact`, `PostCompact`, `TaskCreated`,
`TaskCompleted`, `FileChanged`, `SubagentStart`, `ConfigChange`, `CwdChanged`.

**Release `1.0.0`** once Phase 5 green + 1+ week of rc shakedown + migration
guide validated against ≥1 real 0.79.x factory. **Semver stability promise
engages here.**

## Migration path (v0.79.x → v1.0)

**User-facing:**

1. `/plugin update vsdd-factory@vsdd-factory` pulls the v1.0 prerelease or
   final release.
2. `/vsdd-factory:activate` — **required first step** after install. Detects
   OS+arch, writes the correct `hooks.json` for the platform, verifies the
   dispatcher binary.
3. Fully restart the Claude Code session (hot-upgrades don't reliably rewire
   hooks — documented in getting-started).
4. Existing factories keep working via legacy-bash-adapter on Linux/macOS.
5. Windows users get support only for the 4 native-WASM-ported hooks in v1.0;
   remaining 26 hooks TODO (documented with explicit list).
6. `/vsdd-factory:factory-health` gains a "hook system" diagnostic showing
   dispatcher version, plugin load status, registered hooks, and sink health.

**Operator-facing:**

- `.factory/logs/events-*.jsonl` path preserved as default file sink — zero
  change for factories that don't touch observability config.
- `hooks-registry.toml` and `observability-config.toml` ship with safe
  defaults committed to the plugin; operators override locally only when
  they need to (e.g., adding a Datadog sink).
- `dispatcher-internal.jsonl` is new: always-on self-telemetry, regardless
  of sink config. Never user-visible unless opened directly.
- `.factory/logs/dead-letter-*.jsonl` is new: per-sink drop log when
  `dead_letter` is enabled for that sink.
- Factories using `factory-obs register` continue to work; the dispatcher's
  default file sink produces the same format the collector already tails.

**Contributor-facing:**

- New hook authoring guide: `docs/guide/authoring-hooks.md`
- Porting guide: `docs/guide/porting-bash-hook-to-wasm.md`
- Plugin dev loop: `cargo build --target wasm32-wasip1 --release` + drop
  `.wasm` + update `hooks-registry.toml` + local dispatcher picks it up
  via mtime invalidation — **no dispatcher rebuild**.
- Published `vsdd-hook-sdk` crate provides `#[hook]` macro, types, host fn
  bindings. External plugin authors `cargo add vsdd-hook-sdk` from
  crates.io.

## Non-goals for v1.0

- Porting all 30 bash hooks to native WASM (legacy-bash-adapter suffices for
  the rest; native port backlog is driven by observed latency telemetry
  post-1.0).
- Implementing all 27 Claude Code hook events (only 5 Tier-1 events are in
  scope; the rest are deferred — see Phase 5).
- Hot-reload of the dispatcher binary itself (config files are hot-reloaded
  via mtime; binary requires session restart).
- Third-party plugin marketplace or signing infrastructure (the architecture
  accommodates third-party plugins via crates.io SDK + stable ABI, but we
  don't ship discovery/distribution infra in v1.0).
- WASI preview 2 (stay on p1 for v1.0; revisit in v2.0 when wasi-http and
  wasi-command mature).
- Removal of the local OTel collector / Grafana stack (remains available as
  one sink option — `otel-grpc` — among many).
- Arg-filtering for `exec_subprocess` (capability is binary-level allow-list
  only; arg-level policy is post-1.0 if attack scenarios materialize).
- seccomp / AppArmor / Landlock sandbox integration (v2+ territory).
- Network capability host functions (v1.0 plugins shell out to `curl`/`wget`
  via the subprocess allow-list).

## Open questions (decide before Phase 1 starts)

1. **Binary commit strategy:** ✅ **RESOLVED** — commit binaries directly to
   `plugins/vsdd-factory/hooks/dispatcher/bin/<platform>/`. Release workflow's
   CI matrix produces them; the release chore commit adds them. Never committed
   on feature branches. If repo size becomes a problem later, migration to an
   orphan branch (pattern similar to `factory-artifacts`) is mechanical.
   Rationale: zero install friction beats repo size over the first 3 years
   (~3GB ceiling); works air-gapped; no dependency on Claude Code plugin
   lifecycle hooks; no version-mismatch class of bug.
2. **hook-sdk versioning:** ✅ **RESOLVED** — publish to crates.io as
   `vsdd-hook-sdk` (fallback `factory-hook-sdk` if name taken). Start at
   0.1.0 during alpha; promote to 1.0.0 when vsdd-factory ships v1.0.
   Workspace uses `path` dep internally; CI verifies the crates.io version
   still compiles against the dispatcher. Separately, track host ABI version
   as a dispatcher constant (`HOST_ABI_VERSION: u32 = 1`); breaking host
   ABI changes require major bumps on both SDK and dispatcher; dispatcher
   may support multiple ABI versions concurrently if backward compatibility
   is needed. Third-party plugin development is progressively supported
   (stable ABI, documented SDK); no active recruitment or distribution
   infrastructure in v1.0.
3. **Dispatch timeout behavior:** ✅ **RESOLVED** (under "most correct" lens).
   - **Enforcement**: epoch-based interruption + fuel metering combined.
     Epoch provides wall-clock-ish timeouts for normal code; fuel guards
     against tight-loop runaway between yield points. This matches how
     production wasmtime deployers (Fastly, Shopify) run untrusted WASM.
   - **Per-plugin defaults**: 5000ms wall-clock via epoch; 10M fuel cap.
     Both overridable in the registry.
   - **On timeout**: drop wasmtime store → emit `internal.plugin_timeout`
     with plugin name/version, fuel consumed, wall-clock elapsed, session_id,
     truncated tool_input, dispatcher_trace_id → apply plugin's `on_error`
     policy. Also: record per-plugin latency histograms so operators can see
     approaching-timeout as a leading indicator.
   - **Host function contracts**: all host functions with blocking potential
     take explicit `timeout_ms` + size bounds in the API. No unbounded
     surface. `exec_subprocess(cmd, args, timeout_ms, max_output_bytes)`,
     `read_file(path, max_bytes, timeout_ms)`, etc. Eliminates the
     "forgot-a-timeout" class of bug.
   - **Execution model**: plugins within the same priority tier run in
     **parallel** (isolated wasmtime stores, tokio tasks); priority tiers
     run **sequentially**. `emit_event()` is queued async and serialized
     through a single writer; within-tier ordering is non-deterministic
     but inter-tier ordering is preserved; timestamps + `dispatcher_trace_id`
     always accurate. Rationale: avoids pathological `sum(timeouts)`
     wall-clock; matches CC's own matcher-group-parallel model; priority
     tiers give operators the explicit primitive for expressing
     dependencies.
   - **exec_subprocess safety**: argv-only (no shell interpretation),
     binary allow-list per plugin (Q4 covers the format).
   - **Self-observability**: dispatcher emits its own lifecycle through
     the sink pipeline (`dispatcher.started`, `plugin.loaded`,
     `plugin.invoked`, `plugin.completed`, `plugin.timeout`,
     `plugin.crashed`). We watch ourselves with the same tools we watch
     the factory.
   - **Trace correlation**: every stdin payload gets a
     `dispatcher_trace_id`; every event emitted during that payload's
     dispatch carries it. Full tool-call reconstruction possible from
     sink data.
   - **Plugin version**: plugin exports `VERSION_STRING`; dispatcher
     captures it and adds `plugin_version` to every emitted event.
   - **hooks.json resilience**: non-blocking event entries (PostToolUse,
     Stop, SubagentStop, etc.) get `"async": true` so CC doesn't hang if
     the dispatcher itself pathologically misbehaves. Blocking events
     (PreToolUse, PermissionRequest) stay sync so `exit 2` still blocks
     tool calls.
   - **Sync/async mode per plugin**: NOT added in v1.0. Parallel-within-tier
     already gives latency benefit; async mode would sacrifice observability
     and lifecycle guarantees for marginal gain. Revisit if a specific
     plugin need emerges.
4. **exec_subprocess capability default:** ✅ **RESOLVED** — denied by default;
   multi-layer defense. Per-plugin registry declares allowed binaries (resolved
   to full paths at load time to prevent PATH hijacking); shell interpreters
   (`bash`, `sh`, `zsh`, `pwsh`, `cmd`) require explicit
   `shell_bypass_acknowledged` with a justification string; env inheritance is
   allow-list only (`PATH`, `HOME`, `TMPDIR`, `CLAUDE_PROJECT_DIR`, `VSDD_*` by
   default); cwd restricted to declared paths; setuid/setgid binaries refused
   categorically; all calls have explicit `timeout_ms` and `max_output_bytes`
   at the API level (no unbounded overloads). Arg filtering deferred past
   v1.0; seccomp/AppArmor/Landlock integration deferred; network granularity
   handled via "no network host functions in v1.0, plugins shell out to
   `curl`/`wget` through the allow-list."

5. **Platform binary selection strategy:** ✅ **RESOLVED** — neither of the
   two Claude Code features we hoped for exist: (a) no OS/arch variable
   expansion in `hooks.json` `command` fields (only `${CLAUDE_PLUGIN_ROOT}`,
   `${CLAUDE_PROJECT_DIR}`, `${CLAUDE_PLUGIN_DATA}`, and generic env vars);
   (b) no plugin install lifecycle hooks (`plugin.json` has no
   `onInstall`/`postInstall`/`setup`).
   - **Decision**: use the existing activation skill to write a
     platform-specific `hooks.json` at activation time.
   - **Source of truth**: single `hooks.json.template` committed to the repo.
     CI generates `hooks.json.<platform>` variants on each release (5 files
     covering darwin-arm64, darwin-x64, linux-x64, linux-arm64, windows-x64).
     All 5 variants are committed as part of the release chore commit,
     alongside the dispatcher binaries.
   - **Activation skill flow**: detect OS + arch → copy the matching
     `hooks.json.<platform>` → `hooks.json` → verify the dispatcher binary
     exists and is executable → record activated-for-platform state in
     `.claude/settings.local.json`. `/vsdd-factory:deactivate` removes the
     generated `hooks.json`.
   - **`hooks.json` is gitignored**; it's regenerated per-machine at
     activation time.
   - **Tradeoff accepted**: marketplace auto-install no longer makes the
     plugin immediately functional. `/vsdd-factory:activate` is a required
     first step after install. Documented loud-and-clear in the getting-started
     guide. This is the only path without either a Python runtime dep or a
     Claude Code feature that doesn't exist.
   - **Source of truth check**: CI includes a verification step that
     re-generates all 5 `hooks.json.<platform>` files and diffs against the
     committed versions; release fails if drift is detected. Prevents
     hand-edits to the generated files.
6. **Observability sink retries:** ✅ **RESOLVED** — comprehensive per-sink
   resilience model, all config optional (safe defaults).
   - **Retry strategy**: exponential backoff with jitter; 50ms base,
     ×2 multiplier, max 30s, 3 retries default (configurable 0–10).
     Honor HTTP `Retry-After` headers.
   - **Retry-on classification**: network errors, 5xx, 429 → retry;
     4xx → terminal fail (no retry); unknown → retry (conservative).
   - **Bounded queue per sink**: default 1000 events; `drop-oldest` on
     full; `emit_event()` never blocks. Drop policy configurable.
   - **Circuit breaker**: 5 consecutive failures → open for 30s cooldown
     (exponential up to 5min); probe + close on success. Prevents
     repeated-timeout tax when a sink is down.
   - **Dead letter queue**: dropped events written to
     `.factory/logs/dead-letter-<sink>-<date>.jsonl`, 7-day retention,
     operator-inspectable/replayable. Prevents silent data loss.
   - **Self-observability**: every 60s the dispatcher emits
     `internal.sink_health` per sink (queue depth, circuit state,
     sent/dropped/retried counts, last_success_ts, last_failure_ts);
     routed through all healthy sinks.
   - **Internal self-telemetry fallback**: Option B — the dispatcher
     always writes its own internal events (`internal.*`, lifecycle,
     sink health) to a hardcoded local file
     `.factory/logs/dispatcher-internal.jsonl` regardless of configured
     sinks. This is NOT user-visible observability (doesn't violate
     zero-disk remote-only mode for factory-level events); it's a
     dispatcher-self-diagnostic trail. Keeps the dispatcher debuggable
     when all configured sinks are misconfigured or down.
7. **Should Phase 2 (legacy-bash-adapter) ship as v1.0-beta or roll into v1.0?**
   ✅ **RESOLVED** — staged: `1.0.0-beta.N` → `1.0.0-rc.N` → `1.0.0`. Responsible
   path given the scope of first-implementation risk (new WASM host ABI,
   Windows validation, multi-sink state machines, activation flow,
   migration from 0.79.x).
   - **Cadence:**
     - `1.0.0-beta.1`: Phase 1+2 done. Dispatcher + legacy-bash-adapter.
       All 30 existing hooks work on Linux/macOS via the adapter. Windows
       loads the plugin (zero hooks yet until native WASM ports land).
       File + otel-grpc sinks.
     - `1.0.0-beta.2..N`: iteration on beta.1 feedback.
     - `1.0.0-rc.1`: Phase 3+4 done. Native WASM port of 4 high-value
       hooks. All sink types (file/http/otel-grpc/datadog/honeycomb).
       Two weeks of beta with no critical bugs. Windows validated on at
       least one real user machine.
     - `1.0.0-rc.2..N`: iteration on rc.1.
     - `1.0.0`: Phase 5 done. Tier-1 new CC events wired
       (SessionStart, SessionEnd, WorktreeCreate, WorktreeRemove,
       PostToolUseFailure). One week of rc with no critical bugs.
       Migration guide validated. **Semver stability promise engaged
       here, not earlier.**
   - **Operational:** prereleases tagged on main (no separate branch);
     marketplace users get stable by default (latest stable = 0.79.4
     until 1.0.0 ships); betas/RCs are explicit opt-in; breaking changes
     allowed between prerelease tags; dogfood vsdd-factory's own repo on
     each prerelease before promoting to marketplace.
   - **Build pipeline impact:** `bump-version.sh` needs to accept semver
     prerelease format (`1.0.0-beta.1`). Verify or extend as part of
     Phase 1 infra work.
   v1.0-beta makes it clear what's stable vs migrating; slower to the final
   1.0 tag. Roll-in is one release but has a "beta in production" feel.
