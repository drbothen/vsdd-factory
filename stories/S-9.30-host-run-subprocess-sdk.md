## Narrative

- **As a** Tier 2 plugin author implementing `validate-wave-gate-prerequisite` in native WASM
- **I want to** invoke `verify-sha-currency.sh` under capability-gated subprocess control via
  `host::run_subprocess(spec: &SubprocessSpec)`
- **So that** the critical SHA-currency defense-in-depth check is preserved in the WASM port
  (ADR-014 D-9.2 override) without compromising the capability-gated security model

---

## Goal

The orchestrator confirmed in ADR-014 D-9.2 that the `verify-sha-currency.sh` invocation
inside `validate-wave-gate-prerequisite` is a critical defense-in-depth check that MUST be
preserved in the W-16 WASM port. This story implements the required host ABI extension:

1. **`SubprocessSpec` / `SubprocessResult` types** in `crates/hook-sdk/src/host/` exposing
   the structured subprocess call parameters and return value to plugin authors.

2. **`host::run_subprocess(spec: &SubprocessSpec)` SDK wrapper** in `crates/hook-sdk/src/host/run_subprocess.rs`
   that marshals the spec to JSON, calls the `vsdd::run_subprocess` extern shim, and
   unmarshals the `SubprocessResult` or `HostError`.

3. **Dispatcher binding** in `crates/factory-dispatcher/src/host/run_subprocess.rs` that
   enforces `SubprocessCaps` capability gating (binary allowlist, arg allowlist, env
   stripping), executes the binary via `std::process::Command` (no shell), enforces the
   timeout via SIGKILL/TerminateProcess, caps stdout/stderr at declared byte limits, and
   emits the `host.subprocess.exec` telemetry event.

4. **`SubprocessCaps` struct** in `crates/factory-dispatcher/src/registry.rs` (parallel to
   `WriteFileCaps` from S-8.10) with `binary_allowlist`, `arg_allowlist`, `env_allowlist`,
   `max_stdout_bytes`, `max_stderr_bytes`, `max_timeout_ms` fields.

5. **HOST_ABI.md update** documenting the new `vsdd::run_subprocess` host export.

6. **SS-02 Schema Evolution table row** in `.factory/specs/architecture/SS-02-hook-sdk.md`
   citing BC-2.02.013 and confirming `HOST_ABI_VERSION = 1`.

**ABI policy:** D-6 Option A additive extension. `HOST_ABI_VERSION` stays at 1 (same
rationale as S-8.10 `host::write_file` and S-8.30 HookPayload SubagentStop fields).
Existing plugins compiled against SDK 0.2.x that do not import `run_subprocess` continue
to function; wasmtime silently ignores additional host exports.

## Acceptance Criteria

All 24 MUST statements from BC-2.02.013 are traced below. The numbering I-X.Y refers to
BC-2.02.013 Invariant family X, clause Y (1-based clause index within the family).

### AC-1: Capability gating — no block → deny (BC-2.02.013 I-1.1)

A plugin without a `[hooks.<id>.capabilities.run_subprocess]` block in
`hooks-registry.toml` MUST receive `Err(HostError::CapabilityDenied)` on every call to
`host::run_subprocess`. No subprocess is spawned. Test: dispatcher integration test with
`Capabilities { run_subprocess: None, .. }` asserts the binding returns
`codes::CAPABILITY_DENIED` (-1) before any `Command::new` call.

(traces to BC-2.02.013 I-1.1: MUST declare SubprocessCaps)

### AC-2: Allowlist enforcement — binary, args, env strip (BC-2.02.013 I-1.2, I-1.3, I-1.4, I-2.6)

The dispatcher binding MUST enforce all four allowlist checks in order:

**(a) Binary allowlist check (I-1.2):**
`spec.binary`'s resolved absolute path MUST match at least one glob pattern in
`SubprocessCaps.binary_allowlist`. If no pattern matches → `Err(HostError::CapabilityDenied)`.
The check MUST run before any `execvp`-equivalent call. Test: binary `/usr/bin/grep` with
`binary_allowlist = ["/usr/bin/git"]` → `CapabilityDenied`.

**(b) Arg allowlist check (I-1.3):**
Every element of `spec.args` MUST match at least one pattern in `SubprocessCaps.arg_allowlist`.
Partial arg approval is NOT permitted — if ANY single arg fails, the entire call is denied.
Test: `args = ["--version", "--porcelain"]` with `arg_allowlist = ["--version"]` →
`CapabilityDenied` (second arg fails).

**(c) Env stripping (I-1.4):**
Before exec, every env var key in `spec.env` NOT listed in `SubprocessCaps.env_allowlist`
MUST be stripped. The subprocess inherits ONLY env vars whose keys appear in `env_allowlist`.
Test: `spec.env = {"PATH": "/usr/bin", "CLAUDE_PROJECT_DIR": "/proj"}` with
`env_allowlist = ["CLAUDE_PROJECT_DIR"]` → subprocess env contains only `CLAUDE_PROJECT_DIR`.

**(d) Path traversal pre-check (I-2.6, see also AC-3):**
`spec.binary` containing `../` sequences MUST be rejected BEFORE allowlist pattern matching.
The path traversal check is ordered BEFORE the glob match.

(traces to BC-2.02.013 I-1.2, I-1.3, I-1.4, I-2.6)

### AC-3: Security boundaries — no shell, no traversal, abs path, working_dir (BC-2.02.013 I-2.5, I-2.7, I-2.8, I-2.9)

**(a) No shell interpretation (I-2.5):**
The dispatcher MUST execute `spec.binary` directly via `Command::new(binary).args(args)`.
The execution path MUST NOT invoke `/bin/sh -c`, `bash -c`, or any shell interpreter
implicitly. Test: `spec.binary = "/bin/sh"` not in `binary_allowlist` → denied. If
explicitly listed, it is exec'd as `execvp("/bin/sh", args)` — no additional shell wrapping.

**(b) Path traversal rejection (I-2.6 — check before allowlist):**
`spec.binary = "../../etc/passwd"` MUST be rejected with `HostError::CapabilityDenied`
regardless of what patterns are in `binary_allowlist`. The traversal check MUST occur
before the glob match, not after.

**(c) Absolute path or preopened relative (I-2.7):**
`spec.binary` MUST be an absolute path or a path resolvable within WASI preopened dirs.
PATH-based lookup (`$PATH` search for a bare binary name) MUST NOT be performed.
Test: `spec.binary = "git"` (bare name) → if not an absolute path and not under
a preopened dir prefix → `HostError::CapabilityDenied` or `BinaryNotFound`.

**(d) Working directory constraint (I-2.8):**
If `spec.working_dir` is `Some(path)`, it MUST be `CLAUDE_PROJECT_DIR` or a descendant.
Any `working_dir` outside this tree MUST produce `Err(HostError::CapabilityDenied)`.
Test: `working_dir = Some("/etc")` where `CLAUDE_PROJECT_DIR = "/proj"` → `CapabilityDenied`.

**(e) No extra FD inheritance (I-2.9):**
The subprocess MUST NOT inherit file descriptors beyond stdin/stdout/stderr. The
`Command` builder MUST NOT pass through open FDs from the dispatcher. Implemented by:
(a) not using `.inherit()` on any fd except the three standard ones; (b) setting
`Command::stdin(Stdio::null())` (subprocess needs no stdin from plugin); stdout and stderr
captured to pipes.

(traces to BC-2.02.013 I-2.5, I-2.7, I-2.8, I-2.9)

### AC-4: Output bounds — stdout/stderr truncation (BC-2.02.013 I-3.10, I-3.11, I-3.12)

**(a) stdout truncation (I-3.10):**
If the subprocess writes more than `SubprocessCaps.max_stdout_bytes` bytes to stdout,
the dispatcher MUST capture exactly `max_stdout_bytes` bytes and set
`SubprocessResult.truncated = true`. Excess bytes are consumed and discarded (the
subprocess's stdout pipe is not blocked). Test: subprocess writes
`max_stdout_bytes + 1024` bytes → `result.stdout.len() == max_stdout_bytes`,
`result.truncated == true`.

**(b) stderr truncation (I-3.11):**
Same semantics as stdout: if subprocess writes more than `max_stderr_bytes` → capture
exactly `max_stderr_bytes`, set `truncated = true`, discard excess. Test: subprocess
writes `max_stderr_bytes + 1` bytes to stderr → `result.stderr.len() == max_stderr_bytes`,
`result.truncated == true`.

**(c) Truncation at dispatcher (I-3.12):**
The subprocess itself is never aware of the cap — it writes freely; the dispatcher
enforces the cap on the read side. The subprocess MUST NOT receive SIGPIPE from the cap
enforcement (the pipe read side remains open; dispatcher discards excess bytes).
Test: subprocess writes 2× max_stdout_bytes; verify it completes normally (exit 0),
not hung; `result.truncated == true`.

(traces to BC-2.02.013 I-3.10, I-3.11, I-3.12)

### AC-5: Timeout enforcement — SIGKILL, error value, duration accuracy (BC-2.02.013 I-4.13, I-4.14, I-4.15)

**(a) Kill at timeout deadline (I-4.13):**
If the subprocess runs longer than `SubprocessCaps.max_timeout_ms` wall-clock milliseconds
after spawn, the dispatcher MUST send SIGKILL on Unix or call `TerminateProcess` on Windows.
The kill is unconditional — not conditional on subprocess state.

**(b) Return Timeout error (I-4.14):**
After killing, MUST return `Err(HostError::Timeout)`. The error MUST NOT carry partial
stdout/stderr — the `Err(HostError::Timeout)` variant carries no `SubprocessResult`.
Test: subprocess `sleep 10` with `max_timeout_ms = 100` → `Err(HostError::Timeout)` within
200ms of the call; no partial stdout in the error value.

**(c) Duration field accuracy (I-4.15):**
On non-timeout completion, `SubprocessResult.duration_ms` MUST reflect actual wall-clock
elapsed time from spawn to process exit (measured by the dispatcher). It MUST NOT exceed
`SubprocessCaps.max_timeout_ms`. Test: subprocess `sleep 0.05` (50ms) with
`max_timeout_ms = 5000` → `result.duration_ms` is in range [40, 500] (±20% tolerance
for CI runner variance); `result.duration_ms ≤ 5000`.

(traces to BC-2.02.013 I-4.13, I-4.14, I-4.15)

### AC-7: Telemetry — host.subprocess.exec event on every call (BC-2.02.013 I-6.18, I-6.19)

**(a) Event emitted on every call (I-6.18):**
The dispatcher MUST emit a `host.subprocess.exec` telemetry event on every
`host::run_subprocess` call — both successful calls and all error paths
(CapabilityDenied, BinaryNotFound, Timeout, IoError). Event payload MUST include:
`{plugin_id, binary, args_count, exit_code, duration_ms, stdout_bytes, stderr_bytes,
truncated, timed_out}`. On error paths: `exit_code = null`, `duration_ms = actual_elapsed`,
`truncated = false`, `timed_out = true` (if Timeout), or `timed_out = false` (other errors).

Integration test: assert the event sink receives exactly one `host.subprocess.exec` event
on CapabilityDenied path (no spawn), one on successful exec path, and one on Timeout path.

**(b) Event routed through normal sink chain (I-6.19):**
The `host.subprocess.exec` event MUST be dispatched through the existing telemetry sink
chain (file sink / Datadog / Honeycomb per config), identical to how `host.file.read` and
`host.file.write` events are routed. No subprocess-specific sink bypass.
Test: configure file sink; call run_subprocess; assert `host.subprocess.exec` appears in
the file sink output alongside any other events from the call.

(traces to BC-2.02.013 I-6.18, I-6.19)

### AC-8: Error semantics — all four HostError variants are values, not panics; no new codes (BC-2.02.013 I-7.20, I-7.21, I-7.22, I-7.23, I-7.24)

**(a) CapabilityDenied is a value (I-7.20):**
`HostError::CapabilityDenied` MUST be returned as `Result::Err` value. The dispatcher
MUST NOT panic on capability check failure. Test: assert no panic across all denial paths.

**(b) BinaryNotFound is a value (I-7.21):**
If the binary path does not exist or is not executable → `Err(HostError::BinaryNotFound)`.
No panic. Test: `spec.binary = "/usr/bin/nonexistent_binary_xyzzy"` in allowlist →
`HostError::BinaryNotFound`.

**(c) Timeout is a value (I-7.22):**
`Err(HostError::Timeout)` after SIGKILL/TerminateProcess. No panic.

**(d) IoError is a value (I-7.23):**
OS-level failures during spawn, pipe I/O, or wait MUST be `Err(HostError::IoError(io::Error))`.
No panic. Test: remove exec permission from a listed binary path → either `BinaryNotFound`
or `IoError` depending on OS; assert no panic.

**(e) No new error codes (I-7.24):**
`host::run_subprocess` MUST reuse the existing `HostError` variant set. No new negative
i32 codes are introduced. The existing set `{-1 CapabilityDenied, -2 Timeout, -3 OutputTooLarge,
-4 InvalidArgument, -99 InternalError}` is sufficient. `run_subprocess` error code mapping:
- `0` on success (returns SubprocessResult via JSON out-pointer)
- `-1` on CapabilityDenied (any of: no caps block, binary not in allowlist, arg not in allowlist,
  path traversal, working_dir outside CLAUDE_PROJECT_DIR)
- `-2` on Timeout (SIGKILL sent; no partial output)
- `-4` on InvalidArgument (empty binary string, malformed JSON spec)
- `-99` on IoError (spawn failure, pipe error, wait error)
Note: `-3` (OutputTooLarge) is NOT used by run_subprocess — output truncation is handled
within SubprocessResult (truncated flag), not as an error return.

(traces to BC-2.02.013 I-7.20, I-7.21, I-7.22, I-7.23, I-7.24)

**SubprocessCaps struct definition** (pin for implementer — parallel to `WriteFileCaps`
at `registry.rs:85-91`):
```rust
#[derive(Debug, Clone, Default, Deserialize, Serialize, PartialEq, Eq)]
#[serde(deny_unknown_fields)]
pub struct SubprocessCaps {
    /// Glob patterns for permitted binary absolute paths.
    /// Example: ["/usr/bin/git", "*/verify-sha-currency.sh"]
    pub binary_allowlist: Vec<String>,

    /// Glob patterns for permitted argument strings.
    /// Example: ["--version", "show*", "HEAD:*"]
    pub arg_allowlist: Vec<String>,

    /// Environment variable keys the subprocess may inherit.
    /// All other keys are stripped before exec.
    pub env_allowlist: Vec<String>,

    /// Maximum bytes captured from subprocess stdout (excess discarded).
    pub max_stdout_bytes: u64,

    /// Maximum bytes captured from subprocess stderr (excess discarded).
    pub max_stderr_bytes: u64,

    /// Maximum wall-clock milliseconds before SIGKILL (Unix) / TerminateProcess (Windows).
    pub max_timeout_ms: u64,
}
```
Derives match `WriteFileCaps`: `deny_unknown_fields`, `Default`, `Serialize`, `Deserialize`.

**SubprocessSpec and SubprocessResult definitions** (SDK side; shared types):
```rust
/// Passed by plugin to host::run_subprocess as JSON-serialized spec.
#[derive(Debug, Serialize, Deserialize)]
pub struct SubprocessSpec {
    pub binary: String,
    pub args: Vec<String>,
    pub env: HashMap<String, String>,
    pub timeout_ms: u64,
    pub working_dir: Option<String>,  // PathBuf serialized as String for WASM portability
}

/// Returned on successful subprocess completion.
#[derive(Debug, Serialize, Deserialize)]
pub struct SubprocessResult {
    pub exit_code: i32,
    pub stdout: Vec<u8>,
    pub stderr: Vec<u8>,
    pub duration_ms: u64,
    pub truncated: bool,
}
```

**Protocol note:** The SDK wrapper serializes `SubprocessSpec` to JSON and passes a
pointer+length pair to the host via the `vsdd::run_subprocess(spec_ptr, spec_len,
out_ptr, out_len) -> i32` extern. On success (i32 = 0), the host writes the JSON-encoded
`SubprocessResult` into the output buffer. On error, i32 is a negative error code from
the existing set. This **JSON-pointer** protocol differs from `write_file`'s input-pointer
protocol (raw bytes) — SubprocessSpec/Result are complex structs best transported as JSON.

**Target:** `wasm32-wasip1` (WASI Preview 1 per ADR-003). SDK crate compiles for this
target; dispatcher crate compiles for the host platform.

**Forbidden dependencies:** `crates/hook-sdk/` MUST NOT gain a dependency on
`crates/factory-dispatcher/`. Dependency direction is strictly one-way: dispatcher → SDK
(for types). If cargo build after this story shows hook-sdk depending on factory-dispatcher,
the CI build MUST fail.

**HOST_ABI_VERSION strategy (confirmed AS-DEC):** Stay at 1. Additive export does not
require a version bump. D-6 Option B explicitly disallowed in v1.x.

## Architecture Compliance Rules

1. **SS-02 boundary:** All code changes are within `crates/hook-sdk/` and
   `crates/factory-dispatcher/src/host/`. No files outside these two modules are
   modified (except HOST_ABI.md, SS-02-hook-sdk.md spec, and CHANGELOG for
   AC-6 documentation obligations).
2. **Deny-by-default capability model:** `run_subprocess` follows the same
   deny-by-default pattern as `read_file` and `write_file`: no capability block =
   `CAPABILITY_DENIED`. See `read_file.rs` lines 73-77 for the reference pattern.
3. **Memory safety:** Guest memory access for SubprocessSpec deserialization uses the
   dispatcher's existing `memory::read_wasm_bytes` helper (JSON bytes from WASM memory).
   No raw pointer arithmetic.
4. **Error code stability:** No new negative i32 codes introduced. The existing set
   `{-1 CapabilityDenied, -2 Timeout, -3 OutputTooLarge, -4 InvalidArgument, -99 InternalError}`
   is sufficient. OutputTooLarge (-3) is NOT returned by run_subprocess (truncation is
   in-band via SubprocessResult.truncated). See AC-8(e).
5. **HOST_ABI_VERSION invariant:** Both `crates/hook-sdk/src/lib.rs:58` and
   `crates/factory-dispatcher/src/lib.rs:43` MUST still read `= 1` after this story.
6. **Workspace Cargo.toml:** No new crate is added to the workspace. This story modifies
   two existing crates; `Cargo.toml` workspace `members` array is unchanged.
7. **Pure-core extraction:** `binary_allowlist_check`, `arg_allowlist_check`, and `env_strip`
   MUST be extracted as standalone functions per BC-2.02.013 Refactoring Notes — directly
   unit-testable without a full dispatcher context, mirroring `path_allowed` in `read_file.rs`.
8. **No shell interpretation:** `Command::new(binary).args(args)` ONLY. No `cmd.env_clear()`
   bypass (use `env_strip` logic, then set env vars explicitly on the Command). No
   `.arg("-c")` or shell wrapping.

Option (a) is recommended for portability. The implementer documents the chosen approach
in the PR description with rationale.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `spec.binary` is empty string | `Err(HostError::CapabilityDenied)` (empty string fails allowlist match; never progresses to spawn) |
| EC-002 | `spec.binary = "../../etc/passwd"` (path traversal) | `Err(HostError::CapabilityDenied)` — rejected at traversal check BEFORE allowlist match |
| EC-003 | Plugin registry has no `run_subprocess` capability block | `Err(HostError::CapabilityDenied)` on every call; no spawn |
| EC-004 | `binary_allowlist = []` (empty) | `Err(HostError::CapabilityDenied)` — no binary can match an empty glob list |
| EC-005 | `arg_allowlist = []` but `spec.args` is non-empty | `Err(HostError::CapabilityDenied)` — first arg cannot match any pattern |
| EC-006 | `spec.args = []` (empty) with `arg_allowlist = []` | Allowed — no args to check; proceeds to binary resolution and exec |
| EC-007 | `env_allowlist = []` | Subprocess inherits zero env vars; all `spec.env` entries are stripped |
| EC-008 | `env_allowlist = ["CLAUDE_PROJECT_DIR"]`, `spec.env = {"PATH": "/usr/bin"}` | `PATH` is stripped; subprocess env has only CLAUDE_PROJECT_DIR if set |
| EC-009 | Subprocess writes exactly `max_stdout_bytes` bytes | `truncated = false`; full output captured |
| EC-010 | Subprocess writes `max_stdout_bytes + 1` bytes to stdout | `truncated = true`; exactly `max_stdout_bytes` bytes in `SubprocessResult.stdout` |
| EC-011 | Subprocess writes 0 bytes to both stdout and stderr | `stdout = []`, `stderr = []`, `truncated = false`; exit code captured normally |
| EC-012 | Subprocess exits with code 1 (non-zero) | `Ok(SubprocessResult { exit_code: 1, ... })`; non-zero exit is NOT an error at the ABI level |
| EC-013 | Subprocess exceeds `max_timeout_ms` by 1 ms | `Err(HostError::Timeout)`; SIGKILL sent; no partial output returned |
| EC-014 | Binary listed in allowlist does not exist on disk | `Err(HostError::BinaryNotFound)` |
| EC-015 | Plugin compiled against SDK without `run_subprocess` import loads against dispatcher that exports `run_subprocess` | Plugin loads normally; wasmtime ignores the additional host export; no error — ABI-safe per D-6 Option A |
| EC-016 | `spec.timeout_ms` exceeds `caps.max_timeout_ms` | Dispatcher uses `caps.max_timeout_ms` as hard cap; `spec.timeout_ms` is ignored or clamped to `caps.max_timeout_ms` |
| EC-017 | `spec.working_dir` points outside `CLAUDE_PROJECT_DIR` | `Err(HostError::CapabilityDenied)` |
| EC-018 | Shell interpreter binary (e.g. `/bin/bash`) in `binary_allowlist` | Allowed (explicitly listed); exec'd directly with `spec.args` as argv — no additional shell wrapping by the dispatcher |
