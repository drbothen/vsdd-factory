# S-6.01: Add create-adr skill for ADR authoring

> **One-per-file:** Each story lives in its own file.
> Filename: `S-6.01-create-adr-skill.md`
> Epic E-6 is "VSDD Self-Improvement / Tooling Backlog" — skills that improve the
> factory authoring workflow itself. S-6.01 is the first story in this epic.

## Narrative

- **As a** VSDD architect authoring a new architectural decision record
- **I want to** invoke `/vsdd-factory:create-adr` with a title and optional metadata
- **So that** the skill allocates a collision-free ADR ID, scaffolds the file from
  `adr-template.md`, inserts the ARCH-INDEX row, patches any superseded ADR, and
  validates template compliance — eliminating the manual-process pain that was
  re-exposed during the 10-ADR brownfield backfill burst

## Acceptance Criteria

Numbered behavioral assertions. BCs will be authored as part of skill TDD (see
frontmatter note); `(traces to BC-TBD)` placeholders are intentional for a draft
story — the skill implementer promotes these to real BC-6.NN.NNN IDs.

### AC-1 (traces to BC-TBD — ID allocation)

The skill scans `plugins/vsdd-factory/skills/create-adr/../../../.factory/specs/architecture/decisions/ADR-*.md`
filenames AND the ARCH-INDEX "Architecture Decisions" table rows to determine the
highest existing ADR number, then proposes `ADR-<highest+1>` zero-padded to three
digits (e.g., current max is ADR-013, skill proposes ADR-014).

If the same ID appears in the filesystem but is absent from ARCH-INDEX, or vice versa,
the skill reports the inconsistency and refuses to proceed (exit non-zero, no files
written).

If the user supplies an explicit `--id ADR-NNN` override that already exists, the skill
refuses with: "ADR-NNN already exists at decisions/ADR-NNN-*.md. Omit --id to
auto-allocate or choose a free ID."

- **Test:** `test_id_allocation_no_collision()`
- **Test:** `test_id_allocation_refuses_duplicate()`
- **Test:** `test_id_allocation_filesystem_vs_index_mismatch_blocks()`

### AC-2 (traces to BC-TBD — frontmatter scaffold)

The skill renders the new ADR file from `plugins/vsdd-factory/templates/adr-template.md`
populating:

| Field | Value |
|-------|-------|
| `document_type` | `adr` (literal — never changed) |
| `adr_id` | computed value from AC-1 |
| `status` | `proposed` (always — the skill never writes `accepted`) |
| `date` | today's ISO-8601 date (YYYY-MM-DD) |
| `subsystems_affected` | user-supplied array; each entry validated against ARCH-INDEX Subsystem Registry (`SS-01`…`SS-10`); invalid entries cause refusal |
| `supersedes` | user-supplied `ADR-NNN` if provided; validated to exist as `decisions/ADR-NNN-*.md`; null if not provided |
| `superseded_by` | `null` (always at creation — skill never sets this on the new ADR) |

Section bodies (Context, Decision, Rationale, Consequences, Alternatives Considered,
Source/Origin) are rendered as verbatim placeholder text from the template — the skill
does NOT ghost-write prose.

- **Test:** `test_frontmatter_status_always_proposed()`
- **Test:** `test_frontmatter_date_is_today()`
- **Test:** `test_subsystems_validated_against_registry()`
- **Test:** `test_supersedes_validated_to_exist()`

### AC-3 (traces to BC-TBD — supersession bidirectional patch)

If `supersedes: ADR-NNN` is set, the skill patches the frontmatter of the named older
ADR file to set `superseded_by: ADR-<new>`. Both the new ADR file write and the patch
to the older ADR are treated as a single atomic unit: if the patch fails (e.g., old
file missing or frontmatter malformed), neither file is written and the skill exits
non-zero.

The patch only modifies the `superseded_by` line in frontmatter — it does NOT rewrite
any other content of the older ADR.

- **Test:** `test_supersession_patches_old_adr_superseded_by()`
- **Test:** `test_supersession_atomic_rollback_on_patch_failure()`

### AC-4 (traces to BC-TBD — ARCH-INDEX insertion)

The skill inserts one new row in the ARCH-INDEX `## Architecture Decisions` table
immediately after the row for the current highest ADR, in ascending numeric order.

Row format (pipe-aligned to match existing table):

```
| ADR-NNN | <decision-title> | <subsystems_affected joined by ", "> | decisions/ADR-NNN-<slug>.md |
```

The skill derives `<slug>` from the `--title` argument: lowercased, spaces → hyphens,
non-alphanumeric/hyphen characters stripped.

If ARCH-INDEX does not contain the expected `## Architecture Decisions` section header,
the skill exits non-zero without writing any file.

- **Test:** `test_arch_index_row_inserted_in_numeric_order()`
- **Test:** `test_arch_index_row_pipe_aligned()`
- **Test:** `test_arch_index_slug_derivation()`
- **Test:** `test_arch_index_missing_section_blocks()`

### AC-5 (traces to BC-TBD — hand-off pattern, no ghost-writing)

The skill scaffolds placeholder section bodies exactly as written in
`plugins/vsdd-factory/templates/adr-template.md` — bracketed prompt text like
`[2-5 paragraphs] Background, forces driving the decision…` is preserved verbatim.

After writing the file, the skill prints a guidance block to stdout:

```
ADR-NNN scaffolded at: .factory/specs/architecture/decisions/ADR-NNN-<slug>.md

Sections to flesh out:
  - Context      (2-5 paragraphs: background, forces, constraints)
  - Decision     (1-3 paragraphs: the choice itself)
  - Rationale    (2-5 paragraphs: why this, not alternatives)
  - Consequences (Positive / Negative sub-headings)
  - Alternatives Considered (top 2-3 options rejected)
  - Source / Origin (MUST cite implementation evidence for brownfield ADRs)

Recommended next step:
  Spawn architect agent: "Flesh out ADR-NNN sections. File: .factory/specs/architecture/decisions/ADR-NNN-<slug>.md"
```

The skill does NOT invoke the architect agent itself — hand-off is a print statement only.

- **Test:** `test_scaffold_preserves_template_placeholder_text()`
- **Test:** `test_stdout_guidance_block_present()`

### AC-6 (traces to BC-TBD — brownfield mode annotation)

If `--brownfield` flag is supplied (or if `--supersedes` references an existing ADR,
which implies an established design is being formalized), the skill annotates the
`Source / Origin` section header in the scaffolded file with a non-skippable warning
comment:

```markdown
## Source / Origin
<!-- BROWNFIELD: You MUST cite implementation evidence (file:line from crates/ or
     legacy-design-docs/) before this ADR can be accepted. Omitting evidence is a
     template-compliance failure. -->
```

When the `--brownfield` flag is absent and `--supersedes` is absent, this annotation
is omitted.

- **Test:** `test_brownfield_flag_injects_source_annotation()`
- **Test:** `test_supersedes_implies_brownfield_annotation()`
- **Test:** `test_no_brownfield_flag_no_annotation()`

### AC-7 (traces to BC-TBD — final validation gate)

After all file writes complete, the skill executes:

```
plugins/vsdd-factory/bin/validate-template-compliance.sh \
    .factory/specs/architecture/decisions/ADR-NNN-<slug>.md
```

If the script exits with code 0, the skill prints "Template compliance: PASS" and exits 0.

If the script exits non-zero, the skill prints the script's stderr/stdout, prints
"Template compliance: FAIL — ADR-NNN not registered. Fix the issues above and re-run.",
and exits non-zero. The file that failed validation remains on disk (the implementer
can inspect it) but the ARCH-INDEX row is NOT inserted and any supersession patch is
NOT applied — the skill treats a validation failure as if the write never happened.

- **Test:** `test_validation_pass_exits_zero()`
- **Test:** `test_validation_fail_exits_nonzero_no_index_row()`
- **Test:** `test_validation_fail_supersession_not_applied()`

### AC-8 (traces to BC-TBD — atomicity / no partial state)

If any step in the sequence (ID allocation → file write → supersession patch →
ARCH-INDEX insertion → validation) fails, the skill reverts all side-effects already
applied in that invocation:

- If the new ADR file was written, it is deleted.
- If the ARCH-INDEX row was inserted, it is reverted.
- If the old ADR's `superseded_by` was patched, it is reverted to its prior value.

The skill exits non-zero with a message identifying the step that failed and the
revert actions taken. Idempotent re-invocation with the same arguments must succeed.

- **Test:** `test_atomicity_write_fail_no_index_row()`
- **Test:** `test_atomicity_index_insert_fail_file_deleted()`
- **Test:** `test_atomicity_supersession_patch_fail_new_file_deleted()`
- **Test:** `test_idempotent_reinvocation_after_failure()`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `decisions/` directory does not exist | Skill creates it before writing (not an error) |
| EC-002 | ARCH-INDEX has ADR-013 in the table but no `decisions/ADR-013-*.md` file exists | Reported as inconsistency; skill refuses; user must reconcile manually |
| EC-003 | `--title` contains special characters (e.g., `/`, `<`, `>`) | Slug derivation strips them; file is written with sanitized slug; decision title in ARCH-INDEX row uses the original unsanitized title |
| EC-004 | `--supersedes ADR-NNN` where ADR-NNN is already `superseded` (has a `superseded_by` set) | Skill warns "ADR-NNN is already superseded by ADR-MMM" but does NOT block — double-supersession is valid during reorganization |
| EC-005 | `validate-template-compliance.sh` not found or not executable | Skill exits non-zero with message "validate-template-compliance.sh not found at expected path — cannot complete AC-7" |
| EC-006 | `--subsystems` contains an unknown SS-ID (e.g., `SS-99`) | Skill refuses with list of valid SS-IDs from ARCH-INDEX Subsystem Registry |
| EC-007 | Concurrent invocations write the same ADR-NNN simultaneously | Not explicitly guarded; skill detects on file-write collision and exits non-zero — users must serialise invocations in parallel sessions |
| EC-008 | User invokes skill with no arguments | Skill prints usage summary and exits 0 without writing anything |

## Library & Framework Requirements (MANDATORY)

This story produces a SKILL.md (Markdown instruction file) and a commands binding
(also Markdown). There are no compiled language dependencies.

| Tool | Version | Purpose |
|------|---------|---------|
| `validate-template-compliance.sh` | current (no version pin — Bash script in repo) | Final validation gate (AC-7) |
| Bash | >= 3.2 (macOS system shell) | Script invocation for validation step |

**Forbidden Dependencies:**
- The `create-adr` skill MUST NOT call any other skill (no `/vsdd-factory:register-artifact`
  invocation inside the skill) — the ADR registration workflow is self-contained. Calling
  `register-artifact` would create a second INDEX mutation path that could race with the
  skill's own atomic write. If this constraint changes, update this story first.
- The skill MUST NOT import or invoke any Rust crates or WASM plugins — it is a pure
  Markdown instruction file interpreted by the Claude Code agent.
