# S-7.01: Agent prompt updates for spec/anchor/adversary discipline

> **Cluster decision:** Lessons #1 (spec-first), #3 (capability anchor
> justification), and #7 (adversary partial-fix-regression check) are grouped
> here because all three are prompt-level changes to the builder/reviewer agent
> trio (story-writer, product-owner, adversary). They share no tooling
> dependency and can be authored and reviewed together in a single pass.

## Narrative

- **As a** vsdd-factory orchestrator running a spec sub-cycle
- **I want** the story-writer, product-owner, and adversary agents to enforce
  spec-first discipline, capability anchor justification, and partial-fix
  propagation checks at the prompt level
- **So that** the class of findings that generated 8+ adversarial passes in
  S-6.01 (F-001 empty BCs, F-011 anchor gap, F-023 incomplete propagation) is
  structurally prevented rather than rediscovered each cycle

## Acceptance Criteria

### AC-001: Story-writer spec-first gate (traces to BC-5.36.001, BC-5.36.002)

The story-writer agent prompt must include an explicit **Spec-First Gate** rule:

> "A story's `status:` field MUST NOT be set to `ready` if `behavioral_contracts: []`
> is empty (or absent). The status must remain `draft` until a product-owner has
> authored and anchored BCs for this story. The story file must include an
> explicit note in frontmatter comment: 'BC status: pending PO authorship'."

The gate applies to both new stories and status-transition edits. A story may
be written with `behavioral_contracts: []` during decomposition — this is
expected — but the status transition `draft → ready` is blocked until the array
is populated.

- **Test:** Peer review of story-writer.md confirms Spec-First Gate section is
  present with the above rule text (or equivalent semantics).
- **Test:** Story-writer prompt change is reflected in the "Constraints" or
  "Rules" section (whichever is canonical in the agent file structure).

### AC-002: Product-owner capability anchor justification (traces to BC-5.36.003, BC-5.36.004)

The product-owner agent prompt must include a **Capability Anchor Justification**
requirement:

> "Every behavioral contract MUST include a Capability Anchor Justification cell
> in its Traceability section. The justification MUST cite the specific capability
> name and file verbatim from `specs/domain-spec/capabilities.md` — e.g.,
> 'CAP-017 ("Create and manage formal ADR records") per capabilities.md §CAP-017'.
> A bare CAP-NNN with no verbatim citation is insufficient. The adversary is
> explicitly instructed (per adversary policy 5) to treat missing justification
> as a MEDIUM-severity finding."

- **Test:** Peer review of product-owner.md confirms Capability Anchor
  Justification requirement is present in the BC authoring section.
- **Test:** Adversary policy reference to "policy 5" or equivalent label for
  anchor justification enforcement is present in adversary.md.

### AC-003: Adversary partial-fix-regression check (traces to BC-5.36.005, BC-5.36.006)

The adversary agent prompt must add a mandatory **Partial-Fix Regression** review
axis to the Spec Review and Implementation Review sections:

> "For every finding closed in a prior pass (visible via the convergence report or
> fix commit), the adversary MUST verify: (a) the fix is correct and complete in
> the primary file, AND (b) the same pattern exists in sibling files (other BCs
> in the same subsystem, other agent prompts, other template files of the same
> type) and those siblings received the same fix if applicable.
>
> Report any 'fix applied to primary, sibling not updated' as a MEDIUM or HIGH
> finding depending on the blast radius."

This closes F-023 (propagation incomplete between BC files) and F-027 (fix
applied to capabilities.md but sibling edge-cases.md not updated) from
s6.01-pass-1.md.

- **Test:** Peer review of adversary.md confirms Partial-Fix Regression axis is
  present in the Spec Review section (or as a new "Regression Propagation" axis).
- **Test:** The axis wording explicitly mentions "sibling files" and cites the
  class of pattern (same-subsystem BCs, same-type agent prompts).

### AC-004: Agent prompt update atomicity (traces to BC-5.36.007)

All three agent prompt files (story-writer.md, product-owner.md, adversary.md)
must be updated in the same delivery. A partial update where one or two agents
are updated but the third is not constitutes an incomplete delivery and fails
this story.

- **Test:** Deliverable diff shows all three files modified.
- **Test:** validate-template-compliance.sh (if applicable to agent prompt files)
  exits 0 for all three.

### AC-005: No behavioral regression in existing agent rules (traces to BC-5.36.007)

The additions to each agent prompt must be purely additive. Existing rules,
policies, constraints, and workflow steps in story-writer.md, product-owner.md,
and adversary.md must not be removed or semantically weakened.

- **Test:** Diff review confirms no deletions in existing policy sections.
- **Test:** Adversary pass against the updated agents confirms no NEW findings
  introduced by the prompt changes themselves.

## Edge Cases

|----|----------|-------------------|
| EC-001 | A session starts with an agent that has the old prompt (no spec-first gate) cached before this story merges | Prompt changes take effect on the next agent dispatch. In-flight sessions complete with old rules; only new dispatches after merge use updated rules. No rollback needed — prompts are not stateful. |
| EC-002 | Story-writer prompt change causes story-writer to reject a story it would previously have accepted (regression) | AC-005 explicitly requires no semantic weakening. | EC-003 | Product-owner writes a BC after this story ships but before the capability anchor justification requirement is parsed by the PO's model | The PO prompt now contains the requirement verbatim. If a model ignores it, the adversary's updated policy 5 axis catches the omission in the next review pass. Defense-in-depth: prompt + adversary audit. |
| EC-004 | Adversary partial-fix-regression axis produces false positives (flags a "sibling" file that is intentionally different) | The adversary must provide a specific file:line citation (per existing evidence-check rule) and explain WHY the sibling should have the fix. The orchestrator adjudicates. False positives are MEDIUM-severity adversary findings at worst — not blocking. |
| EC-005 | validate-template-compliance.sh is not configured to run against agent prompt .md files | Implementer checks whether the compliance script scope includes agents/. If not, peer review is the enforcement mechanism. Note in Architecture Compliance Rules section. |
| EC-006 | Delivery ships story-writer and adversary updates but omits product-owner (partial delivery) | AC-004 requires all three in the same delivery. A partial delivery fails the story's definition of done. The implementer must not mark the PR ready until all three files are in the diff. |
