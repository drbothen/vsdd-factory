# S-0.2: Release workflow prerelease handling

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 0 — Infrastructure prep
**Milestone:** `1.0.0-beta.1`
**Tier:** A

**Blocked by:** none (parallel with S-0.1)
**Estimated effort:** S

## Goal

The `.github/workflows/Release.yml` workflow currently assumes stable
versions. Extend it to handle prerelease tags (`1.0.0-beta.1`, `1.0.0-rc.2`)
by marking GitHub Releases as pre-release, so users can distinguish stable
from prereleases and marketplace defaults to stable.

## Acceptance criteria

- [ ] Pushing a tag `1.0.0-beta.1` creates a GitHub Release with
  `prerelease: true`
- [ ] Pushing a tag `1.0.0-rc.1` creates a GitHub Release with
  `prerelease: true`
- [ ] Pushing a tag `1.0.0` creates a GitHub Release with `prerelease: false`
  (no regression)
- [ ] Pushing a tag `0.79.5` creates a GitHub Release with
  `prerelease: false` (no regression on existing release stream)
- [ ] Release notes body includes a note about prerelease status at the top
  when applicable
- [ ] "Latest Release" pointer on GitHub continues to point to the most
  recent stable, not prereleases (verified behavior of GH's `latest`
  semantics)

## Tasks

1. Read current `.github/workflows/Release.yml` to identify the
   `softprops/action-gh-release@v2` (or similar) step.
2. Add a conditional computation: `is_prerelease=$(echo "$TAG" | grep -q '-' && echo true || echo false)`
3. Pass `prerelease: ${{ env.IS_PRERELEASE }}` to the action
4. Optionally prepend a prerelease banner to the release body when
   `is_prerelease=true`
5. Test on a throwaway prerelease tag (`0.79.5-test.1` or similar) against a
   fork or a dry-run.

## Artifacts

- `.github/workflows/Release.yml` — conditional prerelease detection + flag

## Contracts

- **Prerelease detection**: any tag matching `*-*` is treated as prerelease.
  This matches semver where prerelease is signaled by a hyphen after the
  version core.
- **GH Release "prerelease" field**: set to `true` for prerelease tags,
  `false` for stable tags. Marketplace tools reading from GH Releases rely
  on this field.

## Test plan

- Manual: create a test tag on a fork, observe the resulting Release is
  marked prerelease.
- Manual after S-0.1 and S-2.8: first `1.0.0-beta.1` tag produces a
  correctly-flagged prerelease on GH.
- No bats coverage needed (YAML workflow; tested via real-run observation).

## Notes

`softprops/action-gh-release@v2` supports `prerelease: true` per its docs.
If we're using a different action, adjust accordingly.

GH Release semantics: "Latest" is the most recent non-prerelease, regardless
of publish order. This is exactly what we want — marketplace `/plugin install`
gets 0.79.4 until 1.0.0 is tagged, even as prereleases ship in between.
