# S-4.4: Per-sink retry + circuit breaker

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-1.8, S-4.1
**Estimated effort:** M

## Goal

Implement the shared resilience runtime that all HTTP-based sinks
(`sink-http`, `sink-datadog`, `sink-honeycomb`, `sink-otel-grpc`) use
for retry and circuit-breaker behavior. Per Q6 resolution:
exponential backoff with jitter; retry classification by error type;
circuit breaker to isolate persistently-failing sinks; emits
`internal.sink_*` events so operators can observe sink health.

## Acceptance criteria

- [ ] `ResilienceLayer` struct wraps any async HTTP submission, applies
  retries + circuit breaker per sink's config
- [ ] Exponential backoff: `initial_backoff_ms * (multiplier ** attempt)
  + jitter`, capped at `max_backoff_ms`
- [ ] Default: 3 attempts, 50ms base, ×2 multiplier, 30s cap, 10% jitter
- [ ] Retry classification:
  - Network error, 5xx, 429, timeout → retry
  - 4xx other than 429, malformed payload → terminal fail (no retry)
  - Configurable overrides per sink (`retry_on_4xx`)
- [ ] `Retry-After` header honored for 429 responses
- [ ] Circuit breaker: N consecutive failures → open for cooldown period
  (exponentially extending); probe + close on success
- [ ] Default: 5 failures, 30s cooldown, ×2 multiplier, 5min cap
- [ ] Emits `internal.sink_circuit_opened`,
  `internal.sink_circuit_closed`, `internal.sink_error` events
- [ ] Non-blocking: resilience layer runs async, never blocks the
  `emit_event` path
- [ ] Per-sink state tracked independently (two sinks can have
  different circuit states)
- [ ] Integration tests simulate various failure modes; verify events
  and state transitions

## Tasks

1. Create `crates/sink-common/` for shared sink infrastructure.
2. Implement `ResilienceLayer` wrapping a `submit` function:
   ```rust
   pub async fn submit_with_resilience<F, Fut>(
       sink_name: &str,
       attempt_fn: F,
       config: &ResilienceConfig,
       state: &SinkState,
   ) -> Result<()>
   where
       F: Fn(attempt: u32) -> Fut,
       Fut: Future<Output = Result<(), SubmitError>>,
   ```
3. Error classification helper:
   ```rust
   pub enum SubmitError {
       Network(String),       // always retry
       Http4xx(u16, String),  // don't retry unless retry_on_4xx
       Http5xx(u16),          // always retry
       RateLimited(Option<Duration>),  // retry honoring duration
       Timeout,               // retry
       Malformed,             // don't retry
   }
   ```
4. Circuit breaker state machine:
   - `Closed` → normal submission
   - `Open` → reject immediately, emit queue-drop behavior; set a
     timer to transition to HalfOpen after cooldown
   - `HalfOpen` → allow ONE attempt; if succeeds → Closed, if fails →
     Open with extended cooldown
5. Emit `internal.sink_*` events on state changes.
6. Per-sink isolation — each `SinkState` is independent.
7. Integration tests:
   - Fake sink that returns 500 N times → verify N retries + final
     terminal error
   - Fake sink that returns 429 with Retry-After → verify honored
   - Fake sink that goes down for 60s → circuit opens, probe succeeds
     when back up
   - Two sinks, one fails → verify other is unaffected

## Contracts

**`ResilienceConfig` (from TOML):**

```rust
pub struct ResilienceConfig {
    pub max_attempts: u32,             // default 3
    pub initial_backoff_ms: u64,       // default 50
    pub max_backoff_ms: u64,           // default 30000
    pub multiplier: f64,               // default 2.0
    pub jitter: bool,                  // default true
    pub retry_on_4xx: bool,            // default false
    pub circuit_breaker: CircuitBreakerConfig,
}

pub struct CircuitBreakerConfig {
    pub enabled: bool,                 // default true
    pub failure_threshold: u32,        // default 5
    pub initial_cooldown_ms: u64,      // default 30000
    pub max_cooldown_ms: u64,          // default 300000
    pub cooldown_multiplier: f64,      // default 2.0
}
```

**Emitted events:**

```
internal.sink_error:
    { sink_name, attempt, error_class, message, will_retry: bool }

internal.sink_circuit_opened:
    { sink_name, consecutive_failures, cooldown_ms }

internal.sink_circuit_closed:
    { sink_name }

internal.sink_retried:
    { sink_name, attempt, next_backoff_ms }
```

All routed through the normal sink pipeline (to other healthy sinks
and to `dispatcher-internal.jsonl`).

## Notes

Cooldown extension across repeated failures is important — if a sink is
down for 10 minutes, we shouldn't probe every 30s (that's 20 wasted
requests). Exponential cooldown extends the probe interval naturally.

Circuit-breaker on a half-open probe that fails extends the cooldown;
circuit doesn't re-try immediately.

The resilience layer is generic over the HTTP client — designed to be
reusable for any future sink type. Don't couple it to `reqwest`
specifically; use a trait / closure so tests can inject fakes.

Be careful with `internal.sink_error` events — avoid infinite recursion
(error emitting its own emission error). The router must tag
`internal.*` events specially and never retry them.
