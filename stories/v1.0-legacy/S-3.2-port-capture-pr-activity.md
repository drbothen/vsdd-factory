# S-3.2: Port capture-pr-activity to WASM

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 3 — High-value port to native WASM
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-2.8, S-3.4
**Estimated effort:** M

## Goal

Rewrite `capture-pr-activity` from bash to native WASM. More complex
than commit-activity because it handles multiple `gh pr` subcommands
(`create`, `merge`, `close`) with different payload shapes and emits
different event types (`pr.opened`, `pr.merged`, `pr.closed`). Also
computes open→merge duration from event history.

## Acceptance criteria

- [ ] `capture-pr-activity.wasm` compiles and runs
- [ ] Behavior parity with bash version:
  - Fires on PostToolUse Bash with `gh pr create|merge|close`
  - Parses PR number, repo, URL from stdout
  - For `gh pr merge --squash`: also computes `open_to_merge_duration`
    by reading prior `pr.opened` event from the file sink (if present)
  - Emits `pr.opened`, `pr.merged`, `pr.closed` with appropriate fields
- [ ] Open→merge duration computation: reads from current day's file
  sink (via `read_file` host fn with bounded size), finds matching
  `pr.opened` event, computes delta
- [ ] Graceful handling when file sink has rotated past the pr.opened
  event (duration computation skipped, not a failure)
- [ ] Registered in `hooks-registry.toml`
- [ ] Event parity tests on 20+ real `gh pr ...` stdout samples
- [ ] Latency < 10ms typical (file read adds time vs commit-activity)
- [ ] Works on all 5 platforms

## Tasks

1. Create `crates/hook-plugins/capture-pr-activity/`.
2. Port the bash logic:
   - Regex match `gh pr (create|merge|close|reopen)` subcommand
   - Subcommand-specific parsers for stdout JSON or URL patterns
   - For merge: `read_file(".factory/logs/events-<today>.jsonl",
     max_bytes=10_000_000)` — scan for matching `pr.opened` by pr_number
   - Compute duration if found, skip silently if not
3. Emit events matching existing schemas.
4. Update `hooks-registry.toml`.
5. Unit + event-parity tests.
6. Ensure capabilities include `read_file` for the duration lookup.

## Contracts

**Events (unchanged from bash version):**

```
pr.opened:  { pr_number, pr_repo, pr_url, author?, branch?, base? }
pr.merged:  { pr_number, pr_repo, pr_url, merge_strategy,
              open_to_merge_duration_seconds? }
pr.closed:  { pr_number, pr_repo, pr_url, reason? }
```

**File sink lookup contract:**

If the `capture-pr-activity` plugin is registered, its registry entry
must include `read_file` capability for the `.factory/logs/` path:

```toml
[hooks.capabilities.read_file]
paths = ["${CLAUDE_PROJECT_DIR}/.factory/logs"]
max_bytes = 10485760   # 10MB
```

Host enforces that read_file is only allowed under that path.

## Test plan

- Unit: 20+ fixture `gh pr X` stdouts; verify extracted fields
- Event parity: same fixtures through bash + WASM
- Duration test: synthesize a pr.opened event in a test file, invoke
  plugin with a matching pr.merged stdout, verify duration field
- Duration-missing test: no prior pr.opened; verify event still emits,
  just without duration
- Latency benchmark < 10ms

## Notes

The bash version reads the file via `grep | jq`. The WASM port does the
same work in Rust — parse JSONL line by line, match on `type == "pr.opened"
&& pr_number == target`.

Alternative design: store pr.opened timestamps in a small state file
(e.g., `.factory/logs/pr-state.jsonl`) maintained by the plugin itself.
Simpler lookup, but adds state management. Defer to post-1.0 if we care.

Event parity is slightly trickier here because the bash version might
emit fields in a specific order; WASM emits via host function which
normalizes. `jq --sort-keys` before diffing to account for this.
