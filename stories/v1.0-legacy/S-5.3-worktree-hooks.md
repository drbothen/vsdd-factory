# S-5.3: WorktreeCreate / WorktreeRemove hook wiring

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 5 — New Claude Code hook events
**Milestone:** `1.0.0`
**Tier:** G
**Blocks:** S-5.7
**Blocked by:** S-4.8
**Estimated effort:** M

## Goal

Wire `WorktreeCreate` and `WorktreeRemove` events. WorktreeCreate
auto-registers new worktrees with the observability stack — this is
exactly the automation that the v0.78.0 multi-factory register CLI was
reaching for. WorktreeRemove emits `worktree.removed` and cleans up
observability registration.

## Acceptance criteria

- [ ] Dispatcher handles `WorktreeCreate` and `WorktreeRemove` events
- [ ] `worktree-create-handler` WASM plugin:
  - Emits `worktree.created` event with worktree_path, branch,
    parent_repo
  - If factory-obs is running locally: calls `factory-obs register
    <worktree_path>` via `exec_subprocess` (best-effort; graceful if
    not running)
  - Verifies the worktree has the expected `.factory/` structure
- [ ] `worktree-remove-handler` WASM plugin:
  - Emits `worktree.removed` event
  - Calls `factory-obs unregister <worktree_path>` (best-effort)
- [ ] Registered in `hooks-registry.toml`
- [ ] Integration test: CC-simulated WorktreeCreate → correct event +
  registration attempt; WorktreeRemove → correct event + unregistration
- [ ] Works cross-platform

## Tasks

1. Create two plugin crates: `worktree-create-handler`,
   `worktree-remove-handler`.
2. Implement event emission.
3. Implement `factory-obs` registration call via `exec_subprocess` with
   allow-list `["factory-obs"]`.
4. Handle factory-obs absence gracefully — capability error → emit

5. Update registry + hooks.json.template.
6. Tests.

## Contracts

**Events:**

```json
{
  "type": "worktree.created",
  "worktree_path": "/path/to/.worktrees/story-X",
  "branch": "feature/X",
  "parent_repo": "/path/to/main/repo",
  "registered_with_obs": true | false
}

{
  "type": "worktree.removed",
  "worktree_path": "...",
  "unregistered_from_obs": true | false
}
```

**Capability declarations in registry:**

```toml
[hooks.capabilities.exec_subprocess]
binaries = ["factory-obs"]
```

## Notes

Claude Code emits WorktreeCreate/Remove when agent worktree isolation
creates/removes worktrees (see v2.0+ of CC; verify event payload
format at implementation time).

This obviates the v0.78.0 manual register CLI for NEW worktrees.
Existing worktrees (created before the upgrade) still need manual
registration via `factory-obs register`.

The registration call is synchronous during WorktreeCreate — the new
worktree is usable IMMEDIATELY after registration. Don't block
indefinitely if factory-obs is slow; 3s timeout_ms on the exec.
