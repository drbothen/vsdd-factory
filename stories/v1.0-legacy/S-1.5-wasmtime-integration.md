# S-1.5: wasmtime integration + epoch/fuel enforcement

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-1.6, S-2.1, S-3.1, S-3.2, S-3.3
**Blocked by:** S-1.1, S-1.2, S-1.4
**Estimated effort:** L

## Goal

Wire wasmtime into the dispatcher so it can instantiate a `.wasm` plugin,
run its `_start` entry point with a stdin payload, and enforce time + fuel
limits. This is where plugin isolation actually happens: each invocation
gets a fresh wasmtime `Store`, its own epoch deadline, its own fuel cap.
A plugin that exceeds either limit is interrupted cleanly, its result
recorded as a timeout, and the dispatcher moves on.

## Acceptance criteria

- [ ] Shared `wasmtime::Engine` with epoch interruption + fuel consumption
  enabled
- [ ] Dispatcher spawns an epoch-increment timer (background tokio task
  bumping the engine's epoch every N ms; N = 10)
- [ ] `load_plugin(path) -> Result<Module>` — compiles and caches modules
  by path + mtime (hot-reload-able)
- [ ] `invoke_plugin(module, payload, limits) -> PluginResult` — creates
  a Store, sets epoch deadline, sets fuel, pipes stdin, runs `_start`,
  catches traps
- [ ] Epoch-timeout produces `PluginResult::Timeout { elapsed_ms,
  fuel_consumed }`
- [ ] Fuel-exhaustion produces `PluginResult::Timeout` (same variant;
  `fuel_consumed >= cap` disambiguates)
- [ ] Panic/trap produces `PluginResult::Crashed { trap: String }`
- [ ] Normal completion produces `PluginResult::Ok { exit_code: i32,
  result: HookResult, elapsed_ms, fuel_consumed }`
- [ ] Fuel consumed recorded for latency histogram (even on successful
  plugins)
- [ ] Dispatcher emits `plugin.invoked` (start), `plugin.completed` (end),
  `plugin.timeout`, `plugin.crashed` events appropriately
- [ ] Integration test: plugin that runs normally, plugin that hangs in a
  loop, plugin that panics, plugin that exhausts fuel — all handled
- [ ] No memory leak across 10,000 invocations (verify with a loop test)

## Tasks

1. Create `crates/factory-dispatcher/src/engine.rs` — global lazily-
   initialized `wasmtime::Engine` with `Config::epoch_interruption(true)`
   and `Config::consume_fuel(true)`.
2. Create `crates/factory-dispatcher/src/plugin_loader.rs` — module cache
   keyed by `(plugin_path, mtime)`. Compile on first use, reuse on repeats.
3. Create `crates/factory-dispatcher/src/invoke.rs`:
   - `invoke_plugin(module, payload, limits, host_ctx) -> PluginResult`
   - Build `Linker` (from S-1.4), attach `HostContext`
   - Create `Store` with `HostContext`
   - Set `store.set_epoch_deadline(now + limits.timeout_ms / epoch_tick_ms)`
   - Set `store.add_fuel(limits.fuel_cap)`
   - Instantiate module against linker
   - Find `_start` export, call it
   - Match result: Ok / Trap / Timeout-trap (fuel or epoch)
4. Spawn epoch-tick tokio task in `main` that increments engine epoch
   every 10ms. Task is a long-running future; it's the only background
   work the dispatcher does per invocation.
5. Emit lifecycle events at start/completion/error. Each event carries
   `plugin_name`, `plugin_version`, `dispatcher_trace_id`, `elapsed_ms`,
   `fuel_consumed`.
6. Integration tests with handcrafted WASM plugins:
   - `hello-plugin`: emits an event, returns Continue
   - `hang-plugin`: infinite loop — must timeout
   - `fuel-hog-plugin`: tight arithmetic loop — must exhaust fuel
   - `panic-plugin`: calls `panic!()` — must Crashed
   - `block-plugin`: returns `HookResult::Block`
7. Benchmark: invoke `hello-plugin` 10K times, verify p50 latency < 5ms,
   p99 < 20ms, no leaked memory.

## Contracts

**`PluginResult` (consumed by S-1.6):**

```rust
pub enum PluginResult {
    Ok {
        hook_result: HookResult,          // from plugin
        elapsed_ms: u64,
        fuel_consumed: u64,
    },
    Timeout {
        cause: TimeoutCause,              // Epoch | Fuel
        elapsed_ms: u64,
        fuel_consumed: u64,
    },
    Crashed {
        trap_string: String,
        elapsed_ms: u64,
        fuel_consumed: u64,
    },
}
```

**Module cache contract:**

```rust
pub trait ModuleCache {
    fn get_or_compile(&self, path: &Path) -> Result<Module>;
    // mtime-invalidated internally; caller doesn't manage staleness
}
```

**Epoch tick rate:** 10ms. Timeout resolution = 10ms. A 5000ms timeout
becomes `set_epoch_deadline(500)`. Document this; plugin authors should
expect ~10ms jitter on timeouts.

## Notes

`wasmtime::Engine` is expensive to create (~100ms). Share one process-wide.
`wasmtime::Module` compilation is mildly expensive (~10ms typical); cache
by path+mtime so repeated invocations in the same process amortize.

Epoch-based interruption only fires at WASM yield points. A plugin with
no yield points (tight numeric loop) relies on fuel to exit. This is
exactly the hybrid we committed to in Q3.

Fuel exhaustion and epoch deadline both manifest as wasmtime traps. Inspect
the trap's cause to disambiguate. If neither — it was a real plugin panic
(call `panic!()` or hit an unreachable) — record as Crashed.

We do NOT support preview-2 component-model plugins yet (see ADR-003).
Modules are preview-1 WASI commands only.
