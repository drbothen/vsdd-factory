# S-3.3: Port block-ai-attribution to WASM

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 3 — High-value port to native WASM
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-2.8, S-3.4
**Estimated effort:** S

## Goal

Rewrite `block-ai-attribution` from bash to native WASM. The third port
and the strategically important one — this is a PreToolUse/Bash gate
(blocks commits containing AI attribution). Validates that the
dispatcher correctly routes PreToolUse events through WASM plugins and
that `HookResult::Block` propagates to Claude Code as exit 2.

## Acceptance criteria

- [ ] `block-ai-attribution.wasm` compiles and runs
- [ ] Behavior parity with bash version:
  - Fires on PreToolUse Bash
  - Regex-matches `git commit` subcommand
  - Scans commit message body for forbidden patterns:
    - `Co-Authored-By:` referencing Claude/Anthropic/GPT/OpenAI/Gemini/Google AI
    - `Generated with Claude Code`, `Generated by AI`
    - `noreply@anthropic.com`, `noreply@openai.com`
  - Returns `HookResult::Block("AI ATTRIBUTION BLOCKED: <reason>")` on match
  - Returns `HookResult::Continue` otherwise
- [ ] Dispatcher's on_error=block handling converts Block to exit 2
- [ ] Registered in `hooks-registry.toml` with `on_error = "block"`
- [ ] Emits `hook.block` event with reason code
  (`ai_attribution_coauthored`, `ai_attribution_generated`) on block
- [ ] Event parity tests across 30+ fixture commit messages (forbidden
  and allowed)
- [ ] End-to-end test: real Claude Code session tries to commit with
  Co-Authored-By → blocked
- [ ] Latency < 3ms (small regex set, no I/O)

## Tasks

1. Create `crates/hook-plugins/block-ai-attribution/`.
2. Port bash regex patterns to Rust `regex` crate.
3. Implement `on_hook`:
   - Extract command from tool_input.command
   - Check for `git commit` — early return Continue if not
   - Run each forbidden pattern against the command string
   - On first match: emit `hook.block` event, return Block
4. Update `hooks-registry.toml`:
   ```toml
   [[hooks]]
   name = "block-ai-attribution"
   event = "PreToolUse"
   tool = "Bash"
   plugin = "hook-plugins/block-ai-attribution.wasm"
   priority = 10  # runs early (pre-commit)
   on_error = "block"
   ```
5. Unit tests for every pattern.
6. End-to-end manual test.

## Contracts

**Block event emitted:**

```json
{
  "type": "hook.block",
  "hook": "block-ai-attribution",
  "matcher": "Bash",
  "reason": "ai_attribution_coauthored" | "ai_attribution_generated",
  "command": "<first 100 chars, redacted>"
}
```

**Return behavior:**

- Non-match: `HookResult::Continue`
- Match: `HookResult::Block(human_message)` — dispatcher treats as
  exit 2 via on_error policy

## Notes

This is the hook that validates both halves of the v0.79.x matcher bug
are now routed correctly. The bash version's PreToolUse Bash matcher
worked; our new dispatcher should reproduce that behavior via
matcher-free routing + internal tool filtering.

The regex patterns need to be case-insensitive. Use `regex` crate's
`(?i)` inline flag.

Message format for the user-visible block: match the bash version's
output ("AI ATTRIBUTION BLOCKED: ...") so operators get consistent
messaging across the transition.

The current bash version uses `set -euo pipefail` + `exit 2` semantics
on block. The WASM version returns `HookResult::Block` which the
dispatcher translates to exit 2 via the priority-tier aggregation in
S-1.6.
