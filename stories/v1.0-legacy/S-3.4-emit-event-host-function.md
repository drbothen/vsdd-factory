# S-3.4: emit_event as host function refactor

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 3 — High-value port to native WASM
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-3.1, S-3.2, S-3.3 (port stories consume the host fn)
**Blocked by:** S-1.4 (host function scaffolding)
**Estimated effort:** S

## Goal

Complete the `emit_event` host function's implementation (S-1.4 stubs
it) so WASM plugins can emit events through the same pipeline that
sinks consume. Also retire the bash `emit-event` binary's role as a
standalone hook tool — it remains as a shell-script-invocable
compatibility bridge during the adapter window, but the new canonical
path is the host function.

## Acceptance criteria

- [ ] `emit_event(type, fields)` host function fully implemented
- [ ] Plugin-side SDK provides an ergonomic `emit_event` wrapper (from
  S-1.3)
- [ ] Event enrichment applied consistently (dispatcher_trace_id,
  session_id, plugin_name, plugin_version, ts, ts_epoch, schema_version)
- [ ] Events reach every enabled sink per routing filters
- [ ] Backpressure: mpsc channel bounded (1024 default); on full,
  drop-oldest + emit `internal.emit_event_queue_full`
- [ ] Non-blocking: plugin returns in <1ms regardless of sink health
- [ ] The bash `emit-event` script is annotated as "compatibility layer;
  new code should use the SDK's emit_event"
- [ ] Integration test: WASM plugin emits 100 events; verify all land
  in file sink + otel-grpc sink (as configured)
- [ ] Event field override protection: plugin attempting to set
  `dispatcher_trace_id` or `session_id` itself → dispatcher overrides
  with canonical values

## Contracts

**Host ABI signature:**

```
emit_event(type_ptr: u32, type_len: u32,
           fields_ptr: u32, fields_len: u32) -> ()
```

Where `fields` is a sequence of length-prefixed (u32-BE) UTF-8 strings:
key1_len, key1, value1_len, value1, key2_len, key2, ...

**Plugin-facing SDK wrapper:**

```rust
pub fn emit_event(event_type: &str, fields: &[(&str, &str)]) {
    // Serializes fields, calls host ABI
}
```

**Dispatcher enrichment (applied server-side, not overridable):**

- `dispatcher_trace_id`
- `session_id`
- `plugin_name`
- `plugin_version`
- `ts` (ISO-8601)
- `ts_epoch` (unix seconds)
- `schema_version` (always 1 for v1.0)

These override any plugin-provided values silently. The plugin-provided
attempt is logged to `dispatcher-internal.jsonl` at `warn` level.

**Queue semantics:**

- Bounded channel depth: 1024 events
- On full: drop oldest; emit `internal.emit_event_queue_full` (throttled
  to 1/sec)
- Non-blocking from plugin perspective

## Notes

The bash `emit-event` script remains usable for scripts invoked via
`exec_subprocess` from the legacy-bash-adapter. It writes to the same
file sink path the new pipeline does, so events land in one place.

Post-1.0, when the adapter retires, `emit-event` can be deleted.

Event enrichment as server-side override is a security boundary —
plugins cannot forge their identity. This is a foundation for any future
audit-logging or compliance work.

Drop-oldest vs drop-newest: chose oldest because dropped old events are
less valuable than dropped new events during a burst (operator cares
about the most recent signal). Configurable per-sink in S-4.4 for
sinks that prefer the opposite.
