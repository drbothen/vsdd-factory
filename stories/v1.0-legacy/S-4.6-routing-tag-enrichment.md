# S-4.6: Per-sink routing filters + tag enrichment

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-1.8 (Sink trait)
**Estimated effort:** S

## Goal

Implement the routing-filter and tag-enrichment features every sink
supports: routing filters let operators send specific event types to
specific sinks (e.g., audit events to an audit-dedicated file, metrics
to Datadog); tag enrichment adds sink-specific fields to events as they
leave that sink (e.g., `env:prod` tag only on the prod Datadog sink).

## Acceptance criteria

- [ ] Each `Sink` trait impl supports a `route` filter in config
- [ ] Filter matches by `event_types` allowlist (default: all)
- [ ] Filter matches by `severity` threshold (future-ready; not used by
  any event in v1.0 but schema supports it)
- [ ] Unmatched events skip the sink entirely (not dropped — just not
  sent there)
- [ ] Each sink supports a `tags` list of `key:value` strings
- [ ] Tags merged into every event's attributes before submission
- [ ] Tag keys don't override dispatcher-enrichment keys (identity fields
  are protected)
- [ ] Integration test: two file sinks, one with `event_types =
  ["commit.made"]`, one unfiltered; verify routing correctness
- [ ] Integration test: datadog sink with `tags = ["env:prod"]` +
  honeycomb sink with `tags = ["env:prod", "source:dev"]`; verify each
  sink's events have its own tags

## Tasks

1. Extend `SinkConfig` in `sink-core` to include `route` + `tags`.
2. Extend `Sink` trait's default `accepts(event)` impl to consult
   `route`.
3. In the router (dispatcher side), before handing an event to a sink,
   apply the sink's tags (clone event, merge tags).
4. Tag key collision handling: silently drop conflicting keys that
   would override `dispatcher_trace_id`, `session_id`, `plugin_name`,
   `plugin_version`, `ts`, `ts_epoch`, `schema_version`.
5. Integration tests per the criteria.

## Contracts

**Config:**

```toml
[[sinks]]
name = "audit-archive"
type = "file"
path = "/var/log/factory/audit-{date}.jsonl"
tags = ["category:audit"]

[sinks.audit-archive.route]
event_types = ["hook.block", "pr.merged", "commit.made"]
# severity = "warn"                     # not used in v1.0, reserved
```

**Event-type matching:**

- If `event_types` is absent → match all events
- If `event_types` is present → match only listed types
- Matching is exact-string (no wildcards); deliberate keep-it-simple
- Future extension: regex match via `event_types_regex`

**Tag format:**

`["key:value", "key:value", ...]`

Parsed into `HashMap<String, String>` at config load; applied as
event attributes at submission time.

## Test plan

- Unit: event_types filter matches expected subset
- Unit: tag merge respects identity-field protection
- Integration: multi-sink scenario with different routes + tags,
  verify each sink receives correctly-filtered and -enriched events

## Notes

Routing filters were covered in Q6 resolution but implementation lives
here. The filter runs in the router, not in the sink driver itself —
keeps driver code focused on protocol concerns.

Tags could conflict with original event fields. Policy: sink tags take
precedence EXCEPT for the 7 identity fields. Document this clearly in
the observability-sinks.md doc so operators don't accidentally
shadow legitimate event data.

Severity threshold is scaffolded but unused — no factory event emits a
`severity` field in v1.0. Included in the schema so post-1.0 work (e.g.,
error categorization) can use it without a schema bump.
