## Goal

Write the public semver commitment doc at
`docs/guide/semver-commitment.md`. States exactly what's stable after
1.0, what isn't, and how we handle breaking changes. Gate 18 of the
1.0.0 success criteria requires this to be published before the tag.

## Acceptance criteria

- [ ] `docs/guide/semver-commitment.md` exists and is complete
- [ ] Specifies the stability boundary:
  - **Stable (breaking change = major bump)**: plugin ABI
    (`HOST_ABI_VERSION`), `vsdd-hook-sdk` public API, `hooks-registry.toml`
    schema, `observability-config.toml` schema, event schema (event
    types + top-level fields)
  - **Possibly-breaking (minor bump at most)**: event field values (new
    fields allowed, renames require major), plugin filesystem layout
    inside the plugin package, internal dispatcher behavior that
    doesn't affect plugins
  - **Not covered**: bash-adapter script paths (those may move as we
    port hooks), internal config defaults (may tune), `emit-event` bash
    binary behavior (remains compatible but not versioned)
- [ ] Specifies deprecation process:
  - Deprecated features marked in docs + SDK
  - Remain functional for at least one minor version
  - Removed only at the next major version
- [ ] Specifies the breaking-change process:
  - RFC / design-doc first in `.factory/specs/`
  - Migration guide alongside the major release
  - Deprecation window (at least one minor release) for affected APIs
- [ ] Linked from `docs/guide/index.md`, CHANGELOG's 1.0.0 entry,
  release notes

## Contracts

**Stability boundaries (canonical list):**

| Layer | Stable at 1.0? | Breaking change requires |
|---|---|---|
| `HOST_ABI_VERSION` (WASM plugin ABI) | ✅ | major bump on dispatcher + SDK |
| `vsdd-hook-sdk` public API | ✅ | major bump on SDK |
| `hooks-registry.toml` schema | ✅ | major bump |
| `observability-config.toml` schema | ✅ | major bump |
| Event type names + top-level schema | ✅ | major bump |
| Event field additions (new fields only) | ❌ (compatible) | — |

| hooks.json structure (CC-driven) | N/A | Claude Code's contract |

| Bash hook script internals | ❌ | change freely during 1.x |

## Notes

Semver commitment is a product promise, not just a technical rule.
Operators plan around it; breaking changes have a cost to them. Err on
the side of more stability, not less.

The `HOST_ABI_VERSION` is the most-critical layer because a plugin
compiled against SDK 1.0 must run on dispatcher 1.5 without recompile.
If we can't guarantee that, the whole "zero-recompile to add a hook"
promise breaks.

Post-1.0, we maintain the 1.x line with security and bug fixes; new
features go in 1.x minor bumps; 2.0 (WASI preview 2 migration?) is
intentionally distant.
