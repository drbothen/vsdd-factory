# S-2.6: Activation skill integrates with real hooks.json variants

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.8
**Blocked by:** S-0.3 (detection logic), S-0.4 (hooks.json.<platform>
  variants exist in repo), S-2.4 (release workflow commits binaries)
**Estimated effort:** S

## Goal

Take the platform-detection logic from S-0.3 and the CI-generated
`hooks.json.<platform>` variants from S-0.4, and wire them into the
existing `/vsdd-factory:activate` skill so activation actually produces
a working `hooks.json` for the user's platform. This is the
integration story that completes the activation-skill-driven platform
selection strategy from Q5.

## Acceptance criteria

- [ ] `/vsdd-factory:activate` skill detects platform (from S-0.3 logic)
- [ ] Copies the matching `hooks.json.<platform>` to `hooks.json`
- [ ] Verifies the dispatcher binary exists at the expected path and is
  executable (`test -x` on Unix, `Test-Path` on Windows)
- [ ] If binary is missing, fails with a clear error and restoration
  instructions
- [ ] Records `activated_platform` + `activated_plugin_version` +
  `activated_at` in `.claude/settings.local.json`
- [ ] Re-activation with a drift (different platform detected) warns and
  requires explicit confirmation
- [ ] `/vsdd-factory:deactivate` removes the generated `hooks.json` and
  clears persisted state
- [ ] Integration test: fresh checkout of the plugin, `activate` on
  Linux, verifies hooks.json has correct absolute paths
- [ ] Integration test: same on macOS, verifies platform swap
- [ ] Skill emits activation-related events via emit-event (or
  dispatcher-internal.jsonl if the dispatcher is already running): 
  `factory.activated`, `factory.deactivated`, `factory.platform_drift_warning`
- [ ] Documentation in `docs/guide/getting-started.md` updated to note
  that `/vsdd-factory:activate` is now a REQUIRED first step after
  install (not optional)

## Contracts

**Activation side effects:**

1. `plugins/vsdd-factory/hooks/hooks.json` = copy of
   `hooks.json.<detected-platform>`
2. `.claude/settings.local.json` under `vsdd-factory`:
   ```json
   {
     "activated_platform": "<platform>",
     "activated_plugin_version": "<version>",
     "activated_at": "<iso-8601>"
   }
   ```
3. Emit `factory.activated` event (to whatever observability is live)

**Deactivation side effects:**

1. `hooks.json` removed (gitignored, so no git side effect)
2. Persisted state in `settings.local.json` cleared
3. Emit `factory.deactivated` event

**Drift detection (re-activation):**

```
if previously_activated_platform != currently_detected_platform:
    warn operator with message explaining the drift
    require explicit `force=true` flag or prompt to proceed
```

Scenario: user activated on Mac, SSH'd into a Linux server with the same
factory mounted. Detection sees Linux; persisted state says Mac. Drift
warning surfaces; user confirms "yes, I want to re-platform this factory."

## Test plan

- Unit: missing binary → verify clear error
- Unit: mock drift → verify warning behavior
- Integration (Linux): activate from scratch; verify hooks.json is
  linux-x64 variant with correct dispatcher path
- Integration (macOS): same on macOS
- Integration: deactivate, then re-activate, verify idempotent

## Notes

This story is the integration point of several Tier A stories. It's
small in code but critical in correctness — if this is broken, the
plugin doesn't work on first install, which kills the user experience.

Windows testing lands in S-2.7 (regression test suite validation) once
we have access to a Windows environment.

Consider: should activate also run `/vsdd-factory:factory-health` as
its final step? It would catch issues like missing binaries
immediately. Propose: yes, and include the result in the activation
event.

The skill should be safe to re-run (idempotent). If it was activated
yesterday and you run it again today, it should simply re-verify and
exit cleanly.
