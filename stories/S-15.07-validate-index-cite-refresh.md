# S-15.07: validate-index-cite-refresh WASM Hook

**Epic:** E-12 — Context Resolvers + Platform Stories (brownfield-backfill wave, S-15.03 PRIORITY-A M2)
**Delivery order:** M2 Wave-1 (first of four M2 WASM hook stories; S-15.11 → S-15.09 → S-15.14 follow)
**Depends on:** none (zero blocking dependencies per architect adjudication Q2)
**Blocks:** none (S-15.11, S-15.09, S-15.14 reference this story's crate structural patterns but do not
  formally depend on it via `depends_on`)
**Priority:** P1 (closes D-405(c) + D-429(b); highest-visibility M2 deliverable per architect)

## Narrative

- **As a** state-manager or agent editing `ARCH-INDEX.md` in a fix burst
- **I want to** receive an immediate PostToolUse block when the ARCH-INDEX body cites a stale version
  of any 4-index file (BC-INDEX, VP-INDEX, STORY-INDEX, or ARCH-INDEX itself), and additionally
  when STATE.md or INDEX.md trajectory-bearing cells disagree with the live 4-index versions
- **So that** the class of cross-cell stale-cite drift findings that the F5 engine-discipline cycle
  hammered against across 30+ passes (D-405(c), D-429(b)) is caught mechanically at write time
  rather than discovered by the adversary N bursts later

## Why This Exists

The F5 engine-discipline cycle (passes 25–74) generated a persistent class of adversary findings:
the ARCH-INDEX body cites of BC-INDEX/VP-INDEX/STORY-INDEX versions would become stale the moment
any of those indexes was bumped in a fix burst, but the ARCH-INDEX cite was only updated when the
adversary caught it. D-405(c) codified the problem; D-429(b) extended it to cross-cell sibling
sweep (STATE.md trajectory-bearing cells and INDEX.md Convergence Status cells must agree on the
same 4-index versions that ARCH-INDEX cites). Manual sweeps never held — every cycle boundary
regenerated new stale-cite drift.

This hook makes that entire failure class structurally impossible at PostToolUse time: any write to
ARCH-INDEX.md that leaves a stale version cite is blocked before the edit lands in the conversation.

## Goal

Deliver a production-grade WASM hook (`validate-index-cite-refresh`) that PostToolUse-blocks any
Edit/Write to `ARCH-INDEX.md` containing a stale version citation for any of the four canonical
factory indexes, and additionally cross-validates that STATE.md and INDEX.md cells agree with those
live versions (D-429(b) sibling sweep).

## Behavioral Contract

### BC-5.39.003: validate-index-cite-refresh hook MUST block on stale 4-index version citations in ARCH-INDEX, STATE.md, and INDEX.md

**Subsystem:** SS-05 (Engine Governance — per-story adversary workflow and discipline gate hooks)
**Capability:** E-12 Engine Governance (automation sub-capability: stale-cite prevention)
**BC ID:** BC-5.39.003
**File:** `.factory/specs/behavioral-contracts/ss-05/BC-5.39.003.md`

#### Preconditions

1. A PostToolUse Edit/Write event has fired on a file matching `ARCH-INDEX.md`.
2. The dispatcher has invoked the `validate-index-cite-refresh` WASM plugin with the write payload.
3. The four canonical index files are readable via `host::read_file`:
   - `.factory/specs/behavioral-contracts/BC-INDEX.md`
   - `.factory/specs/verification-properties/VP-INDEX.md`
   - `.factory/stories/STORY-INDEX.md`
   - `.factory/specs/architecture/ARCH-INDEX.md`
4. `host::read_file` returns content within the 65536-byte cap and 5000 ms timeout.
5. STATE.md (`.factory/STATE.md`) and the brownfield cycle INDEX.md
   (`.factory/cycles/v1.0-brownfield-backfill/INDEX.md`) are readable for D-429(b) cross-cell check.

#### Postconditions

1. If ALL of the following hold, the hook emits `HookResult::Continue` (pass):
   - Every version string extracted from ARCH-INDEX body matching `BC-INDEX v\d+\.\d+`,
     `VP-INDEX v\d+\.\d+`, `STORY-INDEX v\d+\.\d+`, or `ARCH-INDEX v\d+\.\d+` equals the
     `version:` frontmatter field of the corresponding index file.
   - Every version string extracted from STATE.md matching the same 4-index patterns agrees
     with the corresponding live index frontmatter value.
   - Every version string extracted from INDEX.md Convergence Status cells agrees likewise.
2. If ANY cited version string in ARCH-INDEX body is older (numerically less) than the live index
   version, the hook emits `HookResult::BlockWithFix` with:
   - A human-readable message naming the stale file, the cited version, and the live version.
   - A remediation action telling the author to update the ARCH-INDEX cite to the live version.
3. If ANY cross-cell version string in STATE.md or INDEX.md disagrees with the live index
   frontmatter, the hook emits `HookResult::BlockWithFix` citing the specific cell and the
   discrepancy.
4. If `host::read_file` returns an error (timeout, capability denied, file not found), the hook
   emits `HookResult::Continue` and logs a warning via `host::log_warn` — a missing index file
   is not itself a structural violation at write time.
5. Version comparison is NUMERIC not lexicographic: `v2.9` is older than `v2.10`. Both major
   and minor parts are parsed as integers.

#### Invariants

1. The hook NEVER writes to any file. It is read-only at the host level (no `write_file` capability
   in registry entry).
2. The hook fires on PostToolUse only — it never prevents a write; it signals after the write
   completes. Stale-cite detection is post-write; the dispatcher records the block signal for
   the author to act on.
3. Version extraction regex is anchored to the 4 canonical index file names:
   `(BC-INDEX|VP-INDEX|STORY-INDEX|ARCH-INDEX) v(\d+)\.(\d+)` — no other index name patterns
   are matched.
4. The hook does not validate ARCH-INDEX changelog entries or frontmatter changelog arrays —
   only the body prose version-cite strings.
5. The fuel budget constraint: all `host::read_file` calls together MUST complete within the
   `timeout_ms` budget declared in hooks-registry.toml (5000 ms total). Five files are read;
   each individual read uses max_bytes=65536, timeout_ms=2000. If any single read exceeds its
   budget, the hook falls through to `HookResult::Continue` (fail-open).

#### Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | ARCH-INDEX body contains no 4-index version-cite strings | Hook emits Continue — no cite means no stale cite |
| EC-002 | Cited version equals live version exactly | Hook emits Continue — no staleness |
| EC-003 | Cited version is newer than live version (impossible normally but possible in test) | Hook emits Continue — only older cites are stale |
| EC-004 | BC-INDEX file is not readable (CapabilityDenied or file missing) | Hook emits Continue + log_warn; does not block on read failure |
| EC-005 | ARCH-INDEX body cites BC-INDEX v1.05 when live BC-INDEX is v2.24 | Hook emits BlockWithFix: "ARCH-INDEX cites BC-INDEX v1.05 but live version is v2.24 — update cite to v2.24" |
| EC-006 | STATE.md trajectory cell cites STORY-INDEX v3.28 when live STORY-INDEX is v3.31 | Hook emits BlockWithFix citing the cross-cell mismatch |
| EC-007 | Version string is malformed (e.g. `BC-INDEX vX.Y`) | Hook skips the malformed cite and logs a warning; does not block on parse failure |
| EC-008 | Hook triggered on a write to a file that is NOT ARCH-INDEX.md (file_pattern mismatch) | Hook is never invoked — hooks-registry.toml file_pattern limits trigger to ARCH-INDEX.md writes |

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | ARCH-INDEX body contains no 4-index version-cite strings (e.g. a purely structural edit) | Hook emits Continue — no cite means no stale cite; vacuously passing |

## Implementation Detail

### Hook Logic Specification

The implementer MUST implement `src/lib.rs` with the following logical structure. No line-number
citations (TD-VSDD-091 anti-volatile-pin); function names and behavioral anchors only.

#### Version extraction

```
fn extract_index_cites(content: &str) -> Vec<(IndexName, u32, u32)>
```

Scan `content` for the regex pattern `(BC-INDEX|VP-INDEX|STORY-INDEX|ARCH-INDEX) v(\d+)\.(\d+)`.
For each match: parse group-2 as u32 (major), group-3 as u32 (minor). Return all matches.
On parse failure for any individual match, skip and log_warn — do not abort.

Note: if the workspace `Cargo.toml` does not already carry the `regex` crate, the implementer
MUST use a Rust stdlib hand-rolled scan rather than adding a new dependency. Check
`grep -r '"regex"' Cargo.toml Cargo.lock` before adding.

#### Live version reading

```
fn read_live_version(index_name: IndexName) -> Option<(u32, u32)>
```

Map `IndexName` to the canonical path:
- `BC-INDEX` → `.factory/specs/behavioral-contracts/BC-INDEX.md`
- `VP-INDEX` → `.factory/specs/verification-properties/VP-INDEX.md`
- `STORY-INDEX` → `.factory/stories/STORY-INDEX.md`
- `ARCH-INDEX` → `.factory/specs/architecture/ARCH-INDEX.md`

Call `host::read_file(path, 65536, 2000)`. On `Err(e)`: log_warn with path and error; return
`None`. On success: scan first 2048 bytes of content for `^version: "(\d+)\.(\d+)"`. Parse
major and minor as u32. Return `Some((major, minor))`. On parse failure: return `None`.

#### Version comparison

```
fn is_stale(cited: (u32, u32), live: (u32, u32)) -> bool
```

Returns `true` if `cited.0 < live.0 || (cited.0 == live.0 && cited.1 < live.1)`.
This is strictly numeric less-than. Version equality (`cited == live`) is NOT stale.
Version where cited > live is treated as NOT stale (impossible in normal operation).

#### Cross-cell check (D-429(b))

```
fn cross_cell_check(live_versions: &HashMap<IndexName, (u32,u32)>) -> Vec<Violation>
```

Read STATE.md via `host::read_file(".factory/STATE.md", 65536, 2000)`. On error: log_warn, return
empty vec (fail-open). Extract all `(BC-INDEX|VP-INDEX|STORY-INDEX|ARCH-INDEX) v(\d+)\.(\d+)`
matches from STATE.md content. For each match where the IndexName exists in `live_versions`:
call `is_stale(cited, live_versions[name])`. If stale: append Violation with source="STATE.md".

Repeat for `host::read_file(".factory/cycles/v1.0-brownfield-backfill/INDEX.md", 65536, 2000)`.
On error: log_warn, skip (fail-open). Extract same patterns. Append violations with source="INDEX.md".

**Note on INDEX.md path hardcoding:** The path
`.factory/cycles/v1.0-brownfield-backfill/INDEX.md` is hardcoded for the brownfield cycle
currently in-flight. This is a known limitation: when the cycle rotates, this hook will need
updating. This is acceptable for M2 scope. If the adversary flags this as HIGH severity, the
implementer should surface to the orchestrator for a scope decision rather than silently deferring.

#### Block message format

When `violations.len() > 0`:

```
HookResult::BlockWithFix {
    message: format!(
        "validate-index-cite-refresh: {} stale cite(s) found:\n{}",
        violations.len(),
        violations.iter().map(|v| format!(
            "  [{}] {} cites {} v{}.{} but live version is v{}.{}. Update cite to v{}.{}.",
            v.source, v.location, v.index_name, v.cited.0, v.cited.1,
            v.live.0, v.live.1, v.live.0, v.live.1
        )).collect::<Vec<_>>().join("\n")
    ),
    fix_hint: "Update the stale version cite(s) listed above to the current index versions.",
}
```

### Performance Considerations

Five `host::read_file` calls are made per PostToolUse event on ARCH-INDEX.md:
1. BC-INDEX.md (for ARCH-INDEX cite check AND cross-cell live version)
2. VP-INDEX.md
3. STORY-INDEX.md
4. STATE.md (D-429(b))
5. INDEX.md (D-429(b))

The ARCH-INDEX.md read is free — the dispatcher already has it from the event payload.

At 65536 bytes cap per read, all five reads fit well within the 5000 ms hook timeout. ARCH-INDEX
is ~300 KB at maximum growth; the cap at 65536 bytes covers its frontmatter and first portion of
body where version cites typically appear. If ARCH-INDEX grows beyond 65536 bytes and version
cites appear late in the file, the implementer MUST increase the cap or use a streaming scan.
The recommended approach: use 131072 (128 KB) for ARCH-INDEX reads to cover typical growth.

Fuel budget: WASM hooks run in a sandboxed VM with finite fuel. The five-file read pattern
is the same as `validate-state-index-status-coherence` (already in production) which completes
within budget. No fuel budget changes required.
