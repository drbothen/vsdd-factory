# S-10.04: ADR-015 Wave 1 — Trace propagation and lifecycle event types (D-15.4 + D-15.3)

**Epic:** E-10 — Single-stream OTel-aligned event emission (ADR-015)
**Wave:** Wave 1
**Depends on:** S-10.01
**Blocks:** S-10.05
**Estimated effort:** M (5 pts, ~2 days)

## Narrative

- **As a** vsdd-factory operator debugging a multi-plugin hook invocation
- **I want** `VSDD_TRACE_ID` and `VSDD_PARENT_SPAN_ID` injected unconditionally into
  every subprocess call, and visible lifecycle events when host fields are overridden
  or block paths are taken
- **So that** I can trace causality across plugin boundaries and be alerted when plugin
  code tries to stamp host-owned fields

## Acceptance Criteria

### AC-001: VSDD_TRACE_ID and VSDD_PARENT_SPAN_ID injected into every exec_subprocess call (traces to ADR-015 D-15.4) (traces: BC-1.11.001 postcondition 1)

The dispatcher injects `VSDD_TRACE_ID` and `VSDD_PARENT_SPAN_ID` into the subprocess
environment for every `exec_subprocess` call, unconditionally — regardless of what any
plugin manifest's `env_allowlist` declares.

The `env_allowlist` semantics are UNCHANGED: it remains a per-plugin allowlist for
OTHER environment variables. `VSDD_TRACE_ID` and `VSDD_PARENT_SPAN_ID` flow through
as dispatcher-side invariants, not as allowlist entries.

**Falsifiable test:** An integration test invokes a subprocess plugin and asserts that
the subprocess environment contains both `VSDD_TRACE_ID` and `VSDD_PARENT_SPAN_ID`,
even when the plugin's manifest `env_allowlist` is empty.

### AC-002: Subprocess span chain rule implemented (traces to ADR-015 D-15.4 §Span chain rule) (traces: BC-1.11.001 postcondition 2 — span chain rule invariants)

When a subprocess plugin emits events:
- `trace_id` = inherited `VSDD_TRACE_ID` (unchanged across the hop)
- `parent_span_id` = the invoking plugin's `span_id` (passed as `VSDD_PARENT_SPAN_ID`)
- `span_id` = a new UUIDv4 generated by the subprocess plugin at startup

**Falsifiable test:** Two chained subprocess invocations produce events where the
inner invocation's `parent_span_id` equals the outer invocation's `span_id`, and both
share the same `trace_id`.

### AC-003: vsdd.internal.host_field_override.v1 emitted when plugin supplies host-owned field (traces to ADR-015 D-15.3 §Host-field override visibility) (traces: BC-1.12.005 postcondition 5)

When a plugin supplies a value for a host-owned field (Resource attributes or
per-event identity fields), the dispatcher:

1. Emits a `vsdd.internal.host_field_override.v1` lifecycle event rate-limited to
   once per unique `(plugin.name, field_name)` pair per dispatcher invocation.
   The event carries an `affected.plugin.name` attribute.
2. Stamps the DOMAIN EVENT itself with `event.host_overrides: [<list of overridden field names>]`.
3. Writes a `stderr` warning: `[vsdd-dispatcher] WARN: plugin '<plugin.name>' supplied
   host-owned field '<field_name>'; host-stamped value takes precedence.`

**Falsifiable test:** A test plugin that sets `service.name = "rogue"` in its
`emit_event` payload produces: (a) `vsdd.internal.host_field_override.v1` in the
output stream; (b) the domain event contains `event.host_overrides: ["service.name"]`;
(c) the domain event's `service.name` is still `"vsdd-factory"` (host value wins);
(d) stderr contains the warning line.

Rate-limit test: calling from the same plugin twice with the same field name produces
only ONE `vsdd.internal.host_field_override.v1` event per invocation.

### AC-004: vsdd.block.plugin_blocked.v1 emitted on HookResult::Block (traces to ADR-015 D-15.3 §Block path audit trail) (traces: BC-1.12.006 postcondition 1)

When a plugin returns `HookResult::Block`, the dispatcher emits a
`vsdd.block.plugin_blocked.v1` event with `outcome=blocked`, `plugin.name`, and
`hook.tool_name` before exiting. The event appears in `events-*.jsonl`.

**Falsifiable test:** A test plugin that returns `HookResult::Block` produces a
`vsdd.block.plugin_blocked.v1` event in `events-*.jsonl` with the correct
`outcome`, `plugin.name`, and `hook.tool_name` values.

### AC-005: vsdd.internal.host_id_fallback.v1 emitted when host.id reaches terminal default (traces to ADR-015 D-15.2.c — implemented in S-10.03 but tested here as a lifecycle event type) (traces: BC-1.12.004 postcondition 2 — vsdd.internal.* prefix → lifecycle in registry)

This AC validates the lifecycle event contract established in S-10.03 AC-003 by
confirming the event type exists in the `event.category` registry as `"lifecycle"`.
The `vsdd.internal.host_id_fallback.v1` prefix `vsdd.internal.*` maps to `lifecycle`
per the D-15.2 registry.

**Falsifiable test:** Unit test: `categorize("vsdd.internal.host_id_fallback.v1")` = `"lifecycle"`.

### AC-006: New vsdd.internal.* event types categorized correctly (traces to ADR-015 D-15.2 §event.category taxonomy registry) (traces: BC-1.12.004 postcondition 2)

The compile-time registry (from S-10.03) correctly categorizes all three new lifecycle
event types introduced by ADR-015:

- `vsdd.internal.host_field_override.v1` → `lifecycle`
- `vsdd.internal.host_id_fallback.v1` → `lifecycle`
- `vsdd.internal.event_name_deprecated.v1` → `lifecycle` (used in Wave 3; registry entry added now)

And the audit event type:
- `vsdd.block.plugin_blocked.v1` → `audit`

**Falsifiable test:** Unit test asserts each of the four event names maps to the
correct category.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Plugin supplies multiple host-owned fields in one emit_event call | One `vsdd.internal.host_field_override.v1` event per unique field name (rate-limited); `event.host_overrides` lists all overridden fields |
| EC-002 | Same plugin overrides the same field twice in one dispatcher invocation | Only ONE `vsdd.internal.host_field_override.v1` per `(plugin.name, field_name)` pair per invocation (rate-limit) |
| EC-003 | Subprocess plugin does not read VSDD_PARENT_SPAN_ID | The env var is still injected; it is the subprocess's responsibility to use it |
| EC-004 | VSDD_TRACE_ID already set in the outer environment | Dispatcher uses the existing value rather than generating a new one; trace ID is consistent across the invocation chain |
| EC-005 | Block event FileSink write fails | Fallback to debug file + stderr warning (per S-10.02 AC-003 semantics); `HookResult::Block` exit is NOT delayed |

## Architecture Compliance Rules

Per ADR-015 D-15.4:
- `VSDD_TRACE_ID` and `VSDD_PARENT_SPAN_ID` are DISPATCHER-SIDE INVARIANTS, not
  per-plugin allowlist entries. No plugin manifest change is needed or allowed.
- The `env_allowlist` field semantics in `hooks-registry.toml` are UNCHANGED.

Per ADR-015 D-15.3:
- Host-stamped field value ALWAYS wins. No plugin override passes through.
- Override detection rate-limit is per-dispatcher-invocation (not global/persistent).
