# S-4.04: Per-sink retry + circuit breaker

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.05, S-4.07, S-4.08
**Blocked by:** S-1.08, S-4.01
**Estimated effort:** M

## Capability Anchor Justification

Anchoring to **CAP-024: "Per-sink retry, circuit breaker, and dead-letter queue"** per capabilities.md §CAP-024.

CAP-024 states: "S-4.4/4.5: each sink driver retries failed sends with exponential backoff, trips a circuit breaker on sustained failure, and routes dropped events to a `dead-letter-<sink>-<date>.jsonl`." S-4.04 is explicitly cited as S-4.4 in CAP-024 and is the primary story that delivers the retry + circuit breaker portion of this capability. The outcome CAP-024 promises — "a Datadog outage doesn't lose events — they land in the DLQ" — requires the retry and circuit breaker infrastructure this story provides.

## Narrative
- **As a** factory operator
- **I want to** have sinks automatically retry on transient failures and open a circuit breaker on sustained failures
- **So that** temporary backend outages do not cause data loss and do not degrade dispatcher latency

## Goal

Implement the shared resilience runtime that all HTTP-based sinks
(`sink-http`, `sink-datadog`, `sink-honeycomb`, `sink-otel-grpc`) use
for retry and circuit-breaker behavior. Per the design doc Q6 resolution:
exponential backoff with jitter, configurable retry limit, and a
half-open circuit breaker.

## Acceptance Criteria

- [ ] Shared `RetryPolicy` struct: max_retries, base_delay_ms, max_delay_ms, jitter_factor
  (traces to BC-3.03.002 postcondition 1: send failure protocol — drop gRPC client on error; rebuild on next batch. The RetryPolicy wraps this self-healing reconnect-on-error pattern with configurable retry counts)
- [ ] Exponential backoff with jitter: delay = min(base * 2^n + jitter, max)
  (traces to BC-3.03.002 postcondition 1: the "cheapest fix is reconnect-on-error" comment implies stateless reconnect; RetryPolicy adds structured backoff on top of the reconnect pattern)
- [ ] Circuit breaker: opens after N consecutive failures; half-open after cool-off period
  (traces to BC-3.03.002 postcondition 1: self-healing reconnect-on-error is the base pattern; circuit breaker state machine is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-circuit-breaker-state-machine`. See v1.1 BC Candidates table.)
- [ ] Circuit state emitted as `internal.sink_circuit_opened` / `internal.sink_circuit_closed` events
  (traces to VP-012: Sink Failure Affects Only That Sink — circuit state is per-sink; circuit event emission itself is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-circuit-event-emission`. See v1.1 BC Candidates table.)
- [ ] All HTTP sinks use this resilience layer
  (traces to BC-3.03.002 postcondition 1: self-healing reconnect-on-error is the documented pattern; RetryPolicy makes it consistent across all HTTP-based sinks)
- [ ] Integration test: mock server returning 5xx; verify retry + circuit open sequence
  (traces to BC-3.03.002 precondition 1: previously-built client returns `Err` on export; retry layer exercises this path repeatedly until circuit opens; circuit integration itself is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-circuit-breaker-state-machine`)

## v1.1 BC Creation Dependency

S-4.04's primary behavior (RetryPolicy + CircuitBreaker state machine) is
not yet contracted in the SS-03 BC catalog. The 4 v1.1 BC candidates
(BC-3.NN.NNN-retry-policy-exponential-backoff, -circuit-breaker-state-machine,
-circuit-event-emission, -retry-isolation-per-sink) capture this gap. S-4.04
implementation is dependent on at least retry-policy and CB-state-machine BC
creation in v1.1; this is acknowledged spec-side as deferred work.
