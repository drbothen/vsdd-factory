## Narrative

- **As a** vsdd-factory operator running Claude Code sessions
- **I want** session-learning to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** session-end markers are appended to `.factory/sidecar-learning.md` on all
  platforms (including Windows without git-bash), with lower latency, no adapter
  indirection, and no external binary dependency (`bash` is no longer needed)

## Goal

Port `plugins/vsdd-factory/hooks/session-learning.sh` to a native Rust WASM crate at
`crates/hook-plugins/session-learning/`. The hook fires on the Stop event (session end),
checks for `.factory/` directory existence (exits 0 if absent), creates
`.factory/sidecar-learning.md` with a header if it does not exist, and appends a single
line: `- Session ended at <ISO8601-UTC> (awaiting /session-review)`.

Behavior parity is the sole requirement (E-8 D-2): the Rust implementation must reproduce
every postcondition in BC-7.03.076, BC-7.03.077, and BC-7.03.078. No behavior change is
permitted during the port. After the port, the `.sh` file is deleted and the registry
entry updated to reference the native `.wasm`.

**hooks.json note (DRIFT-004):** The dispatcher migrated to hooks-registry.toml (native
format) — DRIFT-004 is already in the AFTER state. There are NO hooks.json entries for
session-learning in the current codebase; no hooks.json deletion is required. The v1.0
hooks.json deletion AC was a phantom-deletion defect (corrected in v1.1). The
post-migration assertion is: hooks-registry.toml entry is updated; session-learning.sh
is deleted.

**Key simplification:** session-learning does NOT call `bin/emit-event` — the bash source
uses only `date` and file I/O. The WASM port replaces `date` with Rust's `chrono` or
`std::time`. No `host::emit_event` calls are needed (and none may be added — parity only).
This makes session-learning the lowest-complexity Tier 1 port.

**File I/O mechanism (empirically locked — D-172):** `host::write_file` is ABSENT
from `crates/hook-sdk/src/host.rs` (verified empirically in D-172). The implementer MUST
use `wasm32-wasip1` `std::fs` (`std::fs::OpenOptions::new().append(true).create(true)`)
for the append operation. Do NOT wait for a SDK extension — std::fs is the correct and
confirmed-working path for WASI file I/O in this crate. This choice is locked; document
it in a code comment in `src/lib.rs`.

## Acceptance Criteria

### AC-001 (traces to BC-7.03.076 postcondition 1)
WASM crate `crates/hook-plugins/session-learning/` exists with `Cargo.toml` targeting
`wasm32-wasip1`, implements the `vsdd_hook_sdk` hook interface, and builds successfully
via `cargo build --target wasm32-wasip1`. Registry entry in `hooks-registry.toml` is
updated: `plugin = "hook-plugins/session-learning.wasm"`, `script_path` removed,
`shell_bypass_acknowledged` removed. Event binding preserved: `event = "Stop"`,
`priority = 910`, `on_error = "continue"`. The `[hooks.capabilities.exec_subprocess]`
sub-block is removed entirely — session-learning requires no subprocess invocations
post-port and must have no `binary_allow` declarations. The entire top-level
`[hooks.capabilities]` block (including `env_allow = [...]`) is also removed — the
native WASM port needs no environment variables, as `std::fs` paths are resolved
relative to cwd, not via env reads.

### AC-002 (traces to BC-7.03.077 postcondition 1)
When `.factory/` directory exists and `sidecar-learning.md` does not exist: WASM plugin
creates `.factory/sidecar-learning.md` with the header block:
```
# Sidecar Learning

Session-end markers for the VSDD factory. Run /session-review to synthesize.

```
(The trailing blank line after the last header sentence is REQUIRED — bats compares
byte-identical content against the bash source output.)
Then appends: `- Session ended at <ISO8601-UTC> (awaiting /session-review)\n`
(where `<ISO8601-UTC>` matches format `%Y-%m-%dT%H:%M:%SZ`, e.g.
`2026-04-30T14:25:00Z`). Exits 0.

### AC-003 (traces to BC-7.03.078 postcondition 1)
When `.factory/` directory does not exist: WASM plugin exits 0 immediately without
creating any files or writing to stderr. No error output. This guard must be the first
check in the plugin body (before any file I/O attempts).

### AC-004 (traces to BC-7.03.077 postcondition 1 — append-only parity)
When `.factory/` exists and `sidecar-learning.md` already has prior content: WASM
plugin appends exactly one new line per invocation (does NOT overwrite or re-add the
header). Multiple consecutive Stop firings each add exactly one marker line. Bats parity
tests at `tests/integration/hooks/session-learning.bats` verify all three behavioral
cases: (a) first invocation creates file + header + one marker; (b) second invocation
appends one more marker, no duplicate header; (c) invocation without `.factory/` exits 0,
no file created. All bats tests pass.

> warm-invocation regression ceiling. No perf measurement or hyperfine comparison is
> required for AC-004 passage. Performance is INFORMATIONAL ONLY for Tier 1.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `.factory/` directory exists but is not writable | WASM plugin attempts write via std::fs; if write returns error, exit non-zero. Rationale: session-learning.sh line 14 sets `set -euo pipefail` — under `set -e`, a `>>` redirect failure causes non-zero exit, NOT silent failure. D-2 parity mandates the same behavior in the WASM port. The dispatcher's `on_error=continue` setting means a non-zero exit from this hook does not block the session; it is logged as a hook failure and execution continues. No stderr warning is required (bash also produces minimal output on `>>` failure). |
| EC-002 | `sidecar-learning.md` exists but contains only a partial header (no trailing blank line) | WASM plugin appends the marker line without re-adding the header; file integrity is preserved |
| EC-003 | Stop event fires during rapid consecutive sessions (< 1s apart) | Each invocation appends a distinct timestamp line; no deduplication required; ISO-8601 precision to seconds is sufficient |
| EC-004 | `sidecar-learning.md` is very large (many previous sessions) | Append-only; no read of full file needed before appending; WASM implementation must not buffer full file into memory; use `OpenOptions::new().append(true)` exclusively |
| EC-005 | WASM hook receives Stop JSON envelope from dispatcher (Stop event may carry session metadata) | session-learning does NOT parse the stdin envelope. The WASM port MUST read stdin to EOF and discard it — do NOT skip the stdin read call entirely. Skipping the read may cause a WASI SIGPIPE-equivalent failure if the dispatcher is still writing to the plugin's stdin pipe. Bats test case: dispatcher writes a large Stop envelope (>64KB; e.g., 65,536 bytes of padding) to plugin stdin; plugin must complete cleanly without dispatcher receiving a write error. Note: pipe-fill threshold is OS-dependent (typically 64KB on Linux, typically 64KB on macOS too). The bats test MUST run on all platforms — do NOT skip on macOS. If the pipe-fill threshold differs on a given platform, parameterize the write-until-blocked threshold rather than skipping the test entirely. A cross-platform portability invariant (the primary motivation for AC-001/2/3/4) must not have a coverage hole on any supported platform. |
