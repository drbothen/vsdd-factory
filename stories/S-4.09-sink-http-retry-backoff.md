# S-4.09: sink-http retry backoff with jitter

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** (none)
**Blocked by:** S-4.01 (sink-http retry loop; shipped at commit 2ebf031)
**Estimated effort:** S

> **Scope note:** S-4.09 delivers the backoff-delay portion of FR-044 for sink-http only.
> Circuit breaker and DLQ remain in S-4.04 (Wave 12) and S-4.05 (Wave 12) respectively.
> S-4.04 builds the shared `RetryPolicy` struct in sink-core; S-4.09's backoff implementation
> will be superseded/generalized by S-4.04's RetryPolicy.

## Capability Anchor Justification

Anchoring to **CAP-024: "Per-sink retry, circuit breaker, and dead-letter queue"** per capabilities.md §CAP-024.

CAP-024 states: "retries failed sends with exponential backoff" as a specific behavioral promise under the per-sink resilience layer. S-4.09 delivers that backoff-delay component for the sink-http driver, using the existing retry loop from S-4.01 as the anchor point. CAP-024 is the correct and only cap for this story — no other CAP governs per-sink retry resilience.

**SS-03 ownership justification:** SS-03 owns all sink driver implementation; sink-http's retry loop lives in `crates/sink-http/`, which is the canonical SS-03 module boundary for HTTP sink behavior.

## Narrative

- **As a** factory operator with a custom HTTP observability backend
- **I want** sink-http to wait an exponentially increasing delay between retries when the backend returns 5xx errors
- **So that** a temporarily unavailable backend recovers without being hammered by immediate retries (thundering-herd prevention)

## Goal

Add exponential backoff with uniform jitter to the existing sink-http retry loop. The implementer adds `base_delay_ms`, `max_delay_ms`, and `jitter_factor` fields to the existing `RetryConfig` struct and wires `tokio::time::sleep` between retry attempts. No new crate. No circuit breaker state machine (that is S-4.04 scope).

## Acceptance Criteria

- [ ] **AC-001:** After receiving a 5xx response on attempt N, sink-http sleeps for
  `delay_ms = min(base_delay_ms * 2^N + jitter_ms, max_delay_ms)` before issuing attempt N+1,
  where `jitter_ms` is drawn uniformly from `[0, base_delay_ms * jitter_factor]`.
  _(traces to BC-3.07.001 postcondition 1: the delay formula is exact and mandatory)_

- [ ] **AC-002:** `delay_ms` is always strictly positive when `base_delay_ms > 0`; even attempt 0
  (the first retry after a 5xx) introduces a nonzero delay floor.
  _(traces to BC-3.07.001 postcondition 2: strictly positive delay requirement)_

- [ ] **AC-003:** `delay_ms` never exceeds `max_delay_ms` regardless of retry attempt number N,
  including when jitter would push it above the cap.
  _(traces to BC-3.07.001 postcondition 3: max_delay_ms cap is hard)_

- [ ] **AC-004:** The backoff sleep occurs on the sink's worker/batch thread. The `submit()` call
  path returns immediately without waiting for the sleep to complete.
  _(traces to BC-3.07.001 postcondition 4: non-blocking submit path; BC-3.03.005 invariant preserved)_

- [ ] **AC-005:** After the sleep, the HTTP request is retried with the same payload as the
  failed attempt (no payload mutation between retry attempts).
  _(traces to BC-3.07.001 postcondition 5: same-payload retry)_

- [ ] **AC-006:** `RetryConfig` construction fails with `ConfigError::InvalidBackoff` if
  `base_delay_ms = 0` or `max_delay_ms < base_delay_ms`. The sink does not start.
  _(traces to BC-3.07.001 invariant 1: construction-time invariant; EC-001, EC-002)_

- [ ] **AC-007:** The jitter PRNG is seeded from a thread-local or per-instance source — not a
  global static. Two concurrent sink instances sending to the same backend produce uncorrelated
  jitter values.
  _(traces to BC-3.07.001 invariant 2: per-instance PRNG required)_

- [ ] **AC-008:** The backoff sleep does not hold the `Mutex<Vec<SinkFailure>>` lock. Failure
  recording and backoff sleeping are provably non-overlapping operations in the implementation.
  _(traces to BC-3.07.001 invariant 3: lock discipline)_

- [ ] **AC-009:** On a full-failure sequence of N total attempts, exactly (N-1) backoff sleeps
  occur. There is no trailing sleep after the final failed attempt before recording failure.
  _(traces to BC-3.07.001 invariant 4: exactly (max_retries - 1) sleeps)_

- [ ] **AC-010:** An integration test using a mock HTTP server that returns 5xx responses
  measures actual wall-clock delay between retry attempts and asserts it falls within the
  expected range `[delay_ms, delay_ms + tolerance_ms]` for at least two consecutive attempts.
  _(traces to BC-3.07.001 canonical test vectors: base=100ms,max=5000ms,jitter_factor=0.5,
  attempt=0 → delay in [100,150]ms; attempt=1 → delay in [200,250]ms)_

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|---------------|
| sink-http/src/retry.rs (new) | crates/sink-http/ | pure-core (backoff formula, jitter computation) |
| sink-http/src/lib.rs (modified) | crates/sink-http/ | effectful-shell (tokio::time::sleep wiring) |

**Pure/effectful boundary:** The backoff delay formula (`min(base * 2^N + jitter, max)`) and
jitter draw are pure computations. The `tokio::time::sleep` call and PRNG seeding are the
only effectful operations introduced by this story. Keep them in `lib.rs` (effectful-shell);
keep the formula in `retry.rs` (pure-core) so it can be unit-tested without async runtime.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `base_delay_ms = 0` (misconfigured) | `ConfigError::InvalidBackoff` at construction; sink does not start (AC-006) |
| EC-002 | `max_delay_ms < base_delay_ms` (misconfigured) | `ConfigError::InvalidBackoff` at construction; sink does not start (AC-006) |
| EC-003 | `max_retries = 1` (single attempt) | No sleep: single attempt fails, failure recorded immediately, no retry loop entered (AC-009) |
| EC-004 | 4xx response received (client error) | No retry, no backoff sleep; failure recorded immediately as non-retryable |
| EC-005 | Network timeout / connection refused | Treated as retriable identical to 5xx; backoff applies (BC-3.07.001 EC-005) |
| EC-006 | Jitter would push delay above max_delay_ms | Clamped to exactly max_delay_ms (AC-003) |
| EC-007 | Concurrent sink instances under same backend | Each instance uses independent PRNG seed; jitter values are uncorrelated (AC-007) |

## Forbidden Dependencies

the PR MUST be rejected. Acceptable: `sink-core` utility types already present in S-4.01.
