# S-14.09: Forensic Marker Namespace Cleanup: Remove F-PN-NNN from Production Source

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** NINTH in E-14. Can be delivered independently.
**Depends on:** (none)
**Blocks:** (none)
**Priority:** P2

## Narrative

- **As a** developer reading production Rust source code in `crates/*/src/`
- **I want** all inline forensic markers (comments referencing `F-P{N}-NNN` adversarial
  finding IDs) to be converted to proper BC/VP references or removed if redundant
- **So that** the codebase reads cleanly without cryptic cycle-internal references
  that are meaningless to future contributors unfamiliar with the adversarial cycle history

## Background

F-P6-004 (pass-6 adversarial finding) identified that forensic marker comments in
production source have proliferated across the codebase. Comments like
`// F-P5-001: ...` or `// F-P3-007 fix` are meaningful within the adversarial cycle
but become noise in the long-term codebase.

**Namespace rule going forward (codified in D-378/D-379):**
- Bare `F-P{N}-NNN` markers in `crates/*/src/` are FORBIDDEN from this story forward
- Cycle-level markers belong in commit messages, BC/VP cross-references, or decision-log
- If a comment references a BC/VP, replace the `F-P{N}-NNN` reference with the BC/VP ID
- If the comment adds no information beyond the code itself, remove it

## Acceptance Criteria

### AC-001: Audit all F-P{N}-NNN markers in production source
- Run: `grep -rn 'F-P[0-9]\+-[0-9]\+' crates/*/src/ plugins/vsdd-factory/hooks/ plugins/vsdd-factory/bin/`
- Document every hit: file, line, marker, whether it has a corresponding BC/VP

### AC-002: Convert markers with BC/VP equivalents
- For each marker that references a BC behavioral property or VP, replace the
  `F-P{N}-NNN` reference with the canonical BC-X.YY.ZZZ or VP-NNN ID
- The replacement comment must be self-contained — a reader unfamiliar with the
  adversarial cycle must understand it from the BC/VP reference alone

### AC-003: Remove redundant markers
- For each marker that adds no information (e.g., `// F-P3-007 fix` with no BC/VP
  and no explanatory text beyond what the code says), remove the comment entirely
- Do not remove comments that contain substantive rationale — only those that are
  pure change-tracking noise

### AC-004: Zero F-P{N}-NNN markers remain in production source after merge
- Post-merge verification: `grep -rn 'F-P[0-9]\+-[0-9]\+' crates/*/src/` returns zero hits
- Exception: references in `//` doc-test examples or test fixture strings that
  intentionally use the pattern as test data (if any — likely none)

### AC-005: Cargo tests still pass
- `cargo test --workspace --all-targets` green after all marker removals/conversions
- `cargo clippy --workspace --all-targets -- -D warnings` clean
