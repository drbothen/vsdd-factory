## Narrative

- **As a** vsdd-factory operator running subagent pipelines
- **I want** handoff-validator to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** the SubagentStop warning gate for empty/truncated subagent results runs on all platforms (including Windows without git-bash), with lower latency and no adapter indirection

## Goal

Port `plugins/vsdd-factory/hooks/handoff-validator.sh` to a native Rust WASM crate at
`crates/hook-plugins/handoff-validator/`. The hook reads SubagentStop JSON from stdin,
extracts `agent_type` and `last_assistant_message`, and emits warning events to stderr if
the result is empty (exit 0) or suspiciously short under 40 non-whitespace characters
(exit 0). The `on_error = "block"` registry setting means dispatcher failures block the
subagent stop; this must be preserved in the registry entry after migration.

Behavior parity is the sole requirement (E-8 D-2): the Rust implementation must reproduce
every postcondition in BC-7.03.042, BC-7.03.043, and BC-7.03.044. No behavior change is
permitted during the port. After the port, the `.sh` file and the hooks.json command entry
for this hook are deleted, and the registry entry updated to reference the native `.wasm`.

## Acceptance Criteria

### AC-001 (traces to BC-7.03.042 postcondition 1)
WASM crate `crates/hook-plugins/handoff-validator/` exists with `Cargo.toml` targeting
`wasm32-wasi`, implements the `vsdd_hook_sdk` hook interface, and builds successfully
via `cargo build --target wasm32-wasi`. Registry entry in `hooks-registry.toml` is
updated: `plugin = "hook-plugins/handoff-validator.wasm"`, `script_path` removed,
`shell_bypass_acknowledged` removed. Event binding preserved: `event = "SubagentStop"`,
`priority = 910`, `on_error = "block"`.

### AC-002 (traces to BC-7.03.042 invariant 1)
`hooks.json` command entry for `handoff-validator` is deleted from all platform-specific
`hooks.json.*` files and the template. `plugins/vsdd-factory/hooks/handoff-validator.sh`
is deleted from the repository.

### AC-003 (traces to BC-7.03.043 postcondition 1)
When `last_assistant_message` (or fallback fields `result`, `output`) resolves to empty
or whitespace-only: WASM plugin calls `host::emit_event` with
`type=hook.block hook=handoff-validator matcher=SubagentStop reason=subagent_empty_result
severity=warn subagent=<agent_type>`, writes
`"handoff-validator: subagent '<agent>' returned an empty result.\n  This is a silent-failure risk — verify before continuing.\n"` to stderr,
exits 0.

### AC-004 (traces to BC-7.03.044 postcondition 1)
When trimmed non-whitespace result length is 1–39 characters: WASM plugin calls
`host::emit_event` with `type=hook.block hook=handoff-validator matcher=SubagentStop
reason=subagent_truncated_result severity=warn subagent=<agent_type> result_len=<N>`,
writes `"handoff-validator: subagent '<agent>' returned only <N> non-whitespace characters.\n  Suspiciously short — verify the subagent completed its task.\n"` to stderr,
exits 0.

### AC-005 (traces to BC-7.03.043 postcondition 1 + BC-7.03.044 postcondition 1)
Bats parity test at `tests/integration/hooks/handoff-validator.bats` covers: (a) empty
result → exit 0 + expected stderr text, (b) result with <40 trimmed chars → exit 0 +
expected stderr text, (c) result with >=40 trimmed chars → exit 0, no stderr warning.
All three bats tests pass. Additionally, a perf measurement is taken for the WASM warm
invocation and compared against the S-8.00 baseline for Tier 1 representative hook;
result within 20% regression ceiling per E-8 epic AC-7.

### AC-006 (traces to BC-7.03.042 invariant 2)
When `jq` is unavailable (jq-missing scenario): bash fallback was `exit 0`. WASM port
is self-contained (no jq dependency); it parses the JSON input natively via the Rust
JSON library. Verify in bats test: hook handles malformed JSON input by exiting 0
(graceful degradation) with no panic.

### AC-007 (traces to BC-7.03.043 postcondition 1)
`bin/emit-event` calls in the former bash script are replaced with `host::emit_event`
Rust calls in the WASM crate. Verify: no reference to `bin/emit-event` remains in
`crates/hook-plugins/handoff-validator/`. `bin/emit-event` binary is NOT removed
(E-8 D-10 — deferred to S-8.28).

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | SubagentStop JSON missing both `last_assistant_message` and fallback fields | Treat result as empty; emit subagent_empty_result warning; exit 0 (same as bash fallback chain: jq -r '.last_assistant_message // .result // .output // empty') |
| EC-002 | `agent_type` field missing from JSON input | Use "unknown" as the agent name in the event and stderr message (per bash: `jq -r '.agent_type // .subagent_name // "unknown"'`) |
| EC-003 | Result is exactly 40 trimmed characters | No warning emitted; exits 0 (threshold is `< 40`, not `<= 40`; verify off-by-one parity with bash `(( LEN < 40 ))`) |
| EC-004 | Result contains only whitespace | Treated as empty (LEN == 0 after `tr -d '[:space:]'`); emits subagent_empty_result; exit 0 |
| EC-005 | host::emit_event unavailable (no CLAUDE_PLUGIN_ROOT in WASM context) | Emit via host fn directly; WASM context is always within the dispatcher which provides host fn surface; graceful no-op if host fn call fails (same as bash OR-true fallback pattern) |
