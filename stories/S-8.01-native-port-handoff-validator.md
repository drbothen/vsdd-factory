## Narrative

- **As a** vsdd-factory operator running subagent pipelines
- **I want** handoff-validator to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** the SubagentStop warning gate for empty/truncated subagent results runs on all platforms (including Windows without git-bash), with lower latency and no adapter indirection

## Goal

Port `plugins/vsdd-factory/hooks/handoff-validator.sh` to a native Rust WASM crate at
`crates/hook-plugins/handoff-validator/`. The hook reads SubagentStop JSON from stdin,
extracts `agent_type` and `last_assistant_message`, and emits warning events to stderr if
the result is empty (exit 0) or suspiciously short under 40 non-whitespace characters
(exit 0). The `on_error = "block"` registry setting means dispatcher failures block the
subagent stop; this must be preserved in the registry entry after migration.

Behavior parity is the sole requirement (E-8 D-2): the Rust implementation must reproduce
every postcondition in BC-7.03.042, BC-7.03.043, and BC-7.03.044. No behavior change is
permitted during the port. After the port, the `.sh` file is deleted, and the registry
entry updated to reference the native `.wasm`. The `hooks.json.*` entries for this hook
do not exist in the current codebase (D-7 already shipped registry-only dispatch per
DRIFT-004); no deletion of hooks.json entries is required.

## Acceptance Criteria

### AC-001 (traces to BC-7.03.042 postcondition 1; traces to BC-2.02.012 postconditions 1-4)
native WASM crate `crates/hook-plugins/handoff-validator/` exists with `Cargo.toml`
targeting `wasm32-wasip1`, implements the `vsdd_hook_sdk` hook interface, and builds
successfully via `cargo build --target wasm32-wasip1`. Registry entry in
`hooks-registry.toml` is updated: `plugin = "hook-plugins/handoff-validator.wasm"`,
`script_path` removed, `shell_bypass_acknowledged` removed. Event binding preserved:
`event = "SubagentStop"`, `priority = 910`, `on_error = "block"`. After T-6, the
hooks-registry.toml entry MUST NOT contain any `[hooks.capabilities]` block — the
entire block is removed (full block removal, not just sub-key removal), because
`exec_subprocess` was its only child and native WASM crates call no subprocesses.
The crate reads SubagentStop JSON from stdin and deserializes it into `HookPayload`
using serde_json; `payload.agent_type`, `payload.subagent_name`,
`payload.last_assistant_message`, and `payload.result` are the typed `Option<String>`
fields per BC-2.02.012 postconditions 1-4.

### AC-002 (traces to BC-7.03.042 invariant 1)
Verify `hooks.json.*` platform files and `hooks.json.template` contain ZERO
`handoff-validator` command entries (this is already the post-DRIFT-004 baseline; the
verification step is a no-op but confirms the precondition holds). Delete
`plugins/vsdd-factory/hooks/handoff-validator.sh` from the repository.

### AC-003 (traces to BC-7.03.043 postcondition 1; traces to BC-2.02.012 postconditions 5-6 + EC-004)
When the assistant message resolves to empty or whitespace-only: native WASM plugin
calls `host::emit_event` with
`type=hook.block hook=handoff-validator matcher=SubagentStop reason=subagent_empty_result
severity=warn subagent=<agent_type>`, writes
`"handoff-validator: subagent '<agent>' returned an empty result.\n  This is a silent-failure risk — verify before continuing.\n"` to stderr,
exits 0.

Agent identity is resolved via the BC-2.02.012 Postcondition 5 canonical fallback chain:
`payload.agent_type.as_deref().or(payload.subagent_name.as_deref()).unwrap_or("unknown")`.

Assistant message is resolved via the BC-2.02.012 Postcondition 6 canonical 2-stage
fallback chain: `payload.last_assistant_message.as_deref().or(payload.result.as_deref()).unwrap_or("")`.
Per BC-2.02.012 EC-004, the handoff-validator-specific `.output` 3rd-arm fallback
(present in `handoff-validator.sh:28` as `.last_assistant_message // .result // .output // empty`)
is intentionally dropped in the WASM port — behavior parity holds for all observed
envelope shapes (Claude Code emits `last_assistant_message` canonically; `result` and
`output` are bash-era defensive fallbacks). When all three bash arms are absent, the
bash chain and the 2-stage Rust chain both produce empty-string, preserving degenerate-case
parity. This divergence is sanctioned by BC-2.02.012 EC-004 and must be cited in the
implementation comment.

### AC-004 (traces to BC-7.03.044 postcondition 1)
When trimmed non-whitespace result length is 1–39 characters: native WASM plugin calls
`host::emit_event` with `type=hook.block hook=handoff-validator matcher=SubagentStop
reason=subagent_truncated_result severity=warn subagent=<agent_type> result_len=<N>`,
writes `"handoff-validator: subagent '<agent>' returned only <N> non-whitespace characters.\n  Suspiciously short — verify the subagent completed its task.\n"` to stderr,
exits 0.

### AC-005 (traces to BC-7.03.043 postcondition 1 + BC-7.03.044 postcondition 1)
Bats parity test at `tests/integration/hooks/handoff-validator.bats` covers all 7 cases
enumerated in T-5: (a) empty result → exit 0 + stderr contains "empty result",
(b) result with 5 non-whitespace chars → exit 0 + stderr contains "non-whitespace
characters", (c) result with 50 non-whitespace chars → exit 0, no stderr warning,
(d) result with exactly 39 non-whitespace chars → exit 0 + stderr warning emitted
(below threshold), (e) result with exactly 40 non-whitespace chars → exit 0, NO stderr
warning (at-or-above threshold), (f) JSON with missing last_assistant_message → exit 0,
stderr warns, (g) malformed JSON → exit 0, no panic. All 7 bats tests pass. Perf
measurement is taken for the WASM warm invocation and recorded in implementation notes
(INFORMATIONAL — non-blocking; Tier 1 hooks are excluded from the 20% regression gate
per E-8 epic AC-7 Goal #6).

### AC-006 (traces to BC-7.03.042 invariant 2)
When `jq` is unavailable (jq-missing scenario): bash fallback was `exit 0` (advisory).
WASM port is self-contained (no jq dependency); it parses the JSON input natively via
serde_json. Verify in bats test: hook handles malformed JSON input by exiting 0
(graceful degradation, advisory-only) with no panic. NOTE: BC-7.03.042 invariant 2
states "1=jq-missing-fail-closed"; handoff-validator chose exit 0 (advisory) over exit
1 (fail-closed). T-1 includes a sub-step to file a BC update via the S-8.00 audit table
to mark Spec-Current = N for invariant 2 with action "Update invariant 2 to acknowledge
handoff-validator chose exit 0 (advisory only) over exit 1 (fail-closed)."

### AC-007 ([process-gap] traces to E-8 D-10 / R-8.07 architecture-compliance)
`bin/emit-event` calls in the former bash script are replaced with `host::emit_event`
Rust calls in the native WASM crate. Verify: no reference to `bin/emit-event` remains in
`crates/hook-plugins/handoff-validator/`. `bin/emit-event` binary is NOT removed
(E-8 D-10 — deferred to S-8.29).

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | SubagentStop JSON missing both `last_assistant_message` and fallback fields (`result`; historically also `output` in bash) | Treat result as empty; emit subagent_empty_result warning; exit 0. In the WASM port, the 2-stage chain `payload.last_assistant_message.as_deref().or(payload.result.as_deref()).unwrap_or("")` resolves to `""` — same degenerate-case behavior as the bash 3-stage chain when all three arms are absent (per BC-2.02.012 EC-004: behavior parity preserved). |
| EC-002 | `agent_type` field missing from JSON input | Use "unknown" as the agent name in the event and stderr message (per BC-2.02.012 Postcondition 5 canonical fallback chain: `payload.agent_type.as_deref().or(payload.subagent_name.as_deref()).unwrap_or("unknown")`) |
| EC-003 | Result is exactly 40 trimmed characters | No warning emitted; exits 0 (threshold is `< 40`, not `<= 40`; verify off-by-one parity with bash `(( LEN < 40 ))`) |
| EC-004 | Result contains only whitespace | Treated as empty (LEN == 0 after `tr -d '[:space:]'`); emits subagent_empty_result; exit 0 |
| EC-005 | host::emit_event host function call fails at runtime (WASM context — no env vars used; file paths resolved through host::read_file path args, not CLAUDE_PLUGIN_ROOT) | WASM context is always within the dispatcher which provides the host fn surface; if host::emit_event returns a non-zero code, the hook logs a best-effort stderr message and exits 0 (graceful degradation, advisory-only — same semantics as the bash OR-true fallback pattern). `host::read_file` errors surface as `HostError::PathNotAllowed` or similar — hook returns HookResult::Error with a structured message. |
