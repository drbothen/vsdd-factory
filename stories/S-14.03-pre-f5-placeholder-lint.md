# S-14.03: Pre-F5 Artifact Lint for Placeholder Frontmatter

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** THIRD in E-14. Forward-looking for pass-2 cycle. Can be
  delivered in parallel with S-14.04.
**Depends on:** (none — independent of other E-14 stories)
**Blocks:** (none)

## Narrative

- **As a** vsdd-factory orchestrator about to dispatch F5 adversarial review
- **I want** a lint check that scans new BC, VP, ADR, and story frontmatter for
  placeholder strings before F5 entry
- **So that** the adversary does not find and report stale placeholders (which the
  orchestrator must fix mid-cycle), and D-355-style overclaims about "no open spec
  contradictions" are prevented by a mechanical gate rather than manual inspection

## Background

F5 pass-1 adversary found that VP-071's `bcs:` frontmatter contained placeholder
values (`BC-5.NN.002`, `"see PO output"`) that were never resolved after F2 spec
evolution. D-355 overstated "no open spec contradictions" because no automated gate
caught these placeholders before F5 dispatch. Similarly, all 6 new BC files had
`input-hash: "[pending-recompute]"` since F2 — the input-hash drift check could not
establish a baseline.

The pattern is structural: F2 spec evolution creates artifacts with placeholder
frontmatter, state-manager does not always recompute all hashes in the same burst,
and no gate blocks F5 dispatch when placeholders remain. This story creates that gate.

**Sources:** F5 pass-1 OBS-5 (F-LOW-4), OBS-6 (F-LOW-5), adv-cycle-pass-1.md lines
611-636. Fix-plan Section 5 PG-3.

## Behavioral Contracts

No existing BCs cover pre-F5 placeholder lint. New BC required.

- **BC-X.NN.001 (to be authored at story implementation start):** The pre-F5 lint
  check MUST scan all new (changed-since-last-cycle) BC, VP, ADR, and story files
  for placeholder strings and block F5 entry if any are found. The scan MUST cover
  at minimum: `BC-S.NN`, `VP-NNN`, `[pending-recompute]`, `TBD`, `to be determined`,
  `see PO output`. Finding any placeholder in a frontmatter field is a gate failure.

## Acceptance Criteria

### AC-001 (traces to PG-3 intent — lint script existence)

A lint script or skill step exists that scans `.factory/` artifact frontmatter for
placeholder strings. The script may be:
- A bats test file (`plugins/vsdd-factory/tests/pre-f5-placeholder-lint.bats`) invoked
  as a pre-F5 gate step
- A new skill (`plugins/vsdd-factory/skills/pre-f5-lint/SKILL.md`) that the
  orchestrator dispatches before F5 entry
- A WASM hook checking frontmatter on write (most proactive but higher complexity)

The implementation approach is decided in OQ-E14-01. This AC is implementation-approach
agnostic.

**Falsifiable test:** The lint mechanism exists on disk at the designated path.
Running the lint mechanism against the v1.0-feature-engine-discipline-pass-1 cycle
artifacts with VP-071's original placeholder `bcs:` values would have produced
≥1 finding.

### AC-002 (traces to OBS-5 — placeholder string coverage)

The lint check scans for at least the following placeholder patterns in frontmatter
fields (YAML front-matter between `---` delimiters):
1. `BC-S.NN` or `BC-X.NN` (placeholder BC IDs with literal "S" or "X" as section digit)
2. `VP-NNN` (three-digit VP ID placeholder pattern that doesn't match `VP-\d{3}`)
3. `[pending-recompute]`
4. `TBD` (case-insensitive, in a frontmatter value context)
5. `see PO output` or `see output`
6. `to be determined` (case-insensitive)
7. `TBD-*` globs (e.g., `TBD-at-implementation`, `TBD-on-full-authoring`) in
   `input-hash` field specifically

**Falsifiable test:** A test fixture artifact with each of the 7 placeholder patterns
in frontmatter is included in the test suite. Running the lint mechanism against the
fixture produces 7 findings (one per pattern). Running it against a clean fixture
(no placeholders) produces 0 findings.

### AC-003 (traces to OBS-6 — input-hash specific coverage)

The lint check specifically flags `input-hash: "[pending-recompute]"` and any
`input-hash` value matching the `TBD-*` glob. The check covers all document types
that have `input-hash` in their schema: BC files, VP files, ADR files, story files,
epic files.

**Falsifiable test:** A test fixture with `input-hash: "[pending-recompute]"` produces
a finding. A test fixture with `input-hash: "abc123def456"` (valid hash) produces 0
findings.

### AC-004 (traces to PG-3 intent — scoped to cycle delta, not all artifacts)

The lint check is scoped to artifacts that are NEW or MODIFIED in the current cycle
(changed since the previous cycle's close-out commit). It does NOT scan all `.factory/`
artifacts on every run (that would produce noise from intentional TBD markers in
deferred stories). The scope is determined by: (a) git diff against the cycle's base
commit, or (b) a manifest of new artifacts registered in the cycle's INDEX.md.

**Falsifiable test:** Running the lint against the full `.factory/` tree with
git-scoping disabled would find many legitimate TBD markers in deferred stories.
Running with git-scoping enabled against only the cycle delta produces findings ONLY
for cycle-new artifacts.

### AC-005 (traces to PG-3 intent — F5 entry gate integration)

The pre-F5 lint is documented as a required entry condition in the F5 phase skill
(or orchestrator MANDATORY STEPS for F5 dispatch). The lint check runs BEFORE the
adversary subagent is dispatched. If the lint produces ≥1 finding, F5 dispatch is
blocked with an actionable error message listing the failing artifacts and their
placeholder fields.

**Falsifiable test:** `grep -n "pre-f5.*lint\|placeholder.*lint\|lint.*placeholder"
plugins/vsdd-factory/skills/phase-f5-scoped-adversarial/SKILL.md` (or orchestrator.md)
returns ≥1 match confirming the lint is a documented pre-dispatch prerequisite.

## Dev Notes

**Recommended implementation:** bats script at
`plugins/vsdd-factory/tests/pre-f5-placeholder-lint.bats` that:
1. Gets the git diff of files changed since the cycle base commit (or lists files
   from `.factory/cycles/<cycle>/INDEX.md`)
2. For each changed file with YAML frontmatter, extracts the frontmatter block
3. Scans the frontmatter block for the 7 placeholder patterns using grep -E
4. Outputs a Markdown table of violations (file, field, placeholder value found)
5. Exits non-zero if ≥1 violation found

This approach requires only bash + standard UNIX tools. No WASM compilation. The
bats test file is both the implementation and the test runner.

**Pattern for BC-ID detection:** The pattern `BC-[SX]\.[0-9]` catches `BC-S.NN`
and `BC-X.NN` placeholder forms. Pattern `BC-[0-9]+\.[0-9]+\.[0-9]+` matches valid
BC IDs. Any `bcs:` frontmatter value containing a match for the placeholder pattern
is a finding.

**Pattern for VP-ID detection:** Valid VP IDs are `VP-\d{3}` (three digits). Any
`vps:` or `verification_properties:` value that does NOT match `VP-\d{3}` but
matches `VP-[A-Z]{3}` or similar is a finding.
