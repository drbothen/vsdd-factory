# S-5.03: WorktreeCreate / WorktreeRemove hook wiring

**Epic:** E-5 — New Hook Events and 1.0.0 Release

## Capability Anchor Justification

Primary: CAP-002 ("Hook Claude Code tool calls with sandboxed WASM plugins") per capabilities.md §CAP-002. This story wires the `WorktreeCreate` and `WorktreeRemove` event types through the dispatcher, implementing a single WASM plugin (`worktree-hooks.wasm`) that emits `worktree.created` and `worktree.removed` telemetry events.

FR anchor: FR-046 ("New Claude Code lifecycle hook events: SessionStart/SessionEnd/WorktreeCreate/WorktreeRemove/PostToolUseFailure") per PRD §2.x — the same FR anchoring S-5.01 (BC-4.04.*) and S-5.02 (BC-4.05.*). This story is S-5.03's contribution to the Tier F BC family under FR-046.

**Option A scoping (proactive — mirrors S-5.01 pass-4 architectural reversal but applied up-front):** v1.0 scope is event emission ONLY. No filesystem auto-registration, no subprocess invocation, ZERO declared capabilities. Scoping decision documented in BC-4.07.001 §"Scoping Decision: Option A (Zero-Capability)". Options B (exec_subprocess config-write CLI) and C (new `write_file` host fn / HOST_ABI bump) are explicitly rejected. Filesystem-write capability deferred to v1.1 — no CAP ID allocated yet; v1.1 capability registry expansion required (provisional name `CAP-NNN-filesystem-write` to be assigned in v1.1).

**Auto-enrichment note (RESERVED_FIELDS):** The 8 RESERVED_FIELDS are set by the host runtime in three sub-groups — NOT by the plugin: 4 enriched-from-HostContext (`dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`; set by `emit_event` host fn) + 3 enriched-from-InternalEvent::now() (`ts`, `ts_epoch`, `schema_version`) + 1 construction-time (`type`; set from the `emit_event` type argument). The plugin must not attempt to set any of them. Wire payload for WorktreeCreate = 10 fields (2 plugin-set + 4 host-enriched + 3 InternalEvent::now() + 1 construction-time); WorktreeRemove = 9 fields (1 plugin-set + 4 + 3 + 1).

**S-5.01 + S-5.02 lessons applied (non-negotiable):**
1. Production schema field names: `name`, `event`, `plugin` (with `hook-plugins/` prefix), `timeout_ms` — NOT `plugin_name`, `event_name`, `plugin_path`, `epoch_budget_ms`.
2. Capability schema is TOML TABLE form: `[hooks.capabilities.read_file]` — NOT array form. (Moot for this story: zero capabilities declared under Option A.)
3. 8 RESERVED_FIELDS plugin must NOT set: 4 enriched-from-HostContext (`dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`) + 3 enriched-from-InternalEvent::now() (`ts`, `ts_epoch`, `schema_version`) + 1 construction-time (`type`).
4. Wire format: `emit_event.rs:49` string-coerces all plugin-set values. All plugin-set fields arrive as strings on the wire.
5. Per-platform hooks.json variant naming: SUFFIX-based (`hooks.json.darwin-arm64`, etc.) — NOT directory-based.
6. Worktree events: the `once` key MUST be absent (not `once: false`, not `once: true` — the key must not exist at all) in hooks.json.template. Worktree events can re-fire on reconnect (EC-001). Defensive omission protects against future Claude Code parser changes. This differs from SessionStart/SessionEnd which use `once: true`.
7. Layer separation (ADR-011): Layer 1 = hooks.json.template → dispatcher binary; Layer 2 = hooks-registry.toml → worktree-hooks.wasm.
8. F-07 Option 1 semantic scope: `subsystems: ["SS-04"]` — behavior verified, not files-touched. hooks.json.template lives in SS-09/SS-07 space; hooks-registry.toml lives in SS-07 space; but the semantic behavior contracted is SS-04 owned.
9. No new SS-01 BCs: dispatcher routing for WorktreeCreate/Remove is absorbed by the existing registry schema BCs (BC-1.01.* field validation) without a new BC.
10. Capability scope minimization: ZERO declared capabilities. No `read_file`, no `exec_subprocess`.

## Narrative
- **As a** factory operator using multiple worktrees
- **I want to** have worktree lifecycle events (`WorktreeCreate`, `WorktreeRemove`) emitted as structured telemetry
- **So that** the observability stack can track worktree lifecycles in real time without manual intervention

## Goal

Wire `WorktreeCreate` and `WorktreeRemove` events through the dispatcher to a single `worktree-hooks.wasm` plugin. The plugin emits `worktree.created` (with `worktree_path` + `worktree_name`) and `worktree.removed` (with `worktree_path`) telemetry events. ZERO capabilities required — all data comes from the incoming envelope (Option A scoping).

## Acceptance Criteria

- [ ] **AC1: Routing path** — Dispatcher routes `WorktreeCreate` and `WorktreeRemove` events to `worktree-hooks.wasm` via the dual-layer routing tables (hooks.json.template Layer 1 → dispatcher → hooks-registry.toml Layer 2 → plugin). Verified by VP-067 `test_bc_4_07_003_hooks_json_template_worktree_entries` + `test_bc_4_07_004_hooks_registry_toml_worktree_entries`.
  - _trace: BC-4.07.003 (Layer 1 — hooks.json.template entries), BC-4.07.004 (Layer 2 — hooks-registry.toml entries)_

- [ ] **AC2: Wire format — WorktreeCreate** — `worktree.created` event emitted with exactly 10 fields on the wire: `worktree_path` (string), `worktree_name` (string) [plugin-set]; `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version` [host-enriched]; `ts`, `ts_epoch`, `schema_version`, `type="worktree.created"` [construction-time]. All plugin-set values are strings per `emit_event.rs:49` coercion. Plugin must NOT set RESERVED_FIELDS. Verified by VP-067 `test_bc_4_07_001_worktree_created_emitted_with_required_fields`.
  - _trace: BC-4.07.001 (10-field wire payload; plugin-set fields; RESERVED_FIELDS prohibition; wire format coercion)_

- [ ] **AC3: Wire format — WorktreeRemove** — `worktree.removed` event emitted with exactly 9 fields on the wire: `worktree_path` (string) [plugin-set]; 4 host-enriched; 4 construction-time with `type="worktree.removed"`. Unknown worktree paths emit normally (no-op). Verified by VP-067 `test_bc_4_07_002_worktree_removed_emitted_with_required_fields` + `test_bc_4_07_002_unknown_worktree_path_emits_normally`.
  - _trace: BC-4.07.002 (9-field wire payload; unknown-worktree no-op)_

- [ ] **AC4: Zero capability** — Plugin does NOT invoke `exec_subprocess` for either event. `exec_subprocess` CountingMock invocation_count == 0 for both WorktreeCreate and WorktreeRemove dispatches. - [ ] **AC6: hooks-registry.toml** — Registry contains exactly one `WorktreeCreate` entry and exactly one `WorktreeRemove` entry, both routing to `hook-plugins/worktree-hooks.wasm` with `name="worktree-hooks"`, `timeout_ms=5000`, NO capability tables, NO `once` field. Verified by VP-067 `test_bc_4_07_004_hooks_registry_toml_worktree_entries`.
  - _trace: BC-4.07.004 (Layer 2 registration; single-crate two-entry design; zero-capability sandbox)_

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | WorktreeCreate or WorktreeRemove fires multiple times for the same worktree_path (e.g., Claude Code reconnect after disconnect) | Plugin is unconditionally stateless; emits on every invocation. The `once` key MUST be absent from hooks.json.template (no Layer 1 dedup) — intentional. Multiple events for the same path are operator-observable. |
| EC-002 | WorktreeRemove for a worktree_path not previously registered (unknown worktree) | Plugin emits `worktree.removed` normally — no error, no abort. The plugin has no registry of known worktrees. Consumer (observability stack) handles unknown-path removal gracefully. |
| EC-003 | Envelope missing `worktree_path` field (WorktreeCreate or WorktreeRemove) | Plugin emits event with `worktree_path = ""` (empty string default). Plugin does not abort. |
| EC-004 | Capability scope check — plugin attempts exec_subprocess despite zero-capability profile | Host fn dispatch returns `CAPABILITY_DENIED` (BC-1.05.022 deny-by-default). Plugin must handle gracefully. CountingMock invocation_count still 0 from the plugin's deliberate path. |

## Architecture Compliance Footnotes

**F-07 Option 1 semantic scope ruling:** `subsystems: ["SS-04"]` only. The plugin behavior contracted by BC-4.07.001–004 is SS-04-owned (worktree event emission semantics). Files touched include SS-07-space (hooks-registry.toml) and SS-09/SS-07-space (hooks.json.template), but those are inputs to the verification, not the semantic scope. **Option A scoping decision footnote:** v1.0 legacy story (v1.0-legacy/S-5.3-worktree-hooks.md) requested "auto-registers new worktrees with the observability stack" via filesystem config write. This is explicitly out of scope for v1.0 because HOST_ABI v1.0 has no `write_file` host fn (verified at `crates/hook-sdk/HOST_ABI.md`). Three options were evaluated and documented in BC-4.07.001:
- Option A (SELECTED): emit events only; observability stack consumes at runtime; zero capabilities.
- Option B (REJECTED): `exec_subprocess` for a config-write CLI tool — scope creep, hard to test, requires new binary, violates S-5.01 lesson 3.
- Option C (REJECTED): add `write_file` host fn to HOST_ABI — major HOST_ABI bump, out of scope for v1.0 release-gate calendar.
This decision is applied proactively before adversarial review, unlike S-5.01 pass-4 architectural reversal (dispatcher-side dedup dropped) which happened reactively at pass-4.

**Pass-1 adversary correction (CRIT-001):** The original Option A footnote (v2.0) stated "CAP-003 (filesystem-write capability) deferred to v1.1" — this mis-anchored to the active observability capability CAP-003. The correct framing is that the filesystem-write capability has no CAP ID at v1.0 (provisional v1.1 name `CAP-NNN-filesystem-write` to be assigned when the v1.1 capability registry is expanded). Cf.
