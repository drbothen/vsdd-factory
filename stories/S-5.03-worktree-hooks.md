## Capability Anchor Justification

Primary: CAP-002 ("Hook Claude Code tool calls with sandboxed WASM plugins") per capabilities.md §CAP-002. This story wires the `WorktreeCreate` and `WorktreeRemove` event types through the dispatcher, implementing WASM plugins that auto-register/deregister worktrees with the observability stack.

FR anchor: FR-007 ("Legacy hook routing compatibility + dispatcher main") per PRD §2.1, extended to new event types. Part of proposed FR-045 ("New Claude Code lifecycle hook events"). Worktree auto-registration is distinct from session lifecycle but shares the same DRIFT-006 gap.

Cross-subsystem note: BC-1.02.005 lifecycle-event envelope invariant applies at SS-01. The worktree auto-registration (filesystem config write) has no existing BCs at any subsystem — all are v1.1 candidates.

## Narrative
- **As a** factory operator using multiple worktrees
- **I want to** have new worktrees automatically registered with the observability stack
- **So that** each worktree gets its own event stream without manual configuration

## Goal

Wire `WorktreeCreate` and `WorktreeRemove` events. WorktreeCreate
auto-registers new worktrees with the observability stack; WorktreeRemove
deregisters them. This is the automation that the v0.78.0 multi-factory
register CLI was attempting to solve.
