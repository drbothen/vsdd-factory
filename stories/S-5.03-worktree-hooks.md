# S-5.03: WorktreeCreate / WorktreeRemove hook wiring

**Epic:** E-5 — New Hook Events and 1.0.0 Release
**Phase:** Phase 5 — New Claude Code hook events
**Milestone:** `1.0.0`
**Tier:** G
**Blocks:** S-5.07
**Blocked by:** S-4.08
**Estimated effort:** M

## Narrative
- **As a** factory operator using multiple worktrees
- **I want to** have new worktrees automatically registered with the observability stack
- **So that** each worktree gets its own event stream without manual configuration

## Goal

Wire `WorktreeCreate` and `WorktreeRemove` events. WorktreeCreate
auto-registers new worktrees with the observability stack; WorktreeRemove
deregisters them. This is the automation that the v0.78.0 multi-factory
register CLI was attempting to solve.

## Acceptance criteria

- [ ] Dispatcher handles `WorktreeCreate` and `WorktreeRemove` event types
- [ ] `worktree.created` event emitted with worktree_path, worktree_name on creation
- [ ] `worktree.removed` event emitted with worktree_path on removal
- [ ] New worktree sink configuration auto-generated on WorktreeCreate
- [ ] `hooks.json` template updated with both handlers (no `"once": true` — can re-fire)
- [ ] Integration test: simulate WorktreeCreate payload; verify event and sink config
