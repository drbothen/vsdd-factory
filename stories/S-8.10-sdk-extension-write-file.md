## Acceptance Criteria

### AC-1: SDK write_file wrapper
`host::write_file(path: &str, contents: &[u8], max_bytes: u32, timeout_ms: u32) -> Result<(), HostError>`
exists in `crates/hook-sdk/src/host.rs` and round-trips a write call through the
dispatcher's `vsdd::write_file` host import. Both `max_bytes` and `timeout_ms` are
mandatory parameters â€” write_file is a bounded host call per BC-2.02.002 (BC-INDEX:153)
which requires a byte cap on all bounded calls.

The underlying FFI declaration (wasm32 target) has the following exact signature:
```
ffi::write_file(path_ptr: *const u8, path_len: u32, contents_ptr: *const u8,
                contents_len: u32, max_bytes: u32, timeout_ms: u32) -> i32
```
This uses the **input-pointer protocol** â€” the SDK wrapper passes guest-owned bytes;
the dispatcher copies via `read_wasm_bytes`. This differs from `read_file`'s
output-pointer protocol (caller provides buffer; host writes into it). Only the
error-code mapping (`HostError::from_code`) is shared with `read_file`.
(traces to BC-2.02.011 postcondition 1 â€” allowlist denial and capability gating)

### AC-2: Dispatcher write_file binding
A new file `crates/factory-dispatcher/src/host/write_file.rs` implements the
host-side wasmtime binding for `vsdd::write_file` with:
- Path validation: same `path_allowed` policy as `read_file.rs` (deny-by-default;
  paths must be in `capabilities.write_file.path_allow`).
- Timeout enforcement: `timeout_ms` parameter accepted for ABI stability (identical
  to `read_file.rs` line 36: "accepted for ABI stability; enforced in S-1.5 via
  epoch interruption").
- Content received via **input-pointer protocol**: `(contents_ptr: u32, contents_len: u32)`
  from guest memory via `memory::read_wasm_bytes`. The dispatcher copies the bytes
  from guest memory. This is the inverse of `read_file`'s output-pointer protocol;
  the FFI calling conventions differ.
- `max_bytes` cap enforced server-side: if `contents_len > max_bytes`, return
  `codes::OUTPUT_TOO_LARGE` before writing.
- Registered in `mod.rs::setup_linker` as `write_file::register(&mut linker)?;`.
(traces to BC-2.02.011 postcondition 2 â€” byte cap exceeded; no bytes written to disk)

## Architecture Compliance Rules

1. **SS-02 boundary:** All changes are within `crates/hook-sdk/` and
   `crates/factory-dispatcher/src/host/`. No files outside these two modules are
   modified (except CHANGELOG and docs/ABI.md for AC-7/AC-8).
2. **Deny-by-default capability model:** `write_file` follows the same
   deny-by-default pattern as `read_file`: no capability block = `CAPABILITY_DENIED`.
   See `read_file.rs` lines 73-77 for the reference implementation pattern.
3. **Memory safety:** Guest memory access uses `memory::read_wasm_bytes` (or the
   equivalent read helper in `memory.rs`). No raw pointer arithmetic.
4. **Error code stability:** No new negative error codes introduced. The existing set
   {-1 CapabilityDenied, -2 Timeout, -3 OutputTooLarge, -4 InvalidArgument, -99 InternalError}
   is sufficient. `write_file` returns: `0` on success; `-1` on capability denial OR
   path traversal denial (matching `read_file` pattern â€” both map to `CAPABILITY_DENIED`);
   `-3` on `OutputTooLarge` (`max_bytes` exceeded); `-4` on UTF-8 path decoding failure;
   `-99` on filesystem io error.
5. **HOST_ABI_VERSION invariant:** Both `crates/hook-sdk/src/lib.rs:58` and
   `crates/factory-dispatcher/src/lib.rs:43` must still read `= 1` after this story.
   CI must verify this (or the story-writer's AC-3 covers it via grep assertion).
6. **Workspace Cargo.toml:** No new crate is added. This story modifies two existing
   crates; `Cargo.toml` workspace `members` array is unchanged.
