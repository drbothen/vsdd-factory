# S-4.07: End-to-end observability integration tests

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.08
**Blocked by:** S-3.01..S-3.04, S-4.01..S-4.06, S-4.10 (cross-sink internal.sink_error emission per BC-3.07.002)
**Estimated effort:** L (bumped from M: scope expanded to cover datadog, honeycomb, full harness spec, CI integration, and test determinism — see Version History)

## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

CAP-003 states: "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`. Outcome: operator sees events in Grafana/Loki or custom endpoint without modifying dispatcher code." S-4.07 is the verification story that confirms CAP-003's end-to-end outcome actually holds under realistic configurations. It exercises the fan-out, the routing filters, the DLQ, and the circuit breaker — the full SS-03 behavioral surface assembled together.

**Cross-subsystem note:** This story also touches SS-01 because the integration tests exercise the full dispatcher pipeline (SS-01 dispatch path → SS-03 sinks).

## Narrative
- **As a** release engineer preparing 1.0.0-rc.1
- **I want to** have end-to-end integration tests covering all sink configurations and event flows
- **So that** rc.1 ships with verified observability behavior under realistic configurations

## Goal

End-to-end validation that the observability system works under
realistic configurations. Exercises: zero-disk mode, hybrid mode,
multi-sink fan-out, routing filters, DLQ, circuit breaker, datadog,
honeycomb, and all sink driver `internal.sink_error` emissions.

## Integration Test Harness — SUT Boundary

S-4.07 tests exercise `Router::submit()` as the System Under Test (SUT) entry point. Tests construct Router IN-TEST (not via main.rs) using:

```rust
let registry = SinkRegistry::from_config(&config);
let router = Router::new(registry.clone());
router.submit(event).await;
```

This is a TEST-CONTROLLED ASSEMBLY pattern. The integration test holds both `router` and a clone of `registry` for oracle access. Oracle assertions may read `registry.sinks()[0].stats()` directly because the registry is the test's local handle, not a production singleton.

**Exceptions to the Router::submit() SUT entry point:**
- **AC-11 (OTLP attribute mapping)** invokes `OtelGrpcSink::send()` directly. Rationale: deep OTLP wire-level verification (resource attributes, scope, log records) is independent of routing decisions; testing through Router would obscure OTLP semantics behind sink-selection logic.
- **AC-12 (cross-sink failure event emission)** injects `SinkFailure` via per-driver test harness. Rationale: AC-12 verifies the contracted internal-channel emission path on failure, which is invoked from the sink driver's failure-recording site (not from Router). Driving via Router would couple AC-12 to routing-layer concerns.

Both deviations are intentional. All other ACs (AC-1..AC-10, AC-13..AC-16) honor the Router::submit() boundary.

**Why test-controlled assembly:** BC-3.04.001 Precondition 1 documents that dispatcher main.rs does not yet call `Router::submit` (Router-into-main wiring is OUT OF SCOPE for this story; tracked as a follow-up). S-4.07 verifies Router behavior at the API boundary, not at the dispatcher-binary boundary.

**S-4.07 dependency note:** Once Router-into-main wiring lands (separate follow-up story), a downstream story may add main.rs E2E integration tests. S-4.07's tests will continue to pass without modification because `Router::submit`'s contract is unchanged.

## Determinism Strategy

All retry/backoff/circuit-breaker tests SHALL use `tokio::time::pause()` + `tokio::time::advance()` for time progression:

- Production sink retry loops MUST use `tokio::time::sleep` (not `std::thread::sleep`) to be controllable by test time injection.
- Real-clock tolerance (±50ms) is NOT used in any S-4.07 test. All retry/backoff/circuit-breaker tests in this story use `tokio::time::pause()` + `tokio::time::advance()` for deterministic time control. Wall-clock measurement is exclusive to S-4.09's AC-10 (sink-http backoff measurement) and is out of scope for S-4.07.
- Tests that use `tokio::time::pause()` must be marked `#[tokio::test]` (not `#[tokio::test(flavor = "multi_thread")]`) to ensure single-threaded time control.
