# S-4.07: End-to-end observability integration tests

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.08
**Blocked by:** S-3.01..S-3.04, S-4.01..S-4.06
**Estimated effort:** M

## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

CAP-003 states: "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`. Outcome: operator sees events in Grafana/Loki or custom endpoint without modifying dispatcher code." S-4.07 is the verification story that confirms CAP-003's end-to-end outcome actually holds under realistic configurations. It exercises the fan-out, the routing filters, the DLQ, and the circuit breaker — the full SS-03 behavioral surface assembled together.

**Cross-subsystem note:** This story also touches SS-01 because the integration tests exercise the full dispatcher pipeline (SS-01 dispatch path → SS-03 sinks). VP-051 (Dispatcher Startup Flow Writes Parseable JSONL with Correct Envelopes) is included as a cross-subsystem VP.

## Narrative
- **As a** release engineer preparing 1.0.0-rc.1
- **I want to** have end-to-end integration tests covering all sink configurations and event flows
- **So that** rc.1 ships with verified observability behavior under realistic configurations

## Goal

End-to-end validation that the observability system works under
realistic configurations. Exercises: zero-disk mode, hybrid mode,
multi-sink fan-out, routing filters, DLQ, circuit breaker.

## Acceptance Criteria

- [ ] Test: zero-disk mode (otel-grpc only; no file sink) — events appear in mock OTLP receiver
  (traces to BC-3.03.007 postcondition 1: shutdown drains and joins the worker thread — zero-disk mode test verifies the otel-grpc sink lifecycle end-to-end; BC-3.03.001 postconditions 1/2/3: batch trigger thresholds fire and deliver events to mock receiver)
- [ ] Test: hybrid mode (file + otel-grpc) — events in both destinations
  (traces to BC-3.05.002 postcondition 1: local-events file has 10 lines; BC-3.05.003 postcondition 1: 10 records arrive at mock OTLP receiver with correct attribute mapping — these two BCs together define the hybrid mode expected outcome)
- [ ] Test: multi-sink fan-out with routing filters — audit events in audit sink, metrics in metrics sink
  (traces to BC-3.05.002 postcondition 1: audit-filtered file has 8 lines — denies plugin.timeout + internal.sink_error; BC-3.01.004 postcondition 1: RoutingFilter allow non-empty = whitelist; BC-3.02.009 postcondition 1: allow=["commit.made"] passes 2 of 3 events through)
- [ ] Test: sink failure → DLQ — mock 5xx; events appear in DLQ file
  (traces to BC-3.03.002 postcondition 1: drop gRPC client on error, rebuild on next batch — mock 5xx exercises this path; BC-3.01.001 postcondition 1: registry operations succeed even when individual sinks fail)
- [ ] Test: circuit breaker — repeated failures → circuit opens; `internal.sink_circuit_opened` emitted
  (traces to VP-012 (Sink Failure Affects Only That Sink) for failure isolation; circuit breaker emission itself is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-circuit-breaker-integration`. See S-4.04/S-4.07 backlogs.)
- [ ] Test: tag enrichment propagation — tags appear on every event in output
  (traces to BC-3.02.010 postcondition 1: tags `env=prod,team=factory` land on every event; BC-3.02.011 postcondition 1: tag does not clobber producer type field — integration verifies both in combination)
- [ ] Test: disabled sink receives no events
  (traces to BC-3.02.012 postcondition 1: enabled=false → no file written, no events accepted)
- [ ] Test: shutdown drains all in-flight events
  (traces to BC-3.02.015 postcondition 1: shutdown() drains pending events; post-shutdown submit is a no-op — verified in BC-3.05.002 postcondition 1 also)
- [ ] All tests green in CI
  (traces to BC-3.04.001 postcondition 1: Router wraps SinkRegistry; submit delegates to registry.submit_all — all tests exercise the assembled Router → SinkRegistry → individual sinks path; BC-3.05.001 postcondition 1: ObservabilityConfig builds the correct sink registry from parsed config)
- [ ] Test config-load with unknown sink type does not fail
  (traces to BC-3.01.002 postcondition 1: unknown sink type warns to stderr but does not fail config load)
- [ ] OTLP LogRecord field mapping correct in integration context
  (traces to BC-3.05.003 postcondition 1: 10 records arrive; `type` → body, `ts_epoch` → `time_unix_nano`; type and ts_epoch are NOT present in attributes — full attribute mapping validated in integration)
- [ ] Integration test: each sink driver (sink-http, sink-otel-grpc, sink-file) emits `internal.sink_error` events on recorded SinkFailure — captured from the internal event channel and asserted for correct `type`, `sink_type`, `sink_name`, `attempt`, and `error_message` fields
  (traces to BC-3.07.002 postcondition 1: mandatory event fields present; postcondition 3: emission failure is silently dropped, sink does not panic — S-4.10 production emission code is the prerequisite; these assertions confirm it fires from each driver in an end-to-end integration context)
