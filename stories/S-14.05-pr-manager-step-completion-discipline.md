# S-14.05: pr-manager Step-4/Step-5 Early-Exit Codification

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** FIFTH in E-14. Forward-looking for pass-2 cycle. Can be delivered
  independently of S-14.02 through S-14.04.
**Depends on:** (none)
**Blocks:** (none)
**Priority:** P1

## Narrative

- **As a** vsdd-factory orchestrator dispatching pr-manager to deliver a story PR
- **I want** pr-manager to reliably complete all steps (Steps 1–7) without exiting
  silently after Step 4 (security review) or Step 5 (pr-reviewer APPROVE)
- **So that** no story PR is created in a half-delivered state where security review
  is marked complete but the PR has not been opened, or where the PR is opened but
  the merge/CI-wait steps are skipped

## Background

During B1/B2 PR runs for the v1.0-feature-engine-discipline-pass-1 fix burst,
pr-manager exhibited two reproducible early-exit failure modes:

**Mode 1:** pr-manager exits after completing Step 4 (security review) without
proceeding to Step 5 (dispatch pr-reviewer). The PR is never opened. The orchestrator
discovers this only when it checks for a PR and finds none. No error is surfaced.

**Mode 2:** pr-manager exits after Step 5 (pr-reviewer returns APPROVE) without
proceeding to Step 6 (open PR via gh pr create) or Step 7 (wait for CI / merge).
The PR review is marked APPROVE but the PR itself is never created.

Both modes violate the "Never terminate mid-flow" invariant documented in
pr-manager's AGENT.md. The root cause appears to be that pr-manager treats APPROVE
from pr-reviewer as a natural completion signal and stops — rather than recognizing
that approval is a gate on the PR creation step, not the final step.

**Source:** PG-6, observed during B1/B2 orchestration. Not in adv-cycle-pass-1.md
directly — this was identified by the orchestrator during B1/B2 PR runs and surfaced
as an additional process gap for B6.

## Behavioral Contracts

No existing BCs cover pr-manager step-completion discipline. New BCs are required.

- **BC-X.NN.001 (to be authored at story implementation start):** pr-manager MUST
  NOT emit a completion signal or allow context termination after Step 4 (security
  review) unless Step 4 returned a blocking finding that requires human intervention.
  Non-blocking Step 4 completion MUST automatically transition to Step 5.
- **BC-X.NN.002 (to be authored at story implementation start):** pr-manager MUST
  NOT emit a completion signal or allow context termination after Step 5 (pr-reviewer
  APPROVE) unless the PR has already been created. APPROVE from pr-reviewer is a
  gate condition on Step 6 (open PR), not a pipeline terminal state.
- **BC-X.NN.003 (to be authored at story implementation start):** pr-manager MUST
  emit a `STEP_COMPLETE: step-N` marker after each step completes successfully.
  The orchestrator (or a monitoring hook) can detect an absent STEP_COMPLETE for
  step 6 or 7 as a sign of early exit and re-dispatch.

## Acceptance Criteria

### AC-001 (traces to PG-6 Mode 1 — no early exit after Step 4)

`pr-manager/AGENT.md` (or equivalent pr-manager workflow file — implementer must
verify exact path) contains explicit language stating:
- Step 4 completion is a GATE on Step 5, not a terminal state.
- pr-manager MUST proceed to Step 5 immediately after Step 4 returns any result
  other than a blocking security finding.
- If pr-manager context is about to terminate after Step 4 (e.g., due to long
  context), it MUST emit a `STEP_INCOMPLETE: step-4-complete, step-5-not-started`
  marker before exiting, and the orchestrator MUST re-dispatch from Step 5.

**Falsifiable test:** `grep -n "STEP_INCOMPLETE\|Step 4.*gate\|Step 5.*must proceed"
plugins/vsdd-factory/agents/pr-manager/AGENT.md` (or pr-manager workflow file) returns
≥1 match. `grep -n "STEP_COMPLETE" plugins/vsdd-factory/agents/pr-manager/AGENT.md`
returns ≥5 matches (one per step 1–5 minimum).

### AC-002 (traces to PG-6 Mode 2 — no early exit after Step 5)

The pr-manager workflow file contains explicit language stating:
- Step 5 APPROVE from pr-reviewer is a GATE on Step 6 (gh pr create), not a
  terminal state.
- pr-manager MUST NOT treat APPROVE as a completion signal.
- The terminal state for pr-manager is: PR opened + CI checks passed + PR merged
  (or merge-pending human approval). Any earlier termination is STEP_INCOMPLETE.

**Falsifiable test:** `grep -n "APPROVE.*gate\|pr create.*after approve\|STEP_COMPLETE.*step-6"
plugins/vsdd-factory/agents/pr-manager/AGENT.md` (or workflow file) returns ≥1 match.
`grep -n "terminal state\|completion.*merged\|merged.*completion"` returns ≥1 match
defining the true terminal state.

### AC-003 (traces to PG-6 — STEP_COMPLETE marker protocol)

pr-manager emits `STEP_COMPLETE: step-N` in its output after each step completes.
The orchestrator (or monitoring hook) checks for the presence of `STEP_COMPLETE: step-6`
and `STEP_COMPLETE: step-7` before considering pr-manager done. If either marker is
absent from pr-manager's output, the orchestrator re-dispatches from the first
missing step.

The STEP_COMPLETE protocol is documented in:
- pr-manager/AGENT.md (what to emit and when)
- Orchestrator MANDATORY STEPS (how to detect and recover from missing markers)

**Falsifiable test:** `grep -n "STEP_COMPLETE" plugins/vsdd-factory/agents/pr-manager/AGENT.md`
returns ≥6 matches (Steps 1-6 minimum). `grep -n "STEP_COMPLETE.*step-7\|step-7.*STEP_COMPLETE"
plugins/vsdd-factory/agents/pr-manager/AGENT.md` returns ≥1 match.

### AC-004 (traces to PG-6 — orchestrator recovery protocol)

The orchestrator MANDATORY STEPS section is updated with a pr-manager completion check:
after dispatching pr-manager, the orchestrator scans pr-manager's output for
`STEP_COMPLETE: step-6` and `STEP_COMPLETE: step-7` markers. If absent, the
orchestrator re-dispatches pr-manager starting from the first missing step. This
protocol prevents the silent Mode 1 and Mode 2 failures.

**Falsifiable test:** `grep -n "STEP_COMPLETE.*step-6\|STEP_COMPLETE.*step-7\|pr-manager.*re-dispatch"
plugins/vsdd-factory/agents/orchestrator/orchestrator.md` returns ≥1 match in the
MANDATORY STEPS section.

### AC-005 (traces to PG-6 — fix-pr-delivery skill parity)

The `fix-pr-delivery` skill (used during adversarial fix bursts, e.g., B1/B2) is
updated with the same STEP_COMPLETE protocol. fix-pr-delivery also uses pr-manager
internally — the same Mode 1/Mode 2 failures apply. If fix-pr-delivery has its own
pr-manager dispatch section, it must be updated with the recovery check per AC-004.

**Falsifiable test:** `grep -n "STEP_COMPLETE"
plugins/vsdd-factory/skills/fix-pr-delivery/SKILL.md` returns ≥1 match confirming
the completion check is present.

## Dev Notes

**OQ-E14-03 resolution (recommended):** This story recommends Option B (STEP_COMPLETE
sentinel) over Option A (inverted control) for the following reasons:
1. Inverted control requires refactoring pr-manager from a monolithic agent workflow
   into a step-by-step callback model — large blast radius, likely requires Lobster
   workflow changes.
2. STEP_COMPLETE sentinel requires only documentation changes (pr-manager AGENT.md
   and orchestrator MANDATORY STEPS) — zero code changes, faster to deliver.
3. The sentinel approach is self-documenting: an absent sentinel is detectable by
   the orchestrator via simple text scan.

The implementer should document this decision in the PR description and note whether
OQ-E14-03 was resolved before implementation began.

**pr-manager file path:** The pr-manager agent file path is uncertain. It may be
`plugins/vsdd-factory/agents/pr-manager/AGENT.md`,
`plugins/vsdd-factory/agents/pr-manager.md`, or
`plugins/vsdd-factory/skills/pr-manager/SKILL.md`. T-0 survey is mandatory before
modifying any file.

**STEP_COMPLETE marker format:** Recommend a consistent format across all agents
that use multi-step workflows:
```
STEP_COMPLETE: step-N — <brief description of what completed>
```
Example: `STEP_COMPLETE: step-4 — security review passed (no HIGH/CRITICAL findings)`
Example: `STEP_COMPLETE: step-5 — pr-reviewer APPROVED`
Example: `STEP_COMPLETE: step-6 — PR #NNN opened at <url>`

The orchestrator checks for `STEP_COMPLETE: step-6` and `STEP_COMPLETE: step-7` in
pr-manager's output. If either is missing after a reasonable timeout (or after
pr-manager reports it is done), re-dispatch.
