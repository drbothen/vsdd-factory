# S-14.08: Resolver Load-Event Observability: plugin.log resolver.loaded Events

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** EIGHTH in E-14. Can be delivered independently.
**Depends on:** (none)
**Blocks:** (none)
**Priority:** P1

## Narrative

- **As a** factory operator running a VSDD pipeline with WASM context resolvers
- **I want** the dispatcher to emit a `resolver.loaded` event to plugin.log when
  each resolver module is successfully loaded at startup
- **So that** I can confirm which resolvers are active, diagnose missing-resolver
  issues, and have an audit trail of resolver initialization

## Background

F-P3-011 (pass-3 adversarial finding) identified that the dispatcher does not emit
observable `resolver.loaded` events to plugin.log when resolver modules are loaded.
Without these events, an operator cannot determine at runtime which resolvers are
active or diagnose a resolver that silently fails to load.

This finding was triaged in D-378 as FOLLOW-UP STORY requiring:
1. Dispatcher code change: emit `resolver.loaded` event to InternalLog per resolver
   successfully loaded (crates/factory-dispatcher or crates/factory-dispatcher-core)
2. Bats test: grep dispatcher-internal log output for `resolver.loaded` events
   after a test run that loads a resolver

## Acceptance Criteria

### AC-001: Locate resolver loading code in dispatcher
- Find the code path in crates/factory-dispatcher* where resolvers are loaded from
  resolvers-registry.toml at startup
- Confirm whether a `resolver.loaded` log event is currently emitted
- If already emitted, verify format matches BC-4.12.001 INV-N (or document new INV)

### AC-002: Emit resolver.loaded event per loaded resolver
- After each resolver module is successfully loaded, emit a structured log event:
  `resolver.loaded { name: <resolver_name>, path: <wasm_path>, version: <abi_version> }`
- Use the dispatcher's existing InternalLog mechanism (consistent with other plugin.log events)
- If a resolver fails to load, emit `resolver.load_failed { name, path, error }` instead

### AC-003: Bats test verifying resolver.loaded in dispatcher log
- In `plugins/vsdd-factory/tests/resolver-integration.bats` (or a new test file),
  add a test that:
  - Runs a dispatcher invocation with a resolver configured in resolvers-registry.toml
  - Captures dispatcher-internal log output (VSDD_INTERNAL_LOG or equivalent)
  - Asserts `resolver.loaded` event appears with the correct resolver name
- Test must pass in `bats plugins/vsdd-factory/tests/resolver-integration.bats`

### AC-004: Cargo tests pass
- `cargo test --workspace --all-targets` remains green
- No new warnings introduced
