# S-12.07: `vsdd-context-resolvers` Crate + WaveContextResolver (First Concrete Resolver)

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** Batch 4 (after S-12.04 and S-12.05 both complete).
**Depends on:** S-12.04 (WASM loading infrastructure must be live), S-12.05 (SDK resolver authoring API must exist)
**Blocks:** S-12.08 (the convergence hook migration consumes this resolver's output)
**Priority:** P0

## Narrative

- **As a** vsdd-factory operator
- **I want** a per-factory `vsdd-context-resolvers` crate that compiles to a WASM artifact, implementing the `WaveContextResolver` which reads `.factory/STATE.md` and `.factory/wave-state.yaml` to produce a `wave_context` object containing the current cycle ID, wave ID, and story list
- **So that** hooks declaring `needs_context = ["wave_context"]` receive an injected `plugin_config.wave_context` object at dispatch time — enabling the convergence hook to read the wave's story list from a dynamically-injected context rather than a static registry config

## Acceptance Criteria

### AC-001 (traces to BC-4.12.002 postcondition 3 — output shape is `ResolverOutput { key: "wave_context", value: Some({...}) }`)

`WaveContextResolver::resolve_impl` returns `ResolverOutput { key: "wave_context".into(), value: Some(json!({ "stories": [...], "wave_id": "...", "cycle_id": "..." })) }` when `.factory/wave-state.yaml` is present and parseable.

**Falsifiable test:** Rust unit test with in-memory test fixtures: call `resolve_wave_context_pure(&input, &wave_state, Some("v1.0-cycle"))` with a valid `WaveState` fixture; assert `output.key == "wave_context"`; assert `output.value.unwrap()["stories"]` is a JSON array; assert `output.value.unwrap()["wave_id"]` is a string; assert `output.value.unwrap()["cycle_id"]` is a string.

### AC-002 (traces to BC-4.12.004 postcondition 3 + BC-4.12.002 postcondition 3 — `value: None` when `wave-state.yaml` absent or unparseable)

When `.factory/wave-state.yaml` is absent (host `read_file` returns error) or returns malformed YAML, `resolve_impl` returns `ResolverOutput { key: "wave_context", value: None }` — NOT a trap, NOT a panic. The resolver handles these as expected operational states.

**Falsifiable test:** Rust unit test: mock `host::read_file` to return error; call `resolve_impl`; assert `output.value.is_none()`. Second test: mock `host::read_file` to return malformed YAML bytes; assert `output.value.is_none()` (graceful parse error handling).

### AC-003 (traces to BC-4.12.002 postcondition 3 — `value: None` when `wave-state.yaml` has no active wave)

When `.factory/wave-state.yaml` is absent, has all waves in terminal states, or has no active wave, `resolve_impl` returns `ResolverOutput { key: "wave_context", value: None }`. When an active wave is found but its stories list is empty, `resolve_impl` returns `ResolverOutput { key: "wave_context", value: Some({stories: [], wave_id, cycle_id}) }` so that BC-4.10.001 EC-001 vacuous-convergence Continue path is reachable in the consumer hook (per S-12.08 pass-1 HIGH-002 fix).

**Falsifiable test:** Rust unit test: `wave-state.yaml` fixture with `waves: []`; assert `output.value.is_none()`.

### AC-004 (traces to BC-4.12.002 postcondition 3 — `value: None` when `wave-state.yaml` schema has missing optional fields)

`wave-state.yaml` may have optional fields. The resolver handles the case where `wave_id` or `cycle_id` fields are missing (using sensible defaults or returning `None` if they are required but absent). Resolver does NOT trap on missing optional fields.

Note: `WaveEntry.wave` is a required `String` field (no `#[serde(default)]`). YAML missing the `wave:` key causes `parse_wave_state` to return `Err` (fail-loud schema). An empty `waves: []` list is the operational proxy for "no active wave" at the resolver level.

**Falsifiable test:** Two tests jointly cover AC-004:
1. `test_parse_wave_state_rejects_missing_required_wave_field`: asserts `parse_wave_state("waves:\n  - stories: [\"S\"]")` returns `Err` — the schema is fail-loud when the required `wave` field is absent.
2. `test_BC_4_12_002_missing_wave_id_no_panic`: asserts `resolve_wave_context_pure` with an empty `waves: []` list returns `value: None` and does NOT panic — this is the operational proxy for schema-incomplete input (resolver never receives partially parsed entries).

### AC-005 (traces to BC-4.12.003 postconditions 1–2 — `path_allow = [".factory/"]` restricts reads)

**DEFERRED to S-12.08:** This AC is now covered by S-12.08's bats end-to-end test harness rather than a unit-level integration test in this crate. Justification: factory-dispatcher's ResolverLoader is not pub; making the resolver crate dev-depend on factory-dispatcher would create a path-circularity. VP-076 §Proof Harness Locations amended in lock-step.

The `resolvers-registry.toml` entry for `wave_context` declares `path_allow = [".factory/"]`. Attempting to read a path outside `.factory/` (e.g., `/etc/passwd`) via `host::read_file` returns `CapabilityDenied` to the resolver, which handles it by returning `ResolverOutput { key: "wave_context", value: None }` (not a trap).

**Falsifiable test (VP-076-A, VP-076-B):** Integration test in S-12.08 bats harness: run resolver with attempt to read `/etc/passwd`; assert `CapabilityDenied` received; assert no `/etc/passwd` content in `plugin_config`; assert `resolver.capability_denied` event emitted.

### AC-006 (traces to BC-4.12.003 postcondition 1 + VP-076-D — reads within `path_allow` succeed)

**DEFERRED to S-12.08:** Same rationale as AC-005. See VP-076 §Proof Harness Locations for amendment details.

When `.factory/wave-state.yaml` exists and is within `path_allow = [".factory/"]`, the resolver reads it successfully. The positive capability case must work.

**Falsifiable test (VP-076-D):** Integration test in S-12.08 bats harness: create a temp project with `.factory/wave-state.yaml` and `.factory/STATE.md`; run `WaveContextResolver` with `project_dir = <temp_project>`; assert `wave_context` key is present in `plugin_config` with non-null value.

### AC-007 (traces to BC-4.12.002 postcondition 2 — `ResolverInput.project_dir` is used to construct paths)

The resolver reads `input.project_dir` (top-level field per BC-4.12.002 PC2, injected by the dispatcher) to construct the absolute path to `.factory/wave-state.yaml` and `.factory/STATE.md`. It does NOT hardcode any path. Note: `project_dir` is a top-level `String` field on `ResolverInput`, NOT inside `plugin_config`.

**Falsifiable test:** Rust unit test: provide `project_dir = "/tmp/test-project"` as the top-level `ResolverInput.project_dir` field; assert that `plugin_config` does NOT contain `project_dir` (confirming it is not in `plugin_config`). See `test_BC_4_12_002_project_dir_is_top_level_field` in `wave_context_test.rs`.

`crates/vsdd-context-resolvers/src/lib.rs` exposes a `pub fn resolve_wave_context_pure(input: &ResolverInput, wave_state: &WaveState, cycle_id: Option<&str>) -> ResolverOutput` function (pure computation layer with mocked file I/O). The `resolve_impl` function handles ALL error paths (missing files, malformed YAML, unexpected schema) by returning `ResolverOutput { value: None }` — NEVER by calling `panic!()`, `unwrap()`, or any other trap-generating code path. The resolver is panic-free by construction.

## Architecture Compliance Rules

Derived from BC-4.12.001–005, ADR-018, and F1-amendment:

1. **Factory-agnostic dispatcher invariant (BC-1.13.001 invariant 1):** The workspace `Cargo.toml` MUST NOT add `vsdd-context-resolvers` as a dependency of `factory-dispatcher`. The two crates are completely independent at compile time. Communication is via the WASM ABI at runtime only.
2. **Pure/effectful separation is mandatory (VP-075):** `resolve_wave_context_pure` must take pre-parsed `WaveState` as a parameter — not call `host::read_file` internally. The proptest exercises the pure function only. Any drift toward `resolve_wave_context_pure` performing I/O is a VP-075 violation.
3. **No `unwrap()` or `expect()` anywhere in production code (BC-4.12.004 invariant 1):** Use `?` or `match` for all error handling. A panic in `resolve_impl` is a WASM trap; WASM traps are caught by BC-4.12.004 but the resolver should never generate them for operational error cases.
4. **`path_allow` must match exactly what is declared in `resolvers-registry.toml`:** The `WaveContextResolver` reads ONLY `.factory/wave-state.yaml` and `.factory/STATE.md`. The `path_allow = [".factory/"]` declaration covers both. Do NOT widen the `path_allow` beyond what the resolver needs.
5. **`serde_yaml` version pin:** Use the version currently in the workspace. Do NOT introduce a new version. If `serde_yaml` is not in the workspace, use `serde` with a YAML-supporting crate that IS in the workspace, or add `serde_yaml` with justification.

## Forbidden Dependencies

`crates/vsdd-context-resolvers/Cargo.toml` MUST NOT depend on:
- `factory-dispatcher` (no circular dependency; WASM artifact loaded dynamically)
- `validate-per-story-adversary-convergence` (or any other hook plugin crate)
- Any crate that pulls in `wasmtime` at compile time (resolvers run INSIDE wasmtime, they don't host it)
- Any crate that assumes a specific OS (must compile to `wasm32-wasip1`)

## Dev Notes

**Wave-state YAML schema:** Read the actual `.factory/wave-state.yaml` in the project before writing the `WaveState` struct. The struct must match the ACTUAL schema, not a hypothetical one. Use `#[serde(default)]` on optional fields. Use `Option<String>` for fields like `gate_status` that may be absent.

**`project_dir` path construction:** `input.project_dir` is the top-level field on `ResolverInput` (NOT inside `plugin_config`). The resolver constructs paths as `format!("{}/.factory/wave-state.yaml", input.project_dir)`. Do NOT use `input.plugin_config["project_dir"]` — that was corrected in pass-1 (F-004).

**WASM cross-compilation:** The `Cargo.toml` needs `[lib] crate-type = ["cdylib"]`. The build command is `cargo build -p vsdd-context-resolvers --target wasm32-wasip1 --release`. The `Makefile` or `build.rs` in the project may have an existing pattern for WASM cross-compilation of hook plugins — follow that pattern.

**R-PLAT-004 bootstrap concern:** This story's Step 4.5 adversary convergence will run AGAINST the resolver platform itself (the convergence hook reads `wave_context.stories` injected by THIS resolver's predecessor, once S-12.08 is complete). For S-12.07's Step 4.5 specifically, the convergence hook has NOT yet been migrated (S-12.08 comes after), so it will still read from static config. The end-to-end test for S-12.08 is the platform validation.

**Bootstrap-flipped Step 4.5:** Run the adversary loop. First pass typically flags: (a) missing graceful-degrade for absent wave-state; (b) wrong path_allow prefix; (c) `unwrap()` calls; (d) proptest trial count. Fix each before the next pass.
