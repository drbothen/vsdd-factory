# S-7.03: TDD Discipline Hardening — Stub-as-Implementation Anti-Pattern Prevention

> **Cluster decision:** Thirteen new BCs across four layers (anti-precedent guard,
> Red Gate density check, tdd_mode frontmatter contract, mutation testing wave-gate)
> are grouped here because all address the single root cause: the TDD Iron Law was
> silently bypassable via stub-as-implementation. The four layers form a mutually
> reinforcing defense: the guard (Layer 1) shapes agent behavior at dispatch;
> the density gate (Layer 2) provides quantitative enforcement; the tdd_mode contract
> (Layer 3) distinguishes legitimate facade work from violations; and the mutation
> wave-gate (Layer 4) closes the quality gap for facade-mode stories.

## Story Statement

vsdd-factory uses its own VSDD process to harden itself. This story is the direct
product of vsdd-factory's own adversarial process catching a process deficiency in
its own running instances.

During Prism Wave 2, 3 of 5 parallel stub-architect dispatches produced stubs with
full business logic instead of `todo!()` bodies. The root cause was precedent
cascade: earlier merged DTU crates (commits `aa706543`, `6d2d005e`) had
pre-implemented stubs, and Wave 2 stub-architects loaded those crates as contextual
templates. By Step 3 (Red Gate), most tests already passed — meaning the
RED_RATIO was near 0.0 — and Step 4 (implementer dispatch) became a no-op. The
TDD Iron Law was silently violated across 3 stories with no blocking signal.

The correct pattern was demonstrated in commit `e86d03f2` (Prism S-2.06
datasource-trait), which contained 5 genuine `todo!()` macros in the algorithmic
core. That stub produced real RED failures at Step 3, enabling a genuine TDD cycle.

This story codifies four structural defenses against recurrence:
1. An **anti-precedent guard** embedded verbatim in the dispatch prompt so agents
   have an explicit counter-instruction when sibling crates are anti-patterns.
2. A **RED_RATIO density gate** between Step 3 and Step 4 that blocks
   implementer dispatch if fewer than 50% of new tests are red.
3. A **tdd_mode frontmatter contract** that formally distinguishes `strict` (full
   TDD cycle required) from `facade` (DTU/mock work where the scaffold IS the
   implementation) so the pipeline can apply the right enforcement per story type.
4. A **mutation testing wave-gate** that applies to facade-mode stories as the
   compensating quality control in place of the Red Gate cycle they legitimately skip.

This is the third E-7 story — alongside S-7.01 (agent prompt spec-first discipline)
and S-7.02 (state-manager defensive sweep) — continuing the self-referential
dogfooding pattern: vsdd-factory's adversarial process catching vsdd-factory's
process gaps, codifying the lessons into plugin source so future cycles benefit
automatically.

## Narrative

- **As a** vsdd-factory orchestrator running a Wave-N story delivery
- **I want** the stub-architect dispatch prompt to contain an anti-precedent guard,
  Step 3 Red Gate to enforce a RED_RATIO ≥ 0.5 density check, stories to declare
  their tdd_mode (strict or facade), and the wave-gate to run mutation testing for
  facade-mode stories
- **So that** the class of TDD Iron Law violations observed in Prism Wave 2 is
  structurally prevented at every future delivery, regardless of what sibling crates
  an agent observes in context

## Acceptance Criteria

### AC-001: todo!() obligation for non-trivial stub bodies (traces to BC-5.38.001 postcondition 1, postcondition 2)

The stub-architect agent, when executing Step 2 for a `tdd_mode: strict` (explicit
or absent) story, must produce `todo!()` or `unimplemented!()` macro bodies for
every non-trivial function body (any function whose eventual implementation will
exceed 3 lines of real logic). No real business logic is permitted in non-trivial
functions in a strict-mode stub commit.

- **Test:** Review of stub-architect.md confirms the todo!() obligation is in the
  Constraints section citing BC-5.38.001.
- **Test:** Stub commit message for a strict-mode story identifies any functions
  that were NOT stubbed with todo!(), with explicit GREEN-BY-DESIGN or
  WIRING-EXEMPT justification per BC-5.38.002 / BC-5.38.003.
- **Test:** `test_stub_architect_uses_todo_for_nontrivial_bodies()` in
  `tdd-discipline-gate.bats` confirms the guard text is present in both dispatch

### AC-002: GREEN-BY-DESIGN exemption for pure data mappings (traces to BC-5.38.002 postcondition 1, postcondition 2, invariant 1)

Pure data mapping functions (enum variant labels, zero-logic constant accessors,
unit struct constructors — meeting all four criteria: zero branching, no I/O, no
calls to non-trivial helpers, body ≤ 3 lines) may be implemented inline in the
stub commit. Each such function must be listed in a `GREEN-BY-DESIGN` section in
the stub commit report. The Red Gate density check (BC-8.29.001) excludes
GREEN-BY-DESIGN tests from the RED_RATIO denominator using the list.

- **Test:** BC-5.38.002 canonical test vectors: `fn column_name(&self) -> &str { "jira_status" }` maps to GREEN-BY-DESIGN in report; `fn compute_risk_score(&self) -> f64 { 0.0 }` does NOT qualify.
- **Test:** `test_green_by_design_excluded_from_red_ratio_denominator()` in bats confirms denominator logic.

### AC-003: WIRING-EXEMPT for framework integration structural minimums (traces to BC-5.38.003 postcondition 1, postcondition 2, postcondition 3, invariant 1)

Framework integration wiring methods required for compilation (e.g., Tower
`Service::poll_ready` returning `Poll::Ready(Ok(()))`) may contain minimal real
code. Handler methods, service call implementations, and gRPC procedure bodies
must remain `todo!()`. Each wiring method must be listed as `WIRING-EXEMPT` in the
stub report and excluded from the RED_RATIO denominator alongside GREEN-BY-DESIGN
entries.

- **Test:** BC-5.38.003 canonical test vectors: Tower `poll_ready` → WIRING-EXEMPT;
  Tower `call()` with business logic → must be `todo!()`.
- **Test:** `test_wiring_exempt_excluded_from_red_ratio_denominator()` in bats.

### AC-006: Red Gate Density Check section in per-story-delivery.md (traces to BC-8.29.001 postcondition 1, postcondition 2, postcondition 3, invariant 1, invariant 5; BC-8.29.002 postcondition 1, postcondition 4; BC-8.29.003 postcondition [both options])

`plugins/vsdd-factory/workflows/phases/per-story-delivery.md` gains a new
**"Red Gate Density Check (BLOCKING before Step 4)"** section inserted between
Step 3 and Step 4. The section must contain:

1. **RED_RATIO formula:** `RED_RATIO = RED_TESTS / (TOTAL_NEW_TESTS - EXEMPT_TESTS)` where EXEMPT_TESTS = GREEN-BY-DESIGN + WIRING-EXEMPT counts.
2. **≥ 0.5 threshold** (BLOCKING). Integer form for precision: `RED_TESTS * 2 >= (TOTAL_NEW_TESTS - EXEMPT_TESTS)`.
3. **GREEN-BY-DESIGN escape hatch** with red-gate-log requirement: when all effective tests are exempt, trigger the full-exception path (BC-8.29.001 invariant 4). This is NOT a vacuous pass — orchestrator must acknowledge in the Red Gate log.
4. **Remediation options A and B** (BC-8.29.003): Option A = roll back stub and re-dispatch with stricter prompt; Option B = accept with `mutation_testing_required: true` frontmatter registered and mutation testing executed at wave gate. Automated orchestrators default to Option A unless `mutation_testing_required: true` is pre-authorized in frontmatter (BC-8.29.003 EC-001).
5. **Log format:** Red Gate log written to `.factory/logs/red-gate-log-<story-id>.md` before any exception-path invocation (BC-8.29.002 invariant 1). Each unexpectedly-GREEN test requires an entry with: test name, result, rationale category (PURE-DATA / FRAMEWORK-WIRING / STRUCTURAL-ASSERTION / PRE-EXISTING-BEHAVIOR / OTHER-JUSTIFIED / UNJUSTIFIED), notes (BC-8.29.002 postcondition 2).

### AC-007: validate-red-ratio.sh PostToolUse hook enforces the blocking gate (traces to BC-8.29.001 invariant 5; BC-8.29.002 postcondition 1, postcondition 4; BC-8.29.003 invariant 1)

A new PostToolUse hook `plugins/vsdd-factory/hooks/validate-red-ratio.sh` enforces
the RED_RATIO blocking gate when the orchestrator is about to dispatch the
implementer. The hook:

1. Triggers on Write/Edit operations to `.factory/logs/red-gate-log-*.md` and
   on story files where `red_ratio:` frontmatter field is written.
2. Reads `red_ratio:` and `exempt_count:` and `total_new_tests:` fields from the
   modified file.
3. Applies integer check: `red_count * 2 >= (total_new_tests - exempt_count)`.
4. Emits `hook.block` on violation (RED_RATIO < 0.5 with no documented exception).
5. Exits 0 when RED_RATIO ≥ 0.5 or when the file records a valid full-exception
   path acknowledgment or an Option B election.

Reference implementation patterns:
- `plugins/vsdd-factory/hooks/validate-novelty-assessment.sh` (trigger pattern,
  jq-based field extraction, structured stderr output)
- `plugins/vsdd-factory/hooks/validate-count-propagation.sh` (exit-code semantics,
  set -euo pipefail structure)

The hook must be registered in `plugins/vsdd-factory/hooks-registry.toml`.

- **Test:** `test_validate_red_ratio_blocks_on_low_ratio()` in bats: write a
  red-gate-log file with `red_ratio: 0.3` and no exception path; hook exits non-zero.
- **Test:** `test_validate_red_ratio_passes_on_sufficient_ratio()` in bats:
  `red_ratio: 0.5` exits 0.
- **Test:** `test_validate_red_ratio_passes_on_option_b_election()` in bats:
  `red_ratio: 0.3`, `remediation: option_b` exits 0.
- **Test:** `test_validate_red_ratio_registered_in_hooks_registry()` in bats:
  grep for `validate-red-ratio` in `hooks-registry.toml` returns non-empty.

### AC-009: facade-mode delivery semantics in per-story-delivery.md (traces to BC-8.30.002 postcondition Step 2 through Wave Gate; invariant 1, invariant 2, invariant 3)

`plugins/vsdd-factory/workflows/phases/per-story-delivery.md` gains a section
describing `tdd_mode: facade` modifications to the standard flow:

- **Step 2 (facade):** Combined scaffold+impl commit allowed; no `todo!()` obligation;
  commit must include `[facade-mode: tdd_mode=facade]` tag.
- **Step 3 (facade):** Test-writer writes spec-anchored fidelity tests post-hoc;
  tests may be GREEN immediately; Red Gate density check is BYPASSED.
- **Step 4 (facade):** May be a no-op if Step 2 produced a complete implementation;
  implementer reviews fidelity tests and confirms correctness.
- **Wave gate (facade):** `cargo mutants -p <crate> --jobs N --timeout 300` is
  REQUIRED for ALL facade crates in the wave; kill rate ≥ 80% (BC-6.21.001,
  BC-6.21.002). This bypass of Red Gate is ONLY valid because mutation testing
  is the compensating control (BC-8.30.002 invariant 2).
- **Fidelity test anchoring:** Every test must trace to a BC (BC-8.30.002
  invariant 3). Unanchored shape-checking is not facade mode.

- **Test:** `test_facade_mode_section_present()` in bats: grep for
  `tdd_mode: facade` or `facade-mode` in per-story-delivery.md returns non-empty.

### AC-010: Mutation Testing section in wave-gate SKILL.md (traces to BC-6.21.001 postcondition 1, postcondition 2, postcondition 3, postcondition 4, invariant 1, invariant 2, invariant 3, invariant 4, invariant 5; BC-6.21.002 postcondition 1, postcondition 2 [Disposition A/B/C], invariant 1 [80% kill rate floor])

`plugins/vsdd-factory/skills/wave-gate/SKILL.md` gains a **Mutation Testing**
section that:

1. Scans all stories in the wave for `tdd_mode: facade` and stories with
   `mutation_testing_required: true` (Option B from BC-8.29.003 EC-001).
2. For each such story's crate, runs:
   `cargo mutants -p <crate> --jobs $(nproc) --timeout 300`
3. Writes mutation report to `.factory/logs/mutation-report-<wave-id>-<crate>.md`
   (committed as part of the wave PR).
4. Gates on ≥ 80% kill rate (integer: `killed * 100 / total >= 80`).
5. When kill rate < 80%, provides disposition table with columns:
   `Mutant | File | Line | Mutation | Disposition (A/B/C) | Notes`.
   - Disposition A: new test committed + re-run confirming kill.
   - Disposition B: dead-code-equivalent explanation with execution condition.
   - Disposition C: explicit waiver naming mutant, file, line, mutation type.
   No blanket waivers accepted.
6. If `cargo-mutants` is not installed, BLOCKS with:
   "cargo-mutants not found — install with `cargo install cargo-mutants`".
7. If wave has zero facade stories AND zero Option B stories: mutation testing
   step is skipped. An explicit log entry noting "no facade stories in wave —
   mutation testing step skipped" is required (not silently omitted).
