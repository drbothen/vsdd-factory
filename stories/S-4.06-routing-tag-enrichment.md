## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

CAP-003 states: "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`." Per-sink routing filters and tag enrichment are the mechanisms by which the "independently configured" portion of CAP-003 is fully realized — without routing filters, all sinks receive all events (no per-sink control), which violates the "independently configured" promise. S-4.06 completes the wiring that makes per-sink independent configuration observable.

## Migration Decision: Tag Enrichment Refactor

**Decision: Option (a) — Refactor (move enrich up to Router layer).**

`FileSink::enrich()` currently implements tag enrichment inline inside the sink driver. This story moves that responsibility up to `Router::submit()` so that:
1. All sink drivers (file, OTel gRPC, future HTTP sinks) get tag enrichment from a single source of truth.
2. Sink drivers stay simple I/O adapters with no enrichment logic.
3. VP-031 and BC-3.02.010/3.02.011 are migrated to the Router scope where the behavior now lives.

Rationale: keeping enrichment in `FileSink` creates N-sink duplication debt as more drivers are added in E-4 stories. The Router layer already owns the dispatch path; enrichment is a natural dispatch-layer concern.

## Narrative
- **As a** factory operator with multiple sinks
- **I want to** configure routing filters and static tags per sink
- **So that** audit events go to an audit file sink and metrics to a metrics sink, with additional context tags added automatically

## Goal

Implement the routing-filter and tag-enrichment features every sink
supports. Routing filters let operators route specific event types to
specific sinks. Tag enrichment adds static key/value tags to every event
passing through a sink.

### BC-3.04.003 — Router::submit silently drops events that fail RoutingFilter

**Title:** Router::submit silently drops events that fail RoutingFilter; no SinkFailure recorded; debug-level log emitted

**Preconditions:**
- Router has a configured sink with a non-empty RoutingFilter
- An event is submitted that does not pass the RoutingFilter

**Postconditions:**
1. The event is not forwarded to that sink's `submit()` call
2. No `SinkFailure` or error event is recorded
3. A `debug`-level log entry with message category `internal.event_filtered` is emitted for each (event, filtering-sink) pair. If N sinks filter the same event, N log entries are emitted (NOT deduplicated). Each entry includes the sink_name and event type.

**Invariants:**
- Silent drop applies per-sink; other sinks without filter restrictions still receive the event

**Edge Cases:**
- EC-001: Event matches no sink filters at all → silently dropped for all sinks; a `debug`-level `internal.event_filtered` log entry is observable per sink that filtered it

**Traces to:** CAP-003 per capabilities.md §CAP-003; supersedes EC-001 phantom default-sink mechanism

### BC-3.04.004 — Router::submit applies RoutingFilter before delegating to each sink

**Title:** Router::submit applies RoutingFilter before delegating to each sink (wired dispatch)

**Preconditions:**
- Router wraps a SinkRegistry with one or more configured sinks
- Each sink optionally has a RoutingFilter in its SinkConfigCommon

**Postconditions:**
1. For each sink: if the sink's RoutingFilter is non-empty and the event fails it, that sink's `submit()` is NOT called
2. For each sink: if the sink's RoutingFilter is empty or the event passes it, that sink's `submit()` IS called with the (optionally tag-enriched) event
3. Tag enrichment from sink `tags` config is applied to the event before `submit()` is called
4. Tag enrichment must NOT overwrite producer-set fields. Producer fields (e.g., `type`, `ts`, `plugin_id`) are immutable through the Router enrichment pass. If a config-tag key collides with a producer field key, the producer field wins.

**Invariants:**
- Router is the single dispatch gate; no call sites bypass RoutingFilter evaluation
- effectful-shell: Router orchestrates; pure filter matching is delegated to `crates/sink-core/src/router_filter.rs`

**Deprecates:** BC-3.04.001 (thin pass-through state; deprecated_by: BC-3.04.004)

**Traces to:** CAP-003 per capabilities.md §CAP-003

### BC-3.06.007 — sink-core::routing_filter_plugin_ids_allow

**Title:** sink-core::routing_filter_plugin_ids_allow — only events from listed plugins pass; empty list = pass-through

**Preconditions:**
- A RoutingFilter is configured with a non-empty `plugin_ids_allow` list
- An event is submitted with a `plugin_id` field

**Postconditions:**
1. If `plugin_ids_allow` is non-empty and the event's `plugin_id` is in the list, the event passes
2. If `plugin_ids_allow` is non-empty and the event's `plugin_id` is NOT in the list (or `plugin_id` is absent), the event is filtered out
3. If `plugin_ids_allow` is empty, all events pass regardless of `plugin_id`

**Invariants:**
- `plugin_ids_allow` filtering is independent of `event_types_allow`/`event_types_deny`; all configured filter fields must pass for the event to be forwarded

**Edge Cases:**
- EC-001: Event has no `plugin_id` field and `plugin_ids_allow` is non-empty → event filtered out (treated as no match)

**Note:** Plugin ID matching is case-sensitive (parallels BC-3.06.006 for `event_types`). No wildcard semantics are defined; the empty-list pass-through (postcondition 3) is the only escape hatch. If wildcard support is desired in v1.1+, register a new BC — do not smuggle via EC.

**Traces to:** CAP-003 per capabilities.md §CAP-003
