# S-4.06: Per-sink routing filters + tag enrichment

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.07, S-4.08
**Blocked by:** S-1.08 (Sink trait)
**Estimated effort:** S

> **Status note:** As of v1.0.0-beta.4, `RoutingFilter` is implemented in `sink-core` and
> the `tags` field exists on the sink config struct, but per-sink wiring is partial —
> filters are parsed but not fully applied in the dispatch path.

## Narrative
- **As a** factory operator with multiple sinks
- **I want to** configure routing filters and static tags per sink
- **So that** audit events go to an audit file sink and metrics to a metrics sink, with additional context tags added automatically

## Goal

Implement the routing-filter and tag-enrichment features every sink
supports. Routing filters let operators route specific event types to
specific sinks. Tag enrichment adds static key/value tags to every event
passing through a sink.

## Acceptance criteria

- [ ] `RoutingFilter` fully applied in sink dispatch path (not just parsed)
- [ ] Static tags merged into every event before sink `submit()`
- [ ] Routing filter supports: include_event_types, exclude_event_types, include_plugins
- [ ] Integration test: two sinks with different filters; verify events routed correctly
