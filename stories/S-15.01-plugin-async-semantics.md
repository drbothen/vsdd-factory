# S-15.01: Plugin async semantics — full implementation (schema v2 + dispatcher partition + plugin classification + envelope flip + CI lint)

**Epic:** E-15 — Plugin Async Semantics — Registry-Layer Partition (single-shot delivery)
**Delivery order:** FIRST and ONLY story in E-15 (ADR-019 §Decision 6 — no phased rollout).
**Depends on:** none
**Blocks:** none
**Priority:** P1 (closes silent-block-bleed bug observed in prism audit 2026-05-07)

## User Story

As a vsdd-factory operator, I want validators with `on_error = "block"` to actually block
tool calls so that silent-block-bleed (observed in prism audit, 2026-05-07, 55 silently
discarded block decisions) cannot recur.

## Description

This story delivers all four interlocking changes defined in ADR-019 atomically:

1. **Registry schema v2** — `hooks-registry.toml` gains per-plugin `async: bool` field;
   `REGISTRY_SCHEMA_VERSION` bumped to 2; `validate()` hard-errors on v1 registries.
2. **Dispatcher partition** — `partition_plugins()` pure function splits matched plugins
   into `sync_group` (await-all, verdict gates Claude Code) and `async_group`
   (fire-and-forget with `ASYNC_DRAIN_WINDOW_MS` drain per DI-019).
3. **Plugin classification** — 9 telemetry-only plugins classified `async = true` in
   `hooks-registry.toml` per BC-7.06.001 Invariant 6; all validator/governance plugins
   with `on_error = "block"` remain `async = false`.
4. **Envelope flip** — all 5 platform `hooks.json.<platform>` variants and
   `hooks.json.template` have `async: true` removed from every event entry per
   BC-9.01.006 and ADR-019 §Decision 1.
5. **CI lint** — three-layer defense enforcing `on_error = "block"` implies `async = false`:
   pre-commit hook, registry-load-time `validate()`, and VP-078 CI bats harnesses.
6. **Four new event types** — `plugin.async_block_discarded`, `dispatcher.schema_mismatch`,
   `dispatcher.registry_invalid`, `plugin.timeout` (async path) catalogued and emitted per
   BC-3.08.001.

User-locked decisions (verbatim per ADR-019): every event sync at envelope (no carve-outs);
no backwards compatibility; no phased rollout; single-shot delivery; `ASYNC_DRAIN_WINDOW_MS`
= 100ms per DI-019.

## Acceptance Criteria

### AC-001 — Registry schema v2 bump (traces to BC-7.06.001 postcondition 1 + BC-1.14.001 precondition 1)

`hooks-registry.toml` top-level header specifies `schema_version = 2`. `REGISTRY_SCHEMA_VERSION`
constant in `registry.rs` is set to `2`. A registry with `schema_version != 2` (including any
v1 registry) produces `E-REG-001` at load time, emits `dispatcher.schema_mismatch` to
`events-*.jsonl`, emits an explicit stderr diagnostic, and exits with code 2 (fail-closed).
This is the explicit named exception to BC-1.08.001 fail-open.

**Falsifiable test:** Rust unit test `registry_v1_rejected()`: load a v1 registry
(`schema_version = 1`) via `Registry::load_from_str()`; assert `Err(RegistryError::SchemaVersion
{ got: 1, expected: 2 })` and dispatcher exits non-zero. Bats integration test (VP-078 Harness 2)
also verifies exit non-zero and `registry_invalid`/`schema_mismatch` in stderr.

### AC-002 — Per-plugin async field with serde-default semantics (traces to BC-7.06.001 postcondition 2 + postcondition 3)

Each `[[hooks]]` entry in `hooks-registry.toml` MAY include `async = true` or `async = false`.
Absence of the field deserializes as `async = false` (via `#[serde(default)]` on `RegistryEntry`).
The field is type-checked: `async = "true"` (string) produces `RegistryError::ParseError`.

**Falsifiable tests** (VP-078 Harness 4 unit tests):
- `async_explicit_true_parsed_as_true()`: explicit `async = true` → `entry.async_flag == true`
- `async_explicit_false_parsed_as_false()`: explicit `async = false` → `entry.async_flag == false`
- `async_absent_defaults_to_false()`: absent field → `entry.async_flag == false`
- `async_string_value_is_parse_error()`: `async = "true"` → `Err`

### AC-003 — Dispatcher partition function (traces to BC-1.14.001 postconditions 1-6 + invariants 1-5)

`partition_plugins(matched_plugins, registry)` partitions matched plugins into `sync_group`
(where `async_flag == false`) and `async_group` (where `async_flag == true`). The function is
pure and deterministic. The partition is disjoint (`sync_group ∩ async_group = ∅`) and total
(`sync_group ∪ async_group = matched`). Async group plugins are spawned only AFTER sync_group
completes. The `dispatcher_exit` is determined exclusively by sync_group results; async_group
exit codes never affect it.

`block_intent := (plugin.exit_code == 2 && plugin.config.on_error == "block")`. Since
Invariant 4 of BC-1.14.001 prohibits any async plugin from having `on_error = "block"`,
`block_intent` is structurally false for any async plugin that exits 2.

**Falsifiable tests** (VP-077 Kani proof harnesses):
- Harness 1: totality + disjointness + union completeness (up to 4 plugins, `kani::any()`)
- Harness 2: async-field respect — `async=true` lands in async_group; `async=false` in sync_group
- Harness 3: exit-code independence — two executions differing only in async_group results produce identical `dispatcher_exit`
- Harness 4: aggregation correctness — `exit = 2` iff any sync plugin has `block_intent = true`; `exit = 0` otherwise

### AC-004 — Sync-group execution and verdict aggregation (traces to BC-1.14.001 postconditions 2-3)

Sync-group plugins run in parallel within each priority tier (per ADR-008: sequential between
tiers, parallel within a tier). The dispatcher awaits all sync-group completions before
computing the verdict. If any sync-group plugin returns `block_intent = true`, dispatcher exits
2. If no sync-group plugin blocks, dispatcher exits 0. Async-group verdict never influences
`dispatcher_exit` under any condition.

**Falsifiable tests** (Rust integration tests):
- `sync_group_all_continue_exits_0()`: two sync plugins return exit 0; assert dispatcher exit 0
- `sync_group_one_blocks_exits_2()`: one sync plugin returns exit 2 with on_error=block; assert dispatcher exit 2
- `async_group_block_does_not_gate_dispatcher()`: sync plugin exits 0; async plugin exits 2; assert dispatcher exits 0

### AC-005 — Async-task drain window (traces to BC-1.14.001 postcondition 4 + DI-019)

After sync_group completes, the dispatcher waits up to `ASYNC_DRAIN_WINDOW_MS` (canonical
value: DI-019, 100ms) for spawned async tasks to emit terminal events to FileSink. Tasks that
complete within the drain window emit cleanly. Tasks that do not complete within the drain
window are forcibly terminated; their pending I/O is discarded. Total dispatcher wall-clock
latency is bounded by `max(sync_plugin_durations_within_slowest_tier) + ASYNC_DRAIN_WINDOW_MS`.

**Falsifiable tests** (VP-079 bats scenarios 1, 4, 5):
- Scenario 1: async plugin exits 2 during drain window → `plugin.async_block_discarded` event emitted and visible in sink
- Scenario 4: async plugin times out within drain window → `plugin.timeout` event emitted
- Scenario 5: async plugin timeout_ms > ASYNC_DRAIN_WINDOW_MS → `plugin.timeout` NOT emitted (truncation)

### AC-006 — CI lint load-time: on_error=block implies async=false (traces to BC-7.06.001 postcondition 5 + invariant 1 + BC-1.14.001 invariant 4)

`registry.rs::validate()` checks that no entry has both `on_error = "block"` and `async = true`.
If violated, dispatcher emits `E-REG-002`, logs `dispatcher.registry_invalid` event with
`offending_plugin` named, emits explicit stderr diagnostic, and refuses to start.

**Falsifiable test** (VP-078 Harness 1 bats + lint_invariant_tests Rust):
- `registry_validate_rejects_block_plus_async()`: entry with `on_error=block` AND `async=true` → `Err(E-REG-002)`
- `registry_validate_accepts_block_without_async()`: block without async field → `Ok`
- `registry_validate_accepts_async_without_block()`: `async=true` with `on_error=continue` → `Ok`
- Bats test: dispatcher fed violating registry exits non-zero; stderr contains `registry_invalid` or `on_error`/`async`

### AC-007 — CI lint pre-commit: on_error=block implies async=false (traces to BC-7.06.001 postcondition 7 + invariant 5 layer 1)

A pre-commit hook (bash script) scans `hooks-registry.toml` for any `[[hooks]]` entry that
contains both `on_error = "block"` and `async = true` within the same entry block. Violations
fail the commit before any registry-load-time or CI checks run. This is Layer 1 of the
three-layer defense per ADR-019 §Decision 4.

**Falsifiable test:** bats test drives the pre-commit script against a fixture registry
containing a violating entry; asserts non-zero exit code and diagnostic output naming the
offending plugin.

### AC-008 — CI lint CI-PR: on_error=block implies async=false (traces to BC-7.06.001 postcondition 6 + invariant 5 layer 3)

VP-078 bats CI integration test (Harness 1) scans the live `plugins/vsdd-factory/hooks-registry.toml`
on every PR and asserts no entry has both `on_error = "block"` and `async = true`. Fails the
build if the invariant is violated. This is Layer 3 of the three-layer defense.

**Falsifiable test:** VP-078 Harness 1 bats test passes cleanly on the shipped v2 registry.
Introducing a violating entry causes Harness 1 to fail with the plugin name in output.

### AC-009 — Telemetry plugin positive classification (traces to BC-7.06.001 invariant 6)

The following 9 plugins MUST be classified `async = true` in the shipped `hooks-registry.toml`:
`capture-commit-activity`, `capture-pr-activity`, `session-start-telemetry`,
`session-end-telemetry`, `worktree-hooks`, `tool-failure-hooks`, `track-agent-start`,
`track-agent-stop`, `session-learning`. All validator/governance plugins with
`on_error = "block"` must remain `async = false` (or absent). The following plugins are
deliberately SYNC (`on_error = continue`) despite not blocking, because they emit
user-visible stderr warnings: `warn-pending-wave-gate`, `regression-gate`.

**Falsifiable test** (VP-078 Harness 3 Rust integration test):
`vp_078_harness_3_telemetry_positive_classification()` loads the live registry from
`plugins/vsdd-factory/hooks-registry.toml` and asserts all 9 named plugins are present
AND have `async_flag == true`. Panic on any absent or misclassified plugin.

### AC-010 — Envelope flip: all platform variants have no async:true (traces to BC-9.01.006 postconditions 1-3)

All 5 per-platform variant files (`hooks.json.darwin-arm64`, `hooks.json.darwin-x64`,
`hooks.json.linux-x64`, `hooks.json.linux-arm64`, `hooks.json.windows-x64`) and
`hooks.json.template` have no event entry with `"async": true`. The `async` key is absent
from every event entry. This applies to all 10 event types enumerated in ADR-019 §Decision 1.

**Falsifiable test:** `grep -r '"async": true' plugins/vsdd-factory/hooks/` returns no matches.
VP-079-adjacent CI lint check (or VP-078 dedicated SS-09 scan per BC-9.01.006 postcondition 5)
fails the build if any `"async": true` is found.

### AC-011 — New event type: plugin.async_block_discarded (traces to BC-3.08.001 postcondition 1 + VP-079 scenario 1)

When an async-group plugin exits 2, the dispatcher emits `plugin.async_block_discarded` to
`events-*.jsonl` with mandatory fields: `type`, `trace_id`, `plugin_name`, `exit_code: 2`,
`timestamp`, `reason: "async_plugin_block_verdict_discarded"`. The `trace_id` is non-null in
nominal operation; `null` is permitted as DI-017 last-resort fallback. The dispatcher exit
code is unaffected.

**Falsifiable test** (VP-079 bats Scenario 1): async plugin exits 2 within drain window;
`assert_fields_present` verifies all 6 mandatory fields; `reason` field equals the literal
`"async_plugin_block_verdict_discarded"`.

### AC-012 — New event type: dispatcher.schema_mismatch (traces to BC-3.08.001 postcondition 2 + VP-079 scenario 2)

When the dispatcher loads a registry with `schema_version != 2`, it emits
`dispatcher.schema_mismatch` to `events-*.jsonl` with mandatory fields: `type`, `trace_id`,
`found_version`, `expected_version: 2`, `timestamp`, `error_code: "E-REG-001"`. Emit-then-exit
is the behavior; the event must reach FileSink before the dispatcher exits.

**Falsifiable test** (VP-079 bats Scenario 2): v1 registry loads → `dispatcher.schema_mismatch`
event found in sink; `assert_fields_present` verifies all 6 mandatory fields; `expected_version == 2`;
`error_code == "E-REG-001"`.

### AC-013 — New event type: dispatcher.registry_invalid (traces to BC-3.08.001 postcondition 3 + VP-079 scenario 3)

When `registry.rs::validate()` finds an entry with `on_error = "block"` AND `async = true`,
it emits `dispatcher.registry_invalid` to `events-*.jsonl` with mandatory fields: `type`,
`trace_id`, `offending_plugin`, `violation: "on_error_block_with_async_true"`, `timestamp`,
`error_code: "E-REG-002"`. Dispatcher refuses to start.

**Falsifiable test** (VP-079 bats Scenario 3): violating registry loads → `dispatcher.registry_invalid`
event found in sink; `offending_plugin` equals the entry name; `error_code == "E-REG-002"`.

### AC-014 — New event type: plugin.timeout async path (traces to BC-3.08.001 postcondition 4 + VP-079 scenario 4)

When an async-group plugin exceeds its `timeout_ms` within the drain window, the dispatcher
terminates the plugin and emits `plugin.timeout` to `events-*.jsonl` with mandatory fields:
`type`, `trace_id`, `plugin_name`, `execution_group: "async"`, `timeout_ms`, `timestamp`.
The dispatcher exit code is unaffected.

**Falsifiable test** (VP-079 bats Scenario 4): async plugin with `timeout_ms = 50` running
a `sleep 60` script; `plugin.timeout` emitted within drain window; `assert_fields_present`
verifies all 6 mandatory fields; `execution_group == "async"`.

### AC-015 — Kani proof: partition function correctness (traces to BC-1.14.001 invariants 1-5 + VP-077 all 6 properties)

All 6 VP-077 Kani proof properties verified: totality, async-field respect, disjointness,
union completeness, exit-code independence from async group, and aggregation correctness.
The `kani::proof` harnesses in `routing.rs` (or a sibling proof module) compile and pass
without counterexamples. The precondition that (`name`, `event`) tuples are unique is
enforced via `kani::assume` per BC-7.06.001 Invariant 7.

**Falsifiable test:** `cargo kani -p factory-dispatcher` completes without counterexamples
for all 4 proof harnesses defined in VP-077.

### AC-016 — Latency canary: sync_group p95 under 500ms (traces to E-15 D-15.4 + ADR-019 §Consequences)

On a representative Edit/Write workload (~30+ matched plugins), the sync_group p95 latency
is measured and must be ≤ 500ms. Timing measurement evidence stored at
`docs/demo-evidence/S-15.01/latency-canary.md`. If p95 > 500ms, the misclassification audit
in T-3h must identify and flip additional heavy plugins to `async = true` before merge.

**Falsifiable test:** The latency canary measurement script is executed and its output
committed to `docs/demo-evidence/S-15.01/latency-canary.md`. The p95 value recorded in
that file must be ≤ 500ms. CI may validate the file exists and the value is present.

## Architecture Compliance Rules

Extracted from ADR-019, ADR-008, and ARCH-INDEX:

1. **Partition function must be pure** (ADR-019 §Decision 3 + VP-077): `partition_plugins()`
   takes only `&[PluginEntry]` and returns `(Vec<PluginEntry>, Vec<PluginEntry>)`. No I/O.
   No global state. No mutex. No async. Verified by Kani.

2. **Async group spawn is post-sync-group** (BC-1.14.001 postcondition 6): The dispatcher
   MUST NOT interleave async task spawning with sync group execution. Async tasks are spawned
   only after `run_tiers(sync_group)` completes. No exceptions.

3. **Sync group uses ADR-008 tier model** (BC-1.14.001 postcondition 2 + VP-001 amended
   scope): Sequential between priority tiers; parallel within each tier. The tier model
   applies exclusively to the sync group. Async group plugins have no tier ordering.

4. **Schema version is a hard gate** (BC-7.06.001 invariant 3 + DI-014): `REGISTRY_SCHEMA_VERSION`
   in `registry.rs` accepts ONLY `schema_version = 2`. No range check. No "≥ 2" logic. No
   version negotiation. Any deviation is a spec violation.

5. **Schema mismatch exits 2 (fail-closed)** (BC-1.08.001 invariant 2 + BC-1.14.001
   Error Paths): This is the ONE exception to the fail-open convention. The dispatcher exits
   2 (not 0) on schema_version mismatch. Exiting 0 would be observationally identical to a
   clean run — reproducing the silent-failure root cause.

6. **Three-layer lint defense is mandatory** (BC-7.06.001 invariant 5 + ADR-019 §Decision 4):
   Pre-commit (Layer 1), registry-load-time `validate()` (Layer 2), CI-PR VP-078 bats (Layer 3).
   Missing any single layer is a spec violation.

7. **Envelope `async` key is permanently retired** (BC-9.01.006 invariant 1 + ADR-019
   §Decision 1): Do not add `"async": false` to envelope entries. The canonical form is
   absence of the key. CI lint checks for `"async": true` presence; `false` is benign but
   not the canonical form.

8. **ASYNC_DRAIN_WINDOW_MS is DI-019, not inline** (BC-1.14.001 §Constant Reference):
   The constant value (100ms) lives in DI-019 and in `constants.rs`. Do NOT inline `100`
   anywhere except the constants module. All other code references `ASYNC_DRAIN_WINDOW_MS`.
