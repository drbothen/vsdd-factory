# S-14.04: Policy-Rubric Auto-Injection for Adversary Dispatch

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** FOURTH in E-14 (after S-14.02, which already updates the F5
  dispatch sequence — this story adds policy rubric to that sequence).
**Depends on:** S-14.02 (the dispatch automation story updates the F5 skill; this
  story extends that updated skill with policy rubric injection)
**Blocks:** (none)
**Priority:** P1

## Narrative

- **As a** vsdd-factory orchestrator dispatching any adversary review (F5, per-story,
  or wave-gate)
- **I want** the adversary's task prompt to automatically include a
  `## Project Policy Rubric` section populated from `.factory/policies.yaml`
  before the adversary subagent fires
- **So that** the adversary evaluates findings against all active governance policies
  without the orchestrator needing to manually remember to inject them, eliminating
  the silent failure mode where policy enforcement is skipped because the dispatch
  prompt didn't include it

## Background

F5 pass-1 adversarial review (F-MED-9 / OBS-4) found that `orchestrator.md:243`
already states: "Before every adversary dispatch, read `.factory/policies.yaml`
(if it exists) and inject all policies into the adversary's task prompt under a
`## Project Policy Rubric` heading." However, the actual F5 dispatch prompt did not
include a pre-injected rubric section — the adversary had to load the policies file
independently. This worked, but silently depends on the adversary's own initiative
rather than a guaranteed injection step.

The gap is that the orchestrator AGENT.md documents the intent but there is no
mechanical enforcement: no skill step, no lobster workflow step, and no validation
that the rubric is present before adversary dispatch fires. Any future dispatch that
doesn't re-read and follow orchestrator.md line 243 will silently skip policies.

**Source:** F5 pass-1 F-MED-9, fix-plan B6 / Section 5 PG-4. The adversary text:
"Policy-rubric injection is not automated. Each dispatch depends on the orchestrator
remembering. Per S-12.01's mandate to reconcile orchestrator MANDATORY STEPS, this
should be an enforced pre-dispatch step — but it isn't."

## Behavioral Contracts

No existing BCs cover policy-rubric auto-injection. New BC required.

- **BC-X.NN.001 (to be authored at story implementation start):** Every adversary
  dispatch step (F5, per-story, wave-gate adversary) MUST include a
  `## Project Policy Rubric` section populated from `.factory/policies.yaml` in the
  task prompt. If `policies.yaml` does not exist, the section is omitted (no error).
  If `policies.yaml` exists but is empty, the section is included with an "No active
  policies" note. The injection MUST occur before the adversary subagent receives the
  prompt — never post-hoc.

## Acceptance Criteria

### AC-001 (traces to F-MED-9 — orchestrator.md policy injection clause is mechanically enforced)

The orchestrator AGENT.md MANDATORY STEPS section (actual path: verify `orchestrator.md`
or `orchestrator/orchestrator.md` or equivalent) contains an explicit pre-dispatch step
that: (a) reads `.factory/policies.yaml`, (b) injects policies under
`## Project Policy Rubric` in the adversary task prompt, (c) proceeds with dispatch.
This step is not an advisory note — it is a MANDATORY STEP with the same enforcement
language as other mandatory steps.

**Falsifiable test:** `grep -n "policies.yaml\|Policy Rubric\|policy.*inject\|inject.*policy"
plugins/vsdd-factory/agents/orchestrator/orchestrator.md` returns ≥1 match in the
MANDATORY STEPS section (not merely a comment or advisory).

### AC-002 (traces to F-MED-9 — F5 phase skill injection step)

The F5 adversarial-refinement skill (updated by S-14.02) contains an explicit
injection step that occurs BEFORE the adversary subagent dispatch step. The injection
step: (a) reads `.factory/policies.yaml`, (b) formats policies under
`## Project Policy Rubric` heading, (c) prepends the rubric to the adversary task
prompt. The step references BC-X.NN.001 (once authored).

**Falsifiable test:** `grep -n "policies.yaml\|Policy Rubric"
plugins/vsdd-factory/skills/phase-f5-scoped-adversarial/SKILL.md` returns ≥1 match
in the step that precedes the adversary dispatch step.

### AC-003 (traces to F-MED-9 — per-story adversary dispatch injection)

The per-story adversary dispatch step in `per-story-delivery.md` Step 4.5 (added by
S-12.01) is updated to include the policy rubric injection. The injection occurs
before the adversary convergence loop fires for any story. The update cites BC-X.NN.001.

**Falsifiable test:** `grep -n "policies.yaml\|Policy Rubric"
plugins/vsdd-factory/workflows/phases/per-story-delivery.md` returns ≥1 match in
the Step 4.5 adversary dispatch block.

### AC-004 (traces to PG-4 intent — graceful no-op when policies.yaml absent)

All dispatch sites that perform policy rubric injection handle the case where
`.factory/policies.yaml` does not exist: the injection step is skipped silently
(no error, no blocked dispatch). This ensures the injection works in new factory
deployments that have not yet registered any policies.

**Falsifiable test:** A bats test fixture with no `policies.yaml` file runs the
F5 dispatch skill (or the injection sub-procedure) and asserts: (a) no error,
(b) the adversary prompt does not contain a malformed `## Project Policy Rubric`
section with empty/null content.

### AC-005 (traces to PG-4 intent — validation hook or static assertion)

A static test (bats or shell assertion) verifies that the adversary dispatch step
in at least the F5 skill and the per-story Step 4.5 are preceded by a policy rubric
injection step. This prevents regression where a future skill update removes the
injection step without detection.

**Falsifiable test:** `bats plugins/vsdd-factory/tests/adversary-dispatch-policy-injection.bats`
(or equivalent) exits 0. The test asserts that the injection step appears before the
dispatch step in the relevant files (grep line-number ordering check).

## Dev Notes

**Policy injection format:** The canonical format per orchestrator.md line 243 is
`## Project Policy Rubric` as the heading, followed by the YAML content of
`policies.yaml` (or a Markdown rendering of it). The adversary needs to understand
the policies, not just see raw YAML. The injection step should render policies as a
bulleted list of policy names and descriptions if possible. The exact format is
at the implementer's discretion; what matters is that the rubric is present and
readable by the adversary.

**Survey scope:** Step T-0 must identify all adversary dispatch sites. Known sites:
(1) F5 phase skill (phase-f5-scoped-adversarial/SKILL.md), (2) per-story-delivery.md
Step 4.5, (3) possibly phase-4-holdout-evaluation skill (check if it dispatches
adversary). If a wave-gate skill dispatches adversary, that is an additional site.
The bats test in T-4 must cover all identified sites.

**Dependencies note:** S-14.02 modifies the F5 skill and orchestrator.md. S-14.04
extends those same files. The implementer must read S-14.02's completed changes
before implementing T-2 and T-1 to avoid conflicts.
