# S-12.03: ContextResolver Trait + ResolverRegistry (in-memory)

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** Batch 2 (parallel with S-12.05, after S-12.06).
**Depends on:** S-12.06 (HOST_ABI docs codify the contract this story implements)
**Blocks:** S-12.04 (WASM loading needs the trait), S-12.07 (resolver impl uses the trait)
**Priority:** P0

## Narrative

- **As a** factory dispatcher
- **I want** a `ContextResolver` trait, `ResolverRegistry`, and an in-memory registry implementation integrated into the dispatch flow
- **So that** the dispatcher can invoke registered resolvers before each hook dispatch and merge their outputs into `plugin_config`, while existing hooks with no `needs_context` declaration see zero behavioral change (zero overhead path)

## Behavioral Contracts

| BC ID | Title | Subsystem | Role in this story |
|-------|-------|-----------|-------------------|
| BC-1.13.001 | Dispatcher loads resolver registry and injects context before each hook dispatch | SS-01 | Primary — defines pre-dispatch injection sequence, `needs_context` field, absent registry = zero resolvers, fail-loud for unknown resolver names |
| BC-4.12.005 | Context-injection merging is additive overlay; duplicate context_key is a registry-load error | SS-04 | Primary — defines merge algorithm, whole-value replacement, `value: None` = key absent, collision events |

## Acceptance Criteria

### AC-001 (traces to BC-1.13.001 invariant 3 — `needs_context` defaults to empty via `#[serde(default)]`)

`crates/factory-dispatcher/src/registry.rs` `RegistryEntry` struct gains a `needs_context: Vec<String>` field annotated with `#[serde(default)]`. Existing `hooks-registry.toml` entries that lack this field parse successfully without error. No `deny_unknown_fields` constraint prevents this field from being optional.

**Falsifiable test:** Rust unit test: deserialize a `RegistryEntry` from TOML/JSON without a `needs_context` field; assert `entry.needs_context == []`. Deserialize an entry with `needs_context = ["foo"]`; assert `entry.needs_context == ["foo"]`.

### AC-002 (traces to BC-1.13.001 postcondition 3 — zero overhead for `needs_context = []`)

When `executor.rs` processes a hooks-registry entry with `needs_context: []` (or absent `needs_context`), the resolver invocation path is skipped entirely. No resolver is called; `plugin_config` passed to `invoke_plugin` equals the static registry config unchanged.

**Falsifiable test:** Rust unit test: construct a `ResolverRegistry` with one registered resolver; dispatch with an entry that has `needs_context: []`; assert the mock resolver's `resolve()` was never called (call count == 0); assert `plugin_config` passed to `invoke_plugin` equals the static config.

### AC-003 (traces to BC-1.13.001 postcondition 4 — resolver invoked for declared resolver name)

When `executor.rs` processes a hooks-registry entry with `needs_context: ["foo"]` and a resolver named `"foo"` is registered in `ResolverRegistry`, the dispatcher calls `foo.resolve(input)` and merges the returned output into `plugin_config` under the key returned by `ResolverOutput.key`.

**Falsifiable test:** Rust unit test: register a mock `ContextResolver` that returns `ResolverOutput { key: "foo", value: Some(json!({"answer": 42})) }`; dispatch with `needs_context: ["foo"]`; assert `plugin_config["foo"]["answer"] == 42` in the payload passed to `invoke_plugin`.

### AC-004 (traces to BC-4.12.005 postcondition 2 — `value: None` leaves key absent)

When a registered resolver returns `ResolverOutput { key: "foo", value: None }`, the key `"foo"` is NOT written to `plugin_config`. The hook receives a `plugin_config` without the key (not `{"foo": null}`).

**Falsifiable test:** Rust unit test: register a mock resolver returning `value: None`; dispatch with `needs_context: ["foo"]`; assert `plugin_config.get("foo").is_none()` in the payload seen by `invoke_plugin`.

### AC-005 (traces to BC-1.13.001 postcondition 6 — unknown resolver name emits `resolver.not_found`)

When a hooks-registry entry has `needs_context: ["unknown"]` but no resolver named `"unknown"` is registered in `ResolverRegistry`, the dispatcher emits a `resolver.not_found` telemetry event (with hook name and missing resolver name) and dispatches the hook WITHOUT context injection for that key. The dispatch does NOT panic or return an error.

**Falsifiable test:** Rust unit test: create an empty `ResolverRegistry`; dispatch with `needs_context: ["unknown"]`; assert: (a) mock telemetry captures a `resolver.not_found` event with `resolver_name == "unknown"`; (b) `invoke_plugin` is called normally; (c) `plugin_config` does not contain `"unknown"`.

### AC-006 (traces to BC-4.12.005 postcondition 1 — additive overlay; static config preserved)

When a resolver returns `ResolverOutput { key: "extra", value: Some(json!({"data": 1})) }` and the static `plugin_config` from `hooks-registry.toml` contains `{"existing": "value"}`, the merged `plugin_config` contains both: `{"existing": "value", "extra": {"data": 1}}`. The static config is not clobbered.

**Falsifiable test:** Rust unit test: static config = `{"existing": "value"}`; resolver returns `key: "extra", value: Some({"data": 1})`; assert merged config == `{"existing": "value", "extra": {"data": 1}}`.

### AC-007 (traces to BC-4.12.005 postcondition 5 — resolver output wins on static key collision; `resolver.merge_collision` event emitted)

When a resolver returns a key that already exists in the static `plugin_config`, the resolver output wins (replaces the static value wholesale). A `resolver.merge_collision` telemetry event is emitted with the key name, static value, and resolver value.

**Falsifiable test:** Rust unit test: static config = `{"foo": "old"}`; resolver returns `key: "foo", value: Some("new")`; assert: (a) merged config `["foo"] == "new"`; (b) `resolver.merge_collision` event emitted with key `"foo"`.

### AC-008 (traces to BC-4.12.005 invariant 1 + VP-075 — merge is a pure function; proptest)

`resolver.rs` exposes a `pub fn merge_resolver_outputs(static_config: Value, resolver_outputs: &[ResolverOutput]) -> Value` pure function. The proptest for VP-075 (200 trials) verifies: calling `merge_resolver_outputs` twice with identical inputs produces identical results.

**Falsifiable test:** `cargo test -p factory-dispatcher resolver_determinism_proptest` passes all 200 trials within 5s timeout. The proptest strategy generates arbitrary `Value::Object` base configs and arbitrary resolver outputs.

### AC-009 (traces to BC-1.13.001 postcondition 7 — resolver invocation order = declaration order)

When `needs_context: ["a", "b"]`, resolver `"a"` is invoked first; its output is merged first. Then resolver `"b"` is invoked and merged. The declaration order is honored.

**Falsifiable test:** Rust unit test: register resolvers `"a"` (returns `key: "a", value: 1`) and `"b"` (returns `key: "b", value: 2`); dispatch with `needs_context: ["a", "b"]`; assert both keys present; assert mock invocation order: `a` before `b`.

### AC-010 (traces to BC-1.13.001 postcondition 5 + invariant 5 — context injection precedes `invoke_plugin`)

The resolver invocation and merge step complete before `invoke_plugin` is called. The hook's `plugin_config` contains the fully merged payload — no partial injection.

**Falsifiable test:** Rust unit test (order-of-operations): use a mock hook invocation that captures the `plugin_config` at the moment `invoke_plugin` is called; assert the resolver key is already present at that moment.

### AC-011 (traces to BC-1.13.001 invariant 2 — empty registry = zero resolvers, no error)

Constructing a `ResolverRegistry::empty()` (or equivalent) and dispatching any event produces no resolver invocation and no error. Absent `resolvers-registry.toml` at startup yields an empty registry via the same code path (to be validated more fully in S-12.04 when WASM loading is added).

**Falsifiable test:** Rust unit test: dispatch with `needs_context: ["any"]` against an empty `ResolverRegistry`; assert: `resolver.not_found` event emitted; `invoke_plugin` called normally; process does not panic.

### AC-012 (traces to BC-4.12.005 postcondition 6 — duplicate `context_key` at registry-load time is fail-loud)

`ResolverRegistry::register(name, resolver)` returns an error (or panics in the builder) if a resolver with the same name has already been registered. No duplicate keys can coexist in a live registry.

**Falsifiable test:** Rust unit test: register resolver `"foo"` twice; assert the second `register()` call returns `Err` (or panics with a clear message). Assert the registry after the failed registration still contains exactly one `"foo"`.

## Architecture Compliance Rules

Derived from BC-1.13.001 invariants, BC-4.12.005 invariants, and ADR-018:

1. **`merge_resolver_outputs` is a pure function:** No I/O, no side effects, no global state. Must be separately unit-testable (the proptest target). Any drift toward effectful merge is a BC-4.12.005 invariant 1 violation.
2. **`ContextResolver` trait is object-safe:** The trait must use `&self` (or `&mut self`) receiver, not `self`. This allows `Box<dyn ContextResolver>` in `ResolverRegistry`.
3. **No WASM loading in this story:** `resolver.rs` must NOT import `wasmtime`. This story is the in-memory layer only. WASM loading is S-12.04's scope. Forbidden: `use wasmtime::*` or any dependency on `wasmtime::Module`.
4. **`#[serde(default)]` on `needs_context`:** Non-negotiable per BC-1.13.001 invariant 3. Do NOT use `Option<Vec<String>>` — use `Vec<String>` with `#[serde(default)]` so absent field deserializes to `[]`, not `None`.
5. **Telemetry events are non-blocking:** Emitting `resolver.not_found` or `resolver.merge_collision` events must not block the dispatch. Use the existing telemetry sink pattern in `executor.rs`.

## Forbidden Dependencies

The following MUST NOT appear in `crates/factory-dispatcher/`'s dependency graph after this story:
- `wasmtime` imports in `resolver.rs` (WASM loading is S-12.04 only)
