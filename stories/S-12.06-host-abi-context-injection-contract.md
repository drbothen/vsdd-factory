## Acceptance Criteria

### AC-001 (traces to BC-4.12.002 postcondition 9 — ABI contract documented in HOST_ABI.md)

`crates/hook-sdk/HOST_ABI.md` contains a top-level section titled `## Context Injection Contract` (or `## Context-Injection Contract`). The section appears after the existing ABI content and is clearly delineated.

**Falsifiable test:** `grep -c "Context Injection Contract\|Context-Injection Contract" crates/hook-sdk/HOST_ABI.md` returns ≥1. `grep -n "RESOLVER_ABI_VERSION" crates/hook-sdk/HOST_ABI.md` returns ≥1 match.

### AC-002 (traces to BC-4.12.001 postconditions 1–5 — resolver lifecycle documented)

The §Context Injection Contract section documents the resolver lifecycle: loaded once at dispatcher startup from `resolvers-registry.toml` via the same mtime-based cache pattern as `plugin_loader.rs`; compiled into `Module` at startup; fresh `Store` per-dispatch invocation; mtime-change triggers cache eviction and recompile.

**Falsifiable test:** `grep -n "startup\|mtime\|Store\|Module" crates/hook-sdk/HOST_ABI.md` returns ≥4 matches within the new section.

### AC-003 (traces to BC-4.12.002 postconditions 1–7 — resolver ABI type shapes documented)

The section specifies the `ResolverInput` type shape (fields: `event_type: String`, `hook_event_name: String`, `agent_type: Option<String>`, `project_dir: String`, `plugin_config: Value`) and the `ResolverOutput` type shape (fields: `key: String`, `value: Option<Value>`). Both types are presented as JSON examples showing their exact structure. The section states `RESOLVER_ABI_VERSION = 1` as a constant versioned independently from `HOST_ABI_VERSION`.

**Falsifiable test:** `grep -n "ResolverInput\|ResolverOutput\|RESOLVER_ABI_VERSION" crates/hook-sdk/HOST_ABI.md` returns ≥5 matches. `grep "event_type\|hook_event_name\|project_dir\|plugin_config" crates/hook-sdk/HOST_ABI.md` returns ≥4 matches in the new section.

### AC-004 (traces to BC-4.12.003 invariants 1–4 — capability model documented)

The section documents the resolver capability model: `path_allow = [...]` declarations in `resolvers-registry.toml`; deny-by-default (empty `path_allow` = no reads allowed); `host::read_file` is the only I/O host function available to resolvers; `host::write_file`, `host::exec_subprocess`, `host::emit_event` are structurally absent from the resolver linker; `host::log` is always available without restriction; capability violations return `CapabilityDenied` (not a trap).

**Falsifiable test:** `grep -n "path_allow\|deny-by-default\|CapabilityDenied\|write_file" crates/hook-sdk/HOST_ABI.md` returns ≥4 matches in the new section. `grep -c "write_file.*not available\|absent.*resolver\|resolvers.*read-only" crates/hook-sdk/HOST_ABI.md` returns ≥1.

### AC-005 (traces to BC-4.12.004 postconditions 1–5 — error/crash isolation documented)

The section documents resolver error isolation: a resolver panic, trap, timeout, or ABI violation does NOT propagate to the dispatcher process; the dispatcher emits `resolver.error` with `resolver_name`, `error_kind`, `error_detail`, and `hook_event_name`; the failed resolver's key is NOT written to `plugin_config`; hook dispatch proceeds without the missing context; the hook is responsible for treating a missing key as an error if required.

**Falsifiable test:** `grep -n "resolver.error\|resolver_name\|error_kind" crates/hook-sdk/HOST_ABI.md` returns ≥3 matches. `grep "dispatch proceeds\|hook.*missing.*key\|absent.*plugin_config" crates/hook-sdk/HOST_ABI.md` returns ≥1 match.

### AC-006 (traces to BC-4.12.005 postconditions 1–8 — merging contract documented)

The section documents the context-injection merging contract: additive overlay (static `plugin_config` preserved; resolver outputs overlaid under their key); whole-value replacement (no deep merge); `value: None` → key absent; `resolver.merge_collision` event on static key collision; duplicate `context_key` in registry is a startup error. A code example shows the JSON before-and-after of a merge operation.

**Falsifiable test:** `grep -n "additive\|merge_collision\|whole.*replace\|value.*None\|key.*absent" crates/hook-sdk/HOST_ABI.md` returns ≥4 matches.

### AC-007 (traces to BC-1.13.001 postconditions 3–6 — `needs_context` mechanism documented)

The section explains the `needs_context` field in `hooks-registry.toml`: a hook declares `needs_context = ["resolver-name-1", ...]`; the dispatcher invokes only the declared resolvers before that hook's dispatch; entries without `needs_context` (or with empty array) skip resolver invocation entirely (zero overhead path); unknown resolver names emit `resolver.not_found` and dispatch proceeds without context.

**Falsifiable test:** `grep -n "needs_context\|resolver\.not_found\|zero.*overhead\|zero overhead" crates/hook-sdk/HOST_ABI.md` returns ≥4 matches in the new section.

### AC-008 (traces to BC-1.13.001 invariant 1 + BC-4.12.002 postcondition 9 — factory-agnostic vocabulary throughout)

The new `§Context Injection Contract` section uses only factory-agnostic vocabulary. No vsdd-factory-specific terms appear: no `wave`, no `wave_context`, no `WaveContextResolver`, no `wave-state.yaml`, no `STATE.md`, no `story`, no `cycle`. The section describes the platform contract that any factory using `factory-dispatcher` would need to implement a resolver.
