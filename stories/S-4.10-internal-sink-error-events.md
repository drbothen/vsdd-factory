# S-4.10: internal.sink_error event emission (cross-sink)

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.07
**Blocked by:** S-4.01 (establishes `SinkFailure` type and failure-recording contract)
**Estimated effort:** M

> **Scope note:** This story adds production-code emission of `internal.sink_error` events
> to all three currently shipped sink drivers (sink-http, sink-otel-grpc, sink-file). It does NOT
> add integration test verification of these events — that is S-4.07 scope. S-4.07 is updated
> to depend on S-4.10 so that its integration tests can verify the new emissions.
>
> This is a cross-cutting concern: the same `internal.sink_error` event schema and emission
> helper must be implemented once in sink-core and used by all three drivers. Distributing
> the implementation across individual sink stories is rejected (see scoping memo).

## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

CAP-003 states: "The dispatcher fans out every internal event to all enabled sink drivers." The `internal.sink_error` event IS an internal event — it is sink-health telemetry that operators need to monitor without polling the `SinkFailure` buffer directly. Emitting it on every sink failure is part of the "fans out every internal event" promise in CAP-003. CAP-010 (always-on dispatcher self-telemetry) was considered but rejected: `internal.sink_error` is per-sink-failure telemetry, not dispatcher-startup telemetry.

**SS-03 ownership justification:** SS-03 owns all sink driver implementation. The emission helper lives in `crates/sink-core/src/events.rs`, and the call sites are in each sink driver crate (`sink-http`, `sink-otel-grpc`, `sink-file`). All are canonical SS-03 modules.

**Dependency anchor:** S-4.10 depends on S-4.01 because S-4.01 established the `SinkFailure` type and the `Mutex<Vec<SinkFailure>>` failure-recording pattern. This is the canonical recording site where emission is added. S-4.01 must be merged (it is, at commit 2ebf031) before S-4.10 can begin.

**Blocks anchor:** S-4.10 blocks S-4.07 because S-4.07's integration tests will assert that `internal.sink_error` events appear when sinks fail. Without S-4.10's production emission code, those assertions cannot pass. S-4.07 was draft when this block was added; the addition is in scope per the scoping memo.

## Stretch-Anchor Disclosure

AC-007 routes `internal.sink_error` events to "the internal-log writer or a dedicated internal event channel" — a receiver that lives in SS-01 (`factory-dispatcher`'s `internal_log.rs`, per BC-1.06.001-010 anchored to FR-006). This story's `subsystems: ["SS-03"]` declaration is intentionally narrow for the following reasons:

1. **Primary subsystem is SS-03 (emission sites).** All NEW code this story adds lives in SS-03 modules: `sink-core/src/events.rs` (event schema), `sink-http/src/lib.rs`, `sink-otel-grpc/src/lib.rs`, and `sink-file/src/lib.rs` (emission call sites). SS-03 is the correct single anchor.
2. **SS-01 is consumed-not-implemented.** S-4.10 does NOT modify `internal_log.rs` or any other SS-01 module. It integrates with SS-01's existing `try_send` API as a stable, already-shipped interface. The SS-01 receiver existed before this story and is unchanged by it.
3. **Sanctioned pattern: consumed-not-implemented (Wave 9 S-2.07 precedent).** S-2.07 (regression-test-validation) set the precedent that a story integrating with an SS-01 API without modifying it uses a body-level disclosure paragraph rather than a dual-subsystem frontmatter anchor. This story follows the same pattern.
4. **AC-007 scope is verification-only.** AC-007 asserts that events do NOT route through `SinkRegistry` (an invariant). Confirming the internal-log channel is the target is a negative assertion, not an implementation task in SS-01.
5. **If SS-01 `internal_log.rs` must be extended to accept the new event type,** that extension should be a separate story anchored to SS-01. S-4.10 proceeds on the assumption that the existing internal channel already accepts arbitrary `InternalEvent` payloads; if that assumption is wrong, the implementer must flag it before starting and escalate to PO.

## Narrative

- **As a** factory operator monitoring a multi-sink observability deployment
- **I want** each sink driver to emit a structured `internal.sink_error` event whenever it records a failure
- **So that** I can observe sink health in real-time via the event stream without polling internal buffers

## Goal

Add fire-and-forget `internal.sink_error` event emission to the failure-recording path of all
three shipped sink drivers (sink-http, sink-otel-grpc, sink-file). Define the event schema
once in `sink-core` and wire the emission helper into each driver's failure recording site.
Emission is non-blocking and non-fatal — a full or closed event bus silently drops the event.

## Acceptance Criteria

- [ ] **AC-001:** When sink-http records a `SinkFailure`, it emits exactly one `internal.sink_error`
  event to the dispatcher's internal event channel with fields: `type="internal.sink_error"`,
  `sink_name` (operator-assigned), `sink_type="http"`, `error_message` (non-empty string),
  `attempt` (u32, 0-indexed retry attempt number).
  _(traces to BC-3.07.002 postcondition 1: mandatory event fields; test vector: sink-http 503 on attempt 0)_

- [ ] **AC-002:** When sink-otel-grpc records a `SinkFailure`, it emits exactly one
  `internal.sink_error` event with `sink_type="otel-grpc"` and the same mandatory fields
  as AC-001.
  _(traces to BC-3.07.002 postcondition 1: cross-sink coverage; test vector: otel-grpc send fails on attempt 0)_

- [ ] **AC-003:** When sink-file records a `SinkFailure`, it emits exactly one `internal.sink_error`
  event with `sink_type="file"` and the same mandatory fields as AC-001.
  _(traces to BC-3.07.002 postcondition 1: cross-sink coverage; test vector: file write fails on read-only dir)_

- [ ] **AC-004:** After emitting `internal.sink_error`, the `SinkFailure` is still recorded into
  `Mutex<Vec<SinkFailure>>` as before. BC-3.01.008 postcondition 1 is preserved — this story
  is additive only and does not remove or replace existing failure recording.
  _(traces to BC-3.07.002 postcondition 2: additive-only; BC-3.01.008 preserved)_

- [ ] **AC-005:** If the internal event channel is full or closed at the time of emission, the
  emission is silently dropped. The sink does NOT panic and does NOT fail to record the
  `SinkFailure`. The failure recording proceeds normally.
  _(traces to BC-3.07.002 postcondition 3: silent drop on channel full/closed; EC-001, EC-002)_

- [ ] **AC-006:** The emission occurs on the same thread/task that records the failure — no
  additional `tokio::spawn` or background task is introduced for emission. Emission is
  synchronous with failure recording (fire-and-forget via `try_send`, not `await`).
  _(traces to BC-3.07.002 postcondition 4: same-thread emission, no new concurrency)_

- [ ] **AC-007:** `internal.sink_error` events are NOT routed through the `SinkRegistry` fan-out.
  They are sent to the internal-log writer or a dedicated internal event channel only. No
  recursive loop can occur (a sink emitting a failure event cannot trigger another sink failure
  which emits another failure event).
  _(traces to BC-3.07.002 invariant 2: no routing through SinkRegistry; no recursive loop)_

- [ ] **AC-008:** A single send failure produces exactly one `internal.sink_error` event. For a
  sink configured with `max_retries=3` that fails on all three attempts, exactly 3
  `internal.sink_error` events are emitted (one per failed attempt), each with the correct
  `attempt` value (0, 1, 2).
  _(traces to BC-3.07.002 invariant 3: one event per failure, not one per batch; test vector:
  sink-http fails 3 times → 3 events with attempt=0,1,2)_

- [ ] **AC-009:** The `sink_name` field in every emitted event matches the operator-configured
  sink instance name from `observability-config.toml`. For unnamed sinks, `sink_name` defaults
  to `"<unnamed>"`.
  _(traces to BC-3.07.002 invariant 4: sink_name matches config; EC-007)_

- [ ] **AC-010:** An integration test verifies that all three sinks (sink-http, sink-otel-grpc,
  sink-file) emit `internal.sink_error` events with a consistent schema when their respective
  failure conditions are triggered. The test captures events from the internal channel and
  asserts `type`, `sink_type`, `sink_name`, `attempt`, and `error_message` are present and
  correctly typed.
  _(traces to BC-3.07.002 canonical test vectors: all three sink types covered with correct field values)_

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|---------------|
| sink-core/src/events.rs (new) | crates/sink-core/ | pure-core (event struct definition + helper construction) |
| sink-http/src/lib.rs (modified) | crates/sink-http/ | effectful-shell (try_send to internal channel) |
| sink-otel-grpc/src/lib.rs (modified) | crates/sink-otel-grpc/ | effectful-shell (try_send to internal channel) |
| sink-file/src/lib.rs (modified) | crates/sink-file/ | effectful-shell (try_send to internal channel) |

**Pure/effectful boundary:** The `SinkErrorEvent` struct and its construction helper are pure
(defined in `sink-core/src/events.rs`). The `try_send` call to the internal channel is the
only effectful operation. Keep struct construction in sink-core (pure); keep the `try_send`
call in each driver's failure-recording site (effectful-shell).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Internal event channel is full at time of emission | Emission silently dropped; `SinkFailure` still recorded; sink continues (AC-005) |
| EC-002 | Internal event channel is shut down (dispatcher shutting down) | Emission silently dropped; `SinkFailure` still recorded; sink continues (AC-005) |
| EC-003 | Failure occurs on final retry attempt (retry exhaustion) | One `internal.sink_error` emitted for the exhaustion attempt; total N events for N failed attempts (AC-008) |
| EC-004 | Sink is disabled (`enabled=false`) | No send attempted, no failure, no `internal.sink_error` event emitted |
| EC-005 | First attempt succeeds after previous failure | No `internal.sink_error` for the successful attempt; prior failure events already emitted |
| EC-006 | Failure message contains non-UTF-8 bytes (OS error) | `error_message` is sanitized via lossy UTF-8 conversion (replacement chars); event still emitted |
| EC-007 | `sink_name` not configured (unnamed sink) | `sink_name` defaults to `"<unnamed>"` (string literal); event still emitted (AC-009) |

## Forbidden Dependencies

`internal.sink_error` events MUST NOT be routed through `SinkRegistry::submit_all()`. If
any code path routes these events through the fan-out registry, the build MUST be rejected
(infinite recursion risk). Use only the internal-log writer channel or a dedicated internal
event channel for emission.
