# S-15.11: validate-burst-log WASM Hook

**Epic:** E-12 — Context Resolvers + Platform Stories (brownfield-backfill wave, S-15.03 PRIORITY-A M2)
**Delivery order:** M2 Wave-2 (second of four M2 WASM hook stories; S-15.09 → S-15.14 follow)
**Depends on:** none (zero blocking dependencies per architect adjudication D-473)
**Blocks:** none (S-15.09, S-15.14 reference this story's crate structural patterns but do not
  formally depend on it via `depends_on`)
**Priority:** P1 (closes D-421(e)+D-438(d)+D-439(a)+D-444(c)+D-446(a)+D-432(e)+D-448(d)(i)+D-443(e)(ii))

## Narrative

- **As a** state-manager or agent writing a burst-log entry in a fix burst
- **I want to** receive an immediate PostToolUse block when the burst-log entry I just wrote is
  structurally incomplete — missing required block types, malformed h2 heading, or a Dim-1
  cardinality mismatch between the headline count and the enumerated file list
- **So that** the class of burst-log structural violations discovered by the adversary in every
  F5 engine-discipline pass (missing blocks, stale h2 format, Dim-1 count drift) is caught
  mechanically at write time rather than discovered by the adversary N bursts later

## Why This Exists

The F5 engine-discipline cycle (passes 25–74+) generated a persistent class of adversary findings
around burst-log structural completeness. D-421(e) codified the canonical `## Burst: ... (YYYY-MM-DD)`
h2 heading form. D-444(c) enumerated the 9 required block types. D-446(a) established the own-burst
completeness gate. D-432(e) + D-448(d)(i) codified Dim-1 cardinality parity. D-443(e)(ii) required
the own-burst h2 to be present in real-time at Commit A.

Manual adherence never held — every cycle produced at least one burst-log finding. This hook makes
the entire structural-completeness failure class impossible at PostToolUse time: any write to
burst-log.md that leaves a structurally incomplete entry is blocked before the edit lands.

## Goal

Deliver a production-grade WASM hook (`validate-burst-log`) that PostToolUse-blocks any Edit/Write
to a `burst-log.md` file containing:
- A latest h2 heading that does not match the canonical `## Burst: ... (YYYY-MM-DD)` pattern
- A burst entry missing any of the 9 required block types (D-444(c))
- A Dim-1 headline integer that does not match the count of files in the enumerated list (D-432(e))

## Behavioral Contract

### BC-5.39.004: validate-burst-log hook MUST block on structurally incomplete burst-log entries

**Subsystem:** SS-05 (Engine Governance — per-story adversary workflow and discipline gate hooks)
**Capability:** E-12 Engine Governance (automation sub-capability: burst-log structural completeness)
**BC ID:** BC-5.39.004
**File:** `.factory/specs/behavioral-contracts/ss-05/BC-5.39.004.md`

#### Preconditions

1. A PostToolUse Edit/Write event has fired on a file whose path contains `burst-log.md`.
2. The dispatcher has invoked the `validate-burst-log` WASM plugin with the write payload.
3. The burst-log file content is readable from the write payload or via `host::read_file`.
4. `host::read_file` returns content within the 65536-byte cap and 5000 ms timeout.

#### Postconditions

1. If ALL of the following hold, the hook emits `HookResult::Continue` (pass):
   - The latest h2 heading in the file matches the pattern `^## Burst: .+\(\d{4}-\d{2}-\d{2}\)`.
   - All 9 required block types are present in the latest burst entry: Parent-commit, Adversary
     verdict, Files touched (Dim-1), Codifications, Dim-2 Attestation, Dim-5 Attestation,
     Dim-6 Attestation, Dim-7 Attestation, Closes.
   - The Dim-1 headline integer equals the count of files in the Dim-1 enumerated list.
2. If the latest h2 heading does not match the canonical format, the hook emits
   `HookResult::BlockWithFix` with a message naming the malformed heading and the required format.
3. If any of the 9 required block types is absent from the latest burst entry, the hook emits
   `HookResult::BlockWithFix` with a message naming each missing block type.
4. If the Dim-1 headline integer does not equal the Dim-1 list item count, the hook emits
   `HookResult::BlockWithFix` naming the headline count, the actual list count, and the instruction
   to reconcile them.
5. Multiple violations in one write produce a single `HookResult::BlockWithFix` message enumerating
   all violations.
6. If `host::read_file` returns an error (timeout, capability denied, file not found), the hook
   emits `HookResult::Continue` and logs a warning via `host::log_warn` — a read failure is NOT
   treated as a structural violation (fail-open).

#### Invariants

1. The hook NEVER writes to any file. It is read-only at the host level.
2. The hook fires on PostToolUse only — it never prevents a write; it signals AFTER the write
   completes. The dispatcher records the block signal; the author must correct and re-write.
3. Block type detection uses bold-heading pattern matching:
   `\*\*(Parent-commit|Adversary verdict|Files touched \(Dim-1\)|Codifications|Dim-2|Dim-5|Dim-6|Dim-7|Closes)\*\*`.
   No other block name patterns match.
4. The hook validates ONLY the latest burst entry (from the latest h2 heading to the next h2
   or end-of-file). Prior entries are not re-validated.
5. All `host::read_file` calls are fail-open: read errors produce Continue + log_warn, not Block.

#### Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Burst-log file is newly created with only an h2 heading and no blocks | BlockWithFix naming all 9 missing block types |
| EC-002 | h2 heading format is `## Fix Burst: ...` (wrong prefix, no parenthesized date) | BlockWithFix naming the malformed heading |
| EC-003 | h2 heading format is `## Burst: description (2026-05-16)` — correct | Continue if blocks and Dim-1 also valid |
| EC-004 | 8 of 9 blocks present (Closes missing) | BlockWithFix naming "Closes" as missing block |
| EC-005 | Dim-1 headline says "7 unique files" but list has 9 files | BlockWithFix: "Dim-1 headline states 7 but enumerated list has 9 files — reconcile" |
| EC-006 | Dim-1 headline says "7 unique files" and list has exactly 7 files | Continue (Dim-1 cardinality correct) |
| EC-007 | File content truncated by 65536-byte cap (very long burst-log) | Hook reads what it can; may emit false Continue if missing blocks are past cap. Hook emits log_warn if truncation detected. |
| EC-008 | Multiple burst entries; only the latest (most recent h2) is checked | Hook validates only from latest h2 to next h2/EOF; prior entries ignored |
| EC-009 | `host::read_file` returns HostError::Timeout | Hook emits Continue + log_warn; does not block |
| EC-010 | All 9 blocks present and Dim-1 matches but h2 heading has no parenthesized date | BlockWithFix naming the malformed heading |

## Implementation Detail

### Hook Logic Specification

The implementer MUST implement `src/lib.rs` with the following logical structure. No line-number
citations (TD-VSDD-091 anti-volatile-pin); function names and behavioral anchors only.

#### In-plugin file-path guard

```
fn run_hook(payload: &Payload) -> HookResult {
    if !is_burst_log_target(&payload.file_path) {
        return HookResult::Continue;
    }
    // ... proceed with validation
}

pub fn is_burst_log_target(file_path: &str) -> bool {
    std::path::Path::new(file_path)
        .file_name()
        .and_then(|n| n.to_str())
        == Some("burst-log.md")
}
```

This path-component-strict guard is the canonical Q5/Q6 pattern. Using `Path::file_name()` prevents false-positive fires on paths like `/some/dir/xburst-log.md` where `.ends_with("burst-log.md")` would incorrectly match. The registry entry does NOT contain `file_pattern`.

#### Latest burst extraction

```
fn extract_latest_burst(content: &str) -> Option<&str>
```

Find the last occurrence of a line starting with `## Burst:`. Return the substring from that
line to the next `## ` heading (exclusive) or end-of-file. If no `## Burst:` line exists,
return None — which triggers a BlockWithFix via the h2 validation path.

#### h2 heading validation

```
fn validate_h2_heading(h2_line: &str) -> bool
```

Returns true if `h2_line` matches the pattern `^## Burst: .+\(\d{4}-\d{2}-\d{2}\)$` (end of line
may have trailing whitespace; the parenthesized date must be at the end). Pattern components:
- Prefix: `## Burst: `
- Description: one or more non-empty characters
- Suffix: ` (YYYY-MM-DD)` where YYYY, MM, DD are 4, 2, 2 digit sequences respectively

#### Block presence check (D-444(c))

```
fn check_block_presence(burst_content: &str) -> Vec<&'static str>
```

Scan `burst_content` for each of the 9 required bold-header token strings. The regex anchor is
`\*\*TOKEN\*\*` where TOKEN is one of:
- `Parent-commit`
- `Adversary verdict`
- `Files touched \(Dim-1\)`
- `Codifications`
- `Dim-2`
- `Dim-5`
- `Dim-6`
- `Dim-7`
- `Closes`

Return a Vec of the missing token names. Empty vec means all 9 are present.

Note: The Dim-2/5/6/7 tokens are deliberately short-form — a burst entry with `**Dim-2 Attestation**`
will also match `**Dim-2**` since the search finds the prefix. Implementer should use prefix-match
(contains `**Dim-2`) rather than exact-match to avoid false negatives from attestation-suffix variants.

#### Dim-1 cardinality check (D-432(e) + D-448(d)(i))

```
fn check_dim1_cardinality(burst_content: &str) -> Option<(usize, usize)>
```

1. Find the Dim-1 block start: line matching `\*\*Files touched \(Dim-1\)\*\*` or containing
   `Files touched (Dim-1)`.
2. Extract the headline integer: look for `(\d+) unique files` in the Dim-1 header line or the
   immediately following line.
3. Count list items in the Dim-1 block body: lines starting with `- `, `* `, or `\d+. ` between
   the Dim-1 header and the next `**` bold-header or end-of-burst-content.
4. If headline integer != list count: return Some((headline_count, list_count)).
   If equal or if Dim-1 block is absent (already flagged by block presence check): return None.

Note: The `cited_raw: String` field in the Violation struct should capture the raw Dim-1 headline
text as found in the file. This enables the block message to quote the exact string the author wrote.

#### Block message format

When `violations.len() > 0`:

```
HookResult::BlockWithFix {
    message: format!(
        "validate-burst-log: {} violation(s) in latest burst entry:\n{}",
        violations.len(),
        violations.iter().map(|v| format!("  - {}", v.description)).collect::<Vec<_>>().join("\n")
    ),
    fix_hint: "Fix the violations listed above before re-writing burst-log.md.",
}
```

Individual violation descriptions:
- h2 malformed: `"h2 heading does not match canonical format '## Burst: <desc> (YYYY-MM-DD)'; found: '{cited_raw}'"`
- Missing block: `"Required block '**{block_name}**' not found in latest burst entry (D-444(c))"`
- Dim-1 mismatch: `"Dim-1 cardinality mismatch: headline states {headline_count} unique files but enumerated list has {list_count} items (D-432(e))"`

### Performance Considerations

At most one `host::read_file` call is made per PostToolUse event: the burst-log.md content itself.
The dispatcher's write payload already includes the written content in many implementations — the
implementer should check whether the payload includes content before calling `host::read_file`.

At 65536 bytes cap, the read covers burst-log.md content up to approximately 1000 lines. The
typical burst-log entry is 50-100 lines; a burst-log file of 600+ entries would only have its
most recent entry within the cap if the file is large. In practice, burst-log.md is periodically
compacted, keeping size manageable.

Fuel budget: single-file read + regex scans. Well within the 5000 ms hook timeout.
