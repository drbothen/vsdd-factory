## Capability Anchor Justification

Primary: CAP-002 ("Hook Claude Code tool calls with sandboxed WASM plugins") per capabilities.md §CAP-002. This story wires the `SessionEnd` event through the dispatcher and implements the session-end plugin that emits `session.ended` with session telemetry (duration_ms, tool_call_count, timestamp).

FR anchor: FR-046 ("New Claude Code lifecycle hook events: SessionStart/SessionEnd/WorktreeCreate/WorktreeRemove/PostToolUseFailure") per PRD §2.4. S-5.02 is the second story delivering BC-4.05.001–005, the SessionEnd family of the canonical Tier F BC pattern.

S-5.01 lesson applied: S-5.01 required 14 adversarial passes and a major architectural reversal at pass-4 (BC-1.10.001 and BC-1.10.002 retired; dispatcher-side dedup and bespoke host fn replaced by Layer 1 once-discipline and canonical read_file host fn). S-5.02 applies those lessons from the start: no SS-01 BCs, no new host fns, no dispatcher-side dedup. SessionEnd is simpler than SessionStart — no subprocess, no file reads, envelope-only data — resulting in 3 plugin-set fields (vs. 6 for SessionStart) and zero capability declarations in hooks-registry.toml.

Auto-enrichment note: `session_id`, `dispatcher_trace_id`, `plugin_name`, `plugin_version` are host-enriched by `emit_event` from HostContext (per BC-1.05.012). Plugin must NOT set RESERVED_FIELDS (8 total: 4 host-enriched + 4 construction-time). See BC-4.05.001 Postcondition 2 Notes.

## Narrative
- **As a** factory operator
- **I want to** receive a `session.ended` event when each Claude Code session finishes
- **So that** I can compute session duration, tool-call count, and session-level cost tracking

## Goal

Wire `SessionEnd` through the dispatcher. Implement `session-end-telemetry.wasm` — a lightweight, stateless plugin that emits `session.ended` with `duration_ms`, `tool_call_count`, and `timestamp` sourced from the envelope. No subprocess invocation. No file reads. Pairs with `SessionStart` (S-5.01) to give operators complete session-level telemetry. First-of-five anchor for BC-4.05.* family (mirrors S-5.01 as first-of-five anchor for BC-4.04.*).

## Acceptance Criteria

- [ ] **AC1 — Full routing path wired:** Dispatcher receives `SessionEnd` via `hooks.json.template` → dispatcher binary → `hooks-registry.toml` → `session-end-telemetry.wasm`. Layer 1 entry uses `command` field with dispatcher binary path; Layer 2 entry uses `plugin = "hook-plugins/session-end-telemetry.wasm"`. (BC-4.05.004 + BC-4.05.005)

- [ ] **AC2 — `session.ended` event emitted with correct 11-field wire payload:** Plugin emits `session.ended` with exactly 3 plugin-set fields: `duration_ms` (non-negative decimal string; "0" if envelope's `session_start_ts` absent), `tool_call_count` (non-negative decimal string; "0" if envelope's `tool_call_count` absent), `timestamp` (ISO-8601 UTC ms precision Z suffix). Plugin must NOT set RESERVED_FIELDS: `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version` (host-enriched by `emit_event` from HostContext per BC-1.05.012), `ts`, `ts_epoch`, `schema_version`, `type` (construction-time from `InternalEvent::now()`). Total wire fields: 11 (3+4+4). (BC-4.05.001)

- [ ] **AC3 — No subprocess invocation:** Plugin completes without calling `exec_subprocess` at any point. Integration test CountingMock asserts `exec_subprocess` invocation_count == 0 for every `SessionEnd` dispatch. Plugin completes within 5000ms dispatcher timeout. (BC-4.05.002)

- [ ] **AC4 — hooks.json.template `SessionEnd` entry correct:** Entry uses `command` field routing to `factory-dispatcher` binary (NOT `session-end-telemetry.wasm`); `once: true`; `async: true`; `timeout: 10000`. No `.wasm` filename in `command` field (ADR-011 layer separation). (BC-4.05.004)

- [ ] **AC5 — hooks-registry.toml `SessionEnd` entry correct:** `name = "session-end-telemetry"`, `event = "SessionEnd"`, `plugin = "hook-plugins/session-end-telemetry.wasm"` (with `hook-plugins/` prefix), `timeout_ms = 5000`. NO `[hooks.capabilities.read_file]` table. NO `[hooks.capabilities.exec_subprocess]` table. No `once` field (RegistryEntry schema has none; deny_unknown_fields would reject it). (BC-4.05.005)

- [ ] **AC6 — Integration test covers all 5 BCs via VP-066:** Test in `crates/hook-plugins/session-end-telemetry/tests/integration_test.rs` exercises full wiring path. VP-066 proof harness skeleton covers: `test_bc_4_05_001_session_ended_emitted_with_required_fields`, `test_bc_4_05_001_missing_envelope_fields_emit_zero_defaults`, `test_bc_4_05_002_no_subprocess_invoked`, `test_bc_4_05_003_single_dispatch_produces_single_event`, `test_bc_4_05_004_hooks_json_template_has_session_end`, `test_bc_4_05_005_hooks_registry_toml_has_session_end`. (VP-066)

## Architecture Compliance Notes

**Timeout hierarchy (no subprocess):**
- Dispatcher timeout: `timeout_ms = 5000` (BC-4.05.005)
- Harness timeout: `timeout: 10000` (BC-4.05.004)
- No subprocess timeout — SessionEnd has no subprocess (BC-4.05.002)
- Hierarchy: 5000 < 10000 (two-level; simpler than SessionStart's three-level)

**Capability profile:**
- Zero declared capabilities (simplest possible sandbox)
- Deny-by-default: read_file and exec_subprocess both denied
- No capability tables in hooks-registry.toml entry

**RESERVED_FIELDS (8 — plugin must NOT set these):**
- Host-enriched (4): `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`
- Construction-time (4): `ts`, `ts_epoch`, `schema_version`, `type`

## Capability Anchor Justification Footnote

Anchoring to CAP-002: "Hook Claude Code tool calls with sandboxed WASM plugins" because this story implements a WASM plugin (`session-end-telemetry.wasm`) that hooks the `SessionEnd` Claude Code lifecycle event and emits structured telemetry — which is exactly what CAP-002 defines per capabilities.md §CAP-002.

No SS-01 co-anchor is needed: S-5.01's pass-4 architectural simplification established that the canonical `read_file` host fn and Layer 1 once-discipline cover all routing and capability needs without new SS-01 BCs. SessionEnd is even simpler — no file reads, no subprocess — so the SS-04-only scope is clean.

ADR-011 (dual-hook-routing-tables) provides the architectural foundation for BC-4.05.004 (Layer 1) and BC-4.05.005 (Layer 2). BC-1.05.012 (emit_event enrichment) provides the foundation for the RESERVED_FIELDS treatment in BC-4.05.001.

**F-07 subsystems anchor note (per S-5.01 pass-7 architectural ruling, Option 1 — semantic scope):** VP and story `subsystems: ["SS-04"]` reflects the verified-behavior subsystem (SS-04 plugin) rather than file-ownership subsystems. BC-4.05.004/005 architecture anchors reference SS-07 (hooks-registry.toml file ownership) and SS-09 (activation skill artifacts) as files-touched, but those are not S-5.02's scope claim. F-07 ruling applies here mirroring S-5.01: semantic scope wins over file-ownership scope when they diverge.
