# S-4.05: Dead letter queue implementation

## Capability Anchor Justification

Anchoring to **CAP-024: "Per-sink retry, circuit breaker, and dead-letter queue"** per capabilities.md §CAP-024.

CAP-024 states: "S-4.4/4.5: each sink driver retries failed sends with exponential backoff, trips a circuit breaker on sustained failure, and routes dropped events to a `dead-letter-<sink>-<date>.jsonl`." S-4.05 is explicitly cited as S-4.5 in CAP-024 and delivers the dead-letter queue portion. The outcome CAP-024 promises — "a Datadog outage doesn't lose events — they land in the DLQ" — is directly what this story implements.

**Filename pattern authority:** CAP-024 is the single source of truth for the DLQ filename pattern, extending CAP-024's outcome description (`dead-letter-<sink>-<date>.jsonl`) into the canonical implementation form using BC-3.02.001's `{date}` template placeholder syntax. The semantic mapping is: `<sink>` → `<sink-name>` (operator-assigned sink name); `<date>` → `{date}` (template placeholder substituting to YYYY-MM-DD per BC-3.02.001). All ACs and implementation code use the BC-3.02.001-aligned form: `dead-letter-<sink-name>-{date}.jsonl`.

## Narrative
- **As a** factory operator
- **I want to** have events that exhaust their retries written to a dead letter file
- **So that** I can inspect and replay dropped events rather than losing them silently

## Goal

When events are dropped — because the sink's retry attempts exhausted,
or the outbound queue overflowed — write them to a per-sink dead letter
file. This prevents silent data loss.

### BC-3.07.004 — sink-dlq::dlq_write_failure_observable_via_internal_event

**Subsystem:** SS-03 (Observability Sinks)
**Capability:** CAP-024
**Summary:** When a DLQ file write itself fails (disk full, permission error, etc.), the
failure is emitted as `internal.sink_dlq_failure` to the internal channel, logged to stderr
as fallback, and the dispatcher continues without crashing.

**Postconditions:**
1. `internal.sink_dlq_failure` event emitted to internal channel (NOT SinkRegistry)
2. Error logged to stderr as fallback (for hook-invocation environments where channel is unavailable)
3. Dispatcher continues processing; no panic
4. The `type` literal `internal.sink_dlq_failure` is reserved in the SS-01 internal-event-type registry (`crates/factory-dispatcher/src/internal_log.rs` event-type constants). The constant `INTERNAL_SINK_DLQ_FAILURE` MUST be added in the same commit that ships this BC's emitter.

**Why this is a BC and not just EC-002:** EC-002 previously said "log to stderr; do not crash".
That is lossy under hooks-driven invocation (AGENT-SOUL.md #4: no silent failures). Promoting
to a BC ensures the failure is observable in fan-out, not just in terminal output.

### BC-1.06.011 — prune_old::extends_to_dlq_files

| EC-003 | Path traversal in `sink_name` (e.g., `sink_name="../etc/passwd"` or `sink_name="a/b"`) | Sink fails to start with a `SinkConfigError::InvalidSinkName` error; sanitization rejects any `sink_name` containing path separator characters (`/`, `\`) or relative-path tokens (`..`); DLQ file is never created |
| EC-004 | Two sinks configured with identical `sink_name` | Second sink fails to start with `SinkConfigError::NameCollision`; each sink must have a globally unique name; DLQ files are not shared between sinks |
| EC-005 | Concurrent DLQ writes from worker thread | Single-writer-per-DlqWriter via `Mutex<File>` field on the DlqWriter struct (one Mutex per writer; serializes writes during size-cap rotation transitions); workers acquiring the lock serialize writes; no interleaved JSONL lines |
| EC-006 | Recursive DLQ feedback — `internal.sink_dlq_write` event itself fails and would be routed through a failing sink | `internal.sink_dlq_write` events are routed to the internal channel ONLY, never through SinkRegistry (mirroring BC-3.07.002 invariant); no recursive DLQ write is possible |
| EC-007 | Process crash mid-JSONL-line (partial write) | On next start, partial lines at end of DLQ file are tolerated by replay tools (documented contract); BC-3.02.015 shutdown drain reduces (but does not eliminate) the partial-write window; no data corruption of prior complete lines |

## Configuration

### DLQ Configuration in `observability-config.toml`

```toml
[[sinks]]
name = "my-http-sink"
type = "http"
# DLQ is on-by-default. Set to false to disable for this sink.
dlq_enabled = true  # default; defined as INTERNAL_DLQ_DEFAULT_ENABLED in internal_log.rs (or hard-coded constant for v1.0)

[[sinks]]
name = "my-otel-sink"
type = "otel-grpc"
# DLQ explicitly disabled for this sink
dlq_enabled = false

# DLQ retention is independent of dispatcher-log retention_days (default DEFAULT_RETENTION_DAYS = 30; defined in crates/factory-dispatcher/src/internal_log.rs).
[prune]
dlq_retention_days = 7  # default; defined as INTERNAL_DLQ_DEFAULT_RETENTION_DAYS in internal_log.rs
```

When `dlq_enabled = false` for a sink, retry-exhaustion events for that sink are logged
to stderr and dropped — they are NOT written to any DLQ file. The `internal.sink_dlq_write`
event is also suppressed for that sink.

**Note on `[prune]` section parser:**
The `[prune]` block in `observability-config.toml` is parsed by extending `ObservabilityConfig`
in `crates/factory-dispatcher/src/sinks/mod.rs` with a new `prune: Option<PruneConfig>` field,
where `PruneConfig` is a new struct with field `dlq_retention_days: Option<u32>`. If `[prune]`
is absent or `dlq_retention_days` is unset, the default `INTERNAL_DLQ_DEFAULT_RETENTION_DAYS = 7`
is used (per BC-1.06.011 PC1 + Task 5b). Validation: `dlq_retention_days >= 1` (zero or negative
values are rejected with `SinkConfigError::InvalidPruneConfig`).

> **Size cap:** The 100 MB per-file size cap is a hard-coded constant in v1.0 (not exposed as a TOML knob). Configurability is part of the v1.1 BC candidate `BC-3.NN.NNN-dlq-config-toggle-and-size-cap`.

**Default size cap:** 100 MB per DLQ file. When a file reaches 100 MB before midnight UTC,
a new rotation is triggered immediately (file closed, new `dead-letter-<sink-name>-{date}-<seq>.jsonl`
opened where `<seq>` is a 3-digit incrementing counter starting at 001).
