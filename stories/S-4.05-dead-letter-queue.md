# S-4.05: Dead letter queue implementation

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.07, S-4.08
**Blocked by:** S-4.04 (retries need to exhaust before DLQ)
**Estimated effort:** S

## Capability Anchor Justification

Anchoring to **CAP-024: "Per-sink retry, circuit breaker, and dead-letter queue"** per capabilities.md §CAP-024.

CAP-024 states: "S-4.4/4.5: each sink driver retries failed sends with exponential backoff, trips a circuit breaker on sustained failure, and routes dropped events to a `dead-letter-<sink>-<date>.jsonl`." S-4.05 is explicitly cited as S-4.5 in CAP-024 and delivers the dead-letter queue portion. The outcome CAP-024 promises — "a Datadog outage doesn't lose events — they land in the DLQ" — is directly what this story implements.

## Narrative
- **As a** factory operator
- **I want to** have events that exhaust their retries written to a dead letter file
- **So that** I can inspect and replay dropped events rather than losing them silently

## Goal

When events are dropped — because the sink's retry attempts exhausted,
or the outbound queue overflowed — write them to a per-sink dead letter
file. This prevents silent data loss.

## Acceptance Criteria

- [ ] Per-sink dead letter file at `.factory/logs/dlq/<sink-name>.jsonl`
  (traces to BC-3.02.013 postcondition 1: `take_failures()` returns at least one SinkFailure; no panic — DLQ builds on the same failure-recording path, writing the failure payload to a file instead of only to the in-memory failure list)
- [ ] Events written to DLQ when all retries exhausted
  (traces to BC-3.01.008 postcondition 1: `SinkFailure` with path, reason, ts pushed; consumer drains via `take_failures` — DLQ write is triggered when `take_failures()` surfaces a failure from an exhausted-retry context)
- [ ] Events written to DLQ when outbound queue overflows (queue full)
  (traces to BC-3.01.008 postcondition 1: failure recording is the canonical path; queue-full overflow is a failure that is written to the DLQ before the event is discarded)
- [ ] DLQ file has daily rotation matching file sink rotation
  (traces to BC-3.02.015 postcondition 1: shutdown drains queued events — daily rotation reuses the same rotation pattern as the file sink, ensuring consistency across JSONL outputs)
- [ ] `internal.sink_dlq_write` event emitted for each DLQ write
  (traces to BC-3.01.008 postcondition 1: `SinkFailure` with ts pushed — `internal.sink_dlq_write` event carries the same fields and is emitted on the same path as the in-memory failure record)
- [ ] Integration test: sink with mock server always returning 5xx; verify events in DLQ
  (traces to BC-3.02.013 precondition 1: read-only target dir forces a failure path; analogously, mock 5xx forces all retries to exhaust, triggering DLQ write)
- [ ] DLQ directory auto-created if missing
  (traces to BC-3.02.013 invariant 1: permission errors are observable via the failure log, not by crashing the dispatcher — DLQ directory creation follows the same fail-safe principle)
