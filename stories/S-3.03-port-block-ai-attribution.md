## Capability Anchor Justification

Primary: CAP-008 ("Gate tool calls with pre-execution behavioral checks (PreToolUse hooks)") per capabilities.md §CAP-008. The block-ai-attribution plugin is a PreToolUse gate — it blocks `git commit` commands containing AI attribution patterns before execution, directly implementing the behavioral gating capability.

FR anchor: **Dual — FR-013 (primary) + FR-032 (secondary).** FR-013 ("Legacy bash adapter plugin") per PRD §2.4 covers the bash-to-WASM plugin bridging. FR-032 ("PreToolUse gate hooks") also applies because block-ai-attribution is a PreToolUse gate that implements the gating contract directly — the blocking decision logic is within scope of FR-032. Primary trace is FR-013 (the plugin production story); FR-032 is the functional gating requirement this plugin satisfies. Both are listed in `functional_requirements`. Primary `traces_to` is PRD §FR-013 (F-004 fix).

Note: architecture compliance rules reference BC-1.03.001 (plugin execution lifecycle) and BC-1.01.001 (registry schema) — these are cross-subsystem informational references, not SS-04 anchors.

## Narrative
- **As a** repository maintainer
- **I want to** have `block-ai-attribution` running as native WASM in the PreToolUse/Bash hook
- **So that** commits containing AI attribution strings are blocked before they reach git

## Goal

Rewrite `block-ai-attribution` from bash to native WASM. This is a
PreToolUse/Bash gate — it must return `HookResult::Block` to prevent
the Bash tool from running if the commit message contains AI attribution patterns.
