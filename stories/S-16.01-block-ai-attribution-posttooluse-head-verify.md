# S-16.01: block-ai-attribution — PostToolUse retroactive HEAD verification

**Epic:** E-16 — Hook Plugin Capability Extensions  
**Milestone:** `v1.0.0-rc.17`  
**Depends on:** S-3.03 (merged at rc.1 — block-ai-attribution WASM plugin already exists)  
**Blocks:** S-16.02  
**Priority:** P1  

## Subsystem Anchor Justification

**SS-07 (primary):** SS-07 (Hook Bash Layer) owns this story's scope because the
`hooks-registry.toml` routing table (the plugin's registry entry) is an SS-07
artifact, and the WASM plugin behavior contracts (BC-7.03.094, BC-7.03.001) are
anchored in the SS-07 BC shard directory per ARCH-INDEX Subsystem Registry.

**SS-04 (secondary):** SS-04 (Plugin Ecosystem) is cited because the
`block-ai-attribution` crate is an SS-04 plugin crate (per ARCH-INDEX), and the
`exec_subprocess` capability surface change (new host function usage) is documented
in SS-04-plugin-ecosystem.md v1.4 (F2 amendment).

## Dependency Anchor Justification

**depends_on: ["S-3.03"]** — S-16.01 depends on S-3.03 because S-3.03 created the
`block-ai-attribution` crate, `src/lib.rs`, `src/main.rs`, and the initial
`hooks-registry.toml` entry. The PostToolUse arm extends existing shipped code; the
crate must already exist. S-3.03 is merged at rc.1.

**blocks: ["S-16.02"]** — S-16.01 blocks S-16.02 because S-16.02 depends on the
VP-080 proptest harness (`tests/proptest_detect_attribution.rs`) created in this
story, and the fallback-to-Continue invariant established here sets the precedent for
S-16.02's read_file arm. The two stories share test infrastructure; C-before-B
sequencing is required.

## Narrative

- **As a** vsdd-factory operator
- **I want** the `block-ai-attribution` plugin to also fire as a PostToolUse hook
  that retroactively reads the just-landed commit's body via `git log -1 --format=%B HEAD`
  and runs the existing `detect_attribution` against it
- **So that** any AI attribution text committed via editor flow, `-F -` stdin,
  heredoc injection, or any other bypass vector is detected post-commit and
  immediately surfaced to the agent as a corrective-signal Block with a
  `git reset --soft HEAD~1` remediation, before any push or further work

## Acceptance Criteria

### AC-001 (traces to BC-7.03.001 postcondition 1 — registry shape matches described events and capabilities)

The `block-ai-attribution` entry in
`/Users/jmagady/Dev/vsdd-factory/plugins/vsdd-factory/hooks-registry.toml` is
extended to include `PostToolUse` in its events list alongside the existing
`PreToolUse`. The registry entry must also contain an `[hooks.capabilities.exec_subprocess]`
block with `binary_allow = ["git"]`, `shell_bypass_acknowledged = false`, and
`timeout_ms = 1000` (or the equivalent capability declaration structure per BC-7.03.001
v1.3). The existing `on_error = "block"` and `priority = 20` are preserved unchanged.

**Falsifiable test:** `grep -A20 'name = "block-ai-attribution"' plugins/vsdd-factory/hooks-registry.toml | grep "PostToolUse"` exits 0; `grep -A20 'name = "block-ai-attribution"' plugins/vsdd-factory/hooks-registry.toml | grep 'binary_allow.*git'` exits 0.

### AC-002 (traces to BC-7.03.094 precondition 2 — exact token gate prevents lookalike commands from triggering)

For PostToolUse Bash events where `tool_input.command` is precisely `git commit`
(token after `git` is exactly `commit`, not `commit-tree`, `commit-graph`,
`commit-msg`, or any other multi-word variant), the plugin evaluates PRE-2 as true
and proceeds to PRE-3. For commands where the token after `git` is not exactly
`commit`, the plugin returns `HookResult::Continue` without calling
`vsdd::exec_subprocess`.

**Falsifiable test:**
- `test_BC_7_03_094_lookalike_commit_tree_does_not_exec_subprocess`: PostToolUse event with `command = "git commit-tree abc123 -p def456 -m msg"`, exit=0 → `HookResult::Continue`, `exec_subprocess` mock NOT called.
- `test_BC_7_03_094_lookalike_commit_graph_does_not_exec_subprocess`: PostToolUse event with `command = "git commit-graph write"`, exit=0 → `HookResult::Continue`, `exec_subprocess` mock NOT called.
- `test_BC_7_03_094_exact_commit_token_proceeds_to_subprocess`: PostToolUse event with `command = "git commit -m 'fix'"`, exit=0 → `exec_subprocess` mock IS called.

### AC-003 (traces to BC-7.03.094 precondition 3 — success-only gate; failed commits do not trigger exec_subprocess)

On PostToolUse Bash events where `tool_input.command` satisfies PRE-2 (`git commit`
exact token) AND `tool_response` indicates exit_code != 0 (failed commit — pre-commit
hook rejected it, git returned non-zero), the plugin returns `HookResult::Continue`
without calling `vsdd::exec_subprocess`. There is no retroactive check for a commit
that did not land.

**Falsifiable test:** `test_BC_7_03_094_failed_commit_does_not_exec_subprocess`: PostToolUse event with `command = "git commit -m 'msg'"`, `tool_response.exit_code = 1` → `HookResult::Continue`, `exec_subprocess` NOT called.

### AC-004 (traces to BC-7.03.094 postcondition 1 — attribution detected in HEAD → Block with corrective-signal)

When PRE-1 through PRE-3 are satisfied and `vsdd::exec_subprocess("git", ["log", "-1", "--format=%B", "HEAD"], timeout_ms=1000, max_output_bytes=65536)` returns exit 0 with stdout that causes `detect_attribution(stdout)` to return `Some(attribution)`, the plugin returns:

```
HookResult::block_with_fix(
  "block-ai-attribution",
  <reason from Attribution struct — e.g. "Commit message contains AI attribution: Co-Authored-By: claude">,
  "Run `git reset --soft HEAD~1`, edit the message to remove the attribution lines, and recommit.",
  "ai_attribution_post_commit",
)
```

The `event_code` is `"ai_attribution_post_commit"` for all patterns detected via the
PostToolUse arm, regardless of whether the matched pattern is Co-Authored-By,
Generated-with, or email. This distinguishes PostToolUse detections from
PreToolUse command-string detections in telemetry.

**Falsifiable test:** `test_BC_7_03_094_attribution_in_head_returns_block_with_fix`: PostToolUse event, command=`git commit --allow-empty -m ""`, exit=0; mock `exec_subprocess` returns `Ok(stdout="Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>")` → `HookResult::Block`; assert `event_code == "ai_attribution_post_commit"`; assert fix contains "git reset --soft HEAD~1".

### AC-005 (traces to BC-7.03.094 postcondition 2 — clean HEAD → Continue)

When `vsdd::exec_subprocess` returns exit 0 with stdout that causes
`detect_attribution(stdout)` to return `None` (no attribution patterns), the plugin
returns `HookResult::Continue`. No block event is emitted.

**Falsifiable test:** `test_BC_7_03_094_clean_head_returns_continue`: PostToolUse event, command=`git commit -m "Fix login bug"`, exit=0; mock `exec_subprocess` returns `Ok(stdout="Fix login bug\n")` → `HookResult::Continue`.

### AC-006 (traces to BC-7.03.094 postcondition 3 and invariant 1 — subprocess failure → Continue + telemetry, NEVER Block)

On any `vsdd::exec_subprocess` error variant — `CAPABILITY_DENIED`, `TIMEOUT`,
`OUTPUT_TOO_LARGE`, `NOT_FOUND` (git binary absent), or non-zero git exit code —
the plugin MUST return `HookResult::Continue`. A telemetry event of type
`internal.exec_subprocess_failed` MUST be emitted with fields: `plugin_name`,
`arm`, `error_variant`, `timeout_ms`. Under no circumstances may a subprocess
failure produce `HookResult::Block`.

**Falsifiable test (four cases):**
- `test_BC_7_03_094_subprocess_timeout_yields_continue`: mock returns `Err(TIMEOUT)` → `HookResult::Continue`.
- `test_BC_7_03_094_subprocess_capability_denied_yields_continue`: mock returns `Err(CAPABILITY_DENIED)` → `HookResult::Continue`.
- `test_BC_7_03_094_subprocess_not_found_yields_continue`: mock returns `Err(NOT_FOUND)` → `HookResult::Continue`.
- `test_BC_7_03_094_subprocess_nonzero_exit_yields_continue`: mock returns exit_code=128 (orphan branch) → `HookResult::Continue`.

### AC-007 (traces to BC-7.03.094 postcondition 4 — empty stdout → Continue)

When `vsdd::exec_subprocess` returns exit 0 with zero bytes of stdout (empty
repository, no commits), `detect_attribution("")` returns `None` and the plugin
returns `HookResult::Continue`. The empty-string case is not treated as an error.

**Falsifiable test:** `test_BC_7_03_094_empty_stdout_yields_continue`: mock `exec_subprocess` returns `Ok(stdout="")` → `HookResult::Continue`.

### AC-008 (traces to BC-7.03.094 invariant 5 — single exec_subprocess call per PostToolUse invocation)

The PostToolUse handler MUST issue at most one `vsdd::exec_subprocess` call per
hook firing. There is no retry logic. The implementation is verified by the mock
call-count assertions in AC-004 and AC-006 tests — each test asserts the mock was
called exactly once (or not at all for non-trigger cases).

**Falsifiable test:** `test_BC_7_03_094_exec_subprocess_called_exactly_once_on_match`: PostToolUse event with attribution in HEAD; assert mock call count == 1.

### AC-009 (traces to VP-080 anchor — proptest harness for detect_attribution across TV-001..011)

A proptest harness is added at
`/Users/jmagady/Dev/vsdd-factory/crates/hook-plugins/block-ai-attribution/tests/proptest_detect_attribution.rs`
implementing VP-080's four properties:
- Property 1: each TV-001..011 pattern embedded in arbitrary surrounding context is detected by `detect_attribution`.
- Property 2: arbitrary strings not containing canonical patterns return `None` (no false positives).
- Property 3: boundary inputs (`""`, `"   \n\t  "`, `"Co-Authored"`, `"Co-Authored-By"`) all return `None`.
- Property 4: `detect_attribution` is pure — running 1024 cases in single-threaded context produces no observable external state changes.

The harness uses `proptest 1.6` (workspace version). It runs 1024 cases per
invocation with shrink-to-minimal enabled. The proptest verifies that `detect_attribution`
outcome is source-agnostic: the same patterns are detected whether the input is a
`-m` command-string fragment, a file body read via `-F`, or `git log` stdout — all
three are plain `&str` inputs to the same pure function.

**Falsifiable test:** `cargo test -p block-ai-attribution proptest_detect_attribution` runs all four properties in ≤10s GREEN.

### AC-010 (traces to BC-7.03.094 invariant 6 — detect_attribution remains pure after S-16.01 implementation)

After S-16.01 lands, the `detect_attribution` function in `src/lib.rs` has no I/O
side effects, no global state mutation, and no side effects of any kind. The
PostToolUse handler in `src/post_tool_use.rs` (or equivalent) calls `detect_attribution`
on the stdout string AFTER the subprocess returns — `detect_attribution` itself never
initiates any subprocess or file reads. This is verified by the VP-080 proptest harness
running the pure function in isolation.

**Falsifiable test:** Code review audit: `detect_attribution` body contains no `vsdd::` host function calls; `clippy` with `#[deny(clippy::impure_fn)]` equivalent passes.

### AC-011 (traces to BC-7.03.094 postcondition 1 — end-to-end integration test against a real git commit in a tempdir)

An integration test is added at
`/Users/jmagady/Dev/vsdd-factory/crates/hook-plugins/block-ai-attribution/tests/posttooluse_integration.rs`
that exercises the full PostToolUse path against a real `git commit` in a tempdir
worktree. The test:
1. Initializes a `git init` tempdir.
2. Makes a commit with `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>` in the body.
3. Constructs a PostToolUse `HookPayload` with `tool_input.command = "git commit"` and exit=0.
4. Calls `on_post_tool_use_logic(&payload, &real_subprocess_fn)` (or equivalent interface).
5. Asserts `HookResult::Block` with `event_code = "ai_attribution_post_commit"`.

This confirms end-to-end behavior: the corrective-signal Block reaches the agent on
a real commit with real attribution text.

**Falsifiable test:** `cargo test -p block-ai-attribution posttooluse_integration` GREEN (requires git on PATH; marked `#[ignore]` in CI if git is not guaranteed — document in test file header).

**Architecture note:** The PostToolUse handler is factored into a separate

testability — `on_post_tool_use_logic` receives the subprocess function as a parameter

or `on_post_tool_use_logic` (PostToolUse) based on `hook_event_name`.

## Forbidden Dependencies

`crates/hook-plugins/block-ai-attribution/Cargo.toml` MUST NOT gain a dependency on:
- `factory-dispatcher` (creates circular dependency; dispatcher loads plugins dynamically)
- Any crate that imports `wasmtime` at compile time (plugins run INSIDE wasmtime)
- `tokio` (WASM target does not support async runtime)
- Any OS-specific crate that does not compile to `wasm32-wasip1`

If any of these appear in the dependency graph, the WASM build MUST fail. The
implementer must enforce this via `cargo build -p block-ai-attribution --target wasm32-wasip1 --release`
as the final verification step.

| `detect_attribution(s: &str)` | pure-core | String pattern matching; no I/O, no global state, no side effects; identical behavior regardless of whether `s` originated from `-m` command argument, `-F` file body, or `git log` stdout (INV-6, VP-080) |

## Architecture Compliance Rules

Derived from BC-7.03.094, BC-7.03.001 v1.3, SS-07-hook-bash.md v1.3, and SS-04-plugin-ecosystem.md v1.4:

1. **Fallback-to-Continue on any subprocess error (BC-7.03.094 INV-1, HIGH PRIORITY):** The PostToolUse handler MUST return `HookResult::Continue` on ANY `vsdd::exec_subprocess` error. This is the primary safety constraint. Violation is a P0 false-positive reliability regression.
2. **Exact-token command gate (BC-7.03.094 PRE-2):** `git commit-tree` and `git commit-graph` MUST NOT trigger exec_subprocess. Implement by splitting the command on whitespace, finding the `git` token, and checking that the immediately following token is exactly `"commit"`.
3. **Success-only gate (BC-7.03.094 PRE-3):** exec_subprocess is called ONLY when `tool_response.exit_code == 0`. Failed commits must not trigger the subprocess.
4. **Timeout fixed at 1000ms, max_output_bytes fixed at 65536 (BC-7.03.094 INV-3, INV-4):** These are compile-time constants — NOT runtime-configurable. A PR that introduces a runtime config for these values is a BC violation.
5. **`shell_bypass_acknowledged = false` in registry (BC-7.03.094 INV-3 and DI-005):** The `exec_subprocess` capability for this plugin uses `binary_allow = ["git"]` with `shell_bypass_acknowledged = false`. Setting this to `true` would allow shell interpreter bypass (DI-005 violation) and MUST be rejected by code review.
6. **PostToolUse does NOT undo the commit (BC-7.03.094 description):** The plugin returns Block with a remediation message; it does NOT call `git reset --soft HEAD~1` itself. The agent performs the reset in response to the Block.
7. **detect_attribution remains pure (BC-7.03.094 INV-6, VP-080):** The `detect_attribution` function body MUST NOT gain any `vsdd::` host function calls. The PostToolUse handler calls exec_subprocess and then passes the stdout string to `detect_attribution`.
8. **Event code is `"ai_attribution_post_commit"` (BC-7.03.094 PC-1):** Do not reuse existing codes from the PreToolUse arm (`ai_attribution_coauthor`, `ai_attribution_generated`, `ai_attribution_email`). The PostToolUse arm uses a unified telemetry event code.
