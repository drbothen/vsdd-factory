# S-12.04: WASM Resolver Loading, Lifecycle, and Error Isolation

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** Batch 3 (after S-12.03; before S-12.07).
**Depends on:** S-12.03 (the `ContextResolver` trait and `ResolverRegistry` must exist)
**Blocks:** S-12.07 (the concrete WASM resolver needs the loading infrastructure)
**Priority:** P0

## Narrative

- **As a** factory dispatcher
- **I want** the `ResolverRegistry` to load WASM resolver artifacts from disk at startup (reading `resolvers-registry.toml`), compile each to a `wasmtime::Module`, and invoke them per-dispatch with crash isolation
- **So that** factory-specific context (like wave state) can be injected into hook `plugin_config` via sandboxed WASM plugins, with the guarantee that a misbehaving resolver cannot crash the dispatcher or block the dispatch of other hooks

## Acceptance Criteria

### AC-001 (traces to BC-1.13.001 postcondition 1 + invariant 2 — absent `resolvers-registry.toml` = zero resolvers, no error)

When `resolvers-registry.toml` does NOT exist at the configured registry path, the dispatcher calls `ResolverLoader::load_registry(path)` which returns `Ok(ResolverRegistry::empty())` — NOT an error. The dispatcher starts normally. All existing hooks dispatch without change. No `resolver.load_error` event is emitted for a missing file.

**Falsifiable test:** Rust integration test: start the dispatcher with a `project_dir` that has no `resolvers-registry.toml`; assert dispatcher startup succeeds (no panic, no error return); assert zero `resolver.load_error` events emitted; assert an existing hook dispatches normally.

### AC-002 (traces to BC-1.13.001 postcondition 2 — malformed TOML triggers fail-loud startup error)

When `resolvers-registry.toml` exists but contains a TOML parse error or schema validation error, `ResolverLoader::load_registry` returns `Err(ResolverLoadError::ParseError { detail })`. The dispatcher emits a `resolver.load_error` event with the parse error detail and MUST NOT start with a partial resolver set.

**Falsifiable test:** Rust unit test: `ResolverLoader::load_registry` with TOML `"[[resolvers]\nname = 1"` (malformed TOML); assert `Err(ResolverLoadError::ParseError { .. })` returned; assert dispatcher startup returns error (does not silently proceed with zero resolvers when file is present but broken).

### AC-003 (traces to BC-4.12.001 postcondition 1 — module compiled once; cache hit on second dispatch)

`ResolverLoader::get_or_compile(path)` compiles a WASM artifact once and caches the resulting `Arc<Module>`. A second call with the same path + unchanged mtime returns the cached module without re-calling `Module::from_file`. The cache is keyed by `(canonical_path, mtime)`.

**Falsifiable test (VP-073-A):** Rust integration test: call `get_or_compile(path)` twice; assert the second call returns the same `Arc<Module>` (pointer equality or cache hit counter == 1 via mock).

### AC-004 (traces to BC-4.12.001 postcondition 3 — mtime change triggers cache eviction and recompile)

When the WASM artifact mtime changes (simulated by touching the file), the next call to `get_or_compile` evicts the old module and recompiles from the updated artifact.

**Falsifiable test (VP-073-C):** Rust integration test: compile module; touch `.wasm` file (update mtime); call `get_or_compile` again; assert recompile occurred (compile counter incremented or different module Arc pointer).

### AC-005 (traces to BC-4.12.001 postcondition 2 — per-dispatch `Store` isolation)

Each resolver invocation creates a fresh `Store<HostContext>`. No state persists between invocations via the `Store`. Invoking the same resolver WASM twice in the same process does NOT share a `Store`.

**Falsifiable test:** Rust unit test: invoke a mock WASM resolver twice; assert the `Store` passed to each invocation is a distinct object (no reference sharing). This verifies BC-4.12.001 invariant 2.

### AC-006 (traces to BC-4.12.004 postconditions 1–2 — trap isolation: dispatcher process unaffected by resolver WASM trap)

When a resolver's `resolve()` function traps (e.g., executes `unreachable`), the dispatcher catches the trap, emits `resolver.error` with `error_kind: "trap"`, and proceeds to hook dispatch with the resolver's key absent from `plugin_config`. The dispatcher process does NOT panic or exit.

**Falsifiable test (VP-074 integration):** Rust integration test using the `tests/fixtures/trapping_resolver.wasm` fixture: dispatch with `needs_context: ["trap-resolver"]`; assert dispatch returns `Ok`; assert mock telemetry captures a `resolver.error` event with `error_kind == "trap"`; assert `plugin_config.get("trap-resolver").is_none()`.

### AC-007 (traces to BC-4.12.004 invariant 1 + VP-074 kani — `classify_resolver_trap` is total)

`crates/factory-dispatcher/src/resolver_classify_trap.rs` defines a pure function `pub fn classify_resolver_trap(trap_code: TrapCode) -> ResolverError`. The kani harness verifies:
- Every `TrapCode` variant produces a `ResolverError` without panic
- `classify_resolver_trap` never returns `ResolverError::NotFound`
- The error-path result-wrapping in `invoke_resolver` is panic-free

**Falsifiable test (VP-074 kani):** `cargo kani -p factory-dispatcher --harness proof_classify_trap_is_total` passes without counterexamples.

### AC-008 (traces to BC-4.12.004 postcondition 3 — failed resolver key ABSENT from `plugin_config`)

When a resolver fails (trap, timeout, or ABI violation), the dispatcher does NOT write the resolver's key to `plugin_config`. The key is absent — not `null`, not `{}`, not any default value. The hook receives `plugin_config` without that key.

**Falsifiable test:** Rust integration test (same fixture as AC-006): assert `plugin_config.as_object().unwrap().get("trap-resolver")` is `None` — not `Some(Value::Null)`.

### AC-009 (traces to BC-4.12.004 postcondition 6 — resolver A failure does not skip resolver B)

When `needs_context: ["trap-resolver", "good-resolver"]` and resolver `"trap-resolver"` fails, resolver `"good-resolver"` still executes and its output is merged into `plugin_config` normally.

**Falsifiable test:** Rust integration test: two resolvers — one trapping, one returning `{key: "good", value: Some(json!(1))}`; dispatch; assert `resolver.error` for trapping resolver; assert `plugin_config["good"] == 1`.

### AC-010 (traces to BC-4.12.003 postconditions 1–3 — resolver linker excludes write/exec/emit host functions)

The `Linker<HostContext>` constructed for resolver execution does NOT expose `host::write_file`, `host::exec_subprocess`, or `host::emit_event` as imports. A resolver WASM that references `write_file` fails at module instantiation time (linker error), not at runtime.

**Falsifiable test:** Rust integration test: attempt to instantiate a WASM module that imports `host::write_file` in the resolver linker context; assert the error is a linker error (symbol not found), not a trap. Assert the error is reported as `resolver.load_error`.

### AC-011 (traces to BC-4.12.003 postconditions 2 + VP-076 — CapabilityDenied for out-of-`path_allow` read)

When a WASM resolver calls `host::read_file(path)` where `path` is outside the resolver's declared `path_allow`, the host function returns `CapabilityDenied` to the resolver (not a panic, not a trap). The dispatcher emits `resolver.capability_denied` with resolver name and attempted path. The resolver's output key is absent from `plugin_config` if the resolver handles the error by returning `None` — or the resolver traps, in which case BC-4.12.004 applies.

**Falsifiable test (VP-076):** Rust integration test using `tests/fixtures/path_escaping_resolver.wasm` (attempts to read `/etc/passwd`); dispatch; assert `resolver.capability_denied` event emitted; assert `/etc/passwd` content absent from `plugin_config`.

### AC-012 (traces to BC-1.13.001 postcondition 1 — resolver count logged at startup)

When resolvers load successfully at startup, the dispatcher emits a log entry at startup level: `"Compiled N resolver modules from resolvers-registry.toml"` (where N is the count of successfully compiled modules).

**Falsifiable test:** Rust unit test: load a `resolvers-registry.toml` with 2 valid resolver entries; assert the startup log contains a line matching `"Compiled 2 resolver modules"`.

### AC-013 (traces to BC-4.12.001 postcondition 6 — failed module compilation behavior)

When a `resolvers-registry.toml` entry references a `.wasm` file that fails to compile (`Module::from_file` error — e.g., invalid WASM bytes, missing file), the dispatcher emits `resolver.load_error` with the resolver name and compilation error detail.

**Falsifiable test:** Rust unit test: `resolvers-registry.toml` with a resolver pointing to a non-existent `.wasm` path; assert `resolver.load_error` event emitted with the missing path in `error_detail`.

## Architecture Compliance Rules

Derived from BC-4.12.001 invariants, BC-4.12.003 invariants, BC-4.12.004 invariants, and ADR-018:

1. **Compilation is startup-time only (BC-4.12.001 invariant 1):** No WASM compilation inside `executor.rs`'s dispatch loop. Compilation happens ONLY in `resolver_loader.rs` at startup or on mtime-change. A code review finding any `Module::from_file` call in `executor.rs` is a blocker.
2. **Store-per-call isolation (BC-4.12.001 invariant 2):** `Store<HostContext>` must be created inside `invoke_resolver` for each call. No `Store` stored as a field of `ResolverRegistry` or `ResolverLoader`.
3. **Resolver linker is distinct from hook linker:** The `Linker<HostContext>` for resolver execution MUST NOT include `write_file`, `exec_subprocess`, or `emit_event`. If the existing linker setup in `host/mod.rs` is used for both hooks and resolvers, a separate configuration path must be added for resolver execution.
4. **No `unwrap()` on resolver invocation result (BC-4.12.004 invariant 1):** `invoke_resolver` must use `?` or `match` — never `.unwrap()` or `.expect()`. The kani harness verifies the pure error path; the integration test verifies the wasmtime boundary.
5. **Absent registry file is NEVER an error (BC-1.13.001 invariant 2):** The code path that handles a missing `resolvers-registry.toml` must return `Ok(ResolverRegistry::empty())`, not `Err(...)`. A code review finding `?` applied to a `File::open` of the registry without a "not found" special case is a blocker.

## Forbidden Dependencies

The `resolver_classify_trap.rs` module MUST be a pure function module with NO side effects. The following are forbidden in that file:
- `use std::fs` — no filesystem access
- `use std::io` — no I/O
- Any `extern crate` beyond `wasmtime` (for `TrapCode`) and the local `ResolverError` type

## Purity Classification

**Module:** `crates/factory-dispatcher/src/resolver/loader.rs`
**Classification:** Impure (performs filesystem I/O to load WASM binaries from `resolvers-registry.toml` paths)

**Pure sub-components:**
- `resolver_classify_trap` — pure function; maps `wasmtime::TrapCode` → `ResolverError`; no I/O; covered by Kani proof

**Impure sub-components:**
- `load_resolver_registry` — reads `resolvers-registry.toml` from disk (startup I/O)
- `instantiate_resolver` — invokes wasmtime linker + store (side-effectful WASM instantiation)
- `invoke_resolver` — calls WASM export (external side effect via sandboxed WASM execution)

**Test strategy:** Pure sub-components tested via Kani (VP-073); impure components tested via bats integration tests against WASM test fixtures.

## Dev Notes

**mtime-cache pattern:** Read `plugin_loader.rs` first. The mtime-cache pattern stores `HashMap<(PathBuf, SystemTime), Arc<Module>>`. The `get_or_compile` function: (1) stat the file to get mtime; (2) check cache; (3) on miss, compile `Module::from_file(&engine, path)?`; (4) insert into cache; (5) return `Arc<Module>`. Wrap the HashMap in `Mutex<_>` for thread-safety per BC-4.12.001 EC-003.

**Fixture WASM authoring:** The `trapping_resolver.wasm` fixture needs to be a valid WASM module that exports a `resolve()` function which immediately traps. The simplest implementation is a raw WASM text format file compiled with `wasm-tools`:
```wat
(module (func (export "resolve") (param i32 i32) (result i64) unreachable))
```
Compile with `wasm-tools parse trapping_resolver.wat -o tests/fixtures/trapping_resolver.wasm`. Include the `.wat` source in `tests/fixtures/` for auditability.

**Fuel/epoch budget for resolvers:** Set to 25% of the hook budget per F1-amendment §S-12.04 sketch. Check the current hook fuel limit in `host/mod.rs` or `executor.rs` and set resolver budget proportionally.

**Bootstrap-flipped Step 4.5:** This is the most technically dense story in the batch. The adversary will make multiple passes. The kani VP-074 proof must pass cleanly before the adversary will converge. Plan for 3–4 passes. The convergence hook (S-12.02) is now live and will enforce ≥3 NITPICK_ONLY passes mechanically.
