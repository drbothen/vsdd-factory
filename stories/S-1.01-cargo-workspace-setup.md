# S-1.01: Cargo workspace + CI scaffolding

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.0 (BLOCKER — gates all of Tier B.x)

**Estimated effort:** M

## Narrative
- **As a** Rust developer contributing to vsdd-factory
- **I want to** have a properly scaffolded Cargo workspace with CI
- **So that** all dispatcher, SDK, and plugin crates share a coherent build graph and can be developed in parallel

## Goal

Establish the Cargo workspace at the repo root so all Rust crates
(dispatcher, SDK, hook-plugins, sink drivers) share a coherent build
graph, dependency set, and CI pipeline. This is a pure-plumbing story —
no business logic — but it's the single blocker preventing Tier B.x from
fanning out in parallel.

## Acceptance criteria

- [ ] `Cargo.toml` at repo root declares a workspace with members
  `crates/factory-dispatcher`, `crates/hook-sdk`, and a stub plugin
  `crates/hook-plugins/capture-commit-activity`
  _(traces to BC-1.07.005 invariant: plugin crate uses `crate-type = ["cdylib"]` + wasm32-wasip1 target — the scaffold this story creates is the build prerequisite for that invariant)_
- [ ] Each member crate has a minimal `Cargo.toml` + placeholder `lib.rs`
  or `main.rs` that compiles cleanly
  _(traces to BC-1.08.003 precondition: `tokio` must be present with correct feature flags; workspace.dependencies pins this)_
- [ ] Workspace uses `resolver = "2"`
- [ ] Shared dependency versions pinned in `[workspace.dependencies]` —
  minimum: `serde`, `serde_json`, `toml`, `anyhow`, `thiserror`,
  `tokio`, `tracing`, `tracing-subscriber`, `wasmtime`, `clap`
  _(traces to BC-1.07.003 precondition: `toml` crate available for Registry::load round-trip; BC-1.08.003 invariant: single tokio version pinned workspace-wide)_
- [ ] `rust-toolchain.toml` pins a specific stable Rust version (e.g., `1.83.0`)
- [ ] `.github/workflows/ci.yml` gains a `cargo-check` job that runs
  `cargo check --workspace --all-targets` + `cargo fmt --check` +
  `cargo clippy --workspace --all-targets -- -D warnings`
  _(traces to BC-1.07.004 postcondition: registry-generation script is idempotent — CI enforces clean build on each run)_
- [ ] Existing bats CI (1177 tests) continues to run unchanged
- [ ] `target/` is gitignored
- [ ] Workspace builds clean on darwin-arm64, linux-x64 at minimum
  _(traces to BC-1.07.006 postcondition: every legacy registry entry has plugin_config.script_path — only reachable if wasm plugin crates build correctly on target platforms)_

## Contracts

**Workspace layout contract (consumed by S-1.02 through S-1.09):**

```
Cargo.toml                                  # workspace root
crates/
  factory-dispatcher/                       # the binary
  hook-sdk/                                 # the SDK
  hook-plugins/
    <per-plugin>/                           # each a separate crate
```

**Rust version pin:** all crates target the same stable Rust version
declared in `rust-toolchain.toml`.

**Dependency version contract:** shared deps have versions pinned in
`[workspace.dependencies]`. Member crates reference them with `workspace = true`.

## Notes

`crate-type = ["cdylib"]` + `wasm32-wasip1` target for plugin crates is
non-negotiable — required by wasmtime to instantiate them as modules.

## Capability Anchor Justification

Anchoring to **CAP-002: "Hook Claude Code tool calls with sandboxed WASM plugins"** per capabilities.md §CAP-002.

> "Every Claude Code tool invocation (Bash, Edit, Write, etc.) triggers the dispatcher, which routes to matching WASM plugins by event type and tool-regex. Plugins run in wasmtime with bounded fuel + epoch timeout."

This story establishes the Cargo workspace structure that makes the WASM-sandboxed dispatcher buildable. Without `crates/factory-dispatcher` (the WASM runtime host) and `crates/hook-sdk` (the plugin ABI crate) compiling as a workspace, CAP-002 cannot be delivered. The workspace manifest, `rust-toolchain.toml`, `[workspace.dependencies]` version pins, and the `wasm32-wasip1` plugin crate scaffold are the physical prerequisites for every downstream dispatcher story.

Secondary: the workspace also scaffolds `crates/hook-sdk` (CAP-009: "Author and publish WASM hook plugins using the Rust SDK") but CAP-002 is the primary because the dispatcher binary is the central artifact.
