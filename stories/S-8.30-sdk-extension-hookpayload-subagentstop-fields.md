## Narrative

- **As a** vsdd-factory hook plugin author
- **I want to** access `payload.agent_type`, `payload.subagent_name`, `payload.last_assistant_message`, and `payload.result` as typed `Option<String>` fields on `HookPayload`
- **So that** native WASM ports of the four SubagentStop bash hooks (handoff-validator, pr-manager-completion-guard, track-agent-stop, validate-pr-review-posted) can perform typed field projection without re-parsing raw stdin JSON

## Goal

Claude Code's SubagentStop events carry agent identity and assistant-message content as **top-level JSON fields** — not nested inside `tool_input` or `tool_response`. The four existing bash hooks read these fields via jq:

```bash
# All four hooks (handoff-validator.sh:27, pr-manager-completion-guard.sh:25,
# validate-pr-review-posted.sh:21, track-agent-stop.sh:22)
AGENT=$(echo "$INPUT" | jq -r '.agent_type // .subagent_name // "unknown"')
RESULT=$(echo "$INPUT" | jq -r '.last_assistant_message // .result // empty')
# handoff-validator additionally: .last_assistant_message // .result // .output
```

The current `HookPayload` struct (as of `crates/hook-sdk/src/payload.rs:16-53`) does **not** model these fields. Native WASM ports of the SubagentStop hooks currently must re-parse the raw `tool_input` Value — defeating the purpose of the typed-projection model — or cannot access the fields at all.

This story extends `HookPayload` **additively** with four `#[serde(default)] Option<String>` fields, providing:

1. **Backward-compatible deserialization** — missing fields (PreToolUse, etc.) default to `None` via `#[serde(default)]`.
2. **jq-`//`-equivalent null-as-advance semantics** — serde_json maps JSON `null` to `None` for `Option<T>`, mirroring jq's `//` operator.
3. **Typed fallback chains** — canonical Rust expressions for resolving agent identity and assistant-message content, directly parallel to the bash patterns.

**BC anchor:** BC-2.02.012 ("HookPayload SubagentStop fields: top-level envelope schema for agent_type, subagent_name, last_assistant_message, result") — authored by PO in the D-183 Phase A burst, status `ready`.

**ABI policy:** D-183 sealed Path 1 (additive HookPayload extension). HOST_ABI_VERSION stays at 1 per D-6 Option A — no breaking change, no version bump needed. This is the same policy applied by S-8.10 v1.1 (write_file additive SDK extension, BC-2.02.011).

**Precedent:** S-8.10 established the pattern for additive HookPayload-adjacent extensions under D-6 Option A. S-8.30 follows the same structural template (field addition with `#[serde(default)]`, backward-compat tests, HOST_ABI_VERSION stays at 1, HOST_ABI.md documentation update).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Field present in envelope with JSON null (e.g., `"agent_type": null`) | `payload.agent_type = None`; fallback chain advances to `subagent_name`. Per BC-2.02.012 EC-001. Test in AC-4. |
| EC-002 | Field absent entirely from envelope | `payload.agent_type = None` via `#[serde(default)]`; fallback chain advances. Per BC-2.02.012 EC-002. |
| EC-003 | All four SubagentStop fields absent or null | Agent identity resolves to `"unknown"` (final `.unwrap_or("unknown")`); assistant-message resolves to `""`. Per BC-2.02.012 EC-003. Test in AC-4. |
| EC-004 | handoff-validator 3-stage assistant-message chain (`last_assistant_message \| result \| output`) | `handoff-validator.sh:28` uses `.last_assistant_message // .result // .output // empty` — a 3-stage chain including `output` as third fallback. The canonical 2-stage chain in BC-2.02.012 Postcondition 6 does NOT include `output`. **This story does NOT add `output` to HookPayload.** S-8.01 (handoff-validator native port) MUST cite BC-2.02.012 EC-004's divergence path when adding the third arm. If S-8.01 requires `output: Option<String>` on HookPayload, that is a separate additive extension for S-8.01 to author; it follows the same `#[serde(default)]` pattern and does not bump HOST_ABI_VERSION. |
| EC-005 | Non-SubagentStop event with no SubagentStop fields | All four fields = `None`. Plugins that guard on `event_name == "SubagentStop"` before accessing these fields are safe. Per BC-2.02.012 EC-005. |
| EC-006 | SubagentStop envelope with unexpected additional top-level fields | `HookPayload` does not use `#[serde(deny_unknown_fields)]`; additional fields are silently ignored. Per BC-2.02.012 EC-006. |
| EC-007 | Field present with non-string type (e.g., `"agent_type": 123`) | serde_json returns a deserialization error. This is acceptable — the envelope schema requires string values; the dispatcher should never send this. Considered an envelope schema violation, not a HookPayload logic error. Defensive note: implementer may add a test asserting this produces a clear error, but no recovery logic is required. |
| EC-008 | Field present with empty string `"agent_type": ""` | `payload.agent_type = Some("")`. The fallback chain does NOT advance — `Some("")` is "present and valid" in Rust `Option` semantics. Note: this diverges from jq's `//` operator, which advances on `null` and `false` but NOT on empty string. Both Rust and jq behave the same here: an empty string does not advance the fallback. Bash hook jq `.agent_type // .subagent_name // "unknown"` with `agent_type: ""` yields `""` (not `"unknown"`), consistent with Rust `Some("").as_deref().or(...).unwrap_or("unknown")` returning `""`. |

## Architecture Compliance Rules

1. **SS-02 boundary:** All code changes are within `crates/hook-sdk/src/payload.rs` and `crates/hook-sdk/HOST_ABI.md`. No files outside this module are modified by this story (no dispatcher changes, no plugin changes).
2. **`#[serde(default)]` convention:** All optional HookPayload fields use `#[serde(default)]`, not `Option<T>` with explicit `#[serde(skip_serializing_if)]`. This is the existing convention in the struct (confirmed: `tool_name`, `tool_input`, `tool_response`, `plugin_config` all use `#[serde(default)]`).
3. **No `#[serde(deny_unknown_fields)]`:** HookPayload does NOT use `deny_unknown_fields` — additional envelope fields must be silently ignored (BC-2.02.012 EC-006). Do not add this attribute.
4. **HOST_ABI_VERSION = 1 invariant:** Neither `crates/hook-sdk/src/lib.rs:58` nor `crates/factory-dispatcher/src/lib.rs:43` may change. CI must confirm `HOST_ABI_VERSION: u32 = 1` in both after merge.
5. **Field names are immutable:** The field names `agent_type`, `subagent_name`, `last_assistant_message`, `result` are locked by BC-2.02.012 Invariant 4. Do not rename, alias, or abbreviate. Changes require updating all four bash hook scripts and the BC itself.
6. **Canonical fallback chains are normative:** Story specs S-8.01, S-8.02, S-8.03, S-8.05 MUST use the canonical expressions from BC-2.02.012 Postconditions 5-6. Deviations require an explicit divergence rationale (see EC-004 for the handoff-validator 3-stage case).
7. **No new crate dependencies:** This story modifies no `Cargo.toml` files. `payload.rs` uses only `serde::{Deserialize, Serialize}` and `serde_json::Value` — already in scope. No new imports are required for the four `Option<String>` fields.
