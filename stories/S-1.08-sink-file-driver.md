# S-1.08: sink-file driver

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation (feeds into Phase 4 sink system)
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.07, S-4.07
**Blocked by:** S-1.01, S-1.07
**Estimated effort:** M

## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

This story implements the `file` sink driver — the first and most foundational of the multi-sink drivers that CAP-003 describes. CAP-003 specifies: "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`." S-1.08 delivers the file sink driver, which is the primary minimum-viable sink required for beta.1 and the implementation vehicle for the JSONL-append-to-disk behavior referenced throughout SS-03 behavioral contracts.

## Narrative
- **As a** factory operator
- **I want to** have events appended to JSONL files on disk automatically
- **So that** I can inspect the event stream without any external tooling

## Goal

Implement the `file` sink type — JSONL append to a configurable path.
This is the minimum-viable sink required for beta.1 and preserves the
current `.factory/logs/events-*.jsonl` behavior that existing tools depend on.

## Acceptance Criteria

- [ ] `crates/sink-file` crate implements the `Sink` trait from `crates/sink-core`
  (traces to BC-3.06.005 postcondition 1: minimal `SinkConfigCommon` TOML defaults `enabled=true`)
- [ ] Events appended as newline-delimited JSON to the configured file path
  (traces to BC-3.02.008 postcondition 1: 3 sequential events produce 3 lines in submission order; BC-3.01.005 postcondition 1: SinkEvent serializes flat — fields expand to top-level JSON keys, not nested under a wrapper object)
- [ ] Daily rotation supported: file renamed to `<base>.<date>.jsonl` at midnight UTC
  (traces to BC-3.02.001 postcondition 1: `{date}` substitutes to YYYY-MM-DD)
- [ ] File created on first write if missing; parent dirs auto-created
  (traces to BC-3.02.007 postcondition 1: nested non-existent parent dirs are `mkdir_p`'d on first write; also BC-3.01.006 postcondition 1: path template substitutes `{date}`, `{name}`, `{project}` and rejects unknown placeholders)
- [ ] `submit(event)` returns immediately (non-blocking); writes happen on dedicated thread
  (traces to BC-3.01.007 postcondition 1: `DEFAULT_QUEUE_DEPTH = 1000`, overflow increments `queue_full_count` instead of blocking the producer)
- [ ] `flush()` drains the queue and waits for all writes to complete
  (traces to BC-3.02.015 postcondition 1: shutdown drains pending events; post-shutdown submit is rejected)
- [ ] Integration test: submit 1000 events, flush, verify all written to disk
  (traces to BC-3.05.002 postcondition 1: registry fans events to file sinks with filter and tag enrichment; also BC-3.02.014 postcondition 1: queue_depth=2 + 500 submitted events increments backpressure counter)
- [ ] Routing filter correctly includes/excludes event types
  (traces to BC-3.02.009 postcondition 1: `allow=["commit.made"]` passes 2 of 3 events through)
- [ ] Static tag enrichment applied per sink config without overwriting producer fields
  (traces to BC-3.02.010 postcondition 1: configured tags `env=prod,team=factory` land on every event; BC-3.02.011 postcondition 1: tag with `key="type"` does NOT clobber producer's type field)
- [ ] `enabled=false` sinks drop every event and do not write to disk
  (traces to BC-3.02.012 postcondition 1: no file written, no events accepted)
- [ ] Read-only target directory records a `SinkFailure` without panicking
  (traces to BC-3.02.013 postcondition 1: read-only target dir → `SinkFailure` recorded without panic; also BC-3.01.008 postcondition 1: `SinkFailure` with path, reason, ts pushed; consumer drains via `take_failures`)
- [ ] Config deserializes from minimal TOML with `queue_depth` defaulting
  (traces to BC-3.02.016 postcondition 1: minimal TOML config parses with `queue_depth` defaulting)
- [ ] `{project}` template with `project_dir = None` yields empty basename
  (traces to BC-3.02.005 postcondition 1: `None` project_dir → empty basename)
- [ ] Unbalanced brace `{` without closing `}` treated as literal text
  (traces to BC-3.02.006 postcondition 1: opening `{` without closing `}` is treated literally)
- [ ] `ObservabilityConfig` with file sink section builds a working `SinkRegistry`
  (traces to BC-3.05.001 postcondition 1: `ObservabilityConfig` with file sink section builds file sink from parsed config)
