# S-12.08: Migrate convergence hook to consume `plugin_config.wave_context.stories` — closes F-P2-001

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** Batch 5 (final story; after S-12.07).
**Depends on:** S-12.07 (the `WaveContextResolver` must produce `wave_context` output)
**Blocks:** F5 pass-2 fix burst (F-P2-001 formally closed by this story's demo evidence)
**Priority:** P0

## Narrative

- **As a** vsdd-factory operator running wave-gate dispatch
- **I want** the `validate-per-story-adversary-convergence` hook to read the story list from `plugin_config["wave_context"]["stories"]` (injected by `WaveContextResolver`) rather than depending on static registry config
- **So that** F-P2-001 is formally closed: the convergence hook is operationally effective in production (it now receives the actual wave story list at dispatch time, not an absent key from static config)

## Acceptance Criteria

### AC-001 (traces to BC-4.10.001 postcondition 1 — hook reads story list from `plugin_config["wave_context"]["stories"]`)

`crates/hook-plugins/validate-per-story-adversary-convergence/src/lib.rs` `RealCallbacks::list_stories` reads the story list from `plugin_config["wave_context"]["stories"]` (a JSON array of story ID strings). The `extract_stories_from_config` function (or its successor) uses this path instead of any static config path.

**Falsifiable test:** Rust unit test: inject a `plugin_config` with `{"wave_context": {"stories": ["S-12.03", "S-12.04"]}}` via `FakeCallbacks`; assert `extract_stories_from_wave_context(&plugin_config)` returns `Ok(vec!["S-12.03", "S-12.04"])`.

### AC-002 (traces to BC-4.10.001 postcondition 2 — absent `wave_context` key causes Block with explicit message)

When `plugin_config["wave_context"]` is absent (resolver failed or was not wired), the hook returns `HookResult::Block` with:
- `code: "WAVE_CONTEXT_MISSING"`
- `block_with_fix` message explaining that `needs_context = ["wave_context"]` is required in `hooks-registry.toml` and that `WaveContextResolver` must be registered

The hook MUST NOT `HookResult::Continue` in this case (no silent degrade for missing wave_context).

**Falsifiable test:** Rust unit test: inject `plugin_config = {}` (empty, no `wave_context` key); assert `HookResult::Block` returned with `code == "WAVE_CONTEXT_MISSING"`.

### AC-003 (traces to BC-4.10.001 postcondition 2 — wrong type for `wave_context.stories` causes Block)

When `plugin_config["wave_context"]["stories"]` is present but not a JSON array (e.g., it is a string or object), the hook returns `HookResult::Block` with `code: "WAVE_CONTEXT_SCHEMA_ERROR"` and an explanatory message.

**Falsifiable test:** Rust unit test: inject `plugin_config = {"wave_context": {"stories": "not-an-array"}}`; assert `HookResult::Block` with `code == "WAVE_CONTEXT_SCHEMA_ERROR"`.

### AC-004 (traces to BC-4.10.001 postcondition 1 + VP-071 kani — existing block invariant preserved)

The existing VP-071 kani harnesses still pass without counterexamples after the `extract_stories_from_config` refactor. The block invariant (missing state file, insufficient passes, non-NITPICK classification) is unchanged. Only the SOURCE of the story list changes (now from `wave_context.stories` instead of static config).

**Falsifiable test:** `cargo kani -p validate-per-story-adversary-convergence` all 6 harnesses pass without counterexamples after the refactor.

### AC-005 (traces to BC-1.13.001 postcondition 4 — `needs_context = ["wave_context"]` in `hooks-registry.toml`)

The `validate-per-story-adversary-convergence` entry in `plugins/vsdd-factory/hooks-registry.toml` gains:
```toml
needs_context = ["wave_context"]
```
All other fields in the entry are unchanged. Existing hooks in `hooks-registry.toml` that do NOT declare `needs_context` continue to function without modification (backward compatibility per BC-1.13.001 invariant 3).

**Falsifiable test:** `grep -A10 'validate-per-story-adversary-convergence' plugins/vsdd-factory/hooks-registry.toml | grep "needs_context"` returns a match. `cargo test -p factory-dispatcher test_existing_hooks_no_needs_context` passes (existing hooks still parse without the field).

### AC-006 (traces to BC-1.13.001 postcondition 3 + BC-4.12.005 postcondition 1 — static `plugin_config` from registry entry preserved)

The hook's existing static `plugin_config` fields in `hooks-registry.toml` (if any) remain intact after the `wave_context` injection. The resolver output is ADDITIVE per BC-4.12.005 — it does not replace the existing static config.

**Falsifiable test:** Rust unit test: static `plugin_config = {"existing_key": "value"}` merged with `wave_context = {"stories": [...]}`; assert both keys present in the config received by the hook (`existing_key` preserved, `wave_context` added).

### AC-007 (traces to BC-4.12.005 postcondition 5 — `FakeCallbacks` in tests uses synthetic `wave_context` payload)

All existing unit tests that previously passed a static config with a `stories` field are updated to pass a `wave_context` injection payload instead: `plugin_config = {"wave_context": {"stories": ["S-FAKE-001"], "wave_id": "w1", "cycle_id": "test-cycle"}}`. Tests must construct this payload directly in the test setup (synthetic injection — no real filesystem reads in unit tests).

**Falsifiable test:** `cargo test -p validate-per-story-adversary-convergence` — all existing unit tests pass after the refactor; tests call `extract_stories_from_wave_context` directly (the removed `list_stories` callback is no longer referenced in any test).

### AC-008 (traces to BC-1.13.001 + BC-4.10.001 — bats end-to-end integration test closes F-P2-001)

A new bats integration test `plugins/vsdd-factory/tests/resolver-integration.bats` exercises the complete production pipeline:
1. Seeds `.factory/STATE.md` (with `current_cycle = "test-cycle-001"`)
2. Seeds `.factory/wave-state.yaml` (with active wave, stories: `["S-FAKE-001", "S-FAKE-002"]`)
3. Seeds `.factory/cycles/test-cycle-001/S-FAKE-001/adversary-convergence-state.json` (unconverged: `passes_clean: 1`)
4. Seeds `.factory/cycles/test-cycle-001/S-FAKE-002/adversary-convergence-state.json` (converged: `passes_clean: 3, last_classification: "NITPICK_ONLY"`)
5. Invokes the dispatcher with a synthetic `SubagentStop` event (matching the wave-gate dispatch pattern)
6. Asserts the hook returns `Block` (because S-FAKE-001 is unconverged) with code `CONVERGENCE_PASSES_INSUFFICIENT`

This test PROVES the end-to-end producer (WaveContextResolver) → consumer (convergence hook) pipeline works in production. This test formally closes F-P2-001.

**Falsifiable test:** `bats plugins/vsdd-factory/tests/resolver-integration.bats` exits 0 with the end-to-end test case passing.

### AC-009 (traces to BC-4.10.001 postcondition 2 — bats integration test: converged-story case returns Continue)

The same bats integration test includes a second case: all stories in `wave-state.yaml` are converged (all have `passes_clean >= 3` and `last_classification == "NITPICK_ONLY"`). Assert the hook returns `Continue`.

**Falsifiable test:** The `bats resolver-integration.bats` second test case seeds all stories as converged and asserts `HookResult::Continue` from the dispatcher.

### AC-010 (traces to BC-4.10.001 + F-P2-001 + F-P2-008 — old graceful-degrade fallback path removed)

The old `extract_stories_from_config` path that silently returned `Err` (or `Continue`) when the stories key was absent from static config is REMOVED. It is replaced by the fail-loud path (AC-002). The hook no longer has a "stories not configured → graceful Continue" code path.

**Falsifiable test:** Code review: `grep "graceful.*degrade\|static.*config.*stories\|extract_stories_from_config" crates/hook-plugins/validate-per-story-adversary-convergence/src/lib.rs` returns zero matches for the old fallback path. The function signature or behavior is changed so the old path cannot be accidentally re-introduced.
