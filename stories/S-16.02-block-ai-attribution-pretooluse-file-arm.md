## Narrative

- **As a** vsdd-factory operator
- **I want** the `block-ai-attribution` plugin's PreToolUse handler to also parse
  `-F <path>` / `--file=<path>` / `--file <path>` flags from `git commit` commands,
  read the commit message file via `vsdd::read_file`, and run `detect_attribution`
  against its contents
- **So that** AI attribution text in a separate commit message file is caught
  _before_ the commit lands (faster feedback than the PostToolUse safety net in
  S-16.01), while the existing command-string scan remains the first priority and
  the file-read arm only activates when the command string is attribution-free

## Acceptance Criteria

### AC-001 (traces to BC-7.03.001 postcondition 1 — registry shape extended with read_file capability)

The `block-ai-attribution` entry in
`/Users/jmagady/Dev/vsdd-factory/plugins/vsdd-factory/hooks-registry.toml` is
extended to include an `[hooks.capabilities.read_file]` block in the PreToolUse
section with:
- `path_allow = ["**/.git/COMMIT_EDITMSG", "/tmp/**", "/var/folders/**", "<project-root>/**"]`
- `max_bytes = 65536`

The `<project-root>/**` entry is a template token; its resolution semantic is: the
dispatcher substitutes the project root path at capability-check time using the
`cwd` field from the hook envelope. The plugin does NOT perform this substitution
itself — the dispatcher's `read_file` host function owns path resolution (INV-6).
Any path outside this allowlist causes `vsdd::read_file` to return
`CAPABILITY_DENIED`.

**Falsifiable test:** `grep -A15 'capabilities.*read_file\|read_file' plugins/vsdd-factory/hooks-registry.toml | grep 'path_allow'` exits 0; `grep 'max_bytes.*65536' plugins/vsdd-factory/hooks-registry.toml` exits 0.

### AC-002 (traces to BC-7.03.095 precondition 3 and invariant 2 — command-string scan runs first; short-circuit preserved)

The existing `detect_attribution(command)` scan on the command string runs FIRST.
If it returns `Some(attribution)`, the plugin blocks immediately (per existing
BC-7.03.003/004 behavior) and `vsdd::read_file` is NOT called. The file-read arm
activates only when the command-string scan returns `None`.

**Falsifiable test:** `test_BC_7_03_095_command_string_attribution_short_circuits_file_read`: PreToolUse event with `command = "git commit -m 'Co-Authored-By: Claude...' -F /tmp/msg.txt"` → `HookResult::Block` via command-string arm; `read_file` mock NOT called.

### AC-003 (traces to BC-7.03.095 precondition 4 and invariant 7 — all three -F flag forms parsed correctly)

When the command-string scan returns `None`, the plugin parses the command for all
three flag forms:
- `-F <path>` (short form, space-separated)
- `--file=<path>` (long form, equals-separated, no space between `=` and path)
- `--file <path>` (long form, space-separated)

A pure helper function `parse_message_file_path(command: &str) -> Option<String>`
handles all three forms and is independently unit-tested. Order-independent parsing
handles quoted paths.

**Falsifiable test (three cases):**
- `test_BC_7_03_095_parse_short_F_flag`: `parse_message_file_path("git commit -F /tmp/msg.txt")` → `Some("/tmp/msg.txt")`.
- `test_BC_7_03_095_parse_long_file_equals`: `parse_message_file_path("git commit --file=/tmp/msg.txt")` → `Some("/tmp/msg.txt")`.
- `test_BC_7_03_095_parse_long_file_space`: `parse_message_file_path("git commit --file /tmp/msg.txt")` → `Some("/tmp/msg.txt")`.

### AC-004 (traces to BC-7.03.095 postcondition 1 — attribution in file → Block)

When `parse_message_file_path` returns `Some(path)` AND the path is not `-` (stdin
sentinel) AND `vsdd::read_file(path, max_bytes=65536)` succeeds AND
`detect_attribution(file_contents)` returns `Some(attribution)`, the plugin returns:

```
HookResult::block_with_fix(
  "block-ai-attribution",
  <reason from Attribution struct>,
  "Edit the commit message file to remove the attribution lines, then retry.",
  "ai_attribution_file_arm",
)
```

The `event_code` is `"ai_attribution_file_arm"` for all patterns detected via this
arm, regardless of which attribution pattern matched. This distinguishes file-arm
detections from command-string-arm detections in telemetry.

**Falsifiable test:** `test_BC_7_03_095_attribution_in_file_returns_block`: PreToolUse event with `command = "git commit -F /tmp/msg.txt"`; mock `read_file("/tmp/msg.txt", 65536)` returns `Ok("Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>")` → `HookResult::Block`; assert `event_code == "ai_attribution_file_arm"`; assert fix message contains "Edit the commit message file".

### AC-005 (traces to BC-7.03.095 postcondition 2 — clean file → Continue)

When `detect_attribution(file_contents)` returns `None` (no attribution patterns in
the file), the plugin returns `HookResult::Continue`.

**Falsifiable test:** `test_BC_7_03_095_clean_file_returns_continue`: mock `read_file` returns `Ok("Fix login bug\n")` → `HookResult::Continue`.

### AC-006 (traces to BC-7.03.095 postcondition 3 and invariant 1 — read_file failure → Continue + telemetry, NEVER Block)

On any `vsdd::read_file` error variant — `CAPABILITY_DENIED`, `NOT_FOUND`,
`OUTPUT_TOO_LARGE` (file > 65536 bytes), or `IO_ERROR` — the plugin MUST return
`HookResult::Continue`. A telemetry event of type `internal.read_file_failed`
MUST be emitted with fields: `plugin_name`, `arm`, `error_variant`, `path`. Under
no circumstances may a read failure produce `HookResult::Block`.

**Falsifiable test (four cases):**
- `test_BC_7_03_095_capability_denied_yields_continue`: mock `read_file` returns `Err(CAPABILITY_DENIED)` → `HookResult::Continue`.
- `test_BC_7_03_095_not_found_yields_continue`: mock returns `Err(NOT_FOUND)` → `HookResult::Continue`.
- `test_BC_7_03_095_output_too_large_yields_continue`: mock returns `Err(OUTPUT_TOO_LARGE)` → `HookResult::Continue`.
- `test_BC_7_03_095_io_error_yields_continue`: mock returns `Err(IO_ERROR)` → `HookResult::Continue`.

### AC-007 (traces to BC-7.03.095 postcondition 4 and precondition 5 — stdin sentinel → Continue, no read_file call)

When `parse_message_file_path` returns `Some("-")` (the stdin sentinel), the plugin
returns `HookResult::Continue` WITHOUT calling `vsdd::read_file`. Stdin-sourced
commit messages cannot be read by `vsdd::read_file`; BC-7.03.094's PostToolUse arm
provides the safety net.

**Falsifiable test:** `test_BC_7_03_095_stdin_sentinel_yields_continue_without_read`: `parse_message_file_path("git commit -F -")` → `Some("-")`; plugin returns `Continue`; `read_file` mock NOT called.

### AC-008 (traces to BC-7.03.095 invariant 5 — single read_file call per hook invocation)

The file-read arm issues at most ONE `vsdd::read_file` call per PreToolUse firing.
Only the first `-F`/`--file` flag path is parsed and read; if multiple such flags
appear in the command (malformed — git accepts only one), the first path wins.
There is no retry logic.

**Falsifiable test:** `test_BC_7_03_095_single_read_file_call_on_match`: assert mock call count == 1 on a command with `–F /tmp/msg.txt`.

### AC-009 (traces to BC-7.03.095 invariant 6 — relative paths passed as-is to read_file)

When the parsed path is relative (does not begin with `/`), it is passed to
`vsdd::read_file` as-is. The dispatcher resolves relative paths against the `cwd`
field from the hook envelope if available; the plugin does NOT perform path
canonicalization or `cwd` resolution itself. This BC is enforced by the absence of
any `std::fs::canonicalize` or `cwd` lookup in the plugin source.

**Falsifiable test:** `test_BC_7_03_095_relative_path_passed_as_is`: `parse_message_file_path("git commit -F relative/path/msg.txt")` → `Some("relative/path/msg.txt")`; mock `read_file("relative/path/msg.txt", 65536)` called with exact relative path.

### AC-010 (traces to BC-7.03.095 — existing BC-7.03.001..005 contract tests pass unchanged)

All existing `cargo test -p block-ai-attribution` tests pass unchanged after
S-16.02 lands. The command-string detection path (PreToolUse `-m` / heredoc) is
unmodified. The short-circuit (INV-2) ensures the new file-read arm adds zero
overhead to the existing path.

**Falsifiable test:** `cargo test -p block-ai-attribution contract_clean_passes contract_pattern_matching contract_hook_macro` GREEN after S-16.02 implementation.

**Architecture note:** The `-F` file-read arm is added to the EXISTING

## Architecture Compliance Rules

Derived from BC-7.03.095, BC-7.03.001 v1.3, SS-07-hook-bash.md v1.3, and SS-04-plugin-ecosystem.md v1.4:

1. **Short-circuit must run before file-read (BC-7.03.095 INV-2, HIGH PRIORITY):** `detect_attribution(command)` on the command string MUST be called FIRST. If it returns `Some`, block immediately and do NOT call `vsdd::read_file`. This preserves zero host-call cost for the `-m`/heredoc common case.
2. **Fallback-to-Continue on any read failure (BC-7.03.095 INV-1, HIGH PRIORITY):** The file-read arm MUST return `HookResult::Continue` on ANY `vsdd::read_file` error. Read errors are handled at the call site into a telemetry + Continue path. This invariant has the same severity as the detection path — a false-positive block on a legitimate commit with a clean message file is a P0 reliability regression.
3. **Narrow path_allow is mandatory (BC-7.03.095 INV-3):** The `read_file` capability MUST use the four-entry narrow allowlist. Widening to `["**/*"]` is a BC violation. The narrow allowlist covers all realistic commit message file locations.
4. **max_bytes fixed at 65536 (BC-7.03.095 INV-4):** Compile-time constant — NOT runtime-configurable.
5. **Single read_file call per invocation (BC-7.03.095 INV-5):** First `-F`/`--file` path wins. No retry. No multi-file scan.
6. **Plugin does NOT canonicalize paths (BC-7.03.095 INV-6):** Relative paths passed as-is to `vsdd::read_file`. The dispatcher's host function owns resolution semantics. Do not add `std::fs::canonicalize` or `cwd` lookup to the plugin.
7. **Three flag forms (BC-7.03.095 INV-7):** `-F <path>`, `--file=<path>`, and `--file <path>` must all be parsed correctly. A PR that handles only one form is a BC violation.
8. **Event code is `"ai_attribution_file_arm"` (BC-7.03.095 PC-1):** Unified code for all file-arm detections regardless of which attribution pattern matched.
9. **No `std::fs` calls (forbidden dependency rule):** The plugin uses `vsdd::read_file` (host fn) ONLY. Direct filesystem access via `std::fs` bypasses capability gating and is a security violation.
