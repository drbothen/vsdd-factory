# S-14.06: BC-4.12.001 Invariant Numbering Gap Cleanup (INV2→INV4)

**Epic:** E-14 — Engine Discipline Pass-2 — Process-Gap Remediation
**Delivery order:** SIXTH in E-14. Can be delivered independently.
**Depends on:** (none)
**Blocks:** (none)
**Priority:** P2 (cosmetic spec quality; not a runtime bug)

## Narrative

- **As a** spec author or implementer reading BC-4.12.001 (ContextResolver Lifecycle Invariant)
- **I want** the invariant numbering to be sequential without gaps
- **So that** cross-references to specific invariant numbers are unambiguous and
  adversarial review does not surface the same cosmetic finding in future passes

## Background

F-P2-009 (pass-2 adversarial finding) identified a numbering gap in BC-4.12.001:
the invariant list jumps from INV2 to INV4, skipping INV3. This is a cosmetic
spec quality issue — not a runtime bug. The gap likely arose when an invariant
was added or removed during spec authoring without renumbering the remaining items.

This finding was triaged in D-378 as FOLLOW-UP STORY (not inline-closed) because
renumbering requires care: any existing cross-references to INV3/INV4/INV5 in
other specs, test files, or story docs must be updated simultaneously.

## Acceptance Criteria

### AC-001: Audit current invariant numbers in BC-4.12.001
- Read BC-4.12.001 and document all current INV-N labels
- Identify the gap (expected: INV1, INV2, [gap], INV4, ...)
- Confirm no invariant is logically missing — the gap is cosmetic numbering only

### AC-002: Renumber invariants to be sequential
- Renumber so INVs are INV1, INV2, INV3, INV4, ... without gaps
- Bump BC-4.12.001 version by minor (e.g., v1.x → v1.(x+1))
- Add changelog entry: "F-P6-002/S-14.06: invariant numbering gap INV2→INV4 corrected to sequential INV1→INV3→..."

### AC-003: Propagate renumbering to all cross-references
- Search all `.factory/` files for references to old INV numbers (e.g., "BC-4.12.001 INV3", "INV4")
- Update any cross-references found (BCs, ADRs, VPs, story specs, session docs)
- Document any cross-references updated in commit message

### AC-004: Verify no content loss
- The substance of each invariant (its actual rule) is unchanged — only the label changes
- Any adversarial pass after delivery must not find the same gap
