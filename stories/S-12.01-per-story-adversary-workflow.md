# S-12.01: Per-Story Adversary Workflow — Step 4.5, Scope Contract, AGENT.md Reconciliation

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** SECOND in cycle (after S-13.01). Pure markdown/YAML edits — no
new Rust infrastructure.
**Depends on:** S-13.01 (path discipline must be live before this story's `.factory/` writes occur)
**Blocks:** S-12.02 (hook needs the workflow's state file schema as its input contract)
**Priority:** P1

## Narrative

- **As a** vsdd-factory orchestrator dispatching story delivery
- **I want** the per-story-delivery workflow to formally require adversary convergence
  (minimum 3 NITPICK_ONLY passes) between Implement and Record demos, with the adversary
  agent's scope explicitly bounded to story diff + spec + anchored BCs
- **So that** no story ships its demo evidence until its adversary gate has been
  mechanically cleared, and the adversary cannot drift into cross-story or architectural
  findings that are out of scope for per-story review

## Previous Story Intelligence

N/A — first story in E-12. This story establishes the workflow contract that
S-12.02's WASM hook will mechanically enforce.

Key design decisions from F1 and F2 that F4 must honor:

1. **Adversary owns convergence assessment:** Per BC-5.39.001 postcondition 8, the
   adversary agent (defined in `agents/adversary.md`) is the convergence assessor.
   No separate convergence-checker agent is introduced.
2. **State file is per-story JSON:** Path is
   `.factory/cycles/<cycle-id>/<story-id>/adversary-convergence-state.json`. This is
   the schema contract that S-12.02's WASM hook reads. The schema must be exactly as
   specified in BC-5.39.001 postcondition 2 — no variation.
3. **Deferred findings do not reset passes_clean:** BC-5.39.001 postcondition 7 and
   invariant 1 are unambiguous. Cross-story and architectural findings go to
   `deferred_findings` and do not affect the convergence counter.
4. **Lobster step naming:** Name the step `adversary-convergence` (matches BC-5.39.001
   architecture anchor). The backup step is `backup-adversary-convergence` (follows the
   `backup-*` naming pattern for state-manager backup steps).

## Architecture Compliance Rules

Extracted from BC-5.39.001, BC-5.39.002, and ADR-017:

1. **Step 4.5 sequencing is strict:** `per-story-delivery.md` must make it impossible
   to interpret as optional. Language must be directive ("MUST NOT execute") not advisory.
2. **State file path is not hardcoded in this story:** The workflow document should
   reference the schema and path pattern (with `<cycle-id>` and `<story-id>` as
   template variables) not a literal path with real IDs.
3. **Lobster `depends_on` chain:** The dependency chain must be:
   `[implement → backup-implement] → [adversary-convergence → backup-adversary-convergence] → [record-demos]`.
   Breaking this chain serializes in the wrong order.
4. **Gate 3 narrowing must not over-restrict:** The narrowing applies to finding scope,
   not to the gate's ability to block. Gate 3 still blocks if deferred findings are
   serious enough. The narrowing is about what findings are appropriate to raise, not
   about removing blocking capability.
5. **Forbidden dependencies:** This story involves no Rust crates. If F4 implementation
   introduces a new Rust file, it is out of scope — escalate.
