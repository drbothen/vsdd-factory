# S-12.05: hook-sdk Resolver-Authoring Extensions

**Epic:** E-12 — Engine Governance — Per-Story Adversarial Convergence Discipline
**Delivery order:** Batch 2 (parallel with S-12.03, after S-12.06).
**Depends on:** S-12.06 (HOST_ABI doc must specify the surface this story exposes)
**Blocks:** S-12.07 (`vsdd-context-resolvers` uses the SDK to author the `WaveContextResolver`)
**Priority:** P0

## Narrative

- **As a** resolver plugin author
- **I want** the `vsdd-hook-sdk` to provide a `resolver-authoring` feature flag with `ResolverInput`, `ResolverOutput`, `ResolverError` types and a `#[resolver]` proc-macro
- **So that** I can write a Rust function `fn resolve_impl(input: ResolverInput) -> ResolverOutput` and have the SDK generate the correct WASM export (`resolve(input_ptr: i32, input_len: i32) -> i64`) — exactly the same ergonomics as the existing `#[hook]` macro, but for the resolver ABI

## Acceptance Criteria

### AC-001 (traces to BC-4.12.002 postcondition 4 — `RESOLVER_ABI_VERSION = 1` constant defined)

`crates/hook-sdk/src/resolver.rs` defines `pub const RESOLVER_ABI_VERSION: u32 = 1`. The constant is re-exported from `crates/hook-sdk/src/lib.rs` under the `resolver-authoring` feature flag.

**Falsifiable test:** Rust unit test: `assert_eq!(vsdd_hook_sdk::RESOLVER_ABI_VERSION, 1u32)`. Compile succeeds only with `features = ["resolver-authoring"]` in `Cargo.toml`.

### AC-002 (traces to BC-4.12.002 postcondition 2 — `ResolverInput` type shape is exact)

`crates/hook-sdk/src/resolver.rs` defines:
```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub struct ResolverInput {
    pub event_type: String,
    pub hook_event_name: String,
    pub agent_type: Option<String>,
    pub project_dir: String,
    pub plugin_config: serde_json::Value,
}
```
The field names are EXACTLY as listed — no renaming, no omission, no additions.

**Falsifiable test:** Rust unit test: JSON round-trip — serialize a `ResolverInput` to JSON; deserialize; assert all fields are preserved with correct types. Assert `agent_type: null` serializes as JSON `null` (not omitted) when `None`.

### AC-003 (traces to BC-4.12.002 postcondition 3 — `ResolverOutput` type shape is exact)

`crates/hook-sdk/src/resolver.rs` defines:
```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub struct ResolverOutput {
    pub key: String,
    pub value: Option<serde_json::Value>,
}
```
Field names are EXACT. `value: None` serializes as `"value": null` in JSON.

**Falsifiable test:** Rust unit test: `ResolverOutput { key: "foo".into(), value: None }` serialized to JSON equals `{"key":"foo","value":null}`. `ResolverOutput { key: "foo".into(), value: Some(json!({"x": 1})) }` serialized equals `{"key":"foo","value":{"x":1}}`.

### AC-004 (traces to BC-4.12.002 invariant 1 — `ResolverInput`/`ResolverOutput` are DISTINCT from `HookPayload`/`HookResult`)

`HookPayload` and `ResolverInput` are separate structs with no `From` / `Into` / `Deref` relationships between them. `HookResult` and `ResolverOutput` are separate structs with no implicit conversions. A function accepting `HookPayload` cannot accept `ResolverInput` without an explicit field-by-field construction.

**Falsifiable test:** Compile-error test (trybuild): attempt to pass a `ResolverInput` where a `HookPayload` is expected; assert the Rust compiler rejects it with a type mismatch. This is a negative compile test.

### AC-005 (traces to BC-4.12.002 postcondition 5 — `#[resolver]` macro generates correct WASM export)

`crates/hook-sdk-macros/src/lib.rs` exports a `#[resolver]` proc-macro. When applied to a function with signature `fn resolve_impl(input: ResolverInput) -> ResolverOutput`, the macro generates a `pub extern "C" fn resolve(input_ptr: i32, input_len: i32) -> i64` export that: (a) reads input bytes from WASM linear memory at `input_ptr..input_ptr+input_len`; (b) deserializes to `ResolverInput`; (c) calls `resolve_impl`; (d) serializes `ResolverOutput` to JSON bytes; (e) writes bytes to WASM memory allocation; (f) returns packed `((ptr as i64) << 32) | (len as i64)`.

**Falsifiable test:** trybuild test: compile a minimal resolver crate using `#[resolver]` on a valid `fn resolve_impl`; assert the crate compiles without error and the generated `resolve` export is present in the WASM exports (`wasm-objdump -x` or equivalent check).

### AC-006 (traces to BC-4.12.002 postcondition 5 — `#[resolver]` macro rejects wrong signature)

If `#[resolver]` is applied to a function with the wrong signature (e.g., wrong return type, wrong parameter type, or multiple parameters), the macro emits a compile-time error with a clear message. It does NOT silently generate wrong WASM.

**Falsifiable test:** trybuild test: apply `#[resolver]` to `fn resolve_impl() -> String` (wrong signature); assert compilation fails with an error message containing "resolve_impl" and "ResolverOutput" (or equivalent diagnostic).

### AC-007 (traces to BC-4.12.002 postcondition 8 — `resolver-authoring` feature flag is opt-in)

`crates/hook-sdk/Cargo.toml` defines a `resolver-authoring` feature that gates the `resolver` module and all its exports. Hook crates that do NOT enable `features = ["resolver-authoring"]` MUST NOT have access to `ResolverInput`, `ResolverOutput`, `RESOLVER_ABI_VERSION`, or `#[resolver]`. The existing hook authoring types (`HookPayload`, `HookResult`, `#[hook]`) are unaffected by the feature flag — they are always available.

**Falsifiable test:** Compile test: a crate with `vsdd-hook-sdk` as dependency (without `resolver-authoring` feature) that attempts `use vsdd_hook_sdk::resolver::ResolverInput;` fails to compile. A crate WITH `resolver-authoring` feature succeeds.

### AC-008 (traces to BC-4.12.002 invariant 2 — `RESOLVER_ABI_VERSION` versioned independently from `HOST_ABI_VERSION`)

Changing `RESOLVER_ABI_VERSION` does not require changing `HOST_ABI_VERSION`, and vice versa. The two constants are defined in different modules (`resolver.rs` vs the existing ABI constant location) and have separate changelog tracking. The story documents this independence in the `crates/hook-sdk/CHANGELOG.md` entry (or equivalent version annotation).

**Falsifiable test:** Rust unit test: `assert_ne!(vsdd_hook_sdk::HOST_ABI_VERSION, vsdd_hook_sdk::RESOLVER_ABI_VERSION as u32)` — no, this would be wrong if both happen to equal 1. Instead: assert both constants EXIST and are independently defined (check separate declaration sites in source).

Actually: **Falsifiable test:** Code review check — `RESOLVER_ABI_VERSION` and `HOST_ABI_VERSION` must be defined in separate source locations (`resolver.rs` vs existing constant location). A grep confirms their distinct definition lines. No alias, no re-export that would couple them.

### AC-009 (traces to BC-4.12.002 postconditions 1–9 — existing Hook trait and `#[hook]` macro are UNCHANGED)

`crates/hook-sdk/src/lib.rs`, `crates/hook-sdk/src/host.rs`, and `crates/hook-sdk-macros/src/lib.rs` existing exports for hook authoring (`HookPayload`, `HookResult`, `HookContext`, `#[hook]`, `HOST_ABI_VERSION`) are unchanged. All existing hook plugin crates compile without modification after this story's changes.

**Falsifiable test:** Run `cargo build -p validate-per-story-adversary-convergence` (the existing hook) without modifying its `Cargo.toml`; assert it compiles successfully (zero errors, zero new warnings attributable to this story).

### AC-010 (traces to VP-075 — serde round-trip determinism)

`ResolverInput` and `ResolverOutput` serde round-trips are deterministic: serialize to JSON and back yields an equal struct. A proptest (100 trials) generates arbitrary `ResolverInput` instances and verifies `deserialize(serialize(input)) == input`.

**Falsifiable test:** `cargo test -p hook-sdk --features resolver-authoring resolver_serde_roundtrip` passes (proptest, 100 trials).

## Architecture Compliance Rules

Derived from BC-4.12.002 invariants and ADR-018 OD-3:

1. **Distinct types — no aliasing:** `type ResolverInput = HookPayload;` is FORBIDDEN. These are separate structs with separate fields. The `ResolverInput.plugin_config` field may share the `serde_json::Value` type with `HookPayload`, but the structs are not related by type aliases or inheritance.
2. **`resolver-authoring` feature is optional:** Running `cargo build -p hook-sdk` WITHOUT the `resolver-authoring` feature must produce zero compile errors. The feature is an opt-in; it does not affect the hook authoring surface.
3. **No `#[hook]` macro changes:** The existing `#[hook]` macro in `hook-sdk-macros/src/lib.rs` must not be modified. Add only the new `resolver_macro.rs` module and export.
4. **`fn resolve_impl` signature is exact per BC-4.12.002 PC5:** Input is `ResolverInput` (not a reference); output is `ResolverOutput` (not `Result`). The macro must emit a compile error for any other signature.
5. **Packed i64 return encoding:** `((ptr as i64) << 32) | (len as i64)` — upper 32 bits are `ptr`, lower 32 bits are `len`. Match BC-4.12.002 PC1 exactly.

## Dev Notes

**`#[resolver]` macro implementation guide:** Read `crates/hook-sdk-macros/src/lib.rs` for the existing `#[hook]` macro. The `#[resolver]` macro is structurally identical except: (a) input type is `ResolverInput` (not `HookPayload`); (b) output type is `ResolverOutput` (not `HookResult`); (c) return encoding is packed i64 (not an exit code); (d) generated export name is `resolve` (not `on_hook`). If the hook macro uses `_start` / WASI conventions, the resolver macro uses the ABI-level `resolve(input_ptr, input_len) -> i64` convention directly.

**Trybuild test patterns:** Create a `tests/ui/` directory (following the trybuild convention). Each `.rs` file in `tests/ui/` that should fail gets a matching `.stderr` file with the expected error. For AC-004 and AC-006, the expected error is `type mismatch` and `expected ResolverOutput` respectively.
