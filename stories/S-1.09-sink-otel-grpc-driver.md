## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md Â§CAP-003.

This story implements the `otel-grpc` sink driver â€” the second sink type in beta.1 scope and the primary integration path with the local Grafana/Loki/Prometheus stack. CAP-003 specifies: "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`." S-1.09 delivers the OTLP/gRPC sink driver that makes events visible in Grafana, which is the observable user outcome CAP-003 guarantees.

## Narrative
- **As a** factory operator running Grafana/Loki/Prometheus
- **I want to** have events forwarded to an OTel collector over OTLP/gRPC
- **So that** the existing local observability stack receives vsdd-factory telemetry without any code changes

## Goal

Implement the `otel-grpc` sink type â€” forwards events to an OTel collector
over OTLP/gRPC. This is the second sink type in beta.1 scope, feeding the
existing local Grafana/Loki/Prometheus stack directly.

## Acceptance Criteria

- [ ] `crates/sink-otel-grpc` implements the `Sink` trait
  (traces to BC-3.01.009 postcondition 1: registry construction succeeds with unreachable endpoint; worker connects lazily)
- [ ] Events batched and exported via OTLP/gRPC to the configured endpoint
  (traces to BC-3.03.001 postcondition 1: size threshold >= 100 records triggers flush; postcondition 2: elapsed >= 5000ms triggers flush; postcondition 3: either trigger is independent)
- [ ] Configurable endpoint (`OTLP_ENDPOINT` or registry config field)
  (traces to BC-3.03.003 postcondition 1: endpoint validated eagerly at constructor; channel built lazily in worker)
- [ ] Configurable batch size and flush interval
  (traces to BC-3.03.001 postcondition 1/2: `DEFAULT_BATCH_SIZE=100` and `DEFAULT_BATCH_INTERVAL_MS=5000` are the independent thresholds)
- [ ] Handles gRPC errors: reconnect on transient errors
  (traces to BC-3.03.002 postcondition 1: drop gRPC client on error; rebuild on next batch â€” self-healing transport)
- [ ] `flush()` exports the current batch and waits for ack
  (traces to BC-3.03.006 postcondition 1: flush is a synchronous oneshot round-trip; producer blocks on `rx.blocking_recv()` until worker acks)
- [ ] Integration test with a mock OTLP receiver verifies batch delivery
  (traces to BC-3.05.003 postcondition 1: 10 events arrive with correct attribute mapping)
- [ ] `submit()` non-blocking; batching happens on dedicated background thread
  (traces to BC-3.03.005 postcondition 1: producer-side `submit` is fully non-blocking via `try_send`; overflow increments `queue_full_count`)
- [ ] Worker thread owns its own current_thread tokio runtime on a dedicated OS thread
  (traces to BC-3.03.004 postcondition 1: worker thread owns its own `current_thread` tokio runtime on a dedicated OS thread)
- [ ] Shutdown drains and joins the worker thread; post-shutdown `submit` is rejected
  (traces to BC-3.03.007 postcondition 1: shutdown drains and joins the worker thread; idempotent post-`accepts` rejection)
- [ ] SinkEvent `type` field maps to OTLP LogRecord body; `ts_epoch` maps to `time_unix_nano`
  (traces to BC-3.03.008 postcondition 1: `type` â†’ body, `ts_epoch` â†’ `time_unix_nano`; BC-3.03.011 postcondition 1: missing `type` â†’ empty body; BC-3.03.012 postcondition 1: missing `ts_epoch` â†’ `time_unix_nano=0`)
- [ ] Non-reserved fields flatten to OTLP `attributes` as string key-value pairs
  (traces to BC-3.03.009 postcondition 1: non-reserved fields flatten to OTLP `LogRecord.attributes`)
- [ ] Nested JSON values serialized to string in OTLP attributes
  (traces to BC-3.03.010 postcondition 1: nested JSON values are stringified in OTLP attributes)
- [ ] Resource attributes merge operator config overrides over auto-detected defaults
  (traces to BC-3.03.013 postcondition 1: operator overrides win over auto-detected defaults)
