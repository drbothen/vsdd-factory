# S-4.02: sink-datadog driver

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.07, S-4.08
**Blocked by:** S-1.08 (Sink trait), S-4.01 (HttpSink pattern reused)
**Estimated effort:** M

## Capability Anchor Justification

Anchoring to **CAP-023: "Ship advanced observability sinks (HTTP, Datadog, Honeycomb)"** per capabilities.md §CAP-023.

CAP-023 states: "Planned sink drivers (S-4.1/4.2/4.3) allow direct event forwarding without a local OTel collector." S-4.02 (Datadog sink) is explicitly cited as S-4.2 in CAP-023. The outcome CAP-023 promises — "an operator forwards events to Datadog with zero local disk footprint" — is directly what this story delivers.

## Narrative
- **As a** factory operator using Datadog
- **I want to** configure a Datadog sink with my API key
- **So that** all factory events appear in Datadog Logs without any additional infrastructure

## Goal

Implement a Datadog-specific sink that sends events via Datadog's Logs
Intake API. Handles the specific payload schema, authentication, regional
endpoints, and rate limiting that Datadog requires.

## Acceptance Criteria

- [ ] `crates/sink-datadog` implements the `Sink` trait
  (traces to BC-3.01.001 postcondition 1: Sink integrates with SinkRegistry no-op base machinery)
- [ ] Sends events to Datadog Logs Intake API (`/api/v2/logs`)
  (traces to BC-3.01.007 postcondition 1: non-blocking submit via `try_send`; batching on background thread)
- [ ] Auth via `DD-API-KEY` header from config
  (traces to BC-3.06.005 postcondition 1: SinkConfigCommon defaults govern config parsing; DD-API-KEY is a required field)
- [ ] Supports regional endpoints (us1, eu1, etc.)
  (traces to BC-3.06.005 postcondition 1: config deserializes via SinkConfigCommon pattern)
- [ ] Maps factory event fields to Datadog schema (ddsource, ddtags, service, message)
  (traces to BC-3.01.007 postcondition 1: events pass through the queue before transformation and delivery)
- [ ] Respects Datadog's 5MB payload limit per batch
  (traces to BC-3.01.008 postcondition 1: failures recorded into `SinkFailure` list when batch limit exceeded)
- [ ] Integration test with mock Datadog endpoint
  (traces to BC-3.01.001 postcondition 1: registry-level operations succeed end-to-end)
