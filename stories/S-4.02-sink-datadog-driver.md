# S-4.02: sink-datadog driver

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.07, S-4.08
**Blocked by:** S-1.08 (Sink trait), S-4.01 (HttpSink pattern reused)
**Estimated effort:** M

## Narrative
- **As a** factory operator using Datadog
- **I want to** configure a Datadog sink with my API key
- **So that** all factory events appear in Datadog Logs without any additional infrastructure

## Goal

Implement a Datadog-specific sink that sends events via Datadog's Logs
Intake API. Handles the specific payload schema, authentication, regional
endpoints, and rate limiting that Datadog requires.

## Acceptance criteria

- [ ] `crates/sink-datadog` implements the `Sink` trait
- [ ] Sends events to Datadog Logs Intake API (`/api/v2/logs`)
- [ ] Auth via `DD-API-KEY` header from config
- [ ] Supports regional endpoints (us1, eu1, etc.)
- [ ] Maps factory event fields to Datadog schema (ddsource, ddtags, service, message)
- [ ] Respects Datadog's 5MB payload limit per batch
- [ ] Integration test with mock Datadog endpoint
