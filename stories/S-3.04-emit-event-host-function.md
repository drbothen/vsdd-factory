## Narrative
- **As a** WASM plugin
- **I want to** call `emit_event()` as a host function and have it flow through the same sink pipeline as dispatcher events
- **So that** plugin-emitted events and dispatcher-emitted events share a single, consistent observability pipeline

## Goal

Complete the `emit_event` host function's implementation (S-1.04 stubs
it) so WASM plugins can emit events through the same pipeline that
sinks consume. Also retire the bash `emit-event` binary's role as a
standalone emitter — all emission flows through the host function.

## Capability Anchor Justification

Anchoring to **CAP-003: "Stream observability events to multiple configurable sinks"** per capabilities.md §CAP-003.

> "The dispatcher fans out every internal event to all enabled sink drivers (file, OTel gRPC; HTTP/Datadog/Honeycomb planned for rc.1). Sinks are independently configured via `observability-config.toml`."

This story completes the `emit_event` host function so plugin-emitted events flow through the same sink pipeline as dispatcher-originated events. The story's central behavior — routing plugin events to configured sinks with enrichment and reserved-field filtering — is CAP-003's "fans out every internal event to all enabled sink drivers" applied to the plugin emission path. The wire format decode functions (`decode_fields`, `decode_single_pair`, truncation guards) are the low-level protocol that makes this fan-out reliable. CAP-002 is the secondary anchor (plugins interacting with the host), but the primary behavioral concern is sink routing parity, which is CAP-003.
