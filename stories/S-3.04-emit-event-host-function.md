# S-3.04: emit_event as host function refactor

**Epic:** E-3 — WASM Port: High-Value Hooks
**Phase:** Phase 3 — High-value port to native WASM
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-3.01, S-3.02, S-3.03 (port stories consume the host fn)
**Blocked by:** S-1.04 (host function scaffolding)
**Estimated effort:** S

> **Status note:** As of v1.0.0-beta.4, the host fn IS implemented (`host/emit_event.rs` +
> `invoke.rs::emit_event`), but bash hooks still call the old `bin/emit-event` shell tool.
> The refactor to fully retire `bin/emit-event` and route all emission through the host fn
> is outstanding.

## Narrative
- **As a** WASM plugin
- **I want to** call `emit_event()` as a host function and have it flow through the same sink pipeline as dispatcher events
- **So that** plugin-emitted events and dispatcher-emitted events share a single, consistent observability pipeline

## Goal

Complete the `emit_event` host function's implementation (S-1.04 stubs
it) so WASM plugins can emit events through the same pipeline that
sinks consume. Also retire the bash `emit-event` binary's role as a
standalone emitter — all emission flows through the host function.

## Acceptance criteria

- [ ] `emit_event()` host function routes events to the configured sinks (not just internal log)
- [ ] Events emitted by plugins appear in the same JSONL files and OTel streams as dispatcher events
- [ ] `bin/emit-event` shell tool deprecated; existing callers migrated to use dispatcher
- [ ] Plugin-emitted events enriched with `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`
- [ ] Integration test: WASM plugin emits event; verify in file sink output
