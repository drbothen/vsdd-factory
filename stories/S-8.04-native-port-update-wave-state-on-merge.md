## Narrative

- **As a** vsdd-factory operator managing wave-based story delivery
- **I want** update-wave-state-on-merge to execute as a native WASM plugin
- **So that** wave-state.yaml is automatically updated when pr-manager completes a
  merge, on all platforms, without a python3 runtime dependency

## Goal

Port `plugins/vsdd-factory/hooks/update-wave-state-on-merge.sh` to a native Rust WASM
crate at `crates/hook-plugins/update-wave-state-on-merge/`. The hook fires on
SubagentStop for pr-manager agents only, and only when a merge completion signal is
detected in the result text. When triggered, it:

1. Extracts the story ID from the result (pattern `S-[0-9]+\.[0-9]+` or `STORY-[0-9]+`).
2. Reads `.factory/wave-state.yaml` (if present) via `vsdd_hook_sdk::host::read_file`.
3. Finds the wave containing the story ID in its `stories` list.
4. Appends the story ID to `stories_merged` (if not already present).
5. Writes the updated YAML back to disk via `vsdd_hook_sdk::host::write_file` (requires
   SDK extension — see BLOCKER DISCLOSURE above).
6. If `stories == stories_merged` (all stories merged), sets `gate_status = "pending"`
   and `next_gate_required = wave_name`; writes a stderr reminder.
7. Emits `vsdd_hook_sdk::host::emit_event` with `type=hook.action
   hook=update-wave-state-on-merge matcher=SubagentStop reason=wave_merge_recorded`
   plus story_id, wave, total, merged, gate_transitioned fields.
8. Always exits 0.

The `on_error = "continue"` registry setting means dispatcher errors do not block;
preserved in the migrated entry. The `timeout_ms = 10000` (longer than other SubagentStop
hooks) reflects the YAML read/write time; preserved.

**Regex precedence bats test cases (concrete input/expected-output, required for deterministic tests):**
- Input `"STEP_COMPLETE: step=8 status=ok"` → expected match (triggers merge path).
- Input `"STEP_COMPLETE: step=8 status=merged"` → expected match (triggers merge path).
- Input `"STEP_COMPLETE: step=8 status=squash_merge"` → expected match (triggers merge path).
- Input `"merge_complete"` → expected match (matches the `merged` alternation arm; port-as-is behavior per OQ-001).
- Input `"squash_complete"` → expected match (matches the `squash.*merge` alternation arm).
- Input `"STEP_COMPLETE: step=9 status=ok"` → expected NO match (step number wrong; exit 0, no mutation).
- Input `"STEP_COMPLETE: step=8 status=failed"` → expected NO match (status not in alternation).

These concrete cases confirm the port-as-is behavior including the precedence quirk where `merged` alone matches. The bats test file must enumerate each case explicitly with a fixture JSON input and assert the correct YAML mutation (or lack thereof).

Bats tests: non-pm agent → no mutation; pm agent with no merge signal → no mutation.

### AC-005 (traces to BC-7.03.086 postcondition 1)
When all stories in a wave are now in stories_merged (stories == stories_merged set):
WASM plugin sets `wave_data.gate_status = "pending"` and
`state.next_gate_required = wave_name`. Writes stderr message:
`"update-wave-state-on-merge: all stories in <wave> merged. gate_status → pending.\n  Run the wave integration gate before starting the next wave.\n"`.
Emits event with `gate_transitioned=true`. Exits 0.

**YAML gate_status four-case truth table** (per F-S804-P1-007 — YAML semantic
trichotomy):

| Case | YAML representation | Serde result (Option<String> + serde(default)) | Action |
|------|---------------------|----------------------------------------------|--------|
| 1 | key absent | `None` | trigger flip to "pending" |
| 2 | key present, YAML null/~ | `None` | trigger flip to "pending" |
| 3 | key present, "not_started" string | `Some("not_started")` | trigger flip to "pending" |
| 4 | key present, any other string | `Some("...")` | DO NOT flip |

Rust struct constraint: `gate_status` field MUST use `Option<String>` with
`#[serde(default)]`, NOT bare `String`. This ensures cases 1 and 2 are handled
identically (both deserialize as `None`).

### AC-006 (traces to BC-7.03.084 postcondition 1 + BC-7.03.085 postcondition 1)
Bats parity tests cover: (a) pm agent + merge signal + story found in wave → YAML
updated, event emitted, exit 0; (b) all stories merged → gate_status=pending, stderr
reminder; (c) story already in stories_merged → no duplicate append, no event emitted,
exit 0 (matches bash source: duplicate path exits silently with no emit call);
(d) wave-state.yaml absent → graceful exit 0, no YAML write;
(e) story_id not found in any wave → exit 0, no mutation;
(f) gate_status=null explicit (`wave-state-gate-null.yaml` fixture) → treated as None,

**Fixture content — `tests/integration/hooks/fixtures/wave-state-gate-null.yaml`** (verbatim):

```yaml
waves:
  - wave: "wave-14"
    stories: ["S-8.04"]

```
The `gate_status: ~` and `current_wave: null` values exercise AC-005 truth table case 2 (key present, YAML null/~) → deserializes as `None` → triggers gate flip to "pending". This is distinct from case 1 (key absent) and case 3 ("not_started"). The bats fixture file must use this exact YAML form; a zero-byte file or a file with only `gate: ~` (missing other keys) exercises a different code path and must be a separate fixture if needed.
