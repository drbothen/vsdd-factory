## Capability Anchor Justification

**Primary anchor:** CAP-022 ("Port hook plugins from bash to native WASM") per
`domain-spec/capabilities.md` §CAP-022. S-8.05 ports `validate-pr-review-posted.sh`
to native WASM, eliminating legacy-bash-adapter indirection for this pr-reviewer
completion-validation hook.

**Stretch-Anchor Disclosure:** S-8.05 anchors to CAP-022 but spans two subsystems:
- **SS-07 Hook Bash Layer** owns `validate-pr-review-posted.sh` and its BC-7.04.040–044
  behavioral contracts (note: BC-7.04.xxx family, not BC-7.03.xxx — this hook is in a
  different BC sub-family). SS-07 is the source-of-truth for this hook's behavior.
- **SS-01 Hook Dispatcher Core** owns the hooks-registry.toml routing and the
  host fn surface used by the native crate.

Both subsystem anchors verified against ARCH-INDEX Subsystem Registry:
- SS-01 Hook Dispatcher Core: canonical name confirmed.
- SS-07 Hook Bash Layer: canonical name confirmed.
(Per POLICY 5 anchor justification requirement + POLICY 6 architecture-is-source-of-truth.)

**Complexity note (Low — 3 pts):** This hook has three independent validation checks
(pr-review.md written, no gh pr comment fallback, formal review posted) plus agent
scoping, aggregating any failures into a single error output. Complexity is low because
each check is a simple regex on the result text. The hooks.json registry shows
`on_error = "continue"` but the hook itself exits 2 when checks fail — this asymmetry
is expected and must be preserved (see BC-7.04.040 registry binding).

## Narrative

- **As a** vsdd-factory operator running pr-reviewer subagents
- **I want** validate-pr-review-posted to execute as a native WASM plugin
- **So that** the pr-reviewer completion guard (requiring pr-review.md + formal GitHub
  review verdict) runs on all platforms without git-bash, with deterministic error
  aggregation and exit 2 blocking behavior

## Goal

Port `plugins/vsdd-factory/hooks/validate-pr-review-posted.sh` to a native Rust WASM
crate at `crates/hook-plugins/validate-pr-review-posted/`. The hook fires for
SubagentStop on pr-reviewer and pr-review-triage agents only, and runs three
independent validation checks against the result text:

1. **Check 1 (pr-review.md written):** result must match
   `pr-review\.md|wrote.*review|review.*written|Write.*pr-review`.
2. **Check 2 (no gh pr comment fallback):** result must NOT match `gh pr comment`
   (using gh pr comment instead of gh pr review is a known failure mode).
3. **Check 3 (formal review posted):** result must match
   `gh pr review|pr review.*posted|review.*posted.*GitHub|APPROVE|REQUEST_CHANGES`.
   Additionally, if check 3 passes but no verdict detected (`approve` or
   `request-changes`), add a "no verdict" error.

If any errors accumulated, the hook emits `host::emit_event` with
`type=hook.block hook=validate-pr-review-posted matcher=SubagentStop
reason=pr_review_not_posted subagent=<agent>`, writes formatted error list to stderr,
writes the remediation instruction, and exits 2.

If all checks pass, exits 0 with no output.

`on_error = "continue"` (registry) means dispatcher crashes do not block; preserved.

### AC-003 (traces to BC-7.04.041 postcondition 1)
WASM plugin exits 0 immediately (no checks run) when agent_type does not match
`pr.reviewer`, `pr_reviewer`, or `pr.review.triage` pattern. Bats test:

### AC-004 (traces to BC-7.04.042 postcondition 1)
Check 1: if result does NOT match `pr-review\.md|wrote.*review|review.*written|Write.*pr-review`,
accumulate error message: `"pr-review.md may not have been written to .factory/code-delivery/"`.

### AC-005 (traces to BC-7.04.043 postcondition 1)
Check 2: if result matches `gh pr comment`, accumulate error message:
`"Used 'gh pr comment' instead of 'gh pr review' — findings won't show as a formal review verdict"`.

### AC-006 (traces to BC-7.04.044 postcondition 1)
Check 3a: if result does NOT match
`gh pr review|pr review.*posted|review.*posted.*GitHub|APPROVE|REQUEST_CHANGES`,
accumulate error: `"No evidence that a formal GitHub review was posted via 'gh pr review'"`.
Check 3b: if result DOES match `gh pr review` but does NOT match
`approve|request-changes|APPROVE|REQUEST_CHANGES`, accumulate error:
`"Review posted but no verdict (--approve or --request-changes) detected"`.

If ERRORS list is non-empty: emit `host::emit_event` (type=hook.block,
reason=pr_review_not_posted, subagent); write `"PR REVIEW POSTING INCOMPLETE:\n"` +
each error as `"  - <error>\n"` to stderr; write remediation instruction; exit 2.
If ERRORS empty: exit 0.

### AC-007 (traces to BC-7.04.040 postcondition 1 + BC-7.04.042/043/044 postcondition 1)
Bats parity tests cover: (a) all three checks pass → exit 0; (b) pr-review.md not
written → exit 2 with error 1; (c) gh pr comment used → exit 2 with error 2;
(d) no formal review → exit 2 with error 3; (e) review posted, no verdict → exit 2
with error 3b; (f) multiple checks fail → exit 2 with all accumulated errors;
(g) non-pr-reviewer agent → exit 0. All bats tests pass.

### AC-008 (traces to BC-7.04.040 postcondition 1)
`bin/emit-event` calls replaced with `host::emit_event`. No reference to `bin/emit-event`
in `crates/hook-plugins/validate-pr-review-posted/`. `bin/emit-event` binary NOT
removed (E-8 D-10). WASM warm-invocation perf measured; within 20% of S-8.00 baseline.
