## Goal

Port `plugins/vsdd-factory/hooks/validate-pr-review-posted.sh` to a native Rust WASM
crate at `crates/hook-plugins/validate-pr-review-posted/`. The hook fires for
SubagentStop on pr-reviewer and pr-review-triage agents only, and runs three
independent validation checks against the result text:

1. **Check 1 (pr-review.md written):** result must match
   `pr-review\.md|wrote.*review|review.*written|Write.*pr-review`.
   Use Rust raw string literal: `r"pr-review\.md|wrote.*review|review.*written|Write.*pr-review"`.
   In a raw string, `\.` is literal backslash-dot, which in the regex crate matches
   a literal `.` character (correct). All patterns are case-sensitive (Rust regex default).
2. **Check 2 (no gh pr comment fallback):** result must NOT match `gh pr comment`
   (using gh pr comment instead of gh pr review is a known failure mode).
3. **Check 3 (formal review posted):** result must match
   `gh pr review|pr review.*posted|review.*posted.*GitHub|APPROVE|REQUEST_CHANGES`.
   Additionally, if check 3 passes but no verdict detected (`approve` or
   `request-changes`), add a "no verdict" error (Check 3b).

**Check 3b semantics (bash parity, line 54):** Check 3b fires when the result text
matches the literal token `gh pr review` (Check 3a passed this branch) AND the result
does NOT match `approve|request-changes|APPROVE|REQUEST_CHANGES`. This gate tests for
the `gh pr review` token specifically — it is NOT a re-test of Check 3a's full
disjunctive pattern. If result matched Check 3a only via `APPROVE` or `pr review.*posted`
(not via the `gh pr review` literal), Check 3b does not fire.

If any errors accumulated, the hook calls `host::emit_event` with
`event_type=hook.block hook=validate-pr-review-posted matcher=SubagentStop
reason=pr_review_not_posted subagent=<agent>`, writes formatted error list to stderr,
writes the remediation instructions (3 lines), and exits 2.

If all checks pass, exits 0 with no output.

`on_error = "continue"` (registry) means dispatcher crashes do not block; preserved.

The `envelope` is accessed exclusively through BC-2.02.012 typed projection on the
`payload: HookPayload` parameter — no raw `envelope.get(...)` calls. Agent identity and
result content are resolved via the canonical fallback chain expressions specified in
BC-2.02.012 Postconditions 5 and 6.

### AC-003 (traces to BC-7.04.041 postcondition 1; traces to BC-2.02.012 postcondition 5)
WASM plugin exits 0 immediately (no checks run) when agent_type does not match
any of the canonical agent patterns. Canonical form is **hyphenated** (matching
bash source glob arms): `pr-reviewer`, `pr_reviewer`, `pr-review-triage`.
Matching mechanism: substring containment check (equivalent to bash `case
"$AGENT_TYPE" in *pr-reviewer*|*pr_reviewer*|*pr-review-triage*)`). Not a glob
or regex — the Rust implementation must use `.contains("pr-reviewer") ||
.contains("pr_reviewer") || .contains("pr-review-triage")`. Dotted variants
(e.g., `pr.reviewer`) are NOT canonical and must NOT be matched. Bats test:
non-pr-reviewer agent (e.g., `agent_type = "code-reviewer"`) → exit 0, no stderr.

Agent identity MUST be resolved via BC-2.02.012 Postcondition 5 canonical fallback chain:
```rust
let agent: &str = payload.agent_type.as_deref()
    .or(payload.subagent_name.as_deref())
    .unwrap_or("unknown");
```
This mirrors `validate-pr-review-posted.sh:21`: `AGENT=$(echo "$INPUT" | jq -r '.agent_type // .subagent_name // "unknown"')`.
The jq `//` null-as-advance semantics are provided for free by serde_json's
`Option<String>` + `#[serde(default)]` deserialization: JSON null deserializes to
`None`, so `Option::or` advances correctly for both absent and null fields
(BC-2.02.012 Invariant 3).

Agent dual-fallback sub-cases for T-5 / AC-003 bats coverage:
- **(g.1)** `agent_type` present, `subagent_name` absent — primary arm of fallback chain used.
- **(g.2)** `agent_type` absent, `subagent_name` present — fallback arm exercised; value
  must still apply all containment checks if the resolved identifier matches a canonical
  pattern.

### AC-006 (traces to BC-7.04.044 postcondition 1)
Check 3a: if result does NOT match
`gh pr review|pr review.*posted|review.*posted.*GitHub|APPROVE|REQUEST_CHANGES`,
accumulate error: `"No evidence that a formal GitHub review was posted via 'gh pr review'"`.

Check 3b: if result matches the literal substring `gh pr review` but does NOT match
`approve|request-changes|APPROVE|REQUEST_CHANGES`, accumulate error:
`"Review posted but no verdict (--approve or --request-changes) detected"`.
Note: this gate tests for the literal `gh pr review` token only — it is NOT a re-test
of Check 3a's full disjunctive pattern. The two conditions are independent: Check 3a
can pass via APPROVE/REQUEST_CHANGES without triggering Check 3b.

If ERRORS list is non-empty: call `host::emit_event` (event_type=hook.block,
reason=pr_review_not_posted, subagent); write `"PR REVIEW POSTING INCOMPLETE:\n"` +
each error as `"  - <error>\n"` to stderr; write remediation instructions (3 lines,
verbatim from bash source lines 65-67):
```
  pr-reviewer MUST: (1) write pr-review.md, (2) spawn github-ops with
  'gh pr review --approve' or 'gh pr review --request-changes --body-file'.
  NEVER use 'gh pr comment' for review verdicts.
```
Then exit 2.
If ERRORS empty: exit 0.
