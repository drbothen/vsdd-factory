# S-15.05: De-flake TC-4/5/7/9 async timing tests via internal-log event observation (TD #67 closure)

**Epic:** E-15 — Plugin Async Semantics — Registry-Layer Partition (single-shot delivery)
**Delivery order:** Fifth story in E-15; TD cleanup following S-15.01 merge.
**Depends on:** S-15.01 (async partition runtime + internal-log event emission must be live)
**Blocks:** none
**Priority:** P2 (test reliability; removes `#[ignore]` from four BC-coverage tests)

## Narrative

- **As a** maintainer of the dispatcher's async-partition test suite
- **I want to** have TC-4, TC-5, TC-7, TC-9 pass deterministically on CI by observing internal-log events instead of wall-clock measurements
- **So that** the BC-1.14.001 fire-and-forget contract is continuously validated rather than masked by `#[ignore]` annotations

## Background

TC-4, TC-5, TC-7, and TC-9 in `crates/factory-dispatcher/tests/full_stack_plugin_invocation.rs`
are currently annotated `#[ignore]` because each asserts a timing-dependent property (wall-clock
window, elapsed_ms sub-millisecond rounding, drain-window upper bound, JoinHandle race) that is
correct in intent but fragile in mechanism on CI. These tests cover BC-1.14.001 Postcondition 4
(fire-and-forget async execution does not block the dispatcher's return to Claude Code) and
EC-012 (terminal events emitted for every async invocation), which are behavioral contracts with
continuous-coverage obligations.

**Architect analysis — per-test root causes:**

| Test | Current mechanism | Fragility | Strategy B fix |
|------|------------------|-----------|----------------|
| TC-4 | `wall_ms <= max_allowed` (wall-clock window) | System load spikes cause false failures | Assert `plugin.invoked` event present in internal log |
| TC-5 | `assert!(*elapsed_ms > 0, ...)` (sub-ms rounding) | Sub-millisecond completion on fast hosts rounds to 0 | Assert `plugin.completed` event with `exit_code == 0` in internal log |
| TC-7 | `total_elapsed <= test_drain + 200ms` (drain-window upper bound) | CI latency variability exceeds 200ms tolerance | Assert `plugin.invoked` event for the async plugin in internal log (drop drain-window timing) |
| TC-9 | `tokio::time::timeout(5s, handle)` JoinHandle wait | JoinHandle race; handle may be dropped before poll | Poll internal log for `plugin.timeout` event, bounded at 8s |

The fix replaces wall-clock assertions with internal-log event observation throughout. A new
`wait_for_log_event` async test helper is added for the polling tests (TC-5, TC-7, TC-9). TC-4
uses a simpler direct log read (the `plugin.invoked` event is synchronously produced at plugin
invocation and is available without polling).

ADR-019 established the async/sync partition at the registry layer; this story hardens the
test coverage of that partition without changing any production code.

## Acceptance Criteria

### AC-001 (traces to BC-1.14.001 Postcondition 4 — async hook does not block dispatcher)
TC-4 (`test_e2e_BC_1_14_001_async_hook_doesnt_block_dispatcher`): `#[ignore]` removed;
wall-clock assertion `wall_ms <= max_allowed` removed; replaced with assertion that
`plugin.invoked` event for `session-start-telemetry` appears in the internal log.

- **Test:** `test_e2e_BC_1_14_001_async_hook_doesnt_block_dispatcher()`

### AC-002 (traces to BC-1.14.001 Postcondition 4 — async output reaches sink when fast)
TC-5 (`test_e2e_BC_1_14_001_async_hook_output_reaches_sink_when_fast`): `#[ignore]` removed;
`assert!(*elapsed_ms > 0, ...)` removed; replaced with assertion that `plugin.completed` event
for `session-start-telemetry` appears in the internal log with `exit_code == 0`.

- **Test:** `test_e2e_BC_1_14_001_async_hook_output_reaches_sink_when_fast()`

### AC-003 (traces to BC-1.14.001 Postcondition 4 — mixed sync/async partition)
TC-7 (`test_e2e_BC_1_14_001_mixed_sync_async_partition_timing`): `#[ignore]` removed;
drain-window assertion `total_elapsed <= test_drain + 200ms` removed; replaced with assertion
that `plugin.invoked` event for the async plugin appears in the internal log.

- **Test:** `test_e2e_BC_1_14_001_mixed_sync_async_partition_timing()`

### AC-004 (traces to BC-3.08.001 Event 4 — plugin.timeout emitted on async timeout)
TC-9 (`test_e2e_BC_1_14_001_async_timeout_emits_plugin_timeout_event`): `#[ignore]` removed;
`tokio::time::timeout(5s, handle)` JoinHandle wait replaced with internal-log poll for
`plugin.timeout` event with `plugin_name == "async-hang-plugin"`, bounded at 8s.

- **Test:** `test_e2e_BC_1_14_001_async_timeout_emits_plugin_timeout_event()`

### AC-005 (traces to BC-1.14.001 EC-012 — terminal events emitted for every async invocation)
A `wait_for_log_event` async test helper exists in the test file (or a shared helpers module),
callable by TC-5, TC-7, TC-9. TC-4 may use a simpler direct read (event is synchronously
produced at plugin invocation).

- **Test:** `wait_for_log_event()` helper verified by TC-5/TC-7/TC-9 passing.

### AC-006 (traces to BC-1.14.001 Postcondition 4 — deterministic CI pass)
All four tests pass `cargo test -p factory-dispatcher` (without `--ignored`) on a clean CI run.

### AC-007 (traces to BC-1.14.001 — latency_canary.rs untouched)
`latency_canary.rs` `#[ignore]` annotations are NOT touched. Those tests are structurally
separate (require `--release` builds and have a different ignore rationale).

### AC-008 (traces to BC-1.14.001 — non-targeted tests unaffected)
No regression in non-ignored tests in `full_stack_plugin_invocation.rs`; TC-1/2/3/6/8 pass
without modification.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `wait_for_log_event` poll times out before event appears (8s bound) | Test fails with clear message: `timeout waiting for event <type> in internal log`; no JoinHandle race |
| EC-002 | `InternalLog::write` BufWriter not flushed before poll reads | If flush semantics require explicit drain, `wait_for_log_event` uses 100ms interval polling to handle delayed flush; implementer must inspect `InternalLog::write` at T-1 to determine if explicit flush gate or polling suffices |
| EC-003 | TC-4 `plugin.invoked` event not yet written when direct read occurs | If synchronous invocation is not strictly synchronous in all CI environments, escalate TC-4 to use `wait_for_log_event` instead of direct read (implementer's decision at T-7) |
| EC-004 | `async-hang-plugin` name differs from registered name in hooks-registry.toml | TC-9 poll filter uses `plugin_name == "async-hang-plugin"`; implementer must verify name matches registry at T-1 |
| EC-005 | Non-targeted tests TC-1/2/3/6/8 fail after helper module addition | Run full `cargo test -p factory-dispatcher` (not just `--ignored`) to catch any module import conflict |
