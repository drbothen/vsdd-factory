## Narrative

- **As a** state-manager or agent writing a STATE.md update in a fix burst
- **I want to** receive an immediate PostToolUse block when the STATE.md content I just wrote
  contains a banner line-count that diverges from the actual line count, a missing dual-margin
  form, or a trajectory-tail with the wrong number of components
- **So that** the recurring class of STATE.md structural violations discovered by the adversary
  in every F5 engine-discipline pass (banner wc-l drift, absent dual-margin, trajectory-tail
  cardinality errors) is caught mechanically at write time rather than discovered by the
  adversary N bursts later

## Why This Exists

The F5 engine-discipline cycle (passes 25–74+) generated a persistent class of adversary findings
around STATE.md structural properties. D-421(c) through D-442(d) codified the banner line-count
discipline across 7 sub-clauses. D-446(c) required the dual-margin form. D-433(e)+D-439(c)+D-451(c)
established trajectory-tail LENGTH=4 as a hard invariant. D-432(b) codified the canonical
trajectory-tail form.

Manual adherence never held — every cycle produced at least one STATE.md structural finding.
This hook makes the entire structural-correctness failure class impossible at PostToolUse time:
any write to STATE.md that violates banner wc-l, dual-margin, or trajectory-tail form is
blocked before the edit becomes the accepted state.

## Goal

Deliver a production-grade WASM hook (`validate-state-structure`) that PostToolUse-blocks any
Edit/Write to `STATE.md` containing:
- A banner line-count claim that diverges from the actual newline count in the written content
- A banner that lacks the canonical dual-margin form (both margins must be present)
- A trajectory-tail string that does not contain exactly 4 `→(\d+)` matches (LENGTH=4)

Phase 1 covers these three structural classes. Phase 2 (M3, S-15.10) will extend the same crate
with additional STATE.md validations.

## Behavioral Contract

### BC-5.39.005: validate-state-structure Phase 1 hook MUST block on banner line-count drift, dual-margin absence, and trajectory-tail cardinality violations in STATE.md

**Subsystem:** SS-05 Pipeline Orchestration (Engine Governance capability class within Pipeline Orchestration subsystem — per-story adversary workflow and discipline gate hooks)
**Capability:** E-12 Engine Governance (automation sub-capability: STATE.md structural completeness)
**BC ID:** BC-5.39.005
**File:** `.factory/specs/behavioral-contracts/ss-05/BC-5.39.005.md`

#### Preconditions

1. A PostToolUse Edit/Write event has fired on a file whose `file_name` path component is exactly
   `STATE.md` (path-component-strict matching via `is_state_md_target` —
   `Path::new(file_path).file_name() == Some("STATE.md")` — NOT suffix-`ends_with`).
2. The dispatcher has invoked the `validate-state-structure` WASM plugin with the write payload.
3. The STATE.md file content is available from the write payload or via `host::read_file`.
4. The dispatcher provides `host::read_file` capability with the registry-declared `path_allow = [".factory"]` scope. The hook calls `host::read_file(file_path, max_bytes=524_288, timeout_ms=2000)` per BC-5.39.005 Precondition 4. The whole-hook registry-level timeout is `timeout_ms = 5000` (hooks-registry.toml, distinct from the per-call file-read timeout of 2000 ms). `max_bytes=524_288` (512 KiB) was set by the F-P5-002 implementer fix.

#### Postconditions

1. If ALL of the following hold, the hook emits `HookResult::Continue` (pass):
   - The banner line-count claim matches the actual newline count in the full content string.
   - The banner contains both margin values in the canonical dual-margin form per D-446(c).
   - The trajectory-tail string contains exactly 4 `→(\d+)` matches.
2. If the banner line-count claim diverges from the actual count, the hook emits
   `HookResult::BlockWithFix` naming the claimed count, the actual count, and the instruction
   to reconcile the banner.
3. If the dual-margin form is absent from the banner, the hook emits `HookResult::BlockWithFix`
   with a message naming the missing margin form and citing D-446(c).
4. If the trajectory-tail does not contain exactly 4 `→(\d+)` matches, the hook emits
   `HookResult::BlockWithFix` naming the actual match count, the required count (4), and
   citing D-433(e)+D-439(c).
5. Multiple violations in one write produce a single `HookResult::BlockWithFix` message
   enumerating all violations.
6. If `host::read_file` returns an error (timeout, capability denied, file not found), the hook
   emits `HookResult::Continue` and logs a warning via `host::log_warn` — a read failure is
   NOT treated as a structural violation (fail-open).

#### Invariants

1. The hook NEVER writes to any file. It is read-only at the host level.
2. The hook fires on PostToolUse only — it never prevents a write; it signals AFTER the write
   completes.
3. The `is_state_md_target` function implements path-component-strict matching:
   `Path::new(file_path).file_name() == Some("STATE.md")`. Using `ends_with("STATE.md")`
   on the raw path string is FORBIDDEN (false-positive on `xSTATE.md`).
4. Line-count computation uses newline character (`\n`) counting on the raw content string.
5. The dual-margin check requires BOTH margin expressions to be present in the banner block.
6. The trajectory-tail check uses global regex `→(\d+)` applied to the trajectory-tail line;
   count of matches must equal 4.
7. All `host::read_file` calls are fail-open: read errors produce Continue + log_warn, not Block.
8. All byte-index slice expressions where multi-byte UTF-8 input is possible (banner narrative
   text may contain em-dash, en-dash, NBSP) MUST use `is_char_boundary()` guards before slicing.

#### Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Banner claims 399 lines but file has 400 newlines | BlockWithFix naming discrepancy |
| EC-002 | Banner claims 400 lines and file has 400 newlines | Continue for line-count check |
| EC-003 | Banner has only one margin (soft-target present, hard-target absent) | BlockWithFix citing missing dual-margin per D-446(c) |
| EC-004 | Trajectory-tail has 3 components (`→9→9→9`) | BlockWithFix: 3 components vs required 4 |
| EC-005 | Trajectory-tail has 5 components (`→9→9→9→9→9`) | BlockWithFix: 5 components vs required 4 |
| EC-006 | All three validations fail | Single BlockWithFix enumerating all 3 violations |
| EC-007 | `host::read_file` returns HostError::CapabilityDenied | Continue + log_warn; fail-open |
| EC-008 | Path is `/some/dir/xSTATE.md` | Continue (is_state_md_target returns false) |
| EC-009 | Banner narrative contains em-dash (multi-byte UTF-8) | No panic; is_char_boundary() guard |
| EC-010 | No trajectory-tail line found in content | BlockWithFix: no trajectory-tail found |

## Implementation Detail

### Hook Logic Specification

The implementer MUST implement `src/lib.rs` with the following logical structure. No line-number
citations (TD-VSDD-091 anti-volatile-pin); function names and behavioral anchors only.

#### In-plugin file-path guard

```
pub fn is_state_md_target(file_path: &str) -> bool {
    std::path::Path::new(file_path)
        .file_name()
        .and_then(|n| n.to_str())
        == Some("STATE.md")
}

fn run_hook(payload: &Payload) -> HookResult {
    if !is_state_md_target(&payload.file_path) {
        return HookResult::Continue;
    }
    // ... proceed with validation
}
```

This path-component-strict guard is the canonical Q5/Q6 pattern. Using `Path::file_name()`
prevents false-positive fires on paths like `/some/dir/xSTATE.md` where `.ends_with("STATE.md")`
would incorrectly match. The registry entry does NOT contain `file_pattern`.

Unit tests MUST include:
- `test_is_state_md_target_positive()` — asserts `.factory/STATE.md` → true
- `test_xstate_md_is_not_target()` — asserts `/some/dir/xSTATE.md` → false (false-positive guard)

#### Banner line-count extraction and comparison

```
fn extract_banner_line_count(content: &str) -> Option<usize>
```

Scan content for the canonical banner line pattern (e.g., a line containing `(\d+) lines` in
the known banner position). Extract the integer. Guard all byte-index slices with `is_char_boundary()`
where banner narrative text (which may contain em-dash or other multi-byte characters) is involved.

```
fn count_newlines(content: &str) -> usize
```

Count `\n` occurrences in the full content string. This is the authoritative actual line count.
The comparison is: `extracted_integer == count_newlines(content)`.

```
fn validate_banner_wc(content: &str) -> Option<Violation>
```

Calls both functions. If `extract_banner_line_count` returns None: return Some(Violation) naming
"no banner line-count found." If claimed != actual: return Some(Violation) with `cited_raw` set
to the raw banner line text (`.trim_end().to_string()`).

#### Dual-margin check

```
fn validate_dual_margin(content: &str) -> Option<Violation>
```

Scan the banner block for both canonical margin patterns. The dual-margin form per D-446(c)
requires two distinct margin expressions: one relative to the soft-target ceiling and one relative
to the hard-target ceiling. Both must be present. If either is absent: return Some(Violation) with
`cited_raw` set to the raw banner block text that was found. Implementer reads the canonical form
from the actual D-446(c) codification in decision-log.md or STATE.md samples before authoring
the regex anchor — do NOT invent the margin form from memory.

#### Trajectory-tail cardinality check

```
fn validate_trajectory_tail(content: &str) -> Option<Violation>
```

Find the trajectory-tail line in STATE.md content (the line containing the `→N→N→N→N` pattern).
Apply global regex `→(\d+)` and collect all matches. If no matches found: return Some(Violation)
naming "no trajectory-tail found; expected `→N→N→N→N` form per D-432(b)." If match_count != 4:
return Some(Violation) with `cited_raw` set to the raw tail line and violation message naming
`match_count` vs required 4 and citing D-433(e)+D-439(c).

#### is_char_boundary() safety

For all expressions of the form `&content[start..end]` where `start` or `end` are computed from
`str::find()`, `rfind()`, byte arithmetic on indices from a scanned position, or any other
non-trivially-ASCII source: the implementer MUST guard:

```rust
if content.is_char_boundary(start) && content.is_char_boundary(end) {
    &content[start..end]
} else {
    // handle gracefully: return None or use a safe default
}
```

Alternatively, use `chars()` iteration or `char_indices()` iteration to avoid byte-index
arithmetic entirely. Document the choice explicitly in the code with a comment.

Unit tests for UTF-8 safety:
- `test_banner_extract_with_em_dash()` — content with `—` (U+2014, 3 bytes) in banner narrative
- `test_banner_extract_with_en_dash()` — content with `–` (U+2013, 3 bytes) in banner narrative

#### Block message format

When `violations.len() > 0`:

```
HookResult::BlockWithFix {
    message: format!(
        "validate-state-structure: {} violation(s) in STATE.md:\n{}",
        violations.len(),
        violations.iter().map(|v| format!("  - {}", v.description)).collect::<Vec<_>>().join("\n")
    ),
    fix_hint: "Fix the violations listed above before re-writing STATE.md.",
}
```

Individual violation descriptions:
- Line-count mismatch: `"banner claims {claimed} lines but actual line count is {actual} — reconcile banner (D-421(c)); cited: '{cited_raw}'"`
- Dual-margin absent: `"dual-margin form absent from banner: both margins must be present per D-446(c); cited: '{cited_raw}'"`
- Tail LENGTH mismatch: `"trajectory-tail has {actual_count} →N components but required LENGTH=4 per D-433(e)+D-439(c); cited: '{cited_raw}'"`
- No tail found: `"no trajectory-tail found in STATE.md content; expected '→N→N→N→N' canonical form per D-432(b)"`

### Performance Considerations

At most one `host::read_file` call is made per PostToolUse event: the STATE.md content itself.
The implementation reads STATE.md content via `host::read_file` to ensure filesystem authority
over potentially-stale payload content.

STATE.md at typical size is 200-600 lines, well within the `524_288`-byte per-call cap set by the F-P5-002 fix. The three validations are all single-pass linear scans requiring no more than 3 regex applications and 1 newline count. Per-call file-read timeout is 2000 ms (BC-5.39.005 Precondition 4); the registry-level whole-hook timeout is 5000 ms (hooks-registry.toml). Fuel budget: minimal, well within the 5000 ms hook timeout.
