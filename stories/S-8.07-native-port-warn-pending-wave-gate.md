**Complexity note (Low — 3 pts):** warn-pending-wave-gate is a read-only advisory hook.
It: (a) checks for `.factory/wave-state.yaml`, exits 0 if absent; (b) parses
wave-state.yaml to find waves with `gate_status: pending`; (c) if any pending waves
found, emits `hook.block severity=warn` via `host::emit_event` and writes a WAVE GATE
REMINDER to stderr; (d) always exits 0.

The primary porting consideration: the bash source uses `python3` for YAML parsing.
The WASM port uses a pure Rust YAML library (serde_yaml 0.9.34) instead of a subprocess.
This is a straightforward substitution — the YAML structure is simple
(`waves.{name}.gate_status`). No new host fns are required (read_file + emit_event).

## Narrative

- **As a** vsdd-factory operator ending Claude Code sessions
- **I want** warn-pending-wave-gate to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** the session-end wave gate reminder runs on all platforms (including Windows
  without python3), with lower latency, no adapter indirection, and no python3 dependency

## Goal

Port `plugins/vsdd-factory/hooks/warn-pending-wave-gate.sh` to a native Rust WASM crate
at `crates/hook-plugins/warn-pending-wave-gate/`. The hook fires on the Stop event,
reads `.factory/wave-state.yaml` via `host::read_file` (exits 0 if absent), parses all
waves looking for `gate_status: pending`, and if any are found: calls `host::emit_event`
with `event_type="hook.block"` and fields slice `[("hook", "warn-pending-wave-gate"),
("matcher", "Stop"), ("reason", "pending_wave_gate_at_session_end"), ("severity", "warn"),
("pending_waves", "<comma-joined>")]`, then writes a WAVE GATE REMINDER block to stderr
listing each pending wave name and invocation hints. Always exits 0.

Behavior parity is the sole requirement (E-8 D-2). The key change vs bash: `python3`
YAML parsing is replaced with `serde_yaml 0.9.34` (pure Rust, no subprocess). The
`_emit()` function calls are replaced with `host::emit_event`. All output format, field
names, and stderr message text must be identical to the bash version.

**Behavior delta vs bash version:** On hosts lacking python3, the bash version silently
exits 0 (python3-absent early-exit guard). The WASM version reads the YAML and emits
warnings normally — python3 is never needed. This is a deliberate behavior change (the
warning was previously suppressed by a transitive Python dependency); the WASM port
restores the intended semantics. E-8 D-2 "behavior-parity-only" is satisfied modulo this
documented exception. v1.1 BC candidate registered for cross-platform parity delta.

### AC-004 (traces to BC-7.03.091 postcondition 2)
Early-exit paths are silent (no stderr, no emit_event call, exit 0). Each path maps to
the corresponding bash branch for parity verification:

- (a) wave-state.yaml absent: exit 0, no output.
  Bash branch: `[[ ! -f .factory/wave-state.yaml ]] && exit 0`
- (b) YAML parse fails or `waves` key absent (null/empty): exit 0, no output.
  Bash branch: `if not state or 'waves' not in state: sys.exit(0)`
  WASM: use `wave_data.get("waves")` returning `None` → exit 0.
- (c) No wave has `gate_status == "pending"`: exit 0, no output.
  Bash branch: jq `select(.gate_status == "pending")` returning empty.
  WASM: `data.get("gate_status").and_then(Value::as_str) == Some("pending")` — use
  explicit string-value extraction (not strict typed deserialization) to avoid panics.

Note: python3-absent early-exit from bash is superseded. WASM always proceeds to the
YAML check; see Goal section for documented behavior delta.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | wave-state.yaml exists but `waves` key is absent from YAML | Exit 0 with no output (same as bash python3 guard: `if not state or 'waves' not in state: sys.exit(0)`) |
| EC-002 | wave-state.yaml exists but is empty (zero bytes) | YAML parse returns None/empty; exit 0 with no output |
| EC-003 | wave-state.yaml has wave entries without `gate_status` key | Wave is not `pending`; skipped in scan; no REMINDER emitted |
| EC-004 | Multiple waves pending → pending_waves field is comma-joined list | e.g., `pending_waves="W-15,W-16"` in emit_event call; stderr REMINDER lists each wave on its own line |
| EC-005 | wave-state.yaml is a valid YAML file but not the expected structure (e.g., a list instead of a map) | Parse error or type mismatch; exit 0 with no output (graceful degradation) |
| EC-006 | host::emit_event returns an error (underlying sink outage) | SDK's host::emit_event returns `()` (no Result); if the dispatcher's underlying sink has an outage, the host swallows it. The WASM plugin proceeds to stderr output regardless. Matches bash `\|\| true` semantics by SDK construction. Best-effort emission. |
| EC-007 | `waves` key present but value is null or an empty map | Exit 0 with no output (no waves to iterate). WASM: `data.get("waves")` returns Some(Value::Null) or Some(Value::Mapping(empty)) → no pending waves found → exit 0. |
| EC-008 | A wave entry where `gate_status` value is non-string (e.g., integer, bool, list) | Value is not equal to string `"pending"`; wave is skipped. WASM: `data.get("gate_status").and_then(Value::as_str)` returns None for non-string values — no match, no REMINDER. |
| EC-009 | CapabilityDenied: registry missing read_file capability declaration | host::read_file returns HostError::CapabilityDenied; WASM plugin exits 0 with no output. Graceful, matching EC-002 absent-file semantics. AC-001 verifies capability declaration is present to prevent this path. |

## Architecture Compliance Rules (MANDATORY)

| Rule | Source | Enforcement |
|------|--------|-------------|
| HOST_ABI_VERSION = 1 unchanged throughout E-8 | E-8 Goal #5; BC-2.01.003; ADR-006 | warn-pending-wave-gate uses read_file + emit_event host fns — both existing. No new host fn needed. |
| BC-anchor strategy is Option C — reuse existing BCs; no new BC-7.02.x migration family | E-8 D-2 | BC-7.03.091/092 are the anchors. No new BC unless S-8.00 audit table shows Spec-Current = N |
| bin/emit-event must NOT be removed until S-8.29 | E-8 D-10; R-8.07 | T-3 replaces bin/emit-event calls with host::emit_event in Rust. Binary at `bin/emit-event` is NOT touched |
| python3 subprocess MUST NOT be used in WASM crate | warn-pending-wave-gate.sh source analysis | exec_subprocess with python3 would require binary_allow = ["python3"] and shell_bypass_acknowledged; this defeats the purpose of the WASM port. Use serde_yaml 0.9.34 instead. |
| Dispatcher routing: native WASM plugins do NOT register in hooks.json | E-8 D-7; DRIFT-004 | AC-002: hooks.json entry absent (verification, not deletion) |
| read_file capability MUST be declared in registry | SDK sandboxing model | T-6 / AC-001: `[hooks.capabilities.read_file] path_allow = [".factory/wave-state.yaml"]` required. Missing declaration → CapabilityDenied at runtime. |
| All commits must follow Conventional Commits | BC-8.22.001 | Commit type for Rust crate: `feat(hook-plugins)`; registry: `chore(registry)`; .sh deletion: `chore(cleanup)`; workspace Cargo.toml: `chore(workspace)` |
| Forbidden dependency: warn-pending-wave-gate WASM crate must NOT depend on legacy-bash-adapter | E-8 D-10 | Build-time enforcement rule. If this module gains a dependency on legacy-bash-adapter, the build MUST fail. |
| WASM compilation target is wasm32-wasip1 (NOT wasm32-wasi) | Project-wide; wasm32-wasi deprecated | wasm32-wasi is deprecated; all hook-plugin crates target wasm32-wasip1. Verified against capture-commit-activity/Cargo.toml and CI config. |
| Use host::read_file only (not std::fs) for YAML access | WASM sandbox model | std::fs is not available in the WASM sandbox. Pure host::read_file is the only correct path for .factory/wave-state.yaml access. |
