# S-1.05: wasmtime integration + epoch/fuel enforcement

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-1.06, S-2.01, S-3.01, S-3.02, S-3.03
**Blocked by:** S-1.01, S-1.02, S-1.04
**Estimated effort:** L

## Narrative
- **As a** dispatcher operator
- **I want to** have plugins run in isolated wasmtime stores with epoch/fuel limits
- **So that** a hanging or resource-exhausting plugin cannot block or crash the dispatcher

## Goal

Wire wasmtime into the dispatcher so it can instantiate a `.wasm` plugin,
run its `_start` entry point with a stdin payload, and enforce time + fuel
limits. Each invocation gets a fresh wasmtime `Store`, its own epoch
deadline, and its own fuel cap.

## Acceptance criteria

- [ ] Shared `wasmtime::Engine` with epoch interruption + fuel consumption enabled
  _(traces to BC-1.04.001 postcondition: Engine built with epoch interruption + fuel metering + reference types enabled)_
- [ ] Dispatcher spawns an epoch-increment timer (background tokio task bumping engine epoch every 10ms)
  _(traces to BC-1.04.002 postcondition: epoch incremented every 10ms; cooperative shutdown via channel; BC-1.04.003 postcondition: timeout_ms_to_epochs rounds up)_
- [ ] `load_plugin(path) -> Result<Module>` — compiles and caches modules by path + mtime
  _(traces to BC-1.09.001 postcondition: cache key is path; invalidation is mtime-driven; BC-1.09.002 invariant: thread-safe via Mutex<HashMap>; BC-1.09.003 invariant: no eviction policy; BC-1.09.004 postcondition: NotFound/Compile/IO error variants)_
- [ ] `invoke_plugin(module, payload, limits) -> PluginResult` — creates Store, sets epoch deadline, sets fuel, pipes stdin, runs `_start`, catches traps
  _(traces to BC-1.03.010 postcondition: plugin_config spliced into HookPayload before invocation; BC-1.03.011 postcondition: WASI exit(N) maps to PluginResult::Ok with exit_code N)_
- [ ] Epoch-timeout produces `PluginResult::Timeout { elapsed_ms, fuel_consumed }`
  _(traces to BC-1.03.001 postcondition: infinite loop killed via epoch; BC-1.03.002 postcondition: tight arithmetic loop killed via fuel)_
- [ ] Fuel-exhaustion produces `PluginResult::Timeout`
  _(traces to BC-1.03.002 postcondition: fuel exhaustion variant of Timeout)_
- [ ] Panic/trap produces `PluginResult::Crashed { trap: String }`
  _(traces to BC-1.03.003 postcondition: unreachable/trap → Crashed with trap string)_
- [ ] Normal completion produces `PluginResult::Ok { exit_code: i32, result: HookResult, elapsed_ms, fuel_consumed }`
  _(traces to BC-1.03.004 postcondition: exit_code 0 + fuel_consumed recorded on success)_
- [ ] Dispatcher emits `plugin.invoked`, `plugin.completed`, `plugin.timeout`, `plugin.crashed` events
  _(traces to BC-1.03.006 postcondition: empty stderr omitted from lifecycle events.
  v1.1 BC candidate: emission of `plugin.invoked`, `plugin.completed`, `plugin.timeout`,
  `plugin.crashed` events themselves is not yet contracted in any SS-01 BC — only the
  stderr omission constraint is. Add explicit lifecycle-event-emission BC in v1.1 PRD revision.)_
- [ ] Integration test: plugin that runs normally, hangs, panics, exhausts fuel — all handled
  _(traces to BC-1.03.005 postcondition: stderr captured per plugin and truncated at 4 KiB with marker)_
- [ ] No memory leak across 10,000 invocations
  _(traces to BC-1.09.003 invariant: no eviction — entries live process lifetime, but Stores are dropped after each invocation)_

## Contracts

**`PluginResult` type (consumed by S-1.06):**

```rust
pub enum PluginResult {
    Ok { exit_code: i32, result: HookResult, elapsed_ms: u64, fuel_consumed: u64 },
    Timeout { elapsed_ms: u64, fuel_consumed: u64 },
    Crashed { trap: String },
}
```

## Capability Anchor Justification

Anchoring to **CAP-011: "Enforce fuel and epoch budgets on plugin execution"** per capabilities.md §CAP-011.

> "Every plugin invocation has a bounded fuel cap (default 10M operations) and epoch deadline (derived from `timeout_ms`). Exceeded limits produce `Timeout{Epoch}` or `Timeout{Fuel}` outcomes, never hung processes."

This story is the direct implementation of CAP-011. It creates `engine.rs` (wasmtime Engine with epoch interruption + fuel consumption enabled), the epoch ticker background task (bumping epoch every 10ms), `invoke.rs` (sets epoch deadline and fuel cap per invocation, catches traps, maps to typed `PluginResult` variants), and `plugin_loader.rs` (mtime-keyed module cache). Without this story, no WASM plugin can be invoked — CAP-002 delivery also depends on it, but CAP-011 is the primary named capability because the fuel/epoch budgeting is the story's central behavioral concern.

| BC-1.03.011 | WASI exit(N) maps to PluginResult::Ok with exit_code N | `invoke.rs` exit code extraction from WASI proc-exit |
| BC-1.04.001 | Engine builds with epoch interruption + fuel + reference types | `engine.rs` Config settings |
