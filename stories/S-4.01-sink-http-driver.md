# S-4.01: sink-http driver

**Epic:** E-4 — Observability Sinks and RC Release
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.02, S-4.03, S-4.04, S-4.07, S-4.08
**Blocked by:** S-1.08 (Sink trait)
**Estimated effort:** M

## Capability Anchor Justification

Anchoring to **CAP-023: "Ship advanced observability sinks (HTTP, Datadog, Honeycomb)"** per capabilities.md §CAP-023.

CAP-023 states: "Planned sink drivers (S-4.1/4.2/4.3) allow direct event forwarding without a local OTel collector." S-4.01 is explicitly cited in CAP-023 as the HTTP sink driver and is the base HTTP infrastructure that the Datadog and Honeycomb sinks (S-4.02/4.03) build on. This story is the primary implementation vehicle for the generic HTTP forwarding path described in CAP-023.

## Narrative
- **As a** factory operator with a custom observability backend
- **I want to** configure a generic HTTP endpoint to receive events as JSON POST batches
- **So that** I can send factory telemetry to any HTTP-capable backend without a vendor-specific sink

## Goal

Implement a generic `http` sink that batches events and POSTs them to a
user-configured endpoint. This is the "bring your own backend" option and
the base HTTP infrastructure that Datadog and Honeycomb sinks build on.

## Acceptance Criteria

- [ ] `crates/sink-http` implements the `Sink` trait
  (traces to BC-3.01.001 postcondition 1: `SinkRegistry::empty()` operations succeed as no-op — the http sink integrates with the same registry machinery)
- [ ] `enabled=false` config results in no events accepted and no HTTP calls made
  (traces to BC-3.06.005 postcondition 1: minimal `SinkConfigCommon` TOML defaults `enabled=true`; disabled variant tested as edge case)
- [ ] Unknown sink `type` field in config logs a warning and does not fail the load
  (traces to BC-3.01.002 postcondition 1: unknown sink type warns to stderr but does not fail config load)
- [ ] `schema_version != 1` in sink config is a hard error at load time
  (traces to BC-3.01.003 postcondition 1: `schema_version != 1` is a hard error)
- [ ] Events batched and POSTed as JSON array to configured URL
  (traces to BC-3.01.001 postcondition 1: Sink integrates with SinkRegistry machinery; non-blocking submit for HTTP sink is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-cross-sink-non-blocking-submit`. See v1.1 BC Candidates table.)
- [ ] Handles HTTP errors: retry on 5xx, drop on 4xx
  (traces to VP-012 postcondition: Sink Failure Affects Only That Sink — failure isolation invariant; cross-sink failure recording generalization is uncontracted — v1.1 BC candidate `BC-3.NN.NNN-cross-sink-failure-recording`. See v1.1 BC Candidates table.)
- [ ] `flush()` sends current batch synchronously
  (traces to BC-3.01.001 postcondition 1: Sink trait flush is part of registry machinery; HTTP-sink-specific flush semantics are uncontracted — v1.1 BC candidate `BC-3.NN.NNN-http-sink-flush-sync`. See v1.1 BC Candidates table.)
- [ ] Integration test with a mock HTTP server verifies batch delivery
  (traces to BC-3.01.001 postcondition 1: registry operations succeed; end-to-end batch POST verified)
- [ ] `HttpSink` struct exposed for reuse by Datadog and Honeycomb sinks
  (traces to BC-3.01.001 invariant 1: same Sink trait interface shared across all sink types)

## v1.1 BC Candidates

| Candidate ID (proposed) | Description | Source |
|---|---|---|
| BC-3.NN.NNN-http-sink-batch-post | HTTP sink POSTs events as JSON array to configured URL; response 2xx = success; 4xx = drop batch and record failure; 5xx = retry via resilience layer | Pre-emptive (Wave 2 AC scan — no BC exists for the HTTP-specific POST semantics) |
| BC-3.NN.NNN-http-sink-auth-header | Configurable auth header (`Authorization: Bearer <token>`) added to every POST request | Pre-emptive (Wave 2 AC scan — auth header config is uncontracted in existing SS-03 BCs) |
| BC-3.NN.NNN-http-sink-flush-sync | `flush()` on HTTP sink completes the current batch POST synchronously and waits for HTTP response | Pre-emptive (Wave 2 AC scan — BC-3.03.006 contracts otel-grpc flush; equivalent HTTP sink flush is uncontracted) |
| BC-3.NN.NNN-cross-sink-non-blocking-submit | Cross-sink generalization: any non-file sink's `submit()` is non-blocking via a bounded mpsc channel; overflow increments per-sink `queue_full_count` without blocking the caller | F-003 gap (BC-3.01.007 is file-sink-scoped; this generalizes the invariant to HTTP/Datadog/Honeycomb sinks) |
| BC-3.NN.NNN-cross-sink-failure-recording | Cross-sink generalization: non-file sink failures (HTTP errors, network timeouts) are recorded into a per-sink `Mutex<Vec<SinkFailure>>` accessible via `take_failures()`; no panic on failure | F-003 gap (BC-3.01.008 is file-sink-scoped; this generalizes failure recording to all sink types) |
