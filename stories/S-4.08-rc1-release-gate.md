### BC-9.01.006 Canonical Test Vector

**BC-9.01.006 — beta-shakedown window enforcement**

**Inputs:**
- Argument: `--tag <last-prerelease-tag>`
- Argument: `--days <integer-day-count, default 14>`
- Argument: `--p0-query <gh-issue-list-query-string>`

**Outputs:**
- Exit 0: window satisfied (>=days elapsed since tag with no in-flight P0 fix)
- Exit 1: window not satisfied (clock not started OR a P0 issue is currently OPEN OR clock advanced from a recent P0 issue closure-timestamp with insufficient elapsed days)
- Exit 2: tag not found / git error
- Exit 3: gh CLI error / network failure
- stderr: human-readable reason for non-zero exit

**Postconditions:**
- PC1: When called with valid tag and >=14 days have elapsed since tag commit timestamp AND no P0 issues with `label:beta-shakedown` are open, exit 0.
- PC2: When called and any P0 issue with `label:beta-shakedown` is currently OPEN at gate-check time, OR was opened during the shakedown window AND not yet closed, exit 1 with stderr indicating clock reset to most recent issue closure (if any) or to tag commit time.
- PC3: When called with non-existent tag, exit 2.
- PC4: When invoked with `--stories <S-N.NN,S-N.NN,...>` and `--days <N>`, the script verifies that all listed merged stories have been in the merge stream for at least N days AND no P0 issues with `label:beta-shakedown` referencing those stories are open. Exit 0 = production exposure satisfied; exit 1 = either insufficient days OR P0 issue active.
- PC5 (clock-reset semantics on issue closure): When a P0 issue with `label:beta-shakedown` is CLOSED, the shakedown clock advances from the closure timestamp (NOT the original tag commit). Specifically: clock_start = max(tag_commit_time, max(closure_timestamp for all closed P0 beta-shakedown issues since tag)). If a P0 issue is CLOSED at day 13 and no other P0 issues are open, the clock effectively resets — the next 14-day window starts from issue closure, not from day 0 of original tag.
