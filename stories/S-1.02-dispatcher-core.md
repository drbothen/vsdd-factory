# S-1.02: factory-dispatcher core (stdin, TOML load, routing)

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-1.04, S-1.05, S-1.06, S-1.07, S-2.01, S-2.03, S-2.07
**Blocked by:** S-1.01
**Estimated effort:** L

## Narrative
- **As a** Claude Code hook system
- **I want to** have a dispatcher that reads stdin JSON, loads the registry, and routes to the right plugins
- **So that** hook events are delivered to matching plugins in priority order

## Goal

Implement the core skeleton of `factory-dispatcher`: read stdin JSON from
Claude Code, parse it, load `hooks-registry.toml` (cached + mtime-invalidated),
filter registered plugins by event + tool match, group them by priority tier,
and hand off to the execution machinery. This story owns the dispatcher's
I/O boundary, config loading, and routing decisions.

## Acceptance criteria

- [ ] `factory-dispatcher` reads stdin JSON and parses Claude Code hook payload
  _(traces to BC-1.02.001 postcondition 1: non-empty event_name and session_id required; BC-1.02.002 postcondition 1: both `event_name` and `hook_event_name` aliases accepted; BC-1.02.003/004/005 postconditions: all envelope types parse correctly)_
- [ ] `dispatcher_trace_id` generated per invocation (UUID v4)
  _(traces transitively to BC-1.08.004 (cwd context for trace_id flows) and
  BC-1.08.005 (plugin_root context); the UUID v4 generation itself is uncontracted —
  v1.1 BC candidate "dispatcher generates UUID v4 for dispatcher_trace_id per
  invocation". See v1.1 BC Candidates table below.)_
- [ ] `hooks-registry.toml` loaded via `toml` crate into strongly typed structs; schema_version field validated (must be 1)
  _(traces to BC-1.01.001 postcondition 1: Err(RegistryError::SchemaVersion) for unknown version; BC-1.01.002 postcondition 1: invalid tool regex fails load; BC-1.01.003 postcondition 1: unknown fields cause parse error; BC-1.01.007 postcondition: minimal registry parses; BC-1.01.011 postcondition: unknown on_error value fails; BC-1.01.014 postcondition: missing path returns NotFound)_
- [ ] Plugin filter: selects entries where `event == payload.event_name` AND (no `tool` field OR tool regex matches `payload.tool_name`)
  _(traces to BC-1.01.005 postcondition 1: event AND tool filter combined; BC-1.01.015 postcondition: empty vec returned on no match; EC-001 from BC-1.01.015: zero-match payload)_
- [ ] Matched plugins grouped by `priority` field; tiers emit in ascending priority order
  _(traces to BC-1.01.006 postcondition 1: ascending priority order, registry order preserved within tier; BC-1.01.013 postcondition: per-entry priority overrides default; BC-1.01.010 postcondition: missing priority falls through to Registry.defaults)_
- [ ] Dispatcher exits 0 by default; exit 2 only when an `on_error = "block"` plugin returned a block result
  _(traces to BC-1.08.001 postcondition 1: exit 0 on non-block errors; BC-1.08.002 postcondition 1: exit 2 iff block_intent recorded)_
- [ ] Structured error envelope on malformed stdin, missing registry, etc. — emit `internal.dispatcher_error` event and exit 0
  _(traces to BC-1.08.001 postcondition 1: registry/payload errors exit 0; BC-1.08.006 postcondition: env_view populated from process env)_
- [ ] Covered by unit tests for every parsing/routing branch
  _(traces to BC-1.01.008/009/012 postconditions: config block defaults, nested tables, capabilities block all round-trip; BC-1.08.003 invariant: current_thread runtime asserted)_

## Tasks

1. Define Rust types: `HookPayload`, `RegistryEntry`, `Registry`.
2. Implement `load_registry(path: &Path) -> Result<Registry>` with TOML parsing + schema_version validation.
3. Implement `match_plugins(registry: &Registry, payload: &HookPayload) -> Vec<&RegistryEntry>`: filter by event and tool regex, drop `enabled == false`.
4. Implement `group_by_priority(matched: Vec<&RegistryEntry>) -> Vec<Vec<&RegistryEntry>>`: sorted tiers.
5. Main entry point: parse stdin, assign `dispatcher_trace_id`, load registry, match + group, call execution layer stub.
6. Error handling: any recoverable error emits `internal.dispatcher_error` via S-1.07 writer stub.
7. Tests: unit coverage for tool-regex matching, event filtering, priority grouping, malformed registry rejection.

## Contracts

**`Registry` Rust type (consumed by S-1.04, S-1.05, S-1.06):**

```rust
pub struct RegistryEntry {
    pub name: String,
    pub event: String,
    pub tool: Option<String>,        // regex; None = all tools
    pub plugin: PathBuf,
    pub priority: u32,               // default 500
    pub enabled: bool,               // default true
    pub timeout_ms: Option<u32>,
    pub fuel_cap: Option<u64>,       // default 10_000_000
    pub on_error: Option<OnError>,
    pub capabilities: Option<Capabilities>,
}
pub enum OnError { Continue, Block }
```

**Routing decision contract:** Execution layer receives `Vec<Vec<&RegistryEntry>>` (tiers ordered ascending by priority).

## Test plan

- Unit tests for registry parse/validate

- Integration test: feed synthetic stdin + registry, verify dispatcher emits right events
- Malformed-input tests: broken JSON, missing registry, schema_version=2

## Notes

Use `serde` with `deny_unknown_fields` on registry types so typos in TOML fail loudly.

`dispatcher_trace_id` is a UUID v4 string; propagated to every emitted event.

## Capability Anchor Justification

Anchoring to **CAP-002: "Hook Claude Code tool calls with sandboxed WASM plugins"** per capabilities.md §CAP-002.

> "Every Claude Code tool invocation (Bash, Edit, Write, etc.) triggers the dispatcher, which routes to matching WASM plugins by event type and tool-regex. Plugins run in wasmtime with bounded fuel + epoch timeout."

This story is the direct implementation of the routing half of CAP-002: it reads the Claude Code hook payload (stdin JSON), loads `hooks-registry.toml`, filters by `event_name` + tool regex, groups by priority tier, and hands off to the execution layer. Without `payload.rs`, `registry.rs`, `routing.rs`, and the `main.rs` orchestration wired together, no hook event can reach any plugin — CAP-002 is entirely blocked.
