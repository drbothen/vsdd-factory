# S-2.2: hooks-registry.toml auto-generation from existing hooks.json

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.7, S-2.8
**Blocked by:** S-2.1
**Estimated effort:** S

## Goal

Auto-generate `hooks-registry.toml` from the existing `hooks/hooks.json`
so the v1.0 dispatcher starts life with full parity to the v0.79.x hook
set. Every entry in the existing hooks.json becomes a `[[hooks]]` entry
in the registry that routes through the `legacy-bash-adapter`. This is
the migration bridge — once this is done, every existing bash hook
works under the new dispatcher without manual TOML authoring.

## Acceptance criteria

- [ ] `scripts/generate-registry-from-hooks-json.sh` (or `.rs`) reads the
  current `hooks/hooks.json` and produces a corresponding
  `hooks-registry.toml`
- [ ] Each `hooks.json` entry becomes a TOML `[[hooks]]` with:
  - `name` = derived from script basename
  - `event` = Claude Code event name
  - `tool` = matcher field from hooks.json (if present)
  - `plugin` = `"hook-plugins/legacy-bash-adapter.wasm"`
  - `priority` = derived from array position (multiples of 10 to leave
    gaps for future native-WASM insertions)
  - `timeout_ms` = from hooks.json's `timeout` field
  - `on_error` = `"continue"` by default, `"block"` for known-gate hooks
    (block-ai-attribution, destructive-command-guard, red-gate, etc.)
  - `[hooks.config] script_path` = the bash script's relative path
  - `[hooks.capabilities.exec_subprocess]` = per-hook if known, else a
    conservative union
- [ ] A curated allow-list of "known-gate" hooks is encoded in the
  generator (since hooks.json today doesn't distinguish gate vs non-gate)
- [ ] Generated TOML passes registry schema validation (S-1.2)
- [ ] Dispatcher boots cleanly with the generated registry
- [ ] Every hook that runs today via hooks.json runs under the new
  registry (regression)
- [ ] Generator is idempotent; re-runs produce byte-identical output
- [ ] Generator is covered by bats tests

## Tasks

1. Inventory the ~30 existing hooks in `hooks/hooks.json`. Categorize each
   as gate (on_error=block) vs telemetry (on_error=continue).
2. Inventory the capabilities each needs (which binaries it spawns).
   Grep for `git`, `gh`, `jq`, `curl`, etc. in each hook script.
3. Write the generator. Input: path to `hooks.json`. Output:
   `hooks-registry.toml`.
4. Write a small bats test harness that runs the generator and snapshots
   the output; breaks if the generated output changes unexpectedly.
5. Run generator; commit the produced `hooks-registry.toml` as the v1.0
   starting registry.
6. Verify the dispatcher (S-1.2) boots against it without errors.
7. Document the generator script in `docs/guide/migrating-from-0.79.md`
   for operators migrating their own factories (if any externally).

## Contracts

**Gate-hooks allow-list (encoded in generator, maintained by hand):**

```
block-ai-attribution             (PreToolUse:Bash)
destructive-command-guard        (PreToolUse:Bash)
protect-secrets                  (PreToolUse:Bash, PreToolUse:Read)
red-gate                         (PreToolUse:Edit|Write)
factory-branch-guard             (PreToolUse:Edit|Write)
protect-vp                       (PreToolUse:Edit|Write)
protect-bc                       (PreToolUse:Edit|Write)
brownfield-discipline            (PreToolUse:Edit|Write)
validate-wave-gate-prerequisite  (PreToolUse:Agent)
validate-pr-merge-prerequisites  (PreToolUse:Agent)
verify-git-push                  (PreToolUse:Bash)
check-factory-commit             (PreToolUse:Bash)
validate-template-compliance     (PostToolUse:Edit|Write — blocks)
validate-input-hash              (PostToolUse:Edit|Write — blocks)
validate-factory-path-root       (PostToolUse:Edit|Write — blocks)
pr-manager-completion-guard      (SubagentStop — blocks)
handoff-validator                (SubagentStop — blocks)
```

Everything else defaults to `on_error = "continue"`.

**Priority tiering convention:**

- 10–99: PreToolUse guards (block before action)
- 100–199: PostToolUse telemetry (fire-and-forget, independent)
- 200–299: PostToolUse validators (may block; run after telemetry so
  observability sees them too)
- 900–999: legacy-bash-adapter catchall (currently unused)

Initial generator assigns priority 100 to everything; manual tuning
happens when we observe real dependencies.

**Capability assignment rules:**

```
if hook invokes git   → capabilities += "git"
if hook invokes gh    → capabilities += "gh"
if hook invokes jq    → capabilities += "jq"
always                → bash + shell_bypass_acknowledged
always                → env_allowlist = [PATH, HOME, TMPDIR, CLAUDE_PROJECT_DIR, VSDD_*]
```

## Test plan

- Generator run on current `hooks.json` produces expected TOML (snapshot
  test; commit the expected output alongside the test).
- `cargo test -p factory-dispatcher -- loads-legacy-registry` — dispatcher
  parses the generated registry without error.
- Full bats suite under the new dispatcher: all hooks that fire today
  continue to fire.
- Generator run twice produces byte-identical output.

## Notes

The generated registry is the starting point. Hand-tuning happens after:
reorder priorities based on dependencies, tighten capabilities, split
into tiers that matter.

This story is mostly mechanical translation. The interesting decisions
(gate-hooks list, capability unions) are encoded in the generator's
maintained tables. Those tables ARE the human-curated layer; the TOML
output is downstream of them.

One curveball: hooks.json has both `matcher: "Bash"` and `matcher: "Edit|Write"`
entries under the same event. The generator should flatten them into
separate TOML `[[hooks]]` entries, each with its own `tool` filter.
