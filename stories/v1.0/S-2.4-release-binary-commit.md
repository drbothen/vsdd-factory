# S-2.4: Binary commit automation in Release workflow

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.8
**Blocked by:** S-2.3 (needs the matrix artifacts)
**Estimated effort:** M

## Goal

When a release tag is pushed, the Release workflow downloads the CI
matrix's platform-specific `factory-dispatcher` binaries and WASM plugin
artifacts, commits them to `plugins/vsdd-factory/hooks/dispatcher/bin/<platform>/`
and `plugins/vsdd-factory/hook-plugins/*.wasm`, and pushes the commit
as part of the release chore. This is how Option A from Q1 actually
works end-to-end — operators cloning the repo at tag `1.0.0-beta.1` get
working binaries for every platform.

## Acceptance criteria

- [ ] Release workflow triggered by semver tag (stable + prerelease)
- [ ] Waits for `build-dispatcher` CI matrix to complete (or re-runs it)
- [ ] Downloads each platform's artifact
- [ ] Commits them to the correct paths with consistent formatting
- [ ] Commits WASM plugins to `plugins/vsdd-factory/hook-plugins/`
- [ ] Regenerates `hooks.json.<platform>` variants via the S-0.4 generator
  (so they're always in sync with the committed binaries)
- [ ] Commit message: `chore: release <version> — bundled binaries and
  plugins` (added to the existing release chore, not a separate commit)
- [ ] Push happens only if all 5 platform builds succeeded
- [ ] Signing: if we use GPG-signed commits, the bot signs the commit
  (out of scope for this story — use the same convention as other
  release commits)
- [ ] Verification step: after push, the workflow checks out the tagged
  commit on a fresh runner, verifies the binaries are executable on that
  runner's platform
- [ ] If a platform build fails, the release fails loudly; partial
  artifact commits are never published

## Tasks

1. Read existing `.github/workflows/Release.yml` structure.
2. Add a `build-artifacts` job that depends on the `build-dispatcher`
   matrix. Uses `actions/download-artifact` to pull all matrix outputs.
3. Add a `commit-artifacts` job:
   - Checkout the tagged commit
   - Copy downloaded artifacts into place
   - Run the S-0.4 generator (`scripts/generate-hooks-json.sh`)
   - Stage all changes (binaries + wasm + generated hooks.json.variants)
   - Amend the release tag's commit, OR (cleaner) create a follow-up
     commit "chore: release <version> artifacts" and retag
4. Decide: amend vs follow-up commit. Amend is simpler but rewrites
   history; follow-up + retag is cleaner but requires force-push of the
   tag. **Recommendation: follow-up commit + retag** since we've done
   tag-moves safely before and it avoids the `--amend` history confusion.
5. Verification step: after push, spawn a fresh matrix verification job
   that checks out the tag and runs `./plugins/vsdd-factory/hooks/dispatcher/bin/<platform>/factory-dispatcher --version`.
6. Update release notes in the GH Release to mention the bundled
   binaries.
7. Dry-run with a test prerelease tag (e.g., `1.0.0-beta.0` or
   `0.79.5-test.1`).

**Binary size budget:** each platform's `factory-dispatcher` should be
under 15MB stripped. Total commit added at each release: ~50-75MB.
Flag if we exceed.

## Notes

Amend-vs-follow-up-commit decision: follow-up + retag because:
1. Amending rewrites history — downstream clones that fetched the old
   tag get confused
2. The release workflow already retags successfully (we did this for
   v0.79.2 → retag after hooks.json test fix)
3. Explicit is better than implicit

Binary stripping: use `strip` (Unix) or default Rust release mode
`opt-level = "s"` via `[profile.release]` in root Cargo.toml. Balance
size vs debugging information — for released binaries, strip fully
(release builds; production failures shouldn't require GDB on the
user's machine).

Repo size growth: ~60MB per release × 52 releases/year × 3 years = 9GB
worst case. Factory-artifacts-style orphan branch migration is the
documented escape hatch if this becomes painful.
