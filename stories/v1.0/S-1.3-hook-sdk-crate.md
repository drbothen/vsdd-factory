# S-1.3: hook-sdk crate (macro, types, bindings)

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.1, S-3.1, S-3.2, S-3.3, and all future plugin authoring
**Blocked by:** S-1.1
**Estimated effort:** L

## Goal

Design and implement the `vsdd-hook-sdk` crate — the library every WASM
plugin depends on to write hooks ergonomically. Provides the `#[hook]`
proc-macro (generates the `extern "C"` entry point + JSON deserialization +
panic boundary), the `HookPayload` / `HookResult` / `Event` types, and the
host-function import declarations. Plugin authors never touch the raw WASI
ABI; they write `#[hook] fn on_hook(payload: HookPayload) -> HookResult`
and the SDK handles the rest.

## Acceptance criteria

- [ ] `vsdd-hook-sdk` compiles as a library with `wasm32-wasip1` target
- [ ] `#[hook]` proc-macro generates the `extern "C" fn _start()` (or WASI
  equivalent) that the dispatcher invokes
- [ ] The macro-generated entry point:
  - Reads stdin to a `Vec<u8>`
  - Deserializes into `HookPayload`
  - Catches panics, converts them to a structured error result
  - Invokes the user's function
  - Serializes `HookResult` and exits with the correct code
- [ ] `HookPayload` type matches the Claude Code stdin schema (event_name,
  tool_name, session_id, tool_input, tool_response)
- [ ] `HookResult` type with variants: `Continue`, `Block(reason: String)`,
  `Error(message: String)`
- [ ] Host function imports declared via `extern "C"` with a stable ABI:
  - `emit_event`, `log`, `read_file`, `exec_subprocess`, `session_id`,
    `dispatcher_trace_id`, `plugin_root`, `plugin_version`, `cwd`, `env`
- [ ] Host function signatures use length-prefixed UTF-8 strings for
  cross-language compatibility; SDK wraps them in ergonomic Rust APIs
- [ ] `HOST_ABI_VERSION` constant matches the dispatcher's (S-1.4)
- [ ] `VERSION_STRING` export convention: plugins export a function that
  returns their semver, used by the dispatcher for event enrichment
- [ ] Published to crates.io as `vsdd-hook-sdk` at `0.1.0` (in S-2.5; this
  story just makes the crate publish-ready)
- [ ] `#[hook]` works on a trivial example (returns `Continue` always)
- [ ] Unit tests cover macro expansion, payload deserialization, result
  serialization, panic boundary

## Tasks

1. Add `hook-sdk-macros` subcrate for the proc-macro (proc-macro crates
   must be standalone; the main `hook-sdk` re-exports).
2. Implement `HookPayload` struct with `serde::Deserialize`. Use
   `serde_json::Value` for `tool_input`/`tool_response` (schema varies).
3. Implement `HookResult` with `serde::Serialize`.
4. Design the host ABI for each host function:
   - Strings passed as `(ptr: u32, len: u32)` tuples (standard WASI FFI)
   - Return values via memory pointer + caller-provided length sentinel
   - Document each function's signature in a `HOST_ABI.md` file
5. Implement `extern "C"` declarations + ergonomic Rust wrappers for each
   host function.
6. Implement `#[hook]` proc-macro:
   - Takes `fn on_hook(payload: HookPayload) -> HookResult`
   - Emits `#[no_mangle] pub extern "C" fn _start() { ... }` wrapping it
   - Include panic=abort boundary so Rust panics become structured errors
7. Implement `VERSION_STRING` export helper (a second `#[export]` macro or
   a convention documented in the authoring guide).
8. Write example plugin: `examples/hello-hook` that logs "hello from
   plugin" via `log()` and returns `Continue`.
9. Write tests using `cargo test` with macro expansion checks + a
   `wasm32-wasip1` build-and-run smoke test.
10. Document the ABI in `crates/hook-sdk/HOST_ABI.md`.

## Contracts

**`#[hook]` macro user-facing contract:**

```rust
use vsdd_hook_sdk::{hook, HookPayload, HookResult};

#[hook]
pub fn on_hook(payload: HookPayload) -> HookResult {
    // Plugin logic here
    HookResult::Continue
}
```

**Host function signature contract (consumed by S-1.4):**

```
emit_event:        (type_ptr, type_len, fields_ptr, fields_len) -> ()
                   // fields encoded as length-prefixed key/value pairs
log:               (level: u32, msg_ptr, msg_len) -> ()
read_file:         (path_ptr, path_len, max_bytes, timeout_ms,
                    out_ptr_out, out_len_out) -> i32 (0 = ok, negative = err)
exec_subprocess:   (cmd_ptr, cmd_len, args_ptr, args_len, timeout_ms,
                    max_output_bytes, result_ptr_out) -> i32
session_id:        (out_ptr, out_cap) -> u32 (bytes written)
dispatcher_trace_id: (out_ptr, out_cap) -> u32
plugin_root:       (out_ptr, out_cap) -> u32
plugin_version:    (out_ptr, out_cap) -> u32
cwd:               (out_ptr, out_cap) -> u32
env:               (name_ptr, name_len, out_ptr, out_cap) -> i32
```

All strings UTF-8. Error codes: 0 = success, negative = specific error
class (OOB memory, capability denied, etc.). Documented in HOST_ABI.md.

**`HOST_ABI_VERSION` constant:**

```rust
pub const HOST_ABI_VERSION: u32 = 1;
```

Plugin's `on_hook` is only invoked by dispatchers with matching (or
back-compat) ABI version.

**`HookPayload` schema (public API):**

```rust
pub struct HookPayload {
    pub event_name: String,
    pub tool_name: String,
    pub session_id: String,
    pub dispatcher_trace_id: String,
    pub tool_input: serde_json::Value,
    pub tool_response: Option<serde_json::Value>,  // None for PreToolUse
}
```

## Test plan

- Unit: macro expansion produces expected code (check with `trybuild` or
  `macrotest`)
- Unit: `HookPayload` deserializes from representative JSON fixtures
- Unit: `HookResult` serializes correctly per variant
- Integration: compile `examples/hello-hook` to `wasm32-wasip1`, run under
  wasmtime with a fixture stdin, verify expected behavior
- Panic boundary: a plugin that panics must exit cleanly with an
  identifiable error result, not abort the wasmtime store

## Notes

The `#[hook]` macro is the entire ergonomic story. If we get this wrong,
authoring hooks feels like C FFI. Spend time getting it right.

Proc-macro crates must be standalone library crates with
`proc-macro = true` in Cargo.toml. The main `hook-sdk` crate re-exports
the macro from the subcrate so users `use vsdd_hook_sdk::hook;` cleanly.

Panic handling: use `std::panic::catch_unwind` inside the macro-generated
`_start`. Capture panic message + backtrace (if available) into a
structured error that gets serialized as `HookResult::Error`.

WASI preview 1 uses a `_start` function convention for WASI commands;
that's what the dispatcher invokes. For WASI preview 2 (component model)
the entry point is a component export — deferred to v2.0 per the spec.
