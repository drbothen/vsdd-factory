# S-2.7: Regression test suite validation

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.8
**Blocked by:** S-1.2, S-1.4, S-1.5, S-1.6, S-1.7, S-1.8, S-2.1, S-2.2
**Estimated effort:** M

## Goal

Validate that the full existing bats suite (1177+ tests) continues to
pass under the new dispatcher with every bash hook routed through
`legacy-bash-adapter`. This is the single most important regression
test for beta.1: if any bash hook behaves differently under the
adapter than it did directly, operators will notice and trust erodes.

## Acceptance criteria

- [ ] Full bats suite runs cleanly against the new dispatcher on Linux
- [ ] Full bats suite runs cleanly on macOS
- [ ] Every hook's behavior is identical to its pre-v1.0 behavior (same
  events emitted, same exit codes, same stdout/stderr)
- [ ] Any test that currently relies on bash-only side effects (e.g.,
  specific file paths, specific env var inheritance) is either:
  - Validated as still correct, or
  - Updated with an explanatory comment + v1.0 note
- [ ] A diff comparing events emitted by pre-v1.0 hooks vs
  adapter-routed hooks shows no unexpected changes
- [ ] Windows: at least a subset of tests (hook-schema validators,
  config parsers — anything not requiring the bash hooks) pass on
  Windows
- [ ] Dispatcher startup latency is under 20ms on a warm cache (one-off
  measurement)
- [ ] Per-hook invocation latency under legacy-bash-adapter is within
  ~30% of direct bash invocation (the 30% is the adapter's wasmtime +
  subprocess-spawn overhead budget)
- [ ] Any regression is documented in a "known regressions" file or
  fixed before the story closes

## Tasks

1. Set up a test harness that runs bats against the new dispatcher:
   - Replace the current `hooks.json` with the new (activation-skill-
     generated) one
   - Run `bats plugins/vsdd-factory/tests/` end-to-end
2. If any test fails, categorize:
   - Adapter bug → fix in S-2.1 (block on this story's completion)
   - Test assumption violation → update test
   - Real regression → document and decide whether to block beta.1
3. Event-parity check: capture events emitted by a sample of hooks
   running pre-v1.0 vs post-v1.0; diff; investigate any differences.
4. Latency benchmark: pick 5 representative hooks, measure invocation
   latency pre vs post; verify within the 30% budget.
5. Windows partial validation: identify bats tests that don't require
   bash-specific behavior (schema validators, config parsers, certain
   jq-only checks) and run them on Windows.
6. Write a "known regressions" section in `docs/guide/migrating-from-0.79.md`
   for anything that intentionally changes (unlikely for beta.1 but
   possible).

## Contracts

**Parity definition:** a v1.0 hook invocation is "parity" with v0.79.x
if:

- Exit code identical given the same stdin
- Events emitted (to the file sink) identical modulo the new fields
  `dispatcher_trace_id`, `plugin_name`, `plugin_version` which are
  dispatcher enrichments (expected difference, not regression)
- Stdout/stderr identical (both captured in emit-event enrichment)
- Side effects on the filesystem identical (files created, modified,
  deleted)
- Timing within the 30% latency budget

**Regression escalation:**

- Category 1 (adapter bug): block S-2.8; fix before beta.1
- Category 2 (test obsolete): update test, document change
- Category 3 (real behavior change): document in known regressions;
  decide with user whether acceptable for beta.1

## Test plan

- Full bats run on Linux + macOS (CI)
- Manual spot-check on a real factory (vsdd-factory's own factory-artifacts
  branch, dogfood)
- Event-parity test: run 10 specific hooks pre-v1.0 + post-v1.0, diff
  the events
- Latency benchmark
- Windows: partial test run (env allowing)

## Notes

This story is the gate between "the plumbing works" and "the factory
still works." bats passing is necessary but not sufficient — we also
need event parity and latency targets.

The 30% latency budget accounts for:
- wasmtime store creation (~1ms)
- WASM instantiation (~1ms)
- Subprocess spawn overhead (adapter → bash, which we already had)
- Extra JSON serialization (we now re-serialize the HookPayload before
  piping to bash; before, it was just passed through)

If we miss the budget significantly, options:
- Optimize the adapter's hot path
- Skip re-serialization (pass raw stdin through unchanged)
- Defer the hook's port to native WASM as higher-priority post-1.0 work

Event parity for the hooks we'll port natively (capture-commit-activity,
etc.) is tested by S-3.x, not here; those hooks get identical-behavior
tests as part of their port.
