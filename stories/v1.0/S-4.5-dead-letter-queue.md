**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)

**Milestone:** `1.0.0-rc.1`

**Estimated effort:** S

## Goal

When events are dropped — because the sink's retry attempts exhausted,
or the outbound queue overflowed — write them to a per-sink dead letter
file. This prevents silent data loss: operators can inspect what
dropped and replay manually. Per Q6 resolution: 7-day retention,
per-sink path, optional (default on for HTTP sinks, off for file sinks).

## Acceptance criteria

- [ ] `DeadLetterWriter` struct per sink (when enabled)
- [ ] Events dropped after retry exhaustion or queue-full land in
  `.factory/logs/dead-letter-<sink_name>-<date>.jsonl`
- [ ] Path format configurable with the same `{date}` / `{name}`
  templating as file sink
- [ ] Daily rotation; retention enforced on dispatcher startup (delete
  files older than `retention_days`, default 7)
- [ ] Per-event drop reason recorded in a wrapper field:
  ```json
  {
    "dropped_at": "2026-04-24T20:13:45-0500",
    "drop_reason": "retries_exhausted" | "queue_full" | "terminal_error",
    "sink_name": "datadog-prod",
    "attempts": 3,
    "event": { ...original event... }
  }
  ```
- [ ] Write path never blocks (non-fatal on disk full; log to
  dispatcher-internal + drop the DLQ write silently)
- [ ] Disable by setting `dead_letter.enabled = false` in sink config
- [ ] Operator CLI helper (post-1.0 backlog) can re-submit from DLQ;
  this story just ensures the data is preserved in replay-able format
- [ ] Integration test: failing sink + 100 events → all 100 appear in
  DLQ with correct metadata

## Contracts

**Config:**

```toml
[sinks.<name>.dead_letter]
enabled         = true
path            = ".factory/logs/dead-letter-{name}-{date}.jsonl"
retention_days  = 7
```

**DLQ entry schema:**

```json
{
  "dropped_at": "2026-04-24T20:13:45-0500",
  "drop_reason": "retries_exhausted" | "queue_full" | "terminal_error",
  "sink_name": "datadog-prod",
  "attempts": 3,
  "last_error": "connection refused",
  "event": { ...original event payload... }
}
```

**Retention:** dispatcher startup iterates `.factory/logs/dead-letter-*.jsonl`,
deletes files where mtime < now() - retention_days. No in-process
timer; startup-only cleanup keeps it simple.

## Test plan

- Unit: DLQ write format
- Unit: retention — seed files with varied ages; verify correct deletion
- Integration: sink with network down → events route to DLQ → DLQ
  file contains all events with correct metadata
- Integration: DLQ write failure (read-only fs) doesn't crash dispatcher

## Notes

DLQ is defense-in-depth, not a primary pipeline. Normal operation never
writes to DLQ. When it does write, it's an incident — surface it via a
health metric (`internal.sink_health` from S-1.7/S-4.4 includes
`events_dropped_to_dlq` count).

7-day retention is a judgment call. Longer = more recovery capacity;
shorter = less disk. Operators can override.

Replay tool: a future CLI (`factory-obs replay-dlq --sink=datadog-prod`)
reads the DLQ and re-submits events. Out of scope for v1.0; the data
being there means replay IS possible without extra work.
