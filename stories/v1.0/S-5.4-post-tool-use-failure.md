## Goal

Wire `PostToolUseFailure` — fires after a tool call fails. Emits
`tool.error` events that feed a new Factory ROI error-rate dashboard
panel (scoped post-1.0 on the dashboard side; event stream is in v1.0).

## Acceptance criteria

- [ ] Dispatcher handles `PostToolUseFailure` events
- [ ] `tool-error-telemetry` WASM plugin emits `tool.error`:
  - `tool_name` (Bash, Edit, Write, etc.)
  - `error_class` (from Claude Code's payload)
  - `error_message` (truncated)
  - `is_interrupt` (true for user-interrupt vs tool-level failure)
  - `tool_use_id`
- [ ] Registered in `hooks-registry.toml`
- [ ] Integration test
- [ ] Works cross-platform

## Contracts

**`tool.error` event:**

```json
{
  "type": "tool.error",
  "tool_name": "Bash",
  "error_class": "timeout" | "exception" | "interrupt" | "permission_denied" | ...,
  "error_message": "...",
  "is_interrupt": false,
  "tool_use_id": "..."
}
```

Error message truncated to 1000 chars to avoid bloating the sink
stream with long stack traces.

## Test plan

- Integration: synthetic PostToolUseFailure payload → verify emitted
  `tool.error` event

## Notes

Dashboard panel for error rate: post-1.0 backlog. This story just
ensures the event stream exists so the dashboard can be built on top.

Error message truncation is conservative; if operators care about full
errors, they can see them in the Claude Code UI directly. The event
stream is for aggregate analysis.

Categorizing `error_class`: rely on Claude Code's payload fields. If
not present, synthesize from the error message using known patterns
(timeout → "timeout", etc.). Defer deeper categorization to post-1.0.
