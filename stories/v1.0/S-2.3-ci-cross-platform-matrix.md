# S-2.3: Cross-platform CI matrix build targets

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.4, S-2.8
**Blocked by:** S-1.1, S-1.2 (needs a buildable dispatcher)
**Estimated effort:** M

## Goal

Extend the CI workflow to build `factory-dispatcher` and all WASM plugins
on all 5 target platforms: darwin-arm64, darwin-x64, linux-x64, linux-arm64,
windows-x64. Every push to main must produce a green build across the
matrix; release tags build the artifacts that S-2.4 commits.

## Acceptance criteria

- [ ] `.github/workflows/ci.yml` has a `build-dispatcher` matrix job with
  5 platform targets
- [ ] Each matrix job installs the Rust toolchain from `rust-toolchain.toml`,
  installs `wasm32-wasip1` target, runs `cargo build --release`
- [ ] Each job runs `cargo test` for dispatcher unit + integration tests
- [ ] Each job builds every WASM plugin in the workspace
- [ ] Windows-x64 uses the windows-latest runner with cross-compilation
  details handled (`x86_64-pc-windows-msvc` target, MSVC Build Tools)
- [ ] Linux-arm64 uses cross-compilation via `cargo-cross` or
  `cross-rs/cross` action
- [ ] Darwin-x64 + arm64 use macos-latest + macos-14 (M1) runners
- [ ] Build failures on ANY platform fail the overall CI
- [ ] Built artifacts uploaded as GitHub Actions artifacts per platform
  (consumed by S-2.4 on release)
- [ ] Build time under 10 minutes per platform (parallelized)
- [ ] Matrix cache keyed on `Cargo.lock` + `rust-toolchain.toml` so
  dependency builds are reused across runs

## Tasks

1. Read existing `.github/workflows/ci.yml` for conventions.
2. Add a `build-dispatcher` job with a 5-entry matrix:
   ```yaml
   matrix:
     include:
       - platform: darwin-arm64
         os: macos-14
         target: aarch64-apple-darwin
       - platform: darwin-x64
         os: macos-latest
         target: x86_64-apple-darwin
       - platform: linux-x64
         os: ubuntu-latest
         target: x86_64-unknown-linux-gnu
       - platform: linux-arm64
         os: ubuntu-latest  # cross-compile
         target: aarch64-unknown-linux-gnu
         use_cross: true
       - platform: windows-x64
         os: windows-latest
         target: x86_64-pc-windows-msvc
   ```
3. Steps per matrix entry:
   - Checkout
   - Install Rust (from `rust-toolchain.toml`)
   - `rustup target add ${{ matrix.target }}`
   - `rustup target add wasm32-wasip1`
   - Restore cargo cache keyed on `hashFiles('Cargo.lock', 'rust-toolchain.toml')`
   - Build: `cargo build --release --target ${{ matrix.target }}`
     (or `cross build ...` if `use_cross`)
   - Test: `cargo test --target ${{ matrix.target }}` (where runnable;
     skip for Linux ARM64 cross-build since we can't run it)
   - Build each WASM plugin: `cargo build --release --target wasm32-wasip1`
   - Upload artifact: `${{ matrix.platform }}/factory-dispatcher[.exe]`
     + all `.wasm` files
4. Gate the overall workflow on all 5 jobs green.
5. Tests: verify CI green on a PR that touches only README. Ensure matrix
   runs are cached so subsequent runs are quick.
6. Document the matrix in `docs/guide/porting-bash-hook-to-wasm.md` for
   plugin authors needing to understand what targets they ship for.

## Artifacts

## Contracts

**Artifact naming convention:**

```
factory-dispatcher-<platform>[.exe]
<plugin-name>.wasm
```

Uploaded as GH Actions artifacts with names matching the target. S-2.4's
release workflow downloads them and commits to the repo.

**Platform tuple canonicalization:**

Must match the strings used in S-0.3 (activation skill detection) and
S-0.4 (hooks.json variants). The five canonical strings:

```
darwin-arm64
darwin-x64
linux-x64
linux-arm64
windows-x64
```

Single source of truth: a `platforms.yaml` file at repo root (or in a
`ci/` subdir) that both CI and activation skill read.

## Test plan

- CI green on a PR touching dispatcher code
- CI green on a PR touching a WASM plugin
- CI green on a PR touching neither (should still run)
- CI fails visibly if a matrix entry fails
- Cache hit on a rebuild without dependency changes

## Notes

ARM64 Linux cross-compilation: `cross-rs/cross` is the standard tool.
Alternative: native ARM64 Linux runner on GH Actions (currently not a
default offering; paid feature). Start with `cross`; upgrade if runner
availability changes.

macos-14 is GH Actions' M1 (Apple Silicon) runner. It's the only way to
get a true arm64 build; macos-latest is still Intel as of 2026 (check
at story time).

Windows build caveats: `wasmtime` with default features may need
`cl.exe` from Visual Studio Build Tools. Document the exact toolchain in
the CI step.

Caching cargo targets across matrix runs saves 5-10 minutes per
full-matrix build. Worth the complexity.
