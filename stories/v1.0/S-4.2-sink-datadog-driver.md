# S-4.2: sink-datadog driver

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-1.8 (Sink trait), S-4.1 (HttpSink pattern reused)
**Estimated effort:** M

## Goal

Implement a Datadog-specific sink that sends events via Datadog's Logs
Intake API. Handles the specific payload schema, authentication, regional
endpoints, and rate limiting that Datadog requires. This is the
motivating use case from the design discussion — operators who want to
ship logs directly to Datadog without running the OTel collector.

## Acceptance criteria

- [ ] `DatadogSink` implements the `Sink` trait
- [ ] Config: `api_key` (required, env var interpolation supported),
  `site` (default `datadoghq.com`; supports EU, US3, US5, etc.),
  `service` (default `vsdd-factory`), `hostname` (auto-detect),
  `tags` (list of key:value strings)
- [ ] POSTs to `https://http-intake.logs.<site>/api/v2/logs`
- [ ] Payload matches Datadog's logs-intake schema
- [ ] Handles rate limits: honors `X-RateLimit-Remaining` +
  `X-RateLimit-Reset` headers
- [ ] Handles 413 Payload Too Large by splitting the batch and retrying
- [ ] Retries/circuit-breaker from S-4.4 apply
- [ ] Integration test: mock Datadog receiver verifies payload format
- [ ] E2E test: run against a real Datadog sandbox (feature-gated behind
  a secret; only runs if `DD_E2E_API_KEY` set)

## Tasks

1. Create `crates/sink-datadog/`.
2. Share HTTP client infrastructure with `sink-http` (possibly factor
   into `sink-common`).
3. Translate internal `Event` to Datadog log record:
   ```
   event.type            → ddsource (or message if no specific source)
   event.ts_epoch        → timestamp (Datadog uses milliseconds)
   all event fields      → attributes
   tags                  → ddtags
   hostname              → host
   service               → service
   ```
4. Implement site → URL mapping.
5. Handle rate limit headers.
6. Payload splitting on 413.
7. Tests against mock receiver with Datadog's actual response shapes.
8. Feature-gated e2e test against real Datadog.

## Contracts

**Config shape:**

```toml
[[sinks]]
name = "datadog-prod"
type = "datadog"
enabled = true
api_key = "${DATADOG_API_KEY}"
site = "datadoghq.com"                   # or "datadoghq.eu", "us3.datadoghq.com", etc.
service = "vsdd-factory"                 # optional
hostname = "auto"                        # auto-detect via gethostname(); "" to omit
tags = ["env:prod", "team:factory"]      # ddtags format

[sinks.datadog-prod.batch]
size        = 100
interval_ms = 5000
```

**Datadog payload format (v2 logs intake):**

```json
[
  {
    "ddsource": "vsdd-factory",
    "ddtags": "env:prod,team:factory,event_type:commit.made",
    "hostname": "operator-mac.local",
    "service": "vsdd-factory",
    "message": "commit.made: abc1234 on main",
    "timestamp": 1777003425000,
    "attributes": {
      "type": "commit.made",
      "commit_sha": "abc1234",
      "branch": "main",
      ...
    }
  }
]
```

## Notes

Datadog's v2 logs intake endpoint is well-documented. Use their curl
examples as reference for headers and payload.

API key auth uses `DD-API-KEY: <key>` header, not `Authorization: Bearer`.

Regional endpoints:
- `datadoghq.com` (US1)
- `datadoghq.eu` (EU1)
- `us3.datadoghq.com`
- `us5.datadoghq.com`
- `ap1.datadoghq.com`

Payload size limit is 5MB uncompressed, 256KB per log entry. Batch sizing
should respect both.

Tags: Datadog uses `key:value` format, comma-separated. Our internal tag
format is already compatible.
