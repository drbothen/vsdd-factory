# S-4.1: sink-http driver

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 4 — Observability sinks
**Milestone:** `1.0.0-rc.1`
**Tier:** E
**Blocks:** S-4.7, S-4.8
**Blocked by:** S-1.8 (Sink trait)
**Estimated effort:** M

## Goal

Implement a generic `http` sink that batches events and POSTs them to a
user-configured endpoint. Not a specific vendor integration — this is
the "bring your own backend" option for operators who want custom
webhooks, internal aggregators, or bespoke logging services.

## Acceptance criteria

- [ ] `HttpSink` implements the `Sink` trait
- [ ] Accepts config: `url`, `auth_header` (optional, template-
  interpolates env vars), `content_type` (default application/json),
  `batch.size` / `batch.interval_ms`
- [ ] POSTs batched events as JSON array (default) or NDJSON (configurable)
- [ ] Retries and circuit breaker come from S-4.4's shared resilience
  runtime; this story implements the driver, not the retry logic
- [ ] Graceful handling of HTTP errors: 4xx = terminal fail, 5xx = retry
  (S-4.4), 429 = retry with Retry-After
- [ ] TLS support via `rustls` (no OpenSSL dependency — keeps binary size
  down and cross-platform-clean)
- [ ] Proxy support via `HTTP_PROXY` / `HTTPS_PROXY` env vars (honor
  standard conventions)
- [ ] Integration test against a local mock HTTP server; verify batch
  size + interval + payload correctness
- [ ] Connection pooling: single `reqwest::Client` reused across sink
  instances

## Tasks

1. Create `crates/sink-http/` crate.
2. Add `reqwest` (with `rustls-tls` feature, no default features) as dep.
3. Implement `HttpSink::new(config)` returning a `Sink` trait object.
4. Spawn async background task for batch accumulation + submission.
5. Handle auth header templating: `"Bearer ${INGEST_TOKEN}"` →
   interpolate from env at config load.
6. JSON vs NDJSON content format toggle.
7. Integration tests with a tokio-based mock HTTP receiver.

## Contracts

**Config shape:**

```toml
[[sinks]]
name = "custom-webhook"
type = "http"
enabled = true
url = "https://events.internal.example.com/ingest"
auth_header = "Bearer ${INGEST_TOKEN}"   # optional
content_type = "application/json"        # optional; "application/x-ndjson" supported
http_method = "POST"                      # optional, default POST

[sinks.custom-webhook.batch]
size        = 100
interval_ms = 5000

# Generic resilience config (S-4.4) applies uniformly
```

**Request format (default JSON):**

```json
[
  { "type": "commit.made", "dispatcher_trace_id": "...", ... },
  { "type": "plugin.invoked", "dispatcher_trace_id": "...", ... },
  ...
]
```

**NDJSON format:** one event per line, for receivers that prefer
streaming parsing.

## Test plan

- Unit: config parse + env var interpolation
- Integration: mock HTTP server + 100 events batched → verify POST
  received with correct headers + body
- Timeout: mock receiver delays response → verify retry policy kicks in
- 4xx: mock receiver returns 401 → verify event marked terminal, routed
  to dead-letter
- 5xx: mock receiver returns 503 → verify retry + backoff

## Notes

`reqwest` with `rustls-tls` and disabled default features keeps the
binary size down substantially (~1.5MB saved vs native-tls). Worth it.

Proxy support is important for corporate environments. Standard env
var convention means we get it "for free" from reqwest.

Auth header templating: keep it simple — only `${ENV_VAR}` syntax for
now. No conditional logic, no nested templating. If users need complex
auth (OAuth, mTLS), they use a more specific sink or wrap with a proxy.
