# S-0.3: Activation skill platform detection

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 0 — Infrastructure prep
**Milestone:** `1.0.0-beta.1`
**Tier:** A (contract-frozen; integration lands in S-2.6)
**Blocks:** S-2.6 (activation skill integration with real hooks.json variants)
**Blocked by:** none (can start in parallel; integration depends on S-0.4)
**Estimated effort:** M

## Goal

Design and stub the platform-detection logic that the existing
`/vsdd-factory:activate` skill will use to select the correct
`hooks.json.<platform>` variant. The skill doesn't yet have platform-aware
behavior; this story establishes the detection algorithm, the supported
platform strings, and how the skill records what it did in
`.claude/settings.local.json`. **Actual copy-in of real variants** is in
S-2.6 (requires the CI generation from S-0.4 and dispatcher binaries from
S-1.x to exist).

## Acceptance criteria

- [ ] Activation skill has a documented algorithm for detecting OS + arch
  on macOS, Linux (x64 + arm64), and Windows
- [ ] Platform strings are normalized to match the design doc's format:
  `darwin-arm64`, `darwin-x64`, `linux-x64`, `linux-arm64`, `windows-x64`
- [ ] Detection gracefully fails with a clear error when platform is
  unsupported (e.g., `freebsd-x64`, `darwin-x86` — neither in our matrix)
- [ ] Skill writes detected platform to
  `.claude/settings.local.json:vsdd-factory.activated_platform` so we can
  detect drift if the user moves the factory to a different machine
- [ ] Skill surfaces a "platform mismatch" warning on re-activation if the
  recorded `activated_platform` doesn't match current detection (e.g., you
  activated on a Mac, now you're SSH'd into a Linux box with the same
  `.factory/`)
- [ ] Dry-run mode documents what it WOULD do without writing anything
- [ ] Added to `plugins/vsdd-factory/tests/skills.bats` (or new
  `activate.bats`)

## Tasks

1. Read existing `/vsdd-factory:activate` skill to understand current
   structure.
2. Implement detection logic:
   - macOS: `uname -s` = `Darwin`; `uname -m` = `arm64` or `x86_64`
   - Linux: `uname -s` = `Linux`; `uname -m` = `x86_64` or `aarch64` (→ arm64)
   - Windows: check `$env:PROCESSOR_ARCHITECTURE` (AMD64 → x64) OR
     `uname -s` starts with `MINGW`/`MSYS`/`CYGWIN`
3. Normalize detected values to the 5 canonical strings (or fail).
4. Extend skill to persist `activated_platform` in
   `.claude/settings.local.json` under a stable key.
5. Add re-activation drift check: compare persisted vs. current.
6. Add bats tests that mock `uname` output and verify detection.
7. Add docs sketch for the error messages (unsupported platform, drift).

## Artifacts

## Contracts

**Detection output contract (consumed by S-2.6):**

```
detect_platform() → {
  "platform": "darwin-arm64" | "darwin-x64" | "linux-x64" |
              "linux-arm64" | "windows-x64",
  "detected_from": { os: "...", arch: "...", raw_uname: "..." },
  "error": null | "unsupported-platform" | "ambiguous"
}
```

**Persisted state schema in `settings.local.json`:**

```json
{
  "vsdd-factory": {
    "activated_platform": "darwin-arm64",
    "activated_at": "2026-04-24T20:00:00-0500",
    "activated_plugin_version": "1.0.0-beta.1"
  }
}
```

## Test plan

- Unit: mock `uname` to return each supported platform tuple; verify
  normalization.
- Unit: mock `uname` for unsupported platforms (e.g., `FreeBSD x86_64`);
  verify error path.
- Unit: pre-populate `activated_platform` in a temp `settings.local.json`
  with a different value than current detection; verify drift warning.
- Integration (S-2.6): full flow including copying a real
  `hooks.json.<platform>` and verifying the binary path.

## Notes

This story is design + stub + tests only. The actual copy of a real
`hooks.json.<platform>` into place depends on S-0.4 (CI generation) and
S-2.6 (integration). But detection algorithm + schema can be frozen now.

See spec section "Migration path (v0.79.x → v1.0)" and Open Question 5
resolution for context on why this is an activation-skill responsibility.

Windows detection via Git Bash / MSYS is the trickiest case; `uname -s`
returns `MINGW64_NT-10.0-...` or similar — regex-match the prefix.
