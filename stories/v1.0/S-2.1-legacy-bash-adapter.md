# S-2.1: legacy-bash-adapter WASM plugin

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 2 — Legacy bash adapter
**Milestone:** `1.0.0-beta.1`
**Tier:** C
**Blocks:** S-2.2, S-2.7, S-2.8
**Blocked by:** S-1.3, S-1.4, S-1.5, S-1.6
**Estimated effort:** M

## Goal

Implement the `legacy-bash-adapter` WASM plugin. It takes a bash script
path via a per-hook configuration parameter, invokes it via
`exec_subprocess()` with the existing stdin payload, propagates stdout/
stderr/exit-code back to the dispatcher. This is how all 30 existing bash
hooks continue to work on Linux/macOS during the v1.0 migration without
requiring any of them to be rewritten in WASM.

## Acceptance criteria

- [ ] `legacy-bash-adapter` compiles to `wasm32-wasip1`
- [ ] Reads a configuration parameter `script_path` (relative to
  `plugin_root()`) from the registry entry's per-hook config
- [ ] Invokes `exec_subprocess("bash", ["-c", script_invocation], ...)`
  with the Q4 safety envelope (shell_bypass_acknowledged = ["bash"])
- [ ] Passes the original stdin `HookPayload` through to the bash script
  unchanged — bash hooks read from stdin as they always did
- [ ] Propagates bash exit code to `HookResult`:
  - exit 0 → `HookResult::Continue`
  - exit 2 → `HookResult::Block(stderr)`
  - other non-zero → `HookResult::Error(stderr)` (routed through
    `on_error` policy)
- [ ] Stdout/stderr forwarded to the dispatcher's log pipeline so existing
  bash hooks that `echo` for debugging still show up
- [ ] A single instance of the adapter handles one bash script path per
  registry entry (not multiple scripts in one adapter invocation)
- [ ] Registry entries referencing the adapter must specify `script_path`
- [ ] Integration tests: adapter runs `true`, adapter runs `false`,
  adapter runs a script that exits 2, adapter runs a hanging script
  (timeout path), adapter runs a script that writes JSON to stdout
- [ ] Works with at least 3 real existing bash hooks (picked for coverage:
  one Edit|Write PostToolUse, one Bash PostToolUse, one SubagentStop)

## Tasks

1. Create `crates/hook-plugins/legacy-bash-adapter/` member crate.
2. Add dep on `vsdd-hook-sdk`.
3. Write `on_hook` that:
   - Extracts `script_path` from per-hook config (passed via a new
     mechanism in the HookPayload — see Contracts below)
   - Resolves full path: `plugin_root().join(script_path)`
   - Re-serializes the HookPayload to JSON (it was already JSON on
     stdin, but we need a string for the subprocess input)
   - Calls `exec_subprocess("bash", [script_path_str],
     timeout_ms=limits.timeout_ms, max_output_bytes=1MB)` passing the
     JSON as stdin
   - Inspects exit code, maps to HookResult
4. Update `HookPayload` in `vsdd-hook-sdk` to include a
   `plugin_config: serde_json::Value` field — per-hook config from the
   registry entry. (Cross-cut into S-1.2 registry and S-1.3 SDK.)
5. Integration tests with fixture bash scripts.
6. Manual verification with real bash hooks picked for coverage.

## Contracts

**Registry entry referencing the adapter:**

```toml
[[hooks]]
name = "validate-template-compliance"
event = "PostToolUse"
tool = "Edit|Write"
plugin = "hook-plugins/legacy-bash-adapter.wasm"
priority = 50
timeout_ms = 10000
on_error = "block"

# Config passed to the adapter as HookPayload.plugin_config
[hooks.config]
script_path = "legacy-hooks/validate-template-compliance.sh"

[hooks.capabilities.exec_subprocess]
binaries = ["bash", "git", "jq"]
shell_bypass_acknowledged = ["bash"]
shell_bypass_reason = "legacy-bash-adapter runs unported hooks"
env_allowlist = ["PATH", "HOME", "TMPDIR", "CLAUDE_PROJECT_DIR", "VSDD_SESSION_ID"]
cwd_allowlist = ["${CLAUDE_PROJECT_DIR}", "${CLAUDE_PROJECT_DIR}/.factory"]
```

**Bash script input contract:**

Scripts read the full Claude Code hook payload from stdin as JSON, exactly
as they do today. No format changes — the adapter is a transparent bridge.

**Exit code mapping contract:**

```
0       → HookResult::Continue
2       → HookResult::Block (reason = stderr first line)
other   → HookResult::Error (message = stderr, routed via on_error policy)
```

Matches the existing bash hook conventions.

## Notes

The adapter is a temporary bridge. Every bash hook we port to native WASM
(Phase 3 and post-1.0 backlog) reduces the adapter's usage. When the last
bash hook is ported, the adapter can be retired.

Adapter does NOT give bash scripts any capability the plugin registry
didn't grant via `capabilities.exec_subprocess` — bash hooks calling `git`
or `jq` still require those binaries in the allow-list. This is an
intentional narrowing from today's unrestricted bash context, forcing the
capability question to become explicit for every hook.

Documentation: add `docs/guide/porting-bash-hook-to-wasm.md` section
explaining when to keep a hook on the adapter vs port it to native WASM
(rule of thumb: high-frequency hooks port for latency wins; rare hooks
stay on adapter indefinitely).
