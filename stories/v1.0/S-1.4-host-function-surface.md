# S-1.4: Host function surface implementation

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-1.5, S-2.1, S-3.1, S-3.2, S-3.3
**Blocked by:** S-1.1, S-1.2 (routing types), S-1.3 (ABI contract)
**Estimated effort:** L

## Goal

Implement the dispatcher-side of every host function the SDK declares:
`emit_event`, `log`, `read_file`, `exec_subprocess`, `session_id`,
`dispatcher_trace_id`, `plugin_root`, `plugin_version`, `cwd`, `env`. Each
one is a wasmtime `Linker` import that plugins call. Enforces capability
policy: `read_file` respects preopens, `exec_subprocess` respects the
per-plugin allow-list, `env` respects the env allow-list, etc. Emits
`internal.capability_denied` events when policy blocks a call.

## Acceptance criteria

- [ ] Each of the 10 host functions implemented and registered with wasmtime
  `Linker`
- [ ] String arguments read from guest memory via `ptr + len` per the ABI
  contract in S-1.3
- [ ] Return values written back to guest memory via caller-provided output
  pointers + length sentinels
- [ ] `emit_event` enqueues events for sink routing (sink routing is S-1.8
  onward; here, just enqueue onto a dispatcher-owned channel)
- [ ] `read_file` rejects paths outside preopened roots with error code;
  emits `internal.capability_denied` with reason `path_not_preopened`
- [ ] `exec_subprocess` enforces:
  - binary name in allow-list (full-path resolved at registry load)
  - no shell interpretation unless `shell_bypass_acknowledged` is set
  - env inherited per `env_allowlist`
  - cwd in `cwd_allowlist`
  - setuid/setgid binaries refused categorically
  - timeout + max_output_bytes from call args
- [ ] `env` respects per-plugin allow-list; unknown vars return empty
- [ ] All functions handle OOB memory access gracefully (bounds-check
  every read/write)
- [ ] Panic in a host function converts to `internal.host_function_panic`
  event; does not crash the dispatcher
- [ ] Each host function covered by unit tests + integration tests with a
  real WASM plugin calling it

## Tasks

1. Create `crates/factory-dispatcher/src/host/` module with one file per
   host function (`emit_event.rs`, `exec_subprocess.rs`, etc.).
2. Common helpers: `read_wasm_string(store, memory, ptr, len) -> Result<String>`
   with bounds-checking; `write_wasm_string(store, memory, out_ptr, out_cap, s)`.
3. Implement each function. Pay special attention to:
   - **`emit_event`**: parse key/value pairs from the fields buffer;
     enrich with `dispatcher_trace_id`, `session_id`, `plugin_name`,
     `plugin_version`, `timestamp`; hand off to sink pipeline (stubbed for
     now; S-1.8/S-1.9 wire sinks).
   - **`exec_subprocess`**: full Q4 safety envelope. Each binary allow-list
     entry was resolved to full path at registry load (see S-1.2 contract);
     use the resolved path at exec time. Capture stdout/stderr; enforce
     `max_output_bytes` by truncating with a marker.
   - **`read_file`**: canonicalize the requested path, verify it's under a
     preopened root, open, bounded read.
4. Implement `setup_linker(engine: &Engine) -> Linker<HostContext>` helper
   that registers all 10 host functions.
5. Integration test: a stub plugin that exercises every host function and
   verifies outcomes.
6. Capability denial events: every denial emits
   `internal.capability_denied` with `reason`, `plugin`, `function`,
   `details`. Routed through normal sink pipeline.

## Contracts

**`HostContext` (stored in wasmtime `Store`):**

```rust
pub struct HostContext {
    pub plugin_name: String,
    pub plugin_version: String,
    pub plugin_root: PathBuf,
    pub session_id: String,
    pub dispatcher_trace_id: String,
    pub capabilities: Capabilities,      // from registry entry
    pub event_sender: mpsc::Sender<Event>, // fed into sink pipeline
    pub cwd: PathBuf,
    pub env_allowlist: Vec<String>,
    pub preopened_paths: Vec<PathBuf>,
}
```

**Event enrichment (applied in `emit_event`):**

Every event automatically gets these fields added, overriding any attempt
by the plugin to set them:
- `dispatcher_trace_id`
- `session_id`
- `plugin_name`
- `plugin_version`
- `ts` (ISO-8601)
- `ts_epoch`
- `schema_version`

Plugins may not spoof these. The dispatcher owns identity.

## Test plan

- Unit per host function: mock `HostContext`, call the function directly,
  verify behavior.
- Integration: a `capability-exerciser.wasm` plugin that calls each host
  function with valid + invalid inputs; verify every success and denial
  path.
- Capability denial: feed a plugin that tries `exec_subprocess` on an
  out-of-allow-list binary; verify denial event + error return.
- OOB memory: feed a plugin that passes `ptr=0xFFFF_FFFF, len=1` for
  string arg; verify graceful error, no crash.
- Panic: feed a plugin that causes a host function to panic (e.g., pass
  non-UTF-8 bytes); verify `internal.host_function_panic` emits + host
  returns error code.

## Notes

The host/guest memory interop is the riskiest part. Bugs here crash the
dispatcher for the user. Every guest-provided pointer must be bounds-checked
against the current WASM memory size (which can grow between calls).

`exec_subprocess` is the capability-heavy one. Re-read the Q4 resolution
and the `hooks-registry.toml` example in the design doc before implementing;
getting this wrong is a security hole.

The `event_sender` on `HostContext` is a tokio mpsc channel (S-1.8 sets
up the consumer). In this story, stub the consumer as a println logger so
the integration tests can observe emit_event behavior.
