# S-1.6: tokio + parallel-within-tier execution

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.1, S-2.7
**Blocked by:** S-1.1, S-1.2, S-1.4, S-1.5
**Estimated effort:** M

## Goal

Implement the execution layer the dispatcher hands tiered plugins to.
Plugins within the same priority tier run concurrently via tokio tasks,
each with its own wasmtime store. Tiers run sequentially so priority
ordering is preserved. Aggregates per-plugin results, applies `on_error`
policies, decides the dispatcher exit code.

## Acceptance criteria

- [ ] `execute_tiers(tiers: Vec<Vec<&RegistryEntry>>, payload, host_ctx)
  -> TierExecutionSummary` implemented
- [ ] tokio runtime initialized in main (`#[tokio::main]` or explicit
  single-threaded runtime)
- [ ] Plugins in the same tier spawned as concurrent tokio tasks, each
  invoking `invoke_plugin` (from S-1.5)
- [ ] Dispatcher awaits all tasks in a tier before starting the next tier
- [ ] Per-tier wall-clock bounded by `max(plugin.timeout_ms)` + 10ms grace
- [ ] On `on_error = "block"` plugin returning a block result: record it,
  continue the rest of the tier (remaining plugins still fire), then set
  final exit code to 2
- [ ] On `on_error = "continue"` plugin failure: log + route `plugin.timeout`
  / `plugin.crashed` event, continue
- [ ] `TierExecutionSummary` includes: per-plugin PluginResult, total
  wall-clock, tier-level block intent, final exit code
- [ ] Integration tests covering: all plugins succeed; one block plugin
  blocks; timeouts in parallel don't cascade; crash in one plugin doesn't
  affect siblings
- [ ] Latency test: 20 plugins × 100ms each → total wall-clock < 150ms
  (should be ~100ms plus overhead, validating parallel execution)

## Tasks

1. Create `crates/factory-dispatcher/src/executor.rs`.
2. Implement `execute_plugin(entry, payload, host_ctx) -> PluginResult`
   as a `async fn` wrapping `invoke_plugin` (S-1.5) via
   `tokio::task::spawn_blocking` — wasmtime is sync, so we isolate it.
3. Implement `execute_tier(tier, payload, host_ctx) -> Vec<(entry, PluginResult)>`:
   - `tokio::try_join_all` or `FuturesUnordered` over plugins
   - Each task has its own HostContext clone (plugin-scoped)
4. Implement `execute_tiers(tiers, payload, base_host_ctx) -> TierExecutionSummary`:
   - Iterate tiers in order
   - After each tier, inspect results for `on_error = "block"` plugins
     returning `HookResult::Block` — record block intent but proceed
   - Aggregate exit code: 2 if any block intent, 0 otherwise
5. Wire into main: replace the stub execution call from S-1.2 with the
   real `execute_tiers`.
6. Tests:
   - `parallel_happy_path`: 5 plugins in one tier, all return Continue
   - `parallel_timeout_isolated`: 1 plugin hangs, 4 succeed; total
     wall-clock ≈ timeout
   - `parallel_crash_isolated`: 1 plugin panics, 4 succeed
   - `multi_tier_ordering`: tier A runs before tier B; tier B plugins
     observe tier A's emitted events
   - `block_continues_tier`: 3 plugins in tier; middle one has `on_error=block`
     and returns Block; all 3 execute; final exit code 2
   - `latency_check`: benchmark asserting max(timeouts) not sum(timeouts)

## Contracts

**`TierExecutionSummary` (consumed by dispatcher main):**

```rust
pub struct TierExecutionSummary {
    pub per_plugin_results: Vec<(RegistryEntryRef, PluginResult)>,
    pub total_elapsed_ms: u64,
    pub block_intent: bool,             // any on_error=block plugin returned Block?
    pub exit_code: i32,                 // 2 if block_intent else 0
}
```

**Execution ordering contract:**

- Tiers execute in ascending priority order (lower priority = earlier)
- Plugins within a tier execute concurrently; ordering between them is
  unspecified
- All events emitted during tier N are guaranteed visible to sinks before
  tier N+1 begins (the sink pipeline flushes at tier boundaries; S-1.8
  enforces)
- `emit_event()` is non-blocking; plugins return without waiting for sink
  delivery

**Isolation guarantee:**

- A panic / timeout / crash in plugin P in tier T does not affect any
  other plugin in tier T, nor any plugin in tier T+1, T+2, ...
- HostContext is cloned per plugin; mutations by one plugin are invisible
  to siblings

## Test plan

- Unit + integration tests listed above
- Benchmark: 20 plugins × 100ms baseline → verify ~100ms parallel
  execution (vs ~2000ms sequential)
- Stress test: 200 plugins in one tier, all succeed; verify tokio handles
  the fanout without exhausting threadpool

## Notes

Wasmtime invocation is synchronous; we use `spawn_blocking` to offload.
tokio's `current_thread` flavor is fine for the dispatcher since we only
have a handful of concurrent plugins typical (tens, not thousands).

An alternative considered: run all plugins on the main task sequentially.
Rejected because `sum(timeouts)` is a real UX problem — see Q3 resolution.

Event ordering: we flush the sink queue at tier boundaries. This means
tier N+1 plugins can observe tier N plugins' side effects via sink
inspection (if any plugin does that, which it shouldn't — but the guarantee
is useful for priorities-mean-dependencies semantics).

Plugins in the same tier must be independent. Document this in
authoring-hooks.md. If plugin A needs plugin B's output, put B in an
earlier tier.
