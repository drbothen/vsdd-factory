## Goal

End-to-end validation that the observability system works under
realistic configurations. Exercises: zero-disk mode, hybrid mode, multi-
instance of the same sink type, routing filters, tag enrichment,
circuit breaker behavior, dead-letter replay. Covers the rc.1 gate
criteria 10–12.

## Acceptance criteria

**Zero-disk mode (rc gate criterion 10):**
- [ ] `observability-config.toml` with only `datadog` sink (no `file`)
- [ ] Run 100 events through the dispatcher
- [ ] Verify `.factory/logs/events-*.jsonl` is NOT created (or is empty)
- [ ] Verify `dispatcher-internal.jsonl` IS created (always-on per Q6)
- [ ] Verify 100 events arrived at Datadog (mock receiver)

**Hybrid mode (rc gate criterion 11):**
- [ ] `file` + `datadog` both enabled
- [ ] Run 100 events
- [ ] Verify file sink wrote all 100
- [ ] Verify datadog received all 100
- [ ] Kill the Datadog receiver mid-stream; verify file keeps writing
  and datadog events land in dead-letter queue after retries

**Circuit breaker (rc gate criterion 12):**
- [ ] Bring a sink down for 10 minutes (mock unreachable)
- [ ] Verify circuit opens after 5 failures
- [ ] Verify events route to DLQ after breaker opens
- [ ] Verify breaker probes periodically; stays open while sink is down
- [ ] Bring sink back up; verify next probe closes the circuit
- [ ] Verify subsequent events submit successfully

**Routing + tagging:**
- [ ] Two file sinks with different `event_types` filters; verify each
  receives only matching events
- [ ] Datadog sink with `tags = ["env:prod"]` vs Honeycomb with
  `tags = ["team:factory"]`; verify each event has its sink's tags

**Port validation:**
- [ ] `capture-commit-activity` (native WASM, S-3.1) emits `commit.made`
  that lands correctly in every configured sink
- [ ] Same for `capture-pr-activity` and `block-ai-attribution`

## Tasks

1. Set up test harness: spawn mock HTTP/gRPC receivers in tokio, bind
   to random ports, pass ports via env to the test dispatcher.
2. Write each scenario as an integration test in
   `crates/factory-dispatcher/tests/observability_e2e.rs`.
3. Include mock receivers for Datadog and Honeycomb payload shapes.
4. Add chaos testing: random failure injection across 10K events,
   verify bounded data loss (DLQ catches drops, no silent loss).
5. Document the tests' role in rc.1 gate check.

## Artifacts

## Contracts

**Gate criteria coverage:**

This story is the implementation of rc.1 gate criteria 10, 11, 12. Each
criterion must have a corresponding named test that `cargo test` runs
and the rc.1 release gate (S-4.8) verifies.

**CI integration:**

Tests run on CI for Linux + macOS. Full coverage on Windows may require
slight adaptation (port binding, temp paths); document any platform
differences.

## Test plan

- Each scenario above runs as `cargo test`
- Tests are idempotent — can re-run without cleanup
- Each test's expected outcome is specific enough that a regression
  fails loudly
- Chaos tests run with a fixed seed for reproducibility; a second run
  with a different seed also passes

## Notes

Mock receivers: a small tokio-based HTTP server that records requests
to an in-memory buffer is sufficient for HTTP-based sinks. For
otel-grpc, use the `tonic` test utilities to bind a real gRPC server.

Timing-sensitive tests (circuit breaker cooldowns) should use fake
clocks where possible (`tokio::time::pause`) to avoid wall-clock waits
in CI.

The rc.1 gate (S-4.8) runs these tests as part of its walkthrough.
Failing any of them blocks the gate.
