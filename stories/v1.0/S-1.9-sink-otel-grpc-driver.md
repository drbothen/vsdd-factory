# S-1.9: sink-otel-grpc driver

**Epic:** [v1.0 Factory Plugin Kit](EPIC.md)
**Phase:** Phase 1 — Dispatcher foundation (feeds into Phase 4 sink system)
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.7, S-4.7
**Blocked by:** S-1.1, S-1.8 (needs the `Sink` trait)
**Estimated effort:** M

## Goal

Implement the `otel-grpc` sink type — forwards events to an OTel collector
over OTLP/gRPC. This is the second sink type in beta.1 scope, and the one
that feeds the existing local Grafana/Loki/Prometheus stack directly
without requiring the filelog receiver pattern (which stays supported via
`sink-file` + existing collector config).

## Acceptance criteria

- [ ] `OtelGrpcSink` implements the `Sink` trait from S-1.8
- [ ] Connects to configured OTLP endpoint (default:
  `http://localhost:4317` — matches current local stack)
- [ ] Events serialized as OTLP log records (using `opentelemetry` crate)
- [ ] Resource attributes populated from config:
  `service.name = "vsdd-factory"`, `host.name`, plus per-sink `tags`
- [ ] Event attributes include full event body (type, dispatcher_trace_id,
  session_id, plugin_name, plugin_version, and all event-specific fields)
- [ ] Retries/failures handled by the runtime (S-4.4); this story just
  implements the submission path
- [ ] Batching: events collected and submitted in batches per
  `batch.size` / `batch.interval_ms` (defaults 100 / 5000ms)
- [ ] Integration test with a local mock OTLP receiver verifies events
  arrive correctly formatted
- [ ] Works against the real OTel collector from the existing
  `observability/docker-compose.yml`

## Tasks

1. Create `crates/sink-otel-grpc/` crate.
2. Add `opentelemetry`, `opentelemetry-otlp`, `tonic` dependencies.
3. Implement `OtelGrpcSink` struct with config:
   - `endpoint: String`
   - `resource_attrs: HashMap<String, String>`
   - `batch_size: usize`
   - `batch_interval: Duration`
4. Async background task consuming from mpsc:
   - Batch events by count or time-interval threshold (whichever first)
   - Convert each event to an OTLP `LogRecord` with attributes
   - Submit batch via `opentelemetry_otlp::new_exporter`
5. Include all event body fields as log-record attributes (flat, not
   nested) for Loki label compatibility.
6. Implement `flush()` — send any pending batch immediately.
7. Register OtelGrpcSink in the SinkRegistry loader (S-1.8).
8. Integration tests:
   - Local mock OTLP receiver (a tokio task bound on a random port,
     records received payloads)
   - Verify 100 events batched correctly
   - Verify flush() empties the batch
   - Verify endpoint-unreachable case enqueues errors for retry runtime
     (S-4.4 implements actual retries)

## Contracts

**Config shape:**

```toml
[[sinks]]
name = "local-grafana"
type = "otel-grpc"
enabled = true
endpoint = "http://localhost:4317"

# Optional; all have defaults
[sinks.local-grafana.resource_attributes]
"service.name"      = "vsdd-factory"
"deployment.env"    = "dev"

[sinks.local-grafana.batch]
size        = 100
interval_ms = 5000

# Tags add to attributes on every emitted record
tags = ["env:dev", "team:factory"]
```

**OTLP record mapping:**

```
event.type                  → body.string_value
event.dispatcher_trace_id   → attribute
event.session_id            → attribute
event.plugin_name           → attribute
event.plugin_version        → attribute
event.ts_epoch              → timestamp (UnixNano)
event.*                     → flat attributes
```

This mapping is stable across sink-otel-grpc versions. Changes require an
explicit SDK/dispatcher version bump.

## Test plan

- Unit: event → OTLP LogRecord conversion produces expected structure
- Unit: batching logic (flush on size, flush on interval)
- Integration: mock receiver, 100-event batch, verify all received
- Integration: endpoint unreachable → events retained in queue (S-4.4

- E2E: run dispatcher against the real `observability/docker-compose.yml`
  OTel collector; verify events appear in Loki via HTTP query

## Notes

The existing `observability/otel-collector-config.yaml` has an OTLP
receiver at `:4317` with logs routed to Loki. That's the reference
deployment the tests should target.

Resource attributes vs record attributes: resource attrs describe "what
service is emitting" (constant per sink); record attrs describe "what
this specific event is about" (per event). Put tags in both where it
makes sense (as a record attr too, so queries work without resource
joins).

Alternatively we could use the opentelemetry SDK's default log-emitter
path. This is simpler but less control over batching; the manual approach
gives us tighter resilience integration with S-4.4.
