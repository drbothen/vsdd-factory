# S-0.05: Documentation scaffolding

**Epic:** E-0 — Infrastructure Prep
**Phase:** Phase 0 — Infrastructure prep
**Milestone:** `1.0.0-beta.1`
**Tier:** A
**Blocks:** none (docs don't block release, but incomplete docs block 1.0.0 gate)
**Blocked by:** none
**Estimated effort:** S

## Narrative
- **As a** plugin author or operator
- **I want to** find structured documentation skeletons for authoring hooks, porting bash, migration, and observability
- **So that** implementation stories have named files and sections to fill in as work lands

## Goal

Create the documentation skeleton for the v1.0 release so contributors
have a stable structure to fill in as implementation stories land. Docs
don't block the technical work, but shipping 1.0.0 without a migration
guide or plugin authoring guide would be a gate failure.

## Capability Anchor Justification

Anchoring to CAP-014 ("Decompose product specs into verified behavioral contracts") per capabilities.md §CAP-014. This story creates structured documentation skeletons — `docs/guide/*.md` with section headings and TODO markers — which are the documentation layer of the vsdd-factory product spec. BC-8.26.001 (story-completeness 14-check audit) and BC-8.22.001 (Conventional Commits) govern the process discipline for this skeleton-only story.

> **Stretch-Anchor Disclosure:** CAP-014's primary mandate is BC-producing spec methodology; documentation scaffolding is a secondary concern. No existing SS-08 BC directly contracts "create docs/guide/ skeleton files." BC-8.26.006 (user-facing-docs deliverable completeness) was initially anchored here but has been intentionally removed (F-003, Wave 8 pass-1): skeleton-only stories are not complete deliverables, so check 5 of story-completeness does not apply. v1.1 candidate BC-8.31.001 (skeleton-shape contract) addresses the uncontracted skeleton creation pattern. BC-8.26.006 remains anchored to S-5.05 and S-5.06 (genuine full deliverables). If a dedicated CAP for user-facing documentation authoring is created in a future cycle, this story should be re-anchored.

## Contracts

**Stub structure per file:**

```markdown
# <Document title>

**Status:** skeleton — filled in by stories S-X.Y and S-X.Z

## <Section 1>

<!-- TODO(S-X.Y): section content -->
```

Stories that implement features replace their own `TODO(S-X.Y)` markers.
A grep of `docs/guide/` for remaining `TODO(S-` markers at release time
reveals incomplete docs.

## Test plan

- Manual: open each skeleton file, verify headings and TODO markers are in place.
- Pre-release gate (S-5.07): grep for `TODO(S-` in `docs/guide/` returns nothing.

## Notes

This is explicitly about documentation SHAPE, not CONTENT. The goal is to
have named files with named sections so that implementation stories know
exactly where their docs land.
