# S-0.05: Documentation scaffolding

**Epic:** E-0 — Infrastructure Prep
**Phase:** Phase 0 — Infrastructure prep
**Milestone:** `1.0.0-beta.1`
**Tier:** A
**Blocks:** none (docs don't block release, but incomplete docs block 1.0.0 gate)
**Blocked by:** none
**Estimated effort:** S

## Narrative
- **As a** plugin author or operator
- **I want to** find structured documentation skeletons for authoring hooks, porting bash, migration, and observability
- **So that** implementation stories have named files and sections to fill in as work lands

## Goal

Create the documentation skeleton for the v1.0 release so contributors
have a stable structure to fill in as implementation stories land. Docs
don't block the technical work, but shipping 1.0.0 without a migration
guide or plugin authoring guide would be a gate failure.

## Acceptance criteria

- [ ] `docs/guide/authoring-hooks.md` exists with section stubs for:
  introduction, SDK overview, plugin structure, host functions,
  capabilities, testing, deployment
- [ ] `docs/guide/porting-bash-hook-to-wasm.md` exists with section stubs
  for: assessment checklist, common patterns, capturing stdin, emitting
  events, testing, gotchas
- [ ] `docs/guide/migrating-from-0.79.md` exists with section stubs for:
  what changed, activation requirement, windows status, troubleshooting
- [ ] `docs/guide/observability-sinks.md` exists with section stubs for:
  supported sink types, file sink, http sink, otel-grpc sink, datadog sink,
  honeycomb sink, multi-sink recipes, resilience tuning
- [ ] Each stub file has a banner indicating status:
  `Status: skeleton — fill in during PhaseN implementation`
- [ ] Top-level docs index references the new files
- [ ] No technical content in the skeletons yet — only structure

## Tasks

1. Identify existing docs conventions in `docs/guide/`.
2. Create the 4 new skeleton files with section-level headings and a status banner.
3. Cross-link from `docs/guide/index.md` (or existing index).
4. Add TODO comments in each section linking to the story that should fill it in.
5. Commit with a clear message establishing the pattern for docs work.

## Contracts

**Stub structure per file:**

```markdown
# <Document title>

**Status:** skeleton — filled in by stories S-X.Y and S-X.Z

## <Section 1>

<!-- TODO(S-X.Y): section content -->
```

Stories that implement features replace their own `TODO(S-X.Y)` markers.
A grep of `docs/guide/` for remaining `TODO(S-` markers at release time
reveals incomplete docs.

## Test plan

- Manual: open each skeleton file, verify headings and TODO markers are in place.
- Pre-release gate (S-5.07): grep for `TODO(S-` in `docs/guide/` returns nothing.

## Notes

This is explicitly about documentation SHAPE, not CONTENT. The goal is to
have named files with named sections so that implementation stories know
exactly where their docs land.
