**Complexity note (Low — 3 pts):** track-agent-start is a pure telemetry hook. It:

extraction → Rust regex crate; `_emit()` → `host::emit_event`. All three replacements

## Narrative

- **As a** vsdd-factory operator running agent pipelines
- **I want** track-agent-start to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** agent.start telemetry events fire on all platforms (including Windows
  without jq), without legacy-adapter indirection overhead, and with no jq dependency

## Goal

Port `plugins/vsdd-factory/hooks/track-agent-start.sh` to a native Rust WASM crate at
`crates/hook-plugins/track-agent-start/`. The hook fires on PreToolUse filtered to the
Agent tool. It reads stdin JSON, extracts `tool_input.subagent_type` and
`tool_input.prompt`, applies two regex patterns to extract a story_id from the prompt
(pattern 1: `S-[0-9]+\.[0-9]+`; pattern 2: `STORY-[0-9]+` as fallback), then calls
`host::emit_event` with `type=agent.start hook=track-agent-start matcher=Agent
subagent=<subagent_type>` (plus optional `story_id=<id>` if extracted). Always exits 0.

Behavior parity is the sole requirement (E-8 D-2). The hook is best-effort: missing jq
(bash), missing JSON fields, and emit_event failures all result in silent exit 0 (no
error output). The WASM port must preserve this best-effort semantics.

After the port, the `.sh` file is deleted, and the registry entry updated to reference
the native `.wasm`. The `hooks.json.*` entries for this hook do not exist in the current
codebase (D-7 already shipped registry-only dispatch per DRIFT-004); no deletion of
hooks.json entries is required — only verification that zero entries exist.

### AC-004 (traces to BC-7.03.080 postcondition 1)
Story ID extraction follows the two-pattern cascade from the bash source:
1. Search prompt for pattern `S-[0-9]+\.[0-9]+` (e.g., `S-8.03`); use first match.
2. If no match, search for `STORY-[0-9]+` (e.g., `STORY-042`); use first match.
3. If neither matches, omit `story_id` field from the event entirely.
When a story_id is found, the emitted event includes `story_id=<extracted_id>`.
Bats test verifies: prompt containing "S-6.07" produces `story_id=S-6.07`;
prompt containing "STORY-042" (no S-N.NN) produces `story_id=STORY-042`;
prompt with no story pattern produces event without story_id field.

### AC-005 (traces to BC-7.03.079 postcondition 1 + BC-7.03.080 postcondition 1)
Bats parity test at `tests/integration/hooks/track-agent-start.bats` covers:
(a) Agent dispatch with subagent=pr-manager, prompt containing "S-6.07" →
    exit 0 + agent.start event with subagent=pr-manager story_id=S-6.07;
(b) Agent dispatch with subagent=implementer, prompt containing "STORY-042" →
    exit 0 + agent.start event with subagent=implementer story_id=STORY-042;
(c) Agent dispatch with subagent=reviewer, prompt with no story pattern →
    exit 0 + agent.start event with subagent=reviewer, no story_id field;
(d) Non-Agent tool_name in envelope → exit 0, no event emitted;
(e) Malformed JSON stdin → exit 0, no event, no panic;
(f) WASM warm invocation time measured via `hyperfine --warmup 3 --runs 10` and
    recorded in implementation notes (INFORMATIONAL — non-blocking; Tier 1 hooks are
    excluded from the 20% regression gate per E-8 epic AC-7 Goal #6).
All bats tests pass.

### AC-006 (traces to BC-7.03.079 invariant 2)
Best-effort semantics preserved: if JSON parse fails, if `tool_name != "Agent"`,
or if `host::emit_event` returns an error, the hook exits 0 with no stderr output.
No error is surfaced to the dispatcher. The emit_event error path uses
`let _ = host::emit_event(...);` — error is dropped, not logged. Bats test verifies:
`host::emit_event` returning Err (mocked in unit test) → exit 0, no stderr, no panic.
Verify in bats integration test (case d + e above).
`bin/emit-event` binary is NOT removed (E-8 D-10 — deferred to S-8.28).
