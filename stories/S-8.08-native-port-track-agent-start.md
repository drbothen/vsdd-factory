## Narrative

- **As a** vsdd-factory operator running agent pipelines
- **I want** track-agent-start to execute as a native WASM plugin instead of through the legacy-bash-adapter
- **So that** agent.start telemetry events fire on all platforms (including Windows
  without jq), without legacy-adapter indirection overhead, and with no jq dependency

## Goal

Port `plugins/vsdd-factory/hooks/track-agent-start.sh` to a native Rust WASM crate at
`crates/hook-plugins/track-agent-start/`. The hook fires on PreToolUse filtered to the
Agent tool. It reads stdin JSON, extracts `tool_input.subagent_type` and
`tool_input.prompt`, applies two regex patterns to extract a story_id from the prompt
(pattern 1: `S-[0-9]+\.[0-9]+`; pattern 2: `STORY-[0-9]+` as fallback), then calls
`host::emit_event` with the following fields (strict E-8 D-2 bash parity):

agent.start emission fields: `type=agent.start`, `hook=track-agent-start`,
`matcher=Agent`, `subagent=<subagent_type>`, and optional `story_id=<extracted-from-prompt>`.
**Strict E-8 D-2 bash parity: NO `agent_id`, NO `tool_name`, NO additive fields.**

This matches the bash source exactly (lines 43-44 of `track-agent-start.sh`):
```
_emit type=agent.start hook=track-agent-start matcher=Agent \
      subagent="$SUBAGENT" ${STORY_ID:+story_id="$STORY_ID"}
```

Always exits 0. Behavior parity is the sole requirement (E-8 D-2). The hook is
best-effort: missing jq (bash), missing JSON fields, and emit_event failures all result
in silent exit 0 (no error output). The WASM port must preserve this best-effort
semantics.

After the port, the `.sh` file is deleted, and the registry entry updated to reference
the native `.wasm`. The `hooks.json.*` entries for this hook do not exist in the current
codebase (D-7 already shipped registry-only dispatch per DRIFT-004); no deletion of
hooks.json entries is required — only verification that zero entries exist.

**Future Work (TD-015):** Per-invocation telemetry correlation — if a downstream sink
emerges that requires correlation across the same Task call (e.g., dashboard wants "all
events from this specific subagent invocation"), this is a post-v1.0 epic with proper SDK
extension `host::invocation_id()`, BC-7.03.080 + BC-7.03.082 schema amendments, and a
coordinated sweep across all `track-agent-*` and `*-completion-guard` hooks. Adding
speculative fields during the E-8 parity port is explicitly out of scope per E-8 D-2.
