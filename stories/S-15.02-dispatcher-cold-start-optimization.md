# S-15.02: Dispatcher cold-start optimization — daemon mode + WASM AOT cache

**Epic:** E-15 — Plugin Async Semantics — Registry-Layer Partition (single-shot delivery)
**Delivery order:** Second story in E-15; follow-up optimization deferred from S-15.01 per ADR-020 §Out of Scope.
**Depends on:** S-15.01 (schema v2 + partition runtime must be live before daemon mode can be layered)
**Blocks:** none
**Priority:** P2 (deferred architectural optimization; S-15.01 ships first)

## User Story

As a Claude Code user, I want hook dispatch to complete in <100ms p95 so my edit/write
actions feel instant rather than introducing a perceptible ~1 second pause.

## Background

### The structural cost problem

Under the current per-invocation binary-spawn architecture (Class A per ADR-020), every
Claude Code hook event forces the platform to spawn a fresh `factory-dispatcher` process.
On macOS arm64, the dominant costs are:

| Cost component | Observed contribution |
|---------------|----------------------|
| macOS fork + exec + dylib load | ~200–400ms |
| WASM cold-start (module load + compile) | ~300–600ms |
| Sync group plugin execution | ~40–100ms (small fraction) |
| **Total p50 (release, N=100)** | **~940ms** |
| **Total p95 (release, N=100)** | **1050ms** |

The plugin execution time itself is not the bottleneck — the structural spawn overhead
dominates. S-15.01 applied the T-3h misclassification audit (10 telemetry plugins
classified `async = true`) and confirmed that the remaining sync_group latency cannot be
reduced further without architectural changes to the spawn model.

ADR-020 §Class A accepted 1500ms as the operative budget for this architecture. ADR-020
§Class B defines the target budget for a persistent-process architecture: **p95 ≤ 50ms**
post-warmup. This story implements the architectural changes required to reach Class B.

### Out-of-scope constraints (locked, must not be changed by this story)

| Constraint | Source |
|-----------|--------|
| WASM plugin ABI | ADR-002 (locked) |
| Registry schema (currently v2) | BC-7.06.001 + ADR-019 §Decision schema_version = 2 |
| hooks.json envelope format | ADR-019 §Decision 1 + BC-9.01.006 (locked) |

This story is a **runtime-infrastructure change only**. No ABI, schema, or envelope
changes are permitted.

## Scope

### In scope

1. **Persistent dispatcher daemon** — a long-lived `factory-dispatcher` process that
   receives hook events over a Unix domain socket (or stdin-based IPC, TBD at design
   time) and processes them in a persistent Tokio runtime. Process spawn and dylib load
   costs are incurred once at daemon startup, not per-event.

2. **WASM AOT pre-compilation cache** — WASM modules are compiled to native code on
   first load. The compiled form is written to a cache file under
   `$HOME/.cache/factory-dispatcher/wasm/` (or `$XDG_CACHE_HOME/factory-dispatcher/wasm/`
   if set). Cache invalidation triggers on module file-hash change. Cache hit eliminates
   WASM cold-start cost (~300–600ms) on subsequent invocations.

### Out of scope

- WASM plugin ABI changes (locked by ADR-002)
- Registry schema changes (locked by hooks-registry.toml schema_version v2 per ADR-019)
- hooks.json envelope changes (locked by ADR-019 §Decision 1 + BC-9.01.006)
- Any behavioral changes to sync_group verdict aggregation (inherited from S-15.01)
- Any behavioral changes to the async drain window (governed by DI-019, unchanged)

## Acceptance Criteria

> **Status:** Initial draft. ACs will be sharpened when this story enters its F1-F4
> cycle and BCs are authored. Trace targets are placeholder until BC IDs are assigned.

### AC-001 — Persistent daemon mode (traces to BC-SS01-daemon-lifecycle — TBD pending BC authorship)

The dispatcher binary can be launched in daemon mode. In daemon mode it binds a Unix
domain socket (or equivalent IPC channel) and enters an event-handling loop. Claude
Code hook events are delivered to the running daemon via the IPC channel rather than
spawning a new process per event. The daemon persists across multiple hook events in a
single Claude Code session.

**Falsifiable test:** Daemon starts, receives 3 sequential PostToolUse events via IPC
socket, processes each correctly, and remains running after all 3 events complete.

### AC-002 — Daemon detection and event forwarding (traces to BC-SS01-daemon-detection — TBD pending BC authorship)

When the dispatcher binary is invoked in single-shot mode (standard per-invocation
spawn) and a daemon is already running, the binary detects the running daemon via the
IPC socket file and forwards the event payload to the daemon for processing. The
per-invocation binary exits after forwarding; the daemon handles execution and emits
the verdict. This eliminates fork + exec overhead on all post-first invocations.

**Falsifiable test:** Two invocations of the dispatcher binary in sequence; first
invocation starts the daemon; second invocation detects the daemon and forwards the
event; both events are processed and verdicts returned correctly.

### AC-003 — WASM AOT cache stores pre-compiled modules (traces to BC-SS01-wasm-aot-cache — TBD pending BC authorship)

On first dispatch of a WASM plugin, the daemon compiles the module to native code and
writes the compiled form to `$HOME/.cache/factory-dispatcher/wasm/<module-hash>.cwasm`
(or equivalent Wasmtime AOT format). On subsequent dispatches of the same module, the
daemon loads the pre-compiled cache file. The cache hit path must complete module
initialization in <10ms (post-warmup).

**Falsifiable test:** First invocation of a WASM plugin records cache-miss and writes
`.cwasm` file. Second invocation records cache-hit; module init time < 10ms measured.

### AC-004 — Cache invalidation on module file-hash change (traces to BC-SS01-wasm-aot-cache — TBD pending BC authorship)

When a WASM plugin binary changes on disk (file-hash changes), the existing `.cwasm`
cache entry is invalidated. The next dispatch recompiles from the new `.wasm` source and
writes a new cache entry. No stale-cache failures occur: a cache entry is never used
when its source `.wasm` has changed.

**Falsifiable test:** Write a WASM plugin, trigger cache creation, replace the `.wasm`
binary with a modified version (different hash), trigger another dispatch. Assert the
old cache entry is not used and a new `.cwasm` is written with the updated hash.

### AC-005 — Class B latency canary: p95 ≤ 50ms post-warmup (traces to BC-SS01-class-b-budget — TBD pending BC authorship; ADR-020 §Class B)

With the daemon running and WASM AOT cache warm (all modules pre-compiled), a
representative PostToolUse / Write envelope dispatched via IPC completes in p95 ≤ 50ms.
Measurement: N=100 events, macOS arm64, release binary, daemon pre-warmed with at least
one prior invocation of each WASM module in the sync_group.

Evidence stored at `docs/demo-evidence/S-15.02/latency-canary.md`.

**Falsifiable test:** Latency canary script measures 100 IPC-dispatched events
post-warmup and asserts p95 ≤ 50ms. The result is committed to the demo-evidence file.

### AC-006 — Graceful daemon shutdown on SIGTERM (traces to BC-SS01-daemon-lifecycle — TBD pending BC authorship)

When the daemon receives SIGTERM, it completes any in-flight event dispatch (up to
a bounded grace period, TBD at design time), closes the IPC socket, removes the socket
file, and exits cleanly. No orphaned processes remain after the daemon exits. No
in-flight events are silently dropped (any in-flight event that cannot complete within
the grace period emits a termination event to FileSink before exit).

**Falsifiable test:** Daemon processing a long-running sync plugin receives SIGTERM;
asserts clean exit (no zombie process), IPC socket file removed, no orphaned child
processes.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Daemon not yet running on first invocation | Single-shot invocation starts the daemon; subsequent invocations forward to it |
| EC-002 | Daemon crash between invocations | Next invocation detects stale socket file, starts a fresh daemon |
| EC-003 | WASM module updated while daemon is running | Cache invalidation detects hash mismatch; recompile before next dispatch |
| EC-004 | Cache directory does not exist | Daemon creates it on first write; no failure on absent directory |
| EC-005 | SIGTERM during in-flight dispatch | Grace-period shutdown; in-flight event completes or emits termination event; clean exit |
| EC-006 | Corrupted `.cwasm` cache file | Daemon detects load failure, falls back to cold-start compile; rewrites valid cache |
| EC-007 | Multiple Claude Code sessions concurrently | IPC socket arbitration TBD; one daemon per user session may be required |
| EC-008 | Cache full / disk out of space | Eviction policy TBD; fallback to cold-start; no hard failure |

## Previous Story Intelligence

**Predecessor:** S-15.01 — Plugin async semantics (schema v2 + partition + envelope flip).

Key lessons to carry forward:

- The binary-spawn canary in S-15.01 confirmed the structural cost breakdown (OS spawn
  ~200–400ms; WASM cold-start ~300–600ms). These are the exact costs this story
  eliminates. The Class A p50=940ms / p95=1050ms measurements from S-15.01 are the
  baseline this story must improve against.
- Wasmtime's caching API was not used in S-15.01; verify Wasmtime version pin in
  `Cargo.toml` before designing the AOT cache (use the exact pinned version, not training
  data estimates).
- The dispatcher exposes NO `--registry` or `--event` CLI flags (lesson from VP-078
  Harness 2 F-P2-007). Registry path comes from `CLAUDE_PLUGIN_ROOT` env var; event
  payload comes from stdin. The daemon IPC design must preserve this env-var-based
  registry loading.
- VP-079 Scenarios 1 and 4 require a sync plugin in fixture registries to keep the
  dispatcher alive during async task execution. Daemon mode may require analogous fixture
  design discipline.

## Architecture Compliance Rules

1. **No ABI changes** (ADR-002): WASM plugin interfaces are frozen. The AOT cache must
   store compiled versions of unmodified `.wasm` binaries. No plugin source changes.

2. **No schema changes** (ADR-019 + BC-7.06.001): `hooks-registry.toml` schema_version
   remains v2. No new registry fields for daemon mode.

3. **No envelope changes** (ADR-019 §Decision 1 + BC-9.01.006): `hooks.json` envelope
   format is frozen. Daemon-mode dispatch must accept the same envelope format as the
   per-invocation binary.

4. **Partition function remains pure** (BC-1.14.001 + VP-077): `partition_plugins()` in
   `partition.rs` is a pure function. Daemon mode does not change this. The daemon's
   event loop calls the same pure partition function.

5. **ASYNC_DRAIN_WINDOW_MS remains DI-019** (BC-1.14.001 + DI-019): The drain window
   constant does not change. Daemon mode does not affect the async drain window.

6. **SS-01 boundary**: All new code lives in `crates/factory-dispatcher/src/`. No new
   SS boundaries created. No cross-crate dependencies on the new daemon module except
   from `main.rs`.
