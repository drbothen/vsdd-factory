# S-7.02: State-manager defensive sweep + count-propagation hook + meta-rule

> **Cluster decision:** Lessons #2 (defensive propagation sweep), #4 (VP
> multi-BC convention enforcement), and #8 (self-referential codification
> meta-rule) are grouped here because all three are about systematic propagation
> discipline and live in the tooling/hooks/rules layer rather than the agent
> prompt layer (S-7.01). They share a common delivery surface: one agent prompt
> edit (state-manager.md), one new Bash hook, one new rule file, and one hook
> registration entry. Grouping them avoids a trivially small S-7.03.

## Narrative

- **As a** vsdd-factory orchestrator maintaining corpus-wide consistency
- **I want** the state-manager agent to run a defensive grep sweep before
  declaring any count change complete, a new lint hook to catch cross-document
  count drift automatically, and a codified rule requiring that every novel
  adversary process catch generates a codification follow-up
- **So that** the class of wasted passes caused by incomplete propagation
  (F-027 — state-manager declared count change complete after updating only 2
  of 4 index files) and the risk of re-discovering the same process gaps in
  future cycles are both structurally prevented

## Acceptance Criteria

### AC-001: State-manager defensive sweep protocol (traces to BC-5.37.001, BC-5.37.002)

The state-manager agent prompt must include a **Defensive Propagation Sweep**
protocol in its update workflow:

> "Before declaring any count-changing update complete (e.g., 'BC count is now
> 1,875'), the state-manager MUST run a corpus-wide grep to identify all files
> that contain the old count as a literal string. The sweep covers at minimum:
> `STATE.md`, `ARCH-INDEX.md`, `BC-INDEX.md`, `VP-INDEX.md`, `STORY-INDEX.md`,
> all `SS-NN-*.md` sharded architecture files, and `prd.md`. Any file still
> containing the old count after the update is a propagation gap — fix it before
> committing."

This closes F-027 from s6.01-pass-1.md (state-manager updated capabilities.md
count but did not grep for sibling files that also embed the count).

- **Test:** Peer review of state-manager.md confirms Defensive Propagation Sweep
  section is present with the file list above.
- **Test:** The protocol is placed in the "STATE.md Update Protocol" section or
  immediately adjacent to count-related update instructions.

### AC-002: validate-count-propagation.sh hook (traces to BC-7.05.001, BC-7.05.002)

A new Bash lint hook `plugins/vsdd-factory/hooks/validate-count-propagation.sh`
must be written with the following behavior:

**Trigger:** Runs as a PostToolUse hook on Write/Edit operations to
`STATE.md`, `ARCH-INDEX.md`, `BC-INDEX.md`, `VP-INDEX.md`, or any `SS-NN-*.md`
architecture file.

**Behavior:**
1. Extract all integer counts from the modified file (regex: `\b\d{2,5}\b` in
   context of known count-bearing patterns: "NNN BCs", "NNN VPs", "NNN stories",
   "Total.*NNN").
2. For each extracted count, grep the corpus (same file list as AC-001) for the
   same count in the same semantic context.
3. If the same count-keyword pair appears in sibling documents at a DIFFERENT
   value, emit a warning:
   ```
   COUNT DRIFT DETECTED: 'NNN BCs' appears as NNN in FILE-A but MMM in FILE-B.
   Run: grep -r "BCs" .factory/specs/ .factory/STATE.md to reconcile.
   ```
4. Exit non-zero only on detected drift (not on absence of the count in sibling
   files — absence is not drift).

**Scope limit:** The hook checks count consistency across known index files. It
does NOT attempt semantic interpretation of counts — it reports drift and lets
the agent reconcile.

- **Test:** `test_count_drift_detected_exits_nonzero()` — simulate STATE.md with
  "1,875 BCs" and ARCH-INDEX.md with "1,863 BCs"; hook must exit non-zero.
- **Test:** `test_count_match_exits_zero()` — same count in both files; hook
  must exit 0.
- **Test:** `test_count_absent_from_sibling_exits_zero()` — count present in
  modified file but absent from sibling; hook must exit 0 (absence is not drift).

### AC-004: lessons-codification.md rule file (traces to BC-8.28.001, BC-8.28.002)

A new rule file `plugins/vsdd-factory/rules/lessons-codification.md` must be
written with the following content:

**Rule:** "Every novel adversary process catch → codification follow-up before
cycle closure."

**Definition of 'novel process catch':** A finding that identifies a gap in an
agent prompt, hook, rule file, or pipeline workflow — as opposed to a gap in a
specific spec artifact (BC, VP, story). Process findings are categorized by the
adversary using the tag `[process-gap]` in the finding's Observations section.

**Required follow-up:** When the orchestrator closes a sub-cycle, it must:
1. Scan the final convergence report for any `[process-gap]` tagged findings.
2. For each such finding, open a follow-up E-N story (in the appropriate
   self-improvement epic) or record a deferred item in the Drift Items table
   with an explicit justification.
3. The cycle is not declared CLOSED until either a story exists or a justified
   deferral is logged.

**Enforcement:** The orchestrator references this rule during cycle-closure
checklist review. The adversary references it when tagging findings.

- **Test:** Rule file exists at the specified path.
- **Test:** Rule file contains the three-step "Required follow-up" protocol.
- **Test:** Rule file contains the `[process-gap]` tag definition.

### AC-005: VP multi-BC convention — template-compliance hook check (traces to BC-7.05.003)

This closes lesson #4 (PARTIAL status in STATE.md). The VP multi-BC convention
(`source_bc=primary BC`, `bcs[]=full list of BCs exercised`) was documented in
commit 7765573. This AC adds automated enforcement:

The existing `validate-template-compliance.sh` must be extended (or a new check
added to it) to verify that VP files with `bcs:` arrays containing more than one
entry also have a `source_bc:` field populated with a single primary BC ID.

Compliance rule: `IF len(bcs) > 1 THEN source_bc must be non-empty AND source_bc must be in bcs[]`.

On violation: emit `TEMPLATE COMPLIANCE WARNING: VP-NNN has multiple bcs[] but
missing or invalid source_bc field.` Exit non-zero.

- **Test:** `test_vp_multi_bc_missing_source_bc_exits_nonzero()` — VP file with
  `bcs: [BC-A, BC-B]` and no `source_bc:` triggers warning and non-zero exit.
- **Test:** `test_vp_multi_bc_with_source_bc_exits_zero()` — VP file with
  `bcs: [BC-A, BC-B]` and `source_bc: BC-A` passes.
- **Test:** `test_vp_single_bc_no_source_bc_required_exits_zero()` — VP file
  with `bcs: [BC-A]` and no `source_bc:` passes (single-BC VPs don't need it).

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | validate-count-propagation.sh triggers during a wave-gate remediation burst where counts legitimately change mid-burst | Hook may fire on intermediate state during the burst (e.g., STATE.md updated but ARCH-INDEX not yet updated). The hook should emit a warning but NOT block the write. False-positive rate during multi-file bursts is accepted; the hook's value is post-burst catch, not intra-burst blocking. If this causes too many false positives, add a `SKIP_COUNT_PROPAGATION=1` env var escape hatch. |
| EC-002 | hooks-registry.toml already has a hook at the same trigger path (duplicate registration) | Implementer checks for existing entries before adding. If a duplicate would result, merge the trigger conditions into the existing entry rather than creating a second entry. |
| EC-003 | lessons-codification.md rule conflicts with an existing rule in `plugins/vsdd-factory/rules/` | Read all 9 existing rule files before writing. If a conflict exists, add a cross-reference note in the new rule and in the conflicting rule. Do NOT silently supersede existing rules. |
| EC-004 | validate-template-compliance.sh extension for VP multi-BC check is called against a non-VP file | The check must be scoped to files matching `VP-*.md` or having `document_type: verification-property` in frontmatter. Other file types must not trigger the VP-specific check path. |
| EC-005 | Corpus grep in AC-001 defensive sweep produces false matches (e.g., "1,863" appears in a changelog as a date or ID) | The sweep regex should be anchored to known count-bearing contexts: "NNN BCs", "NNN VPs", "NNN stories", "Total.*NNN". Raw number matches without context are ignored. The state-manager protocol must specify the anchored regex, not a bare number grep. |
| EC-006 | Rule file `lessons-codification.md` is added to `plugins/vsdd-factory/rules/` but ARCH-INDEX does not list rules individually | ARCH-INDEX lists `plugins/vsdd-factory/rules/` as "(9 files)" in the SS-08 row. This is itself a count-propagation scenario — the defensive sweep (AC-001) and hook (AC-002) should catch it if omitted. |
