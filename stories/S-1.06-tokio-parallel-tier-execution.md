# S-1.06: tokio + parallel-within-tier execution

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.01, S-2.07
**Blocked by:** S-1.01, S-1.02, S-1.04, S-1.05
**Estimated effort:** M

## Narrative
- **As a** dispatcher operator with multiple hooks registered for the same event
- **I want to** have plugins within the same priority tier run concurrently
- **So that** hook latency scales with the slowest plugin in a tier, not the sum of all plugins

## Goal

Implement the execution layer the dispatcher hands tiered plugins to.
Plugins within the same priority tier run concurrently via tokio tasks,
each with its own wasmtime store. Tiers run sequentially so priority
ordering is maintained. A `block`-result in any tier halts downstream tiers.

## Acceptance criteria

- [ ] Plugins within the same tier execute concurrently via `tokio::task::spawn_blocking`
- [ ] Tiers execute sequentially; tier N+1 starts only when all tier N plugins complete
- [ ] If any plugin in a tier returns `Block`, subsequent tiers are skipped
- [ ] Aggregate result: `Continue` if no Block, `Block(reason)` if any plugin blocked
- [ ] Total elapsed time = sum of all tier walls (not sum of all plugin walls)
- [ ] Emits `tier.started` / `tier.completed` events for each tier
- [ ] Handles mix of Ok/Timeout/Crashed results within a single tier gracefully
- [ ] Integration test: 3 tiers with 2 plugins each; verify order and concurrency
