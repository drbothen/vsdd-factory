# S-1.06: tokio + parallel-within-tier execution

**Epic:** E-1 — Dispatcher Foundation
**Phase:** Phase 1 — Dispatcher foundation
**Milestone:** `1.0.0-beta.1`
**Tier:** B.x
**Blocks:** S-2.01, S-2.07
**Blocked by:** S-1.01, S-1.02, S-1.04, S-1.05
**Estimated effort:** M

## Narrative
- **As a** dispatcher operator with multiple hooks registered for the same event
- **I want to** have plugins within the same priority tier run concurrently
- **So that** hook latency scales with the slowest plugin in a tier, not the sum of all plugins

## Goal

Implement the execution layer the dispatcher hands tiered plugins to.
Plugins within the same priority tier run concurrently via tokio tasks,
each with its own wasmtime store. Tiers run sequentially so priority
ordering is maintained. A `block`-result in any tier halts downstream tiers.

## Acceptance criteria

- [ ] Plugins within the same tier execute concurrently via `tokio::task::spawn_blocking`
  _(traces to BC-1.03.008 postcondition: plugins in same tier run concurrently; BC-1.03.012 postcondition: 5-plugin single tier all return Ok)_
- [ ] Tiers execute sequentially; tier N+1 starts only when all tier N plugins complete
  _(traces to BC-1.03.007 postcondition: between-tier order preserved; BC-1.03.015 postcondition: tier 10 plugin executes before tier 100 plugins)_
- [ ] If any plugin in a tier returns `Block`, subsequent tiers are skipped
  _(traces to BC-1.03.009 postcondition: block_intent set only when on_error=block AND plugin asks to block; EC-001 in story edge cases: block in tier 1 skips tier 2+)_
- [ ] Aggregate result: `Continue` if no Block, `Block(reason)` if any plugin blocked
  _(traces to BC-1.03.009 invariant: block_intent is aggregate across all tier results)_
- [ ] Total elapsed time = sum of all tier walls (not sum of all plugin walls)
  _(traces to BC-1.03.014 postcondition: wall < 2s for 4-plugin tier with one 120ms hang — wall = max plugin time, not sum)_
- [ ] Emits `tier.started` / `tier.completed` events for each tier
  _(traces to BC-1.03.016 postcondition: empty tier list yields summary with exit_code=0, block_intent=false)_
- [ ] Handles mix of Ok/Timeout/Crashed results within a single tier gracefully
  _(traces to BC-1.03.013 postcondition: crash does not affect sibling Ok plugins; BC-1.03.014 postcondition: timeout does not cascade to siblings)_
- [ ] Integration test: 3 tiers with 2 plugins each; verify order and concurrency
  _(traces to BC-1.03.015 postcondition: multi_tier_runs_in_priority_order — tier 10 before tier 100)_

## Capability Anchor Justification

Anchoring to **CAP-002: "Hook Claude Code tool calls with sandboxed WASM plugins"** per capabilities.md §CAP-002.

> "Every Claude Code tool invocation (Bash, Edit, Write, etc.) triggers the dispatcher, which routes to matching WASM plugins by event type and tool-regex. Plugins run in wasmtime with bounded fuel + epoch timeout."

This story implements `executor.rs`, the tier orchestration layer: plugins within the same priority tier run concurrently via `tokio::task::spawn_blocking`, and tiers execute sequentially in ascending priority order. The executor is the component that converts the `Vec<Vec<RegistryEntry>>` produced by S-1.02's routing into actual plugin invocations — it is the execution spine of CAP-002's "routes to matching WASM plugins" behavior. A `block` result in any tier halts downstream tiers and sets the `block_intent` that drives the dispatcher's exit code 2.
