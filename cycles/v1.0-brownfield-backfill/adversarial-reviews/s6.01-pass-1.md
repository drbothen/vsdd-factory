### F-007 — Atomicity claim incompatible with POSIX filesystem reality (no acknowledgment)

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** BC (correctness)
- **Artifacts:** BC-6.20.012; AC-8 in S-6.01; VP-058
- **Evidence:** BC-6.20.012 Description (line 33) claims the skill "treats the sequence ... as a single atomic unit. If any step fails, all side-effects already applied in that invocation are reverted." Invariant 1 (line 50): "No partial state is ever left in the repository after a failed invocation". POSIX filesystems offer NO multi-file atomic primitive. The "rollback" pattern (delete + revert + restore) is a *compensating-transaction* pattern, not atomicity. EC-004 (line 62) acknowledges "Rollback of ARCH-INDEX row itself fails" → "WARNING: ARCH-INDEX revert failed — manual cleanup required" — proving rollback is not guaranteed.
- **Issue:** The contract over-promises. A reader will read "atomic" and assume crash-safety (e.g., kill -9 mid-rollback leaves nothing partial). Reality: a process crash between "write new ADR" and "rollback file delete" leaves the new ADR on disk, ARCH-INDEX with the row, but no supersession patch — exactly the partial state Invariant 1 forbids. The BC has no precondition like "process is not killed" and no edge case for "skill crashes mid-rollback".
- **Recommended fix:** Either (a) rename to "best-effort rollback on graceful failure" and add EC-007: "Process killed mid-execution — partial state may persist; user must run `validate-template-compliance.sh` to detect and clean up", OR (b) explicitly enumerate the assumption: "Atomicity assumption: skill runs to completion; SIGKILL or system crash voids this contract."

### F-008 — Concurrency contract under-specified given AC-8 atomicity claim

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** BC / Story
- **Artifacts:** S-6.01 EC-007; BC-6.20.012 EC-005; BC-6.20.001
- **Evidence:** S-6.01 line 279 EC-007: *"Concurrent invocations write the same ADR-NNN simultaneously — Not explicitly guarded; skill detects on file-write collision and exits non-zero — users must serialise invocations in parallel sessions"*. BC-6.20.012 EC-005 (line 63) says "Second invocation detects file collision on write; rolls back; exits non-zero; user serialises". But BC-6.20.001's auto-allocation step (read filesystem, take max+1) is NOT atomic with the file-write step. Two parallel invocations could both compute max=13, both attempt to write ADR-014, and the loser would lose its title/subsystems work even after rollback.
- **Issue:** AC-8 claims atomicity. EC-007 admits no concurrency guard. These contradict. There's also no contract requiring *which* invocation wins (filesystem race is non-deterministic).
- **Recommended fix:** Add precondition to BC-6.20.001 / BC-6.20.012: "Skill assumes serialized invocation; concurrent invocations may produce one success and one rollback, but no rollback may produce mixed-state." Or introduce a lock file (e.g., `.factory/specs/architecture/decisions/.lock`) and add a BC for it.

### F-011 — Slug-derivation rules under-specified for non-ASCII and consecutive-hyphen handling

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** BC
- **Artifacts:** BC-6.20.008 EC-001; AC-4
- **Evidence:** BC-6.20.008 EC-001 (line 60) says "Slug strips `/`, `<`, `>`; becomes `adr-with--special-chars` (consecutive hyphens collapsed or preserved — implementation choice, documented in SKILL.md)". ### F-015 — Idempotency contract has a contradiction

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** BC
- **Artifacts:** BC-6.20.011 EC-004; BC-6.20.012 EC-006; BC-6.20.012 Postcondition 5
- **Evidence:**
  - BC-6.20.012 Postcondition 5 (line 47): "After a full rollback, re-invoking the skill with identical arguments produces the same result as a first-time invocation (idempotency)."
  - BC-6.20.011 EC-004 (line 73): "Re-invocation after validation failure (idempotency) — Skill detects ADR-NNN file already exists (BC-6.20.002 duplicate check); must omit `--id` and let skill clean up or user deletes file manually before re-run"
  - BC-6.20.012 EC-006 (line 64): "Re-invocation after EC-002 (file still on disk from previous failure) — BC-6.20.002 duplicate check triggers; user must delete the leftover file before re-running"
- **Issue:** Postcondition 5 says re-invocation with "identical arguments" produces the same result as first-time. But EC-004 and EC-006 say the opposite — after a validation failure, the file persists (per BC-6.20.011 invariant 3) and re-invocation with the same args triggers BC-6.20.002 duplicate check. So idempotency holds for *full-rollback* paths but NOT for the validation-failure path. The contract's idempotency claim is not bounded.
- **Recommended fix:** Reword BC-6.20.012 Postcondition 5: "After a full rollback (i.e., not validation-failure path), re-invoking the skill with identical arguments produces the same result as a first-time invocation. After a validation-failure rollback (where the file persists), re-invocation requires either deleting the leftover file or invoking with `--id` omitted (which would re-trigger the duplicate detection in BC-6.20.002)." Clarify which path supports identical-args re-invocation.
