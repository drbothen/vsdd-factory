## Part B — New Findings (Pass 2)

### F-020 — ARCH-INDEX subsystem design note has stale BC count for SS-06

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Cross-cutting (POLICY 8 propagation)
- **Artifacts:** `.factory/specs/architecture/ARCH-INDEX.md` line 97
- **Evidence:** ARCH-INDEX line 79 (Subsystem Registry table) says SS-06 has BC count `583` (correct after the +12 BCs were added). But line 97 in the "Deviations from the suggested 10-subsystem template" prose says: *"SS-06 'Skill Catalog' is the largest single BC surface (553 BCs)"*. The 553 figure is stale prose narrative drift.
- **Issue:** A reader who consumes the design-note prose (rather than the table) will get a stale number. Same class of finding as F-002 in PRD.
- **Recommended fix:** Update ARCH-INDEX line 97 from "(553 BCs)" to "(583 BCs)" to match line 79. Optionally rephrase as "(largest single BC surface, see Subsystem Registry table for current count)" to avoid future drift.

### F-021 — VP-INDEX Method Summary uses `property-test` label; VP-059 frontmatter uses `proptest`

- **Severity:** LOW
- **Confidence:** MEDIUM
- **Category:** VP / VP-INDEX (POLICY 9)
- **Artifacts:** `.factory/specs/verification-properties/VP-INDEX.md` line 45; VP-059.md line 18
- **Evidence:** VP-INDEX Method Summary at lines 41-45 reports: unit-test, integration, manual, kani-proof, property-test. VP-059 frontmatter line 18 says `proof_method: proptest`. Label drift.
- **Recommended fix:** Either (a) change VP-INDEX line 45 column to `proptest` to match VP frontmatter literal, or (b) document the mapping explicitly in a leading note ("Method names in this table are normalized: `proptest` → `property-test`").

### F-022 — VP-058 frontmatter `source_bc` singleton hides partial coverage of BC-6.20.011 (F-017 was comment-only)

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** VP / Anchor (POLICY 4)
- **Artifacts:** `.factory/specs/verification-properties/VP-058.md` lines 14-16
- **Evidence:** Pass-1 F-017 fix added a Markdown-comment annotation at line 16: `# Note: VP-058 also covers BC-6.20.011 partially; see body Source Contract section.` But YAML doesn't permit `#` mid-frontmatter as a structured field — it's a comment that the spec-reader human sees but tooling that reads VP frontmatter via YAML parser may strip it. The frontmatter remains literally singleton-anchored on BC-6.20.012, while `bcs: [BC-6.20.011, BC-6.20.012]` (line 37) lists both.
- **Issue:** F-017 was marked applied but the fix is documentation-only; not a structural fix. POLICY 4 (semantic_anchoring_integrity) requires anchors that are semantically correct.
- **Recommended fix:** Two paths: (a) Extend VP frontmatter schema across the spec to support `source_bcs:` (list) — preferred long-term; would require a sweep of all 60 VPs. (b) Acknowledge limitation as a known schema gap in `templates/vp-template.md` until (a) is done. The `bcs:` array is the actual machine-readable trace; `source_bc:` is the legacy singleton. Either route is acceptable for this convergence; pin the convention.

### F-023 — Epic E-6 frontmatter `prd_capabilities: [FR-041]` is type-mismatched

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Anchor (POLICY 4)
- **Artifacts:** `.factory/stories/epics/E-6-tooling-backlog.md` line 5
- **Evidence:** Epic frontmatter: `prd_capabilities: [FR-041]`. The field name is `prd_capabilities` but the value is a Functional Requirement ID. The matched Capability is CAP-017 (per PRD line 974, FR-041 itself anchors to CAP-017). The body table at line 57 also uses `FR-041` in the "Capability ID" column.
- **Issue:** Field name says "capabilities" but value is an FR-ID. POLICY 4 type-correctness violation.
- **Recommended fix:** Change line 5 to `prd_capabilities: [CAP-017]` and add `prd_frs: [FR-041]` as a sibling field; update the body table to split into Capability + FR rows or rename column header. Note: low-impact because no live tooling currently joins on this field, but should be fixed before consistency-validator reads it.
