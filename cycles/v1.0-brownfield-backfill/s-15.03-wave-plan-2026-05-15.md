#### Category G — TD #74 Option (b): WASM Lint Hook for cargo-audit

**TD #74 option (b):** Scan `.factory/cycles/*/td-*-dispatch.md` files for crate references (e.g., `serde_yaml = "0.9"`) and cross-check against `cargo audit` output, blocking if a recommended dependency has known CVEs.

**Automation surface:** PostToolUse Edit/Write on `td-*-dispatch.md` path pattern. WASM plugin reads file content, extracts `[dependencies]` or inline `<crate> = "<version>"` patterns, invokes `cargo audit --json` via host shell (if available) or reads a pre-generated `advisory.json` cache.

**Feasibility:** LOW-MEDIUM. The challenge is that WASM plugins cannot directly invoke `cargo audit` — they run in a sandbox. The two viable approaches are: (a) a pre-commit bash hook outside WASM that runs cargo-audit and writes a cache file, with the WASM hook reading the cache; (b) a pure-WASM heuristic that flags crate names matching a known-bad-list embedded in the binary. Approach (a) is architecturally cleaner but requires a two-part implementation (bash script + WASM reader). Approach (b) is simpler but has false-negative risk for new CVEs.

This is a genuinely scoped, concrete story regardless of approach choice. Effort estimate: multi-day-with-research (the cargo-audit invocation pattern needs an ADR-level decision on sandboxing).
