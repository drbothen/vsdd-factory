### F-1 [CRITICAL]: CAP-030's enumerated 15 fields contradict ADR-015 D-15.2's enumerated 15 fields; BC-1.12.003 inherits the wrong enumeration in its Capability Anchor Justification

**Artifact:** `/Users/jmagady/Dev/vsdd-factory/.factory/specs/domain-spec/capabilities.md` §CAP-030 + `/Users/jmagady/Dev/vsdd-factory/.factory/specs/behavioral-contracts/ss-01/BC-1.12.003.md` Traceability Capability Anchor Justification
**Policy / Axis:** POLICY 4 (semantic_anchoring_integrity), POLICY 7 (BC H1 is title source-of-truth — by spirit), Axis J (CAP-029/CAP-030 newly-authored capability quality), Axis K (capability supersession completeness)
**Issue:** CAP-030 in `capabilities.md` defines a 15-field Resource attribute block with fields: `service.name`, `service.version`, `host.id`, `host.name`, `os.type`, `process.pid`, `process.runtime.name`, `process.runtime.version`, `vsdd.dispatcher.version`, `vsdd.session_id`, `vsdd.worktree_id`, `vsdd.project_root`, `vsdd.config_hash`, `telemetry.sdk.name`, `telemetry.sdk.version`. ADR-015 D-15.2 Resource attributes table defines a different 15-field set: `service.name`, `service.namespace`, `service.instance.id`, `service.version`, `deployment.environment.name`, `host.name`, `host.id`, `os.type`, `process.pid`, `vcs.repository.url.full`, `vcs.repository.name`, `vcs.provider.name`, `vcs.owner.name`, `worktree.id`, `schema_url`. Only 6 fields overlap (`service.name`, `service.version`, `host.id`, `host.name`, `os.type`, `process.pid`). Nine fields are in CAP-030 but NOT in ADR-015 (`process.runtime.name/version`, `vsdd.dispatcher.version`, `vsdd.session_id`, `vsdd.worktree_id`, `vsdd.project_root`, `vsdd.config_hash`, `telemetry.sdk.name/version`). Nine fields are in ADR-015 but NOT in CAP-030 (`service.namespace`, `service.instance.id`, `deployment.environment.name`, `vcs.*` 4 fields, `worktree.id`, `schema_url`). BC-1.12.003 Postcondition 1 implements ADR-015's set faithfully, but BC-1.12.003's Capability Anchor Justification verbatim quotes CAP-030's 15 fields ("`service.name, service.version, host.id, host.name, os.type, process.pid, process.runtime.name, process.runtime.version, vsdd.dispatcher.version, vsdd.session_id, vsdd.worktree_id, vsdd.project_root, vsdd.config_hash, telemetry.sdk.name, telemetry.sdk.version`") to justify CAP-030 anchoring — but the BC's Postcondition 1 lists ADR-015's actual fields. An implementer reading the BC will find the Capability Anchor Justification names fields the BC does NOT specify, and the BC specifies fields the Capability Anchor Justification does NOT name. This is internally contradictory within a single BC and causes the canonical capability registry (capabilities.md) to disagree with the canonical ADR (ADR-015 D-15.2).
**Evidence:** `capabilities.md` §CAP-030 (P1, "15 fields: service.name, service.version, host.id, host.name, os.type, process.pid, process.runtime.name, process.runtime.version, vsdd.dispatcher.version, vsdd.session_id, vsdd.worktree_id, vsdd.project_root, vsdd.config_hash, telemetry.sdk.name, telemetry.sdk.version"); ADR-015 D-15.2 Resource attributes table (rows 1–15: service.name through schema_url); BC-1.12.003 Postcondition 1 table (lists ADR-015's 15 fields); BC-1.12.003 Traceability "Capability Anchor Justification" cell (verbatim quotes CAP-030's 15 fields, NOT BC-1.12.003's own Postcondition 1 list).
**Routing:** architect (capabilities.md amendment) — capabilities.md must be the source-of-truth per POLICY 6 spirit; CAP-030 description must be amended to reflect the actual ADR-015 D-15.2 field set, OR the BC must re-anchor away from CAP-030. The current state propagates a stale anchor description into the BC's own justification text.
**Suggested fix:** Amend CAP-030 in `capabilities.md` to enumerate the actual 15 fields from ADR-015 D-15.2 (or to refer abstractly to "the 15 OTel-aligned Resource attributes per ADR-015 D-15.2 Resource attributes table" without inline enumeration). Then update BC-1.12.003's Capability Anchor Justification to either (a) reference the BC's own Postcondition 1 list, or (b) quote the corrected CAP-030 field list. Mis-anchoring blocks convergence per POLICY 4.

### F-2 [CRITICAL]: BC-1.12.009 has internal four-state vs five-state taxonomy contradiction; "five-state" in H1, Description, Postcondition 4, Architecture Anchors but "four-state" in Description line 59, Postcondition 8, Purity Classification, ADR-row text

**Artifact:** `/Users/jmagady/Dev/vsdd-factory/.factory/specs/behavioral-contracts/ss-01/BC-1.12.009.md`
**Policy / Axis:** POLICY 4 (semantic_anchoring_integrity), POLICY 7 (H1 is title source-of-truth), Axis F (Dual-emit five-state classification — mutually exclusive AND collectively exhaustive)
**Issue:** The BC's H1 (file line 28) declares "five-state event classification (paired-current / paired-deprecated / orphaned-deprecated-half / orphaned-current-half / non-paired)" — five named states. Description line 41 says "the five-state classification taxonomy". Postcondition 4 line 97 says "Any event in `events-*.jsonl` falls into exactly one of FIVE states (excluding malformed, which degrades to orphaned-half — see below)". Architecture Anchors line 243 says "this BC implements the five-state consumer taxonomy". HOWEVER: Description line 59 says "This BC specifies the four-state classification that handles both the Wave 2 state and the post-Wave-3 state." Postcondition 8 line 198 says "the four-state classifier degrades gracefully: States 1–4 are checked first". Purity Classification line 316 says "four-state classification is a pure function". The Traceability ADR row line 307 says "ADR-015 D-15.2.e (dual-emit pair identity contract — five-state classification reduced to four-state consumer taxonomy". An implementer will not know whether to write a 4-state or 5-state classifier, and the test vectors require all five to be distinguishable (Verification Properties row "Five-state classification correctly applied for all five input variants (States 1–4 + non-paired)" — counts 5 by including non-paired). The H1-renaming pass (D-315 F-12 per CHANGELOG) updated the H1 to five-state but did not propagate to all body prose. This is the ADR-015 v1.5 vintage carry-over: the architect renamed four → five for the orphaned-half addition, but the BC still has both terms.
**Evidence:** BC-1.12.009 H1 (line 28: "five-state event classification (paired-current / paired-deprecated / orphaned-deprecated-half / orphaned-current-half / non-paired)"); BC-1.12.009 Description line 41 ("five-state classification taxonomy"), line 59 ("This BC specifies the four-state classification that handles both the Wave 2 state and the post-Wave-3 state"); BC-1.12.009 Postcondition 4 line 97 ("FIVE states"); BC-1.12.009 Postcondition 8 line 198 ("four-state classifier degrades gracefully: States 1–4"); BC-1.12.009 Purity Classification line 316 ("four-state classification is a pure function"); BC-1.12.009 Traceability ADR row line 307 ("five-state classification reduced to four-state consumer taxonomy"); CHANGELOG line referenced in v1.1 ("D-315 F-12 BC-side. Re-anchored to CAP-029 + H1 updated to five-state taxonomy" — exists in S-10.05 CHANGELOG noting this BC's H1 change).
**Routing:** PO (BC body sweep) — propagate the five-state convention through all body prose; replace "four-state classifier" with "five-state classifier" or clarify "four directional states + one non-paired baseline = five total states", whichever the canonical authorial intent is. Pick ONE convention and use it everywhere.
**Suggested fix:** Replace "four-state classification" / "four-state classifier" / "States 1–4" wording at lines 59, 198, 316 with "five-state" language consistent with the H1. Specifically: Description line 59 → "This BC specifies the five-state classification..."; Postcondition 8 line 198 → "the five-state classifier degrades gracefully: States 1–4 are checked first; if no fields match any state, the event is in the non-paired (fifth) state"; Purity line 316 → "five-state classification is a pure function"; ADR row line 307 → simplify to "five-state classification (paired-current, paired-deprecated, orphaned-deprecated-half, orphaned-current-half, non-paired) per ADR-015 D-15.2.e v1.5".
