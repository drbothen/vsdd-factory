### F-P3-003 [HIGH] — Task B.1 hyperfine command underspecified for .wasm

- Location: S-9.00:321-330 (Task B.1)
- Description: B.1 says "Measure cold-start p95 latency using hyperfine --warmup 0 --runs 10 on a real native Tier 1 plugin (e.g., handoff-validator.wasm)". hyperfine runs executables, not raw .wasm files. A plugin's cold-start cost is the dispatcher binary loading and instantiating it. The story does not specify the hyperfine target. An implementer reading "hyperfine on handoff-validator.wasm" literally would either fail (".wasm not executable") or measure the wrong thing.
- Evidence: Task B.1 line 321-330 does not specify a hyperfine command line. AC-7 line 185 says cold-start is measured "using hyperfine --warmup 0 to capture the first-invocation cost" but also doesn't pin the target.
- Suggested fix: Specify exact hyperfine invocation: "hyperfine --warmup 0 --runs 10 'factory-dispatcher --plugin handoff-validator.wasm --stdin-fixture <fixture.json>'" (or whatever the dispatcher CLI is). Document the dispatcher CLI command for plugin invocation if it exists.
- Confidence: HIGH

### F-P3-004 [HIGH] — Task A.0 git show command is invalid for directory tree

- Location: S-9.00:241 (Task A.0)
- Description: A.0 instructs: "run git show v1.0.0:plugins/vsdd-factory/hook-plugins/ | wc -c (or equivalent) to derive the GA bundle size". This command is invalid. git show <ref>:<path> for a directory outputs a tree listing (a few lines of text), not the concatenated bundle bytes. Piping to wc -c would count bytes of the listing, not actual bundle size (~7.2MB target).
- Evidence: Task A.0 line 238-246.
- Suggested fix: Replace with valid procedure: "git checkout v1.0.0 && find plugins/vsdd-factory/hook-plugins -name '*.wasm' -exec wc -c {} + | tail -1" — or — "git ls-tree -lr v1.0.0 plugins/vsdd-factory/hook-plugins/ | awk '{sum+=$4} END {print sum}'". Pin exact command.
- Confidence: HIGH

### F-P3-005 [MEDIUM] — Advisory cap formula does not specify metric (total_bytes vs grand_total_bytes)

- Location: S-9.00:181 (AC-3b) + S-9.00:296 (A.4 schema row advisory_soft_cap_bytes)
- Description: AC-3(b) says "advisory soft cap: cumulative bundle growth ≤ 100% vs pre-W-15 baseline × 2 (≤14MB cumulative target at end of W-17)". The 14MB target presumably matches research §Q3 (~7.2MB × 2 = 14.4MB). However, AC-1 distinguishes total_bytes (excludes dispatcher) from grand_total_bytes (includes dispatcher). The 7.2MB pre-W-15 baseline reading depends on which metric — and per E-8-bash-baseline.json:40, dispatcher_binary_bytes = ~12MB at S-8.00 baseline (already > 14MB). If cap targets grand_total_bytes, the ~12MB dispatcher already swamps the 14MB cap. If cap targets total_bytes (WASM only), then pre-W-15 baseline of 7.2MB doesn't match E-8 measurement of 322KB WASM.
- Evidence: AC-3(b) line 181, A.4 schema line 296, research line 303, E-8-bash-baseline.json line 40 (dispatcher 12,150,752 bytes) and line 39 (all_hook_plugins_wasm_bytes 321,843).
- Suggested fix: Clarify in AC-3(b): "advisory soft cap applies to <metric> (specify: grand_total_bytes or all_hook_plugins_wasm_bytes only)". Reconcile 7.2MB baseline with E-8 measurements: either (a) 7.2MB refers to a different historical artifact (clarify which), or (b) Task A.0 should NOT presume 7.2MB and instead measure the v1.0.0 tag fresh.
- Confidence: HIGH
