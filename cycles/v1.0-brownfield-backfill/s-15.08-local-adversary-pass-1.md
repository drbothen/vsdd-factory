## Part A — Findings

### F-S15.08-LOCAL-P1-001 — CRITICAL: `dim7-dispatched-count-sweep.sh` line-mapping formula is structurally broken for multi-burst files

- **What:** The script tracks burst sections via outer loop on `## Burst:` headings, then the inner `while read FILE_LINE` reads the ENTIRE FILE from line 1 while incrementing CURRENT_LINE from 0. The formula `ACTUAL_LINE_NUM = SECTION_START + CURRENT_LINE - 1` then maps every file line as if it started AT SECTION_START. For any second-or-later burst section, file line 1 is mis-mapped to SECTION_START (e.g., 33), file line 2 to SECTION_START+1 (e.g., 34), etc. The boundary check `[[ ACTUAL_LINE_NUM -lt SECTION_START ]] || [[ -gt SECTION_END ]]` becomes meaningless because all read lines are mapped INTO the section.
- **Where:** `plugins/vsdd-factory/hooks/dim2-gates/dim7-dispatched-count-sweep.sh` — inner while loop, formula at the line `ACTUAL_LINE_NUM=$(( SECTION_START + CURRENT_LINE - 1 ))`.
- **Why it matters:** This script EXISTS to detect META-LEVEL-24-class false-green attestations (D-450(b) Dim-7 sibling-sweep gate). A semantically broken gate produces false-green output that downstream callers TRUST. The bats fixtures coincidentally don't expose the bug (PASS fixture has 2 bursts but mapping happens to align; FAIL fixture's anachronism happens to land where the broken formula still flags it). The library exists to prevent exactly this category of false-green, and the gate itself fails the check.
- **Evidence:**
  ```
  $ wc -l plugins/vsdd-factory/tests/fixtures/dim2-gates/dim7-dispatched-count-sweep-pass/burst-log.md
  $ grep -n "## Burst:" plugins/vsdd-factory/tests/fixtures/dim2-gates/dim7-dispatched-count-sweep-pass/burst-log.md
  ```
  (verification deferred to fix-burst implementer; see Recommended Fix)
- **Recommended fix:** Restructure the inner loop to use `sed -n "${SECTION_START},${SECTION_END}p"` to extract the section's actual lines, then iterate ONLY those lines and offset CURRENT_LINE by SECTION_START. Add a fixture with 3+ bursts where a 3rd-burst Dim-7 cell references a future pass — script must catch this. Route via `implementer`.
- **Tags:** [process-gap] (gates that don't gate)

### F-S15.08-LOCAL-P1-002 — HIGH: `propagation-completeness.sh` parses but ignores per-site `<file-path>:<pattern>` second-column

- **What:** sites.txt second-column pattern is read but unused; script greps for global DERIVED_VALUE on every site rather than the per-site pattern. AC-005 reads naturally as "verify the pattern at each site"; current implementation reduces to "verify DERIVED_VALUE appears at each site path".
- **Where:** `plugins/vsdd-factory/hooks/dim2-gates/propagation-completeness.sh` — site-loop body uses `grep -cF "${DERIVED_VALUE}"` not `grep -cE "${SITE_PATTERN}"`.
- **Why it matters:** Contract-vs-implementation mismatch. POLICY 4 (semantic anchoring integrity). The dead-code per-site pattern column hides bugs because both PASS and FAIL fixtures happen to encode `pattern == DERIVED_VALUE`.
- **Evidence:** Inspect script vs sites.txt fixture. Both PASS and FAIL fixtures use `D-453` as both site pattern and global derived value, which makes the per-site contract un-exercised.
- **Recommended fix (Option A — production-grade default):** Change `grep -cF "${DERIVED_VALUE}"` to `grep -cE "${SITE_PATTERN}"` (or fixed-string variant per AC-005 phrasing). Add a fixture pair where per-site patterns DIFFER from the global derived value to lock in the contract. Route via `implementer`.
- **Recommended fix (Option B — only if architect adjudicates):** Document sites.txt second column as a human-readable anchor (not executable); strip the parsed-but-unused field from the script. Route via `story-writer` for AC-005 amendment + `implementer` for script simplification.
- **Tags:** [process-gap]

### F-S15.08-LOCAL-P1-003 — HIGH: `banner-wc-l-pass/STATE.md` fixture is EOL-sensitive — silent CI break risk

- **What:** PASS fixture banner asserts "actual 23 lines"; file has 24 displayed lines. `wc -l` returns 24 (with trailing newline) or 23 (without). Test currently green only because file lacks trailing newline; any editor save / lint pass / `.gitattributes` `*.md text` rule could insert EOL and silently flip the test red.
- **Where:** `plugins/vsdd-factory/tests/fixtures/dim2-gates/banner-wc-l-pass/STATE.md`.
- **Why it matters:** Production-grade tests must not depend on invisible whitespace. CI brittleness; hard-to-debug failures. FAIL fixture is robust (mismatch is 23 OR 24 ≠ 50) — only PASS fixture is fragile.
- **Evidence:**
  ```
  $ wc -l plugins/vsdd-factory/tests/fixtures/dim2-gates/banner-wc-l-pass/STATE.md
  $ tail -c 1 plugins/vsdd-factory/tests/fixtures/dim2-gates/banner-wc-l-pass/STATE.md | od -c
  ```
- **Recommended fix:** Either (a) commit `.gitattributes` `tests/fixtures/dim2-gates/banner-wc-l-pass/* -text` + an assertion in the bats setup that `wc -l < STATE.md` returns exactly 23 before invoking the script under test; or (b) restructure fixture so the banner cites a number tolerant to ±1. Option (a) preferred. Route via `test-writer`.

### F-S15.08-LOCAL-P1-004 — MEDIUM: bats FAIL-case disjunctive assertions weaken to near-tautology (POLICY 11)

- **What:** Pattern `[[ "$output" == *"FAIL"* ]] || [[ "$output" == *"exit"* ]] || [ "$status" -ne 0 ]` (in 8 of 11 .bats files). The trailing `[ "$status" -ne 0 ]` clause means the FAIL test passes whenever the script exits non-zero, regardless of output content — including crash exits 127/126/139. POLICY 11 (no test tautologies): tests must not pass for the wrong reasons.
- **Where:** `tests/dim2-gates/{freshness-literal-stdout,active-branches-sha-currency,block-label-canonical-form,decision-log-monotonic-rows,dim1-file-count-arithmetic,dim7-dispatched-count-sweep,layer-ordinal-dual-direction,banner-wc-l}.bats`.
- **Why it matters:** Defensive-coding instinct (allow flexible script output formats) defeats the purpose of asserting on output. Implementer DID emit deterministic strings; tests can and should assert on them.
- **Evidence:** `grep -nE '\|\| \[\[' plugins/vsdd-factory/tests/dim2-gates/*.bats`
- **Recommended fix:** Replace each disjunctive FAIL assertion with `[[ "$output" == *"FAIL:"* ]]` (note the colon). Route via `test-writer`.

### F-S15.08-LOCAL-P1-005 — MEDIUM: scripts' executable bit not verified by tests; cross-platform/clone risk

- **What:** No bats meta-test verifies `[ -x "$script" ]`. If `core.fileMode=false` or a Windows clone strips +x, scripts fail with exit 126 (Permission denied). Bats tests interpret that as exit-not-0 / exit-not-1, fail with confusing diagnostic.
- **Where:** all `tests/dim2-gates/*.bats` (no setup chmod) + plugin release tarball flow.
- **Why it matters:** Production-grade scripts deployed via plugin marketplace must preserve mode. Test gap = silent regression vector.
- **Evidence:** `git ls-files --stage plugins/vsdd-factory/hooks/dim2-gates/*.sh` should show 100755 for each.
- **Recommended fix:** Add a meta `.bats` file with one parametric test asserting all 11 scripts have `[ -x ]`. Route via `test-writer` + verification by `devops-engineer` at PR phase.

### F-S15.08-LOCAL-P1-006 — LOW: `block-label-canonical-form.sh` regex too permissive (no trailing colon enforcement)

- **What:** Regex `^\*\*${LABEL}` matches `**Dim-2`, `**Dim-25`, `**Dim-2something:**`. D-444(c) prescribes `**LABEL:**` or `**LABEL (Dim-N):**`. POLICY 13 (HH-N regex-alternation predicates): regex should explicitly enumerate canonical-form variants.
- **Where:** `plugins/vsdd-factory/hooks/dim2-gates/block-label-canonical-form.sh:85`.
- **Why it matters:** D-454(d) tri-way canonical-form alignment (codification = regex = document). Regex side too loose.
- **Recommended fix:** Tighten to `^\*\*${LABEL}[: ]` and add a third bats case where burst-log has `**Dim-2something:**` — script should detect Dim-2 missing. Route via `implementer` + `test-writer`.
