## Findings

### F-P2-001 [HIGH] — Hook registration mechanism is incoherent: `.claude/settings.json` does not carry hook scaffolding; cited S-13.01 precedent registered a WASM plugin

**Evidence:**
- Story S-15.01 lines 242-243 (AC-007), 495-500 (T-3i), 690-696 (Previous Story Intelligence) all claim the new bash hook `scripts/hooks/lint-registry-async-invariant.sh` should be "registered in `.claude/settings.json` as a Claude Code PostToolUse hook ... per S-13.01 precedent"
- Verified `/Users/jmagady/Dev/vsdd-factory/.claude/settings.json` (full file 6 lines): only `enabledPlugins: { "vsdd-factory@claude-mp": true }`. No `hooks` key. No PostToolUse scaffold.
- S-13.01 (the cited precedent) AC-014 line 209-216 + Architecture Mapping line 238: artifact-path-governance hook was registered in `plugins/vsdd-factory/hooks-registry.toml` and built as `validate-artifact-path.wasm` (a native WASM plugin). NOT in `.claude/settings.json`.
- `hooks-registry.toml` dispatches WASM plugins; bash scripts at `scripts/hooks/*.sh` are not its registration target.

**Why HIGH:** Bash script + cited mechanism + cited precedent do not form a coherent registration path. An implementer following T-3i would have no path to execute. Three story locations propagate the same incorrect mechanism.

**Required action:** Architect investigation needed to choose actual mechanism. Options:
- (a) Port lint as native WASM plugin (matches S-13.01 precedent literally)
- (b) Use CI-only enforcement (drop edit-time layer)
- (c) Reference a project-specific bash-hook layer if one exists outside `hooks-registry.toml`

Once mechanism chosen, update T-3i, AC-007, Previous Story Intelligence, and File List in lockstep.
