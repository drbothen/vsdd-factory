# VP-063: RED_RATIO computation correctness — monotonic, bounded [0.0, 1.0], boundary-safe at 0.5 threshold

## Property Statement

For any (red_count: usize, total_count: usize, exempt_count: usize) where total_count >= exempt_count and (total_count - exempt_count) > 0:
- RED_RATIO = red_count as f64 / (total_count - exempt_count) as f64
- RED_RATIO is bounded: 0.0 ≤ RED_RATIO ≤ 1.0
- RED_RATIO is monotonic in red_count: if red_count increases, RED_RATIO does not decrease (all else equal)
- The threshold check `RED_RATIO >= 0.5` is equivalent to `red_count * 2 >= (total_count - exempt_count)` (integer form, no float precision drift)
- When total_count == exempt_count (all tests exempt), RED_RATIO is undefined and the gate triggers the full-exception path, not vacuous pass

- **Primary: BC-8.29.001** — RED_RATIO = RED_TESTS / TOTAL_NEW_TESTS must be ≥ 0.5 before Step 4; integer boundary check required

// File: plugins/vsdd-factory/src/red_gate.rs (or equivalent logic module)

use proptest::prelude::*;

/// RED_RATIO computation with integer arithmetic (no float precision drift)
fn red_ratio_check(red_count: usize, total_count: usize, exempt_count: usize) -> RedGateDecision {
    let effective_total = total_count.saturating_sub(exempt_count);
    if effective_total == 0 {
        return RedGateDecision::FullExceptionPath;
    }
    // Integer threshold: equivalent to red_count / effective_total >= 0.5
    if red_count * 2 >= effective_total {
        RedGateDecision::Pass
    } else {
        RedGateDecision::Block { red_count, effective_total }
    }
}

proptest! {
    /// Property 1: Bounded — RED_RATIO always in [0.0, 1.0] when effective_total > 0
    #[test]
    fn prop_red_ratio_bounded(
        red_count in 0usize..=1000,
        total_count in 0usize..=1000,
        exempt_count in 0usize..=1000,
    ) {
        let exempt_count = exempt_count.min(total_count);
        let red_count = red_count.min(total_count - exempt_count);
        let effective_total = total_count - exempt_count;
        if effective_total > 0 {
            let ratio = red_count as f64 / effective_total as f64;
            prop_assert!(ratio >= 0.0);
            prop_assert!(ratio <= 1.0);
        }
    }

    /// Property 2: Monotonic — increasing red_count does not decrease RED_RATIO
    #[test]
    fn prop_red_ratio_monotonic(
        red_a in 0usize..=500,
        red_b in 0usize..=500,
        effective_total in 1usize..=1000,
    ) {
        let red_a = red_a.min(effective_total);
        let red_b = red_b.min(effective_total);
        if red_a <= red_b {
            prop_assert!(
                red_a as f64 / effective_total as f64 <=
                red_b as f64 / effective_total as f64
            );
        }
    }

    /// Property 3: Boundary safety — integer form matches float form at 0.5
    #[test]
    fn prop_boundary_integer_float_equivalence(
        red_count in 0usize..=1000,
        effective_total in 1usize..=1000,
    ) {
        let red_count = red_count.min(effective_total);
        let float_pass = red_count as f64 / effective_total as f64 >= 0.5;
        let int_pass = red_count * 2 >= effective_total;
        prop_assert_eq!(float_pass, int_pass,
            "float and integer forms disagree at red={}, total={}",
            red_count, effective_total
        );
    }

    /// Property 4: Undefined case — zero effective_total triggers full-exception path
    #[test]
    fn prop_zero_effective_total_is_exception_not_pass(
        total_count in 0usize..=100,
    ) {
        // All tests are exempt
        let result = red_ratio_check(0, total_count, total_count);
        prop_assert!(matches!(result, RedGateDecision::FullExceptionPath));
    }

    /// Property 5: Count consistency — log counts sum correctly
    #[test]
    fn prop_count_consistency(
        red in 0usize..=100,
        green_design in 0usize..=50,
        green_wiring in 0usize..=50,
        unjustified_green in 0usize..=50,
    ) {
        let total = red + green_design + green_wiring + unjustified_green;
        let exempt = green_design + green_wiring;
        let effective = total - exempt;
        // Effective total = red + unjustified_green
        prop_assert_eq!(effective, red + unjustified_green);
    }
}

/// Canonical test vectors from BC-8.29.001
#[test]
fn test_canonical_vectors() {
    assert!(matches!(red_ratio_check(4, 6, 0), RedGateDecision::Pass));       // 4/6 = 0.667
    assert!(matches!(red_ratio_check(3, 6, 0), RedGateDecision::Pass));       // 3/6 = 0.5 exact
    assert!(matches!(red_ratio_check(2, 6, 0), RedGateDecision::Block { .. })); // 2/6 = 0.333
    assert!(matches!(red_ratio_check(4, 6, 2), RedGateDecision::Pass));       // 4/4 = 1.0 (2 exempt)
    assert!(matches!(red_ratio_check(0, 6, 6), RedGateDecision::FullExceptionPath)); // all exempt
    assert!(matches!(red_ratio_check(0, 0, 0), RedGateDecision::FullExceptionPath)); // no tests
}

## Notes

VP-063 verifies the mathematical correctness of the RED_RATIO computation and its boundary semantics. The key insight is that float arithmetic at the 0.5 boundary can produce precision drift (e.g., `3.0 / 6.0` may not be exactly `0.5` in some float representations), so the integer form `red_count * 2 >= effective_total` must be used as the canonical check. Property 3 proves these are equivalent. The `usize` overflow risk (`red_count * 2`) is bounded by the assertion `red_count <= effective_total <= 1000` in the proof harness; production code should use checked arithmetic.
