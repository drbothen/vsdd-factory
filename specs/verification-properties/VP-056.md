# VP-056: on_failure Semantics — retry → escalate → abort Are Correctly Ordered

## Property Statement

The on_failure handling chain follows: retry (up to max_retries) → escalate (to human or next handler) → abort. A step with on_failure: escalate is the workflow default. A gate with fail_action: block explicitly blocks the tool call via exit 2. The orchestrator input-hash drift check fires before Phase 1/2/3/7 human approvals.

## Source Contract

- **BC:** BC-5.01.006 — Failure handling: on_failure: escalate is the workflow default; gate.fail_action: block is the blocking shape
- **Postcondition/Invariant:** No failure silently proceeds to the next step without escalation

## Proof Harness Skeleton

```python
# validate_failure_handling.py
VALID_ON_FAILURE = {'escalate', 'retry', 'abort', 'continue'}
for step in workflow['steps']:
    on_failure = step.get('on_failure')
    if on_failure:
        assert on_failure in VALID_ON_FAILURE, f"Invalid on_failure: {on_failure}"
    # Gate steps must declare fail_action
    if step.get('type') == 'gate':
        assert 'fail_action' in step, f"Gate {step['name']} missing fail_action"
```
