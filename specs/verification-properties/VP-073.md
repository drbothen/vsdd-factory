# VP-073: Resolver-Load Purity

## Property Statement

Loading a `.wasm` resolver artifact (via `ResolverLoader::get_or_compile`) is
**deterministic** and has **no observable side effects at load time**. Specifically:

1. **Determinism:** Given the same `.wasm` bytes (same path, same mtime), two
   sequential `get_or_compile` calls return structurally identical `ResolverModule`
   instances. The module compilation is a pure function of the WASM bytes.

2. **No side-effect invocation:** The load operation does NOT call the resolver's
   exported `resolve()` function. Loading a resolver only compiles its WASM bytes
   into a `wasmtime::Module`; it does not create a `Store`, does not execute any
   WASM instructions, and does not invoke any host functions.

3. **No global state mutation:** The load operation does not read
   `resolvers-registry.toml` again. It does not write to any filesystem path. It
   does not emit any `resolver.*` events.

4. **No-op on cache hit:** If the resolver's WASM path + mtime matches a cached
   entry, `get_or_compile` returns the cached `Module` without any recompilation.
   The cache key uniquely identifies the artifact version.

## Rationale for this VP

Resolver loading is the first step in the resolver lifecycle (BC-4.12.001). If
loading has side effects, the dispatcher's startup sequence becomes nondeterministic.
Specifically: if `get_or_compile` accidentally invoked `resolve()`, a resolver that
reads filesystem state at load time would produce different results depending on
filesystem state at startup versus dispatch time — breaking the determinism invariant
that VP-075 depends on. This VP guards the boundary between "compile artifact" and
"invoke artifact".

## Formal Property Statement (integration test)

```rust
// VP-073 integration test — lives in crates/factory-dispatcher/tests/resolver_load_test.rs
#[cfg(test)]
mod vp_073_resolver_load_purity {
    use factory_dispatcher::resolver_loader::ResolverLoader;
    use std::sync::atomic::{AtomicUsize, Ordering};

    static HOST_FN_CALL_COUNT: AtomicUsize = AtomicUsize::new(0);

    /// VP-073-A: Two sequential get_or_compile calls on the same WASM bytes
    /// return structurally identical modules (determinism).
    #[test]
    fn test_load_is_deterministic() {
        let loader = ResolverLoader::new();
        let wasm_path = test_resolver_wasm_path();  // path to a test .wasm artifact

        let module_a = loader.get_or_compile(&wasm_path)
            .expect("first load must succeed");
        let module_b = loader.get_or_compile(&wasm_path)
            .expect("second load must succeed (cache hit)");

        // Both calls return a module; the second call is a cache hit (no recompilation).
        // Structural identity: both modules export `resolve` with the same signature.
        assert!(module_a.get_export("resolve").is_some());
        assert!(module_b.get_export("resolve").is_some());
        // Cache hit: same Arc reference returned (pointer equality on the inner Module).
        assert!(
            std::ptr::eq(module_a.as_ptr(), module_b.as_ptr()),
            "cache hit must return the same module reference"
        );
    }

    /// VP-073-B: get_or_compile does NOT invoke the resolver's resolve() function.
    /// A test resolver that sets a flag when resolve() is called; the flag must
    /// remain unset after load.
    #[test]
    fn test_load_does_not_invoke_resolve() {
        // Use a test WASM resolver that calls host::set_load_flag() if resolve() is executed.
        // After get_or_compile(), verify the flag was never set.
        HOST_FN_CALL_COUNT.store(0, Ordering::SeqCst);

        let loader = ResolverLoader::new_with_test_host(|call_name| {
            if call_name == "resolve" {
                HOST_FN_CALL_COUNT.fetch_add(1, Ordering::SeqCst);
            }
        });
        let wasm_path = test_resolver_wasm_path();
        let _ = loader.get_or_compile(&wasm_path)
            .expect("load must succeed");

        assert_eq!(
            HOST_FN_CALL_COUNT.load(Ordering::SeqCst), 0,
            "resolve() must not be invoked during module compilation"
        );
    }

    /// VP-073-C: A different mtime for the same path causes recompilation (cache
    /// invalidation), not a stale-cache hit.
    #[test]
    fn test_mtime_change_triggers_recompilation() {
        let loader = ResolverLoader::new();
        let wasm_path = test_resolver_wasm_path();

        let _module_a = loader.get_or_compile(&wasm_path)
            .expect("first load must succeed");

        // Simulate mtime change by touching the file.
        test_utils::touch_file(&wasm_path);

        let _module_b = loader.get_or_compile(&wasm_path)
            .expect("load after mtime change must succeed");

        // No structural assertion on modules; the goal is that the second call
        // succeeds without error (recompilation path is non-panicking).
    }
}
```

## Expected Outcome

All three test cases pass. The `get_or_compile` function is total (no panics on
valid WASM input). The load operation never invokes the resolver's `resolve()`
export. Cache-hit and cache-miss paths both return valid `ResolverModule` instances.

## Related BCs

- BC-4.12.001 — Resolver lifecycle invariant: resolvers are compiled at startup,
  not per-dispatch. Per-dispatch invocations create a fresh `Store`. This VP
