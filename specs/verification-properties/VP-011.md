# VP-011: Sink submit Must Not Block the Dispatcher

## Property Statement

Sink drivers use bounded mpsc channels with try_send. When the channel is full, submit returns immediately (dropping the event and incrementing queue_full_count) rather than blocking. The dispatcher's hot path remains non-blocking regardless of sink throughput.

## Source Contract

- **BC:** BC-3.01.007 — file sink mpsc bounded at default 1000; submit is non-blocking via try_send
- **Postcondition/Invariant:** DI-011 — Sink submit must not block the dispatcher

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| unit-test | Rust unit test with queue_depth=2 | yes — 500 events into depth-2 queue | try_send overflow behavior |

## Proof Harness Skeleton

```rust
#[test]
fn verify_vp_011_submit_nonblocking() {
    let sink = FileSink::with_queue_depth(2);

    // Fill queue beyond capacity without blocking
    let start = Instant::now();
    for _ in 0..500 {
        sink.submit(make_event());
    }
    let elapsed = start.elapsed();

    // Must complete in microseconds, not block for sink drain time
    assert!(elapsed < Duration::from_millis(100));
    assert!(sink.queue_full_count() > 0);
}
```

## Notes

The non-blocking property is enforced by using try_send instead of send. This is a Rust type-level guarantee once the mpsc channel is chosen. The DEFAULT_QUEUE_DEPTH = 1000 is the production setting; tests use depth=2 to trigger backpressure reliably. Both file and otel-grpc sinks share this pattern.
