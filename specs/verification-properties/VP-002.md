# VP-002: Plugin Crash or Timeout Does Not Block Sibling Plugins

## Property Statement

When one plugin in a tier produces a Crashed or Timeout result, all other plugins in the same tier continue execution to completion. The final exit_code is determined by the on_error policy, not by the crashing plugin's existence.

## Source Contract

- **BC:** BC-1.03.013 — crash_does_not_affect_siblings
- **Postcondition/Invariant:** DI-002 — A plugin crash or timeout does not block sibling plugins

## Proof Harness Skeleton

```rust
// Integration test: crates/factory-dispatcher/tests/executor_integration.rs
#[tokio::test(flavor = "current_thread")]
async fn verify_vp_002_crash_isolation() {
    // Three plugins: ok-a, crash (unreachable), ok-b — all same priority
    let summary = execute_tiers(inputs, tiers).await.unwrap();

    // Sibling results must be Ok
    let ok_results: Vec<_> = summary.per_plugin_results.iter()
        .filter(|r| r.plugin_name != "crash")
        .collect();
    assert!(ok_results.iter().all(|r| matches!(r.result, PluginResult::Ok { .. })));

    // Final exit_code not blocked by crash (on_error=continue)
    assert_eq!(summary.exit_code, 0);
}
```

## Amendment 2026-05-07

**Scope narrowed to sync group; async group crash semantics clarified (ADR-019 — v1.0-feature-plugin-async-semantics-pass-1).**

The partition step introduced by ADR-019 creates a two-group execution model. This VP's
crash-isolation invariant applies differently depending on group membership:

**Sync group (this VP's scope, unchanged):**
- A crash or timeout in one sync plugin does not block sibling sync plugins.
- The sync group's `dispatcher_exit` is the aggregate of non-crashing sync plugin
  results per the `on_error` policy. This invariant is unchanged.

**Async group (new clarification — out of this VP's scope):**
- An async group plugin crash or timeout does NOT affect `dispatcher_exit`.
- `dispatcher_exit` is a function only of sync group results (VP-077, Harness 3).
- An async group crash emits a `hook.async_crash` diagnostic event to the event
  stream (SS-03) but does not surface to the user and does not gate Claude Code.
- Async group crashes are fire-and-forget failures; they are not within the scope
  of this VP's sibling-isolation invariant because async plugins have no siblings
  in the blocking execution path.

**Property statement narrowed:**
- **Pre-amendment scope:** all matched plugins M — any plugin crash does not block
  sibling plugins.
- **Post-amendment scope:** sync group plugins only — a crash in one sync plugin
  does not block sibling sync plugins. Async group crash behavior is explicitly
  out of scope (handled by the async spawn path in `engine.rs`).

The existing proof harness (`executor_integration.rs::crash_does_not_affect_siblings`)
remains valid; the integration fixture supplies only sync plugins. No harness change is
required; a comment annotating the sync-group assumption should be added during F4.

See VP-077 for the proof that async group exit codes cannot influence `dispatcher_exit`.
