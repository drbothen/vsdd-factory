# VP-079: Async-Semantics Event Types — Payload Schema Conformance

## Property Statement

For each of the four event types introduced by ADR-019 F2 and catalogued in BC-3.08.001,
when the triggering condition occurs, the dispatcher MUST emit a JSON line to
`events-*.jsonl` that:

1. Carries the mandatory fields specified in BC-3.08.001 (no mandatory field is absent
   or null).
2. Parses as valid JSON without a schema.
3. Has a `type` field matching the exact event-type string.
4. Has a `trace_id` field. The value is non-null in nominal operation; per BC-3.08.001
   Error Paths, `trace_id: null` is acceptable as a last-resort fallback when trace
   context is unavailable at emission time (per DI-017 degraded-mode provision).

The four event types and their mandatory fields are:

| Event type | Mandatory fields |
|------------|-----------------|
| `plugin.async_block_discarded` | `type`, `trace_id`, `plugin_name`, `exit_code`, `timestamp`, `reason` |
| `dispatcher.schema_mismatch` | `type`, `trace_id`, `found_version`, `expected_version`, `timestamp`, `error_code` |
| `dispatcher.registry_invalid` | `type`, `trace_id`, `offending_plugin`, `violation`, `timestamp`, `error_code` |
| `plugin.timeout` (async path) | `type`, `trace_id`, `plugin_name`, `execution_group`, `timeout_ms`, `timestamp` |

Emit-then-crash is acceptable for `dispatcher.schema_mismatch` and
`dispatcher.registry_invalid` (BC-3.08.001 Invariant 2). The event must be written to
`events-*.jsonl` before the dispatcher exits.

5. **ASYNC_DRAIN_WINDOW_MS constraint (DI-019 per BC-1.14.001 v1.4 traceability):** After
   `sync_group` completes, the dispatcher waits up to `ASYNC_DRAIN_WINDOW_MS` (canonical
   value defined in DI-019; default 100ms) for async tasks to emit terminal events before
   exit. DI-019 is the authoritative constant source; BC-1.14.001 PC4 is the dispatch-loop
   partition contract that enforces it. Scenarios 1 and 4 require that async-plugin-triggered
   events (`plugin.async_block_discarded`, `plugin.timeout`) reach FileSink within this drain
   window. Both scenarios therefore need a sync plugin in their fixture registry to hold the
   dispatcher alive for the duration of the async task's execution (since `sync_group` empty
   → dispatcher exits immediately, before any async event can be emitted). The drain window
   guarantees at most a 100ms wait on the async side; `timeout_ms` values below 100ms are
   within the drain window and will have their terminal events captured. Async tasks that
   exceed the drain window are forcibly terminated — their events are NOT emitted (Property 6,
   Scenario 5).

## Source Contract

- **BC:** BC-3.08.001 — Async-semantics event types catalogued in SS-03 with mandatory payload
  schemas (Postconditions 1–4, Invariants 1–4)
- **Domain Invariant:** DI-017 — every emitted event carries a `trace_id`

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| integration | bats (fault-injection) | yes — one triggering scenario per event type | payload field presence, JSON parseability, type string, trace_id present (non-null in nominal mode; null permitted as DI-017 fallback) |

### Why integration (not Kani)

All four event types involve FileSink I/O (writing to `events-*.jsonl`) and are
emitted via the effectful shell of the dispatcher. Kani verifies pure function
properties; it cannot model filesystem writes. The correct verification approach is
end-to-end fault injection: supply the triggering condition (async block verdict, bad
schema version, invalid registry, async plugin timeout) and parse the resulting JSONL
output to assert field presence.

## Proof Harness Skeleton

```bash
#!/usr/bin/env bats
# async-event-schema-conformance.bats
# VP-079: BC-3.08.001 — each of the four async-semantics event types emits
# all mandatory fields per the catalogued wire format.
#
# Invocation surface: dispatcher reads registry from
# ${CLAUDE_PLUGIN_ROOT}/hooks-registry.toml and reads the hook payload
# from stdin (JSON envelope). No --registry or --event CLI flags exist.
# Event capture: VSDD_SINK_FILE (debug builds only) or internal log dir.

PLUGIN_ROOT=""
SINK_FILE=""

setup() {
    PLUGIN_ROOT=$(mktemp -d)
    SINK_FILE=$(mktemp --suffix=.jsonl)
    mkdir -p "$PLUGIN_ROOT/hook-plugins" "$PLUGIN_ROOT/test-fixtures"
    # Minimal stub for legacy-bash-adapter.wasm — test-writer replaces
    # with the real built WASM artifact. Placeholder keeps harness parseable.
}

teardown() {
    rm -rf "$PLUGIN_ROOT"
    rm -f "$SINK_FILE"
}

# Helper: find event line by type in the VSDD_SINK_FILE output
find_event_line() {
    local event_type="$1"
    grep "\"type\":\"${event_type}\"" "$SINK_FILE" 2>/dev/null | tail -1
}

# Helper: assert all listed fields are present in a JSON line.
# trace_id is checked for field presence only — null is permitted as
# DI-017 last-resort fallback per BC-3.08.001 Error Paths.
assert_fields_present() {
    local json_line="$1"
    shift
    for field in "$@"; do
        local val
        val=$(echo "$json_line" | python3 -c \
            "import sys,json; d=json.load(sys.stdin); print(d.get('${field}','__MISSING__'))")
        [ "$val" != "__MISSING__" ] || {
            echo "FAIL: mandatory field '${field}' missing from: $json_line"
            return 1
        }
    done
}

# Dispatch helper: write registry, supply envelope on stdin, run dispatcher.
# $1 = registry toml content (heredoc string)
# $2 = event_name (e.g. "PostToolUse")
run_dispatcher() {
    local registry_content="$1"
    local event_name="$2"
    local tool_name="${3:-Write}"

    printf '%s' "$registry_content" > "$PLUGIN_ROOT/hooks-registry.toml"

    local envelope
    envelope=$(printf '{"hook_event_name":"%s","tool_name":"%s","session_id":"vp079-test","tool_input":{}}' \
        "$event_name" "$tool_name")

    run sh -c "printf '%s' '$envelope' | \
        CLAUDE_PLUGIN_ROOT=\"$PLUGIN_ROOT\" \
        VSDD_SINK_FILE=\"$SINK_FILE\" \
        CLAUDE_PROJECT_DIR=\"$PLUGIN_ROOT\" \
        factory-dispatcher 2>/dev/null"
}

# ---
# Scenario 1: plugin.async_block_discarded
# Trigger: async group plugin returns exit code 2
# Assumption: fixture includes a sync plugin so sync_group is non-empty, keeping
# the dispatcher alive during the async task's execution and the drain window
# (ASYNC_DRAIN_WINDOW_MS canonical value per DI-019; default 100ms). The async plugin
# exits 2 immediately (fast), so its terminal event is emitted well within the drain window.
# Invariant: ASYNC_DRAIN_WINDOW_MS >= timeout_ms + emit_latency for the async plugin.
@test "plugin.async_block_discarded emits all mandatory fields" {
    # Fixture: registry with one sync plugin (exit 0) and one async plugin (exit 2).
    # The sync plugin keeps the dispatcher alive; the async plugin triggers the event.
    printf '%s\n' "exit 0" > "$PLUGIN_ROOT/test-fixtures/exit0.sh"
    printf '%s\n' "exit 2" > "$PLUGIN_ROOT/test-fixtures/exit2.sh"
    chmod +x "$PLUGIN_ROOT/test-fixtures/exit0.sh"
    chmod +x "$PLUGIN_ROOT/test-fixtures/exit2.sh"

    run_dispatcher '
schema_version = 2

[[hooks]]
name = "sync-gate-plugin"
plugin = "hook-plugins/legacy-bash-adapter.wasm"
on_error = "block"
async = false
events = ["PostToolUse"]
priority = 50

[hooks.config]
script_path = "test-fixtures/exit0.sh"

[[hooks]]
name = "test-async-blocker"
plugin = "hook-plugins/legacy-bash-adapter.wasm"
on_error = "continue"
async = true
events = ["PostToolUse"]
priority = 100

[hooks.config]
script_path = "test-fixtures/exit2.sh"
' "PostToolUse"

    local line
    line=$(find_event_line "plugin.async_block_discarded")
    [ -n "$line" ] || {
        echo "FAIL: plugin.async_block_discarded event not found in $SINK_FILE"
        return 1
    }

    assert_fields_present "$line" \
        type trace_id plugin_name exit_code timestamp reason

    local reason
    reason=$(echo "$line" | python3 -c \
        "import sys,json; print(json.load(sys.stdin)['reason'])")
    [ "$reason" = "async_plugin_block_verdict_discarded" ] || {
        echo "FAIL: reason must be 'async_plugin_block_verdict_discarded'; got: $reason"
        return 1
    }
}
```

## False-Positive Scenarios

| Scenario | Expected behavior |
|----------|------------------|
| Normal async plugin run (exit 0) | No `plugin.async_block_discarded` event emitted |
| Valid registry v2 | No `dispatcher.schema_mismatch` event emitted |
| Valid registry with no block+async pairs | No `dispatcher.registry_invalid` event emitted |
| Async plugin completes within timeout_ms | No `plugin.timeout` event emitted |
| `trace_id` unavailable at emission time | Event emitted with `trace_id: null`; this is the DI-017 last-resort fallback permitted by BC-3.08.001 Error Paths — VP-079 does not flag null as a violation, only field absence |
| Async task exceeds ASYNC_DRAIN_WINDOW_MS | No event emitted (truncation expected — dispatcher forcibly terminates the task at drain expiry per DI-019 + BC-1.14.001 PC4); Scenario 5 verifies this non-emission |

## Amendment 2026-05-07 (v1.2 — F2 pass-3 fix burst)

Addresses adversary pass-3 findings F-P3-002 and F-P3-007.

**Root cause (F-P3-002 / F-P3-007):** BC-1.14.001 PC4 specifies that the dispatcher
exits as soon as `sync_group` completes. VP-079 Scenarios 1 and 4 had async-only
fixture registries (empty `sync_group`), so the dispatcher exited immediately after
spawning the async task — before any timeout enforcement, `plugin.timeout` detection,
or `plugin.async_block_discarded` emission could occur. This made both scenarios
structurally untestable under the original BC-1.14.001 PC4 semantics.

**Resolution:** PO amended BC-1.14.001 to v1.3, adding a bounded
`ASYNC_DRAIN_WINDOW_MS` (default 100ms). After `sync_group` completes, the
dispatcher waits up to 100ms for async tasks to emit terminal events before exit.
Async tasks not done by drain expiry are forcibly terminated with no event emission.

**VP-079 changes in this amendment:**

1. **Property Statement — Property 5 added:** Formalizes the ASYNC_DRAIN_WINDOW_MS
   constraint and the consequences for Scenarios 1 and 4. Documents the invariant
   `ASYNC_DRAIN_WINDOW_MS >= timeout_ms + emit_latency` that test fixtures must satisfy.

2. **Scenario 1 fixture fixed:** Added a sync plugin (`sync-gate-plugin`, exit 0) so
   `sync_group` is non-empty. The dispatcher now runs the sync group, then enters the
   100ms drain window during which `test-async-blocker` (exit 2) emits
   `plugin.async_block_discarded`. Explicit assumption documented in test comment.

3. **Scenario 4 fixture fixed:** Added `sync-gate-plugin` (exit 0) for the same
   reason. `slow-async-plugin` has `timeout_ms = 50` — within the 100ms drain window —
   so `plugin.timeout` is emitted before drain expiry. Explicit drain window assumption
   documented.

4. **Scenario 5 added (fault-injection — drain window truncation):** New scenario
   with `timeout_ms = 200ms` (exceeds 100ms drain window). Verifies that `plugin.timeout`
   is NOT emitted when the async task's timeout fires after drain expiry. This tests the
   negative case required by the updated BC-1.14.001 PC4 semantics.

5. **False-Positive Scenarios table:** Added row for "async task exceeds drain window."

6. **Feasibility Assessment:** Added "Drain window dependency" row.

No changes to Scenarios 2 or 3 (schema_mismatch / registry_invalid — those are
sync-path events unaffected by the drain window).
