# VP-079: Async-Semantics Event Types — Payload Schema Conformance

## Property Statement

For each of the four event types introduced by ADR-019 F2 and catalogued in BC-3.08.001,
when the triggering condition occurs, the dispatcher MUST emit a JSON line to
`events-*.jsonl` that:

1. Carries the mandatory fields specified in BC-3.08.001 (no mandatory field is absent
   or null).
2. Parses as valid JSON without a schema.
3. Has a `type` field matching the exact event-type string.
4. Has a `trace_id` field. The value is non-null in nominal operation; per BC-3.08.001
   Error Paths, `trace_id: null` is acceptable as a last-resort fallback when trace
   context is unavailable at emission time (per DI-017 degraded-mode provision).

The four event types and their mandatory fields are:

| Event type | Mandatory fields |
|------------|-----------------|
| `plugin.async_block_discarded` | `type`, `trace_id`, `session_id`, `plugin_name`, `exit_code`, `timestamp`, `reason` |
| `dispatcher.schema_mismatch` | `type`, `trace_id`, `session_id`, `found_version`, `expected_version`, `timestamp`, `error_code` |
| `dispatcher.registry_invalid` (E-REG-002) | `type`, `trace_id`, `session_id`, `offending_plugin`, `violation`, `error_code`, `timestamp` |
| `dispatcher.registry_invalid` (E-REG-003) | `type`, `trace_id`, `session_id`, `offending_plugin`, `offending_event`, `offending_tool`, `violation`, `error_code`, `timestamp` |
| `plugin.timeout` (async path) | `type`, `trace_id`, `session_id`, `plugin_name`, `execution_group`, `timeout_ms`, `timestamp` |

Emit-then-crash is acceptable for `dispatcher.schema_mismatch` and
`dispatcher.registry_invalid` (BC-3.08.001 Invariant 2). The event must be written to
`events-*.jsonl` before the dispatcher exits.

5. **ASYNC_DRAIN_WINDOW_MS constraint (DI-019 per BC-1.14.001 PC4 traceability):** After
   `sync_group` completes, the dispatcher waits up to `ASYNC_DRAIN_WINDOW_MS` (canonical
   value defined in DI-019) for async tasks to emit terminal events before
   exit. DI-019 is the authoritative constant source; BC-1.14.001 PC4 is the dispatch-loop
   partition contract that enforces it. Scenarios 1 and 4 require that async-plugin-triggered
   events (`plugin.async_block_discarded`, `plugin.timeout`) reach FileSink within this drain
   window. Both scenarios therefore need a sync plugin in their fixture registry to hold the
   dispatcher alive for the duration of the async task's execution (since `sync_group` empty
   → dispatcher exits immediately, before any async event can be emitted). The drain window
   guarantees at most `ASYNC_DRAIN_WINDOW_MS` on the async side; `timeout_ms` values below
   `ASYNC_DRAIN_WINDOW_MS` are within the drain window and will have their terminal events
   captured. Async tasks that exceed the drain window are forcibly terminated — their events
   are NOT emitted (this property, Scenario 5).

6. **Production-path emission counter-proof:** For each of the four event types, the
   corresponding `emit_*` function in `crates/factory-dispatcher/src/host/emit_event.rs`
   is invoked from a production code path (not test code) when the triggering condition
   occurs. The five production caller sites are:
   - the `emit_dispatcher_schema_mismatch` call site in `factory_dispatcher::main::run` (SchemaVersion match arm) — SITE_1
   - the `emit_dispatcher_registry_invalid` call site in the `RegistryError::AsyncBlockConflict` match arm of `factory_dispatcher::main::run` (E-REG-002 path) — SITE_2
   - the `emit_dispatcher_registry_invalid` call site in the `RegistryError::DuplicateEntry` match arm of `factory_dispatcher::main::run` (E-REG-003 path, added by F-P8-001) — SITE_5
   - the `emit_plugin_async_block_discarded` call site in the async drain loop of `factory_dispatcher::main::run` — SITE_3
   - the `emit_plugin_timeout_async` call site in the async timeout match arm of `factory_dispatcher::main::run` — SITE_4

   Verified by mutation: removing any single production caller (e.g., commenting out
   the emit-call site) MUST cause at least one of Scenarios 1–5 to fail with "expected
   event not present." If Scenarios 1–5 can all pass GREEN after a production caller is
   removed, VP-079 provides no end-to-end coverage guarantee and MUST be treated as
   insufficient. Scenario 6 (below) is the mechanical counter-proof for this property.

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| integration | bats (fault-injection) | yes — one triggering scenario per event type | payload field presence, JSON parseability, type string, trace_id present (non-null in nominal mode; null permitted as DI-017 fallback) |

### Why integration (not Kani)

All four event types involve FileSink I/O (writing to `events-*.jsonl`) and are
emitted via the effectful shell of the dispatcher. Kani verifies pure function
properties; it cannot model filesystem writes. The correct verification approach is
end-to-end fault injection: supply the triggering condition (async block verdict, bad
schema version, invalid registry, async plugin timeout) and parse the resulting JSONL
output to assert field presence.

## Proof Harness Skeleton

> **Test-infrastructure note:** The `plugin = "hook-plugins/legacy-bash-adapter.wasm"` + `script_path = "test-fixtures/<X>.sh"` pattern in this fixture is test infrastructure used to inject controlled exit codes into the dispatcher under test. The bash stubs are stand-ins for OTHER plugins in the registry, not the production plugin under test (which is itself native WASM per BC-7.06.001 + S-15.01 AC-007). Per project WASM-migration directive, NEW shipped plugins are native WASM; legacy-bash-adapter usage in test fixtures is a transitional convenience that avoids unnecessary per-fixture WASM compilation overhead.

#!/usr/bin/env bats
# async-event-schema-conformance.bats
# VP-079: BC-3.08.001 — each of the four async-semantics event types emits
# all mandatory fields per the catalogued wire format.
#
# Invocation surface: dispatcher reads registry from
# ${CLAUDE_PLUGIN_ROOT}/hooks-registry.toml and reads the hook payload
# from stdin (JSON envelope). No --registry or --event CLI flags exist.
# Event capture: VSDD_SINK_FILE (debug builds only) or internal log dir.

PLUGIN_ROOT=""
SINK_FILE=""

setup() {
    PLUGIN_ROOT=$(mktemp -d)
    SINK_FILE=$(mktemp --suffix=.jsonl)
    mkdir -p "$PLUGIN_ROOT/hook-plugins" "$PLUGIN_ROOT/test-fixtures"
    # Minimal stub for legacy-bash-adapter.wasm — test-writer replaces
    # with the real built WASM artifact. Placeholder keeps harness parseable.
}

teardown() {
    rm -rf "$PLUGIN_ROOT"
    rm -f "$SINK_FILE"
}

# Helper: find event line by type in the VSDD_SINK_FILE output
find_event_line() {
    local event_type="$1"
    grep "\"type\":\"${event_type}\"" "$SINK_FILE" 2>/dev/null | tail -1
}

# Helper: assert all listed fields are present in a JSON line.
# trace_id is checked for field presence only — null is permitted as
# DI-017 last-resort fallback per BC-3.08.001 Error Paths.
assert_fields_present() {
    local json_line="$1"
    shift
    for field in "$@"; do
        local val
        val=$(echo "$json_line" | python3 -c \
            "import sys,json; d=json.load(sys.stdin); print(d.get('${field}','__MISSING__'))")
        [ "$val" != "__MISSING__" ] || {
            echo "FAIL: mandatory field '${field}' missing from: $json_line"
            return 1
        }
    done
}

# Dispatch helper: write registry, supply envelope on stdin, run dispatcher.
# $1 = registry toml content (heredoc string)
# $2 = event_name (e.g. "PostToolUse")
run_dispatcher() {
    local registry_content="$1"
    local event_name="$2"
    local tool_name="${3:-Write}"

    printf '%s' "$registry_content" > "$PLUGIN_ROOT/hooks-registry.toml"

    local envelope
    envelope=$(printf '{"hook_event_name":"%s","tool_name":"%s","session_id":"vp079-test","tool_input":{}}' \
        "$event_name" "$tool_name")

    run sh -c "printf '%s' '$envelope' | \
        CLAUDE_PLUGIN_ROOT=\"$PLUGIN_ROOT\" \
        VSDD_SINK_FILE=\"$SINK_FILE\" \
        CLAUDE_PROJECT_DIR=\"$PLUGIN_ROOT\" \
        factory-dispatcher 2>/dev/null"
}

# Scenario 6: Production-caller mutation counter-proof (Property 6)
# For each of the 5 production emit-caller sites in factory_dispatcher::main::run,
# comment out the emit call in a temporary git worktree, rebuild, and assert at
# least one of Scenarios 1-5 FAILS. The mutation MUST be reverted after each trial.
#
# Caller sites under mutation (stable symbol anchors per TD-031):
#   SITE_1 — emit_dispatcher_schema_mismatch call site in factory_dispatcher::main::run (SchemaVersion match arm)
#   SITE_2 — emit_dispatcher_registry_invalid call site in RegistryError::AsyncBlockConflict match arm of factory_dispatcher::main::run (E-REG-002 path)
#   SITE_3 — emit_plugin_async_block_discarded call site in async drain loop of factory_dispatcher::main::run
#   SITE_4 — emit_plugin_timeout_async call site in async timeout match arm of factory_dispatcher::main::run
#   SITE_5 — emit_dispatcher_registry_invalid call site in RegistryError::DuplicateEntry match arm of factory_dispatcher::main::run (E-REG-003 path)
#
# Implementation note: the test-writer SHOULD prefer cargo-mutants for this proof
# if the mut targets are stable. The bash approach below is the fallback when
# cargo-mutants cannot easily target call-site suppression (vs. function bodies).
#
# cargo-mutants preferred invocation (declarative):
#   cargo mutants --jobs=1 \
#     --filter='emit_dispatcher_schema_mismatch|emit_dispatcher_registry_invalid|emit_plugin_async_block_discarded|emit_plugin_timeout_async' \
#     -- bats tests/bats/async-event-schema-conformance.bats
# Note: the `emit_dispatcher_registry_invalid` pattern in --filter covers both
# SITE_2 (AsyncBlockConflict / E-REG-002) and SITE_5 (DuplicateEntry / E-REG-003)
# since both call sites invoke the same fn name.
# Expectation: ALL mutations are "caught" (at least one test fails per mutation).
# If any mutation survives, Scenarios 1-5 are insufficient and VP-079 fails.
# Mutation target is the bats production-path conformance file (matching Scenarios
# 1-5/7/8 actually invoking the dispatcher binary), NOT the in-process Rust unit
# tests at `event_emission_fault_injection.rs` (those test emit functions in
# isolation per F-P1-002 limitation).
#
# bash worktree approach (imperative fallback):
@test "vp_079_scenario_6_production_caller_mutation" {
    # Locate the production binary source root via git
    local src_root
    src_root=$(git -C "$PLUGIN_ROOT" rev-parse --show-toplevel 2>/dev/null) || {
        # PLUGIN_ROOT is a temp dir; fall back to the real source root
        src_root=$(git rev-parse --show-toplevel 2>/dev/null)
    }
    [ -n "$src_root" ] || {
        echo "SKIP: cannot determine source root; run from within the git working tree"
        skip "source root unavailable"
    }

    local main_rs="$src_root/crates/factory-dispatcher/src/main.rs"
    [ -f "$main_rs" ] || skip "main.rs not found at $src_root"

    # Confirm binary build is current
    run cargo build --bin factory-dispatcher --manifest-path "$src_root/Cargo.toml" 2>&1
    [ "$status" -eq 0 ] || skip "factory-dispatcher build failed; skip mutation trial"

    # Baseline: all 5 scenarios must pass before any mutation
    run bats --filter 'plugin\.' "$(dirname "${BASH_SOURCE[0]}")/async-event-schema-conformance.bats" 2>&1
    local baseline_status="$status"

    # Declare the 5 production caller sites as (line_range fn_pattern site_label) tuples.
    # line_range: single line number (e.g. "133") OR inclusive range (e.g. "142,147")
    #             for multi-line emit calls spanning the call site and its arguments.
    # fn_pattern: unique fn name fragment; used only for diagnostic grep output.
    # site_label: human-readable label for failure messages.
    #
    # NOTE (TD-031): line ranges below are operationally required by sed; spec text
    # uses stable symbol anchors per TD-031. See Property 6 for canonical site descriptions.
    # Line ranges verified by POLICY 4 grep at HEAD (F-P13-001):
    #   133       — emit_dispatcher_schema_mismatch (single-line call)
    #   142,147   — emit_dispatcher_registry_invalid AsyncBlockConflict / E-REG-002
    #   162,167   — emit_dispatcher_registry_invalid DuplicateEntry / E-REG-003
    #   416       — emit_plugin_async_block_discarded (single-line call)
    #   427       — emit_plugin_timeout_async (single-line call)
    SITES=(
        "133 emit_dispatcher_schema_mismatch SITE_1_schema_mismatch"
        "142,147 emit_dispatcher_registry_invalid SITE_2_async_block_conflict"
        "162,167 emit_dispatcher_registry_invalid SITE_5_duplicate_entry"
        "416 emit_plugin_async_block_discarded SITE_3_async_block_discarded"
        "427 emit_plugin_timeout_async SITE_4_async_timeout"
    )

    local entry
    for entry in "${SITES[@]}"; do
        read -r line_range fn_pattern site_label <<< "$entry"

        # Diagnostic: show current call site lines
        grep -n "${fn_pattern}" "$main_rs" | head -3

        # Create mutant: work on a temp copy, then swap in
        local mutant_rs
        mutant_rs=$(mktemp)
        cp "$main_rs" "$mutant_rs"

        if [[ "$line_range" == *,* ]]; then
            # Multi-line emit call: comment ALL lines in the range (F-P13-001)
            sed -i.tmp "${line_range}{ s|^|// MUTANT-SUPPRESSED: | }" "$mutant_rs"
        else
            # Single-line emit call: use line-anchored substitution (F-P13-001)
            sed -i.tmp "${line_range}s|^|// MUTANT-SUPPRESSED: |" "$mutant_rs"
        fi
        rm -f "${mutant_rs}.tmp"

        # Swap in the mutant, rebuild, re-run scenarios 1-5
        cp "$main_rs" "${main_rs}.bak"
        cp "$mutant_rs" "$main_rs"
        rm -f "$mutant_rs"

        run cargo build --bin factory-dispatcher --manifest-path "$src_root/Cargo.toml" 2>&1
        local build_ok="$status"

        local caught=0
        if [ "$build_ok" -eq 0 ]; then
            run bats --filter 'plugin\.|dispatcher\.' \
                "$(dirname "${BASH_SOURCE[0]}")/async-event-schema-conformance.bats" 2>&1
            # If ANY scenario fails, the mutation is caught (status != 0)
            [ "$status" -ne 0 ] && caught=1
        fi

        # Always restore
        cp "${main_rs}.bak" "$main_rs"
        rm -f "${main_rs}.bak"

        # Rebuild clean binary
        cargo build --bin factory-dispatcher --manifest-path "$src_root/Cargo.toml" >/dev/null 2>&1

### §Scenario 7 — EC-012 partial-drain witness (F-P15-004)

**Source:** BC-1.14.001 v1.9 EC-012 (partial-drain witness — events emitted by async tasks that
complete within the drain window reach FileSink; in-flight tasks not completed at drain expiry are
forcibly terminated and their events are NOT emitted).

**Bats reference:** `@test "VP-079 S7: ..."` block in `tests/bats/async-event-schema-conformance.bats`.

**Triggering condition:** Two async plugins are registered; the first completes within the
`ASYNC_DRAIN_WINDOW_MS` window (fast — exits immediately after the sync group completes); the
second exceeds the drain window (slow — `sleep 60`). The fixture asserts that:

1. The fast plugin's terminal event (`plugin.async_block_discarded` or `plugin.timeout`) IS
   present in `events-*.jsonl`.
2. The slow plugin's terminal event is NOT present in `events-*.jsonl` (truncated at drain expiry).

**Properties verified by Scenario 7:**

- Completed-within-drain events are emitted and reach FileSink before dispatcher exit.
- In-flight events (tasks still running at drain expiry) MAY be lost; their absence is
  expected and not a VP-079 violation.
- The drain window boundary is enforced: the dispatcher does not wait indefinitely.

**Fixture note:** Scenario 7 requires a sync plugin to hold the dispatcher alive during the
`sync_group` execution phase so the drain window can activate. Without a sync plugin,
`sync_group` is empty and the dispatcher exits before any async task has time to emit.

### §Scenario 8 — E-REG-003 DuplicateEntry counter-proof (F-P15-004)

**Source:** BC-7.06.001 v1.9 Invariant 7 (fail-closed `DuplicateEntry` with E-REG-003) +
BC-3.08.001 §Event 3 §E-REG-003 §Postconditions (E-REG-003 wire-format extended per
F-P14-001 Path B with `offending_event`, `offending_tool`, `session_id`).

**Bats reference:** `@test "VP-079 S8: ..."` block in `tests/bats/async-event-schema-conformance.bats`.

**Triggering condition:** A registry with two entries sharing an identical `(name, event, tool)`
tuple is supplied. `registry.rs::validate()` detects the duplicate, emits
`dispatcher.registry_invalid` with `error_code = "E-REG-003"`, writes a `[E-REG-003]`-prefixed
line to stderr, and exits with code 2.

**Properties verified by Scenario 8:**

1. Dispatcher exits with exit code 2 (fail-closed per BC-7.06.001 Invariant 7 F-P8-001 +
   BC-3.08.001 E-REG-003 fail-closed classification).
2. `stderr` contains a line prefixed `[E-REG-003]` (BC-7.06.001 Invariant 7 F-P8-001 stderr
   obligation).
3. `events-*.jsonl` contains a `dispatcher.registry_invalid` event with the full E-REG-003
   wire payload (BC-3.08.001 §Event 3 §E-REG-003 §Postconditions mandatory fields per F-P14-001 Path B):
   - `offending_plugin` — name of the duplicating entry
   - `offending_event` — event value of the duplicating `(name, event, tool)` tuple
   - `offending_tool` — tool regex string of the duplicating entry, or `null` when the entry
     has no `tool` filter (wildcard binding)
   - `violation` = `"duplicate_hook_registration"`
   - `error_code` = `"E-REG-003"`

**E-REG-002 vs E-REG-003 field asymmetry (per BC-3.08.001 §E-REG-003 §Postconditions F-P14-001):** Scenario 3
verifies E-REG-002 and does NOT assert `offending_event` or `offending_tool` — those fields are
absent in the E-REG-002 wire format because AsyncBlockConflict is an intra-entry violation (a
single entry has `on_error=block` and `async=true`; no second entry is involved). Scenario 8
verifies E-REG-003 and MUST assert all three tuple fields (`offending_plugin`, `offending_event`,
`offending_tool`) because DuplicateEntry is an inter-entry violation where the
`(name, event, tool)` tuple uniquely identifies the duplicating entry.

**Cargo-mutants coverage note:** The mutation filter `emit_dispatcher_registry_invalid` in
Scenario 6 covers both SITE_2 (E-REG-002 / AsyncBlockConflict) and SITE_5 (E-REG-003 /
DuplicateEntry) since both call sites invoke the same fn name. Scenario 8 provides independent
end-to-end coverage for the E-REG-003 production path such that a mutation removing SITE_5 causes
Scenario 8 to fail with "expected event not present."
