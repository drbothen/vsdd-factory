# VP-076: Resolver-Capability Confinement

## Property Statement

A resolver attempting to read a filesystem path outside its declared `path_allow`
entries MUST:

1. **Receive `CapabilityDenied`** from `host::read_file`. The host function returns
   the `CapabilityDenied` error code (matching the same `-1` code as for hooks per
   DI-004). The resolver does NOT observe the file contents — not even a partial read.

2. **Not silently succeed.** The file's contents are never written to the resolver's
   linear memory. The `host::read_file` call returns before any file I/O occurs.

3. **Propagate the error to the dispatcher.** The dispatcher surfaces the
   capability-denied resolver invocation as `ResolverError::CapabilityDenied`. This
   is logged as a `resolver.capability_denied` event. The resolver's context key is
   NOT merged into `plugin_config`.

4. **Emit an audit event.** The dispatcher emits `resolver.capability_denied` with
   the resolver name, the attempted path, and the `path_allow` list from the
   resolver's registry entry. This mirrors the `hook.capability_denied` event
   emitted for hook capability denials (DI-004).

## Source Contract

- **BC:** BC-4.12.003 — Resolver capability model: `path_allow` declarations; deny-by-default; read-only
- **Postcondition/Invariant:** PC1 (deny-by-default: paths outside `path_allow` return `CapabilityDenied`); PC2 (no file contents exposed to resolver on denial); audit event emitted on each denial (DI-004 extension to resolvers)

## Rationale for this VP

Capability confinement is a first-class security property of the vsdd-factory
sandbox model (DI-004, VP-004, VP-021). Extending this guarantee to resolvers is
not optional: a resolver that silently reads `/etc/passwd` (or any file outside
its declared `path_allow`) would be an information-disclosure vulnerability. The
`WaveContextResolver` is declared with `path_allow = [".factory/"]` — it must be
impossible for it to read files outside that prefix, even if the resolver's WASM
code attempts to do so.

The integration test uses a real WASM resolver that deliberately attempts to read
`/etc/passwd` — a path that is guaranteed to be outside any `.factory/` prefix on
all supported platforms. This is the exact test pattern used for hook capability
confinement (VP-004 precedent).

## Formal Property Statement (integration test)

```rust
// VP-076 integration test — lives in S-12.08 bats harness
// (see §Proof Harness Locations for deferral rationale)
#[cfg(test)]
mod vp_076_capability_confinement {
    use factory_dispatcher::resolver::{ResolverRegistry, ResolverError};
    use factory_dispatcher::executor::Executor;

    /// VP-076-A: A resolver attempting to read /etc/passwd (outside path_allow)
    /// receives CapabilityDenied and does NOT observe file contents.
    #[test]
    fn test_resolver_cannot_read_outside_path_allow() {
        let registry = ResolverRegistry::from_toml_str(r#"
            [[resolvers]]
            name = "confined-test-resolver"
            plugin = "tests/fixtures/path_escaping_resolver.wasm"
            path_allow = [".factory/"]
        "#).expect("registry parse must succeed");

        let executor = Executor::new_with_resolvers(registry);
        let entry = mock_registry_entry_with_needs_context(vec!["confined-test-resolver"]);
        let payload = mock_hook_payload_with_project_dir("/tmp/test-project");

        let dispatch_result = executor.dispatch(&entry, &payload);
        assert!(
            dispatch_result.is_ok(),
            "dispatch must succeed even when resolver attempts capability escape"
        );

        let config = dispatch_result.unwrap().plugin_config;
        assert!(
            config.get("confined-test-resolver").is_none(),
            "capability-denied resolver key must be absent from plugin_config"
        );
    }

    /// VP-076-D: A resolver IS permitted to read a path within its declared path_allow.
    #[test]
    fn test_resolver_can_read_within_path_allow() {
        let project_dir = test_utils::create_temp_project_with_wave_state();

        let registry = ResolverRegistry::from_toml_str(&format!(r#"
            [[resolvers]]
            name = "wave_context"
            plugin = "hook-plugins/vsdd-context-resolvers.wasm"
            path_allow = [".factory/"]
        "#)).expect("registry parse must succeed");

        let executor = Executor::new_with_resolvers(registry);
        let entry = mock_registry_entry_with_needs_context(vec!["wave_context"]);
        let payload = mock_hook_payload_with_project_dir(&project_dir);

        let dispatch_result = executor.dispatch(&entry, &payload);
        assert!(dispatch_result.is_ok(), "dispatch must succeed");

        let config = dispatch_result.unwrap().plugin_config;
        assert!(
            config.get("wave_context").is_some(),
            "wave_context resolver key must be present when read succeeds"
        );
    }
}
```

## Expected Outcome

All four integration test cases pass:

- VP-076-A: The `CapabilityDenied` return path prevents file access for out-of-`path_allow` paths.
- VP-076-B: No file contents from the forbidden path appear in `plugin_config` or the resolver output.
- VP-076-C: A `resolver.capability_denied` event is emitted with `resolver_name` and `attempted_path`.
- VP-076-D: A legitimate read within `path_allow` succeeds (no over-restriction).
