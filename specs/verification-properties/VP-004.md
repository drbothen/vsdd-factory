# VP-004: Capability Denial Produces Return Code AND Audit Event

## Property Statement

Every capability-gated host function (exec_subprocess, read_file, env) atomically returns CAPABILITY_DENIED (-1) to the WASM caller AND emits an internal.capability_denied event to the internal log. A return code without an audit event, or an audit event without the return code, constitutes a violation.

## Source Contract

- **BC:** BC-1.05.001 — exec_subprocess denies when no exec_subprocess capability
- **Postcondition/Invariant:** DI-004 — Capability denial always produces both a return code AND an audit event

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| unit-test | Rust unit tests in host/*.rs | yes — deny paths | all three cap-gated host fns |

## Proof Harness Skeleton

```rust
#[test]
fn verify_vp_004_capability_denial_is_atomic() {
    let mut events = vec![];
    let ctx = bare_context_with_event_capture(&mut events);

    // Attempt exec without capability
    let result = exec_subprocess(&ctx, "git", &[], 100, 1024);

    // Return code must be CAPABILITY_DENIED
    assert_eq!(result, Err(CAPABILITY_DENIED));

    // Audit event must also have been emitted
    assert!(events.iter().any(|e| e.event_type == "internal.capability_denied"));
}
```

## Notes

The audit-event half of this invariant (emit + return code atomicity) is harder to verify in isolation because event emission is a side effect on the Store context. The unit tests capture this via mock/spy pattern on the emitted event buffer. A kani proof could verify the control-flow path but not the side-effect emission; unit tests remain the primary vehicle.
