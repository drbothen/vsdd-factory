# VP-078: CI Lint Invariant — `on_error = "block"` Implies `async = false`

## Property Statement

**Property 1 — Negative classification (invariant):** For every `[[hooks]]` entry in
`plugins/vsdd-factory/hooks-registry.toml`:

**If `on_error = "block"` then `async` is either unset or explicitly `false`.**

Equivalently: no entry may have both `on_error = "block"` and `async = true`.

This invariant is enforced at three layers (defense in depth):

1. **Parse time** — `registry.rs` `validate()` emits a hard error if any entry
   violates the invariant. The dispatcher refuses to start.
2. **Pre-commit** — a lint hook (native WASM plugin per BC-7.06.001 + S-15.01 AC-007 — `lint-registry-async-invariant.wasm`) scans
   `hooks-registry.toml` and fails the commit if any entry violates the invariant.
3. **CI on every PR** — the same lint runs in CI and fails the build.

**Property 2 — Positive classification (BC-7.06.001 Invariant 6):** The following
nine telemetry-only plugins MUST be present in the registry AND carry `async = true`:
`capture-commit-activity`, `capture-pr-activity`, `session-start-telemetry`,
`session-end-telemetry`, `worktree-hooks`, `tool-failure-hooks`, `track-agent-start`,
`track-agent-stop`, `session-learning`. Verified at CI time by Harness 3. A future
engineer flipping any of these to `async = false` would silently degrade user-facing
latency without triggering Property 1's lint.

- **ADR:** ADR-019 — Plugin async semantics belong at the registry layer (§Decision 4)
- **BC:** BC-7.06.001 — Per-plugin async-field schema and CI lint invariant

### Why integration (not Kani)

This is a config-file constraint, not a pure function property. The invariant is about
the content of a TOML file at rest — a relationship between two fields in each
`[[hooks]]` entry. Kani verifies program logic over bounded inputs; it is not
appropriate for file-content constraints. A CI lint (bats test or bash scan) is the
correct tool: it reads the actual registry file used in production, making it a
live contract test rather than a synthetic unit test.

The runtime-side enforcement (parse-time check in `registry.rs`) is verified by a
Rust unit test, not by Kani, because the parse path has I/O dependencies (file
reading) that are injected via fixture strings in the unit test.

## Proof Harness Skeleton

### CI lint (bats)

```bash
#!/usr/bin/env bats
# hooks-registry-lint.bats
# Invariant: on_error=block => async != true

REGISTRY="plugins/vsdd-factory/hooks-registry.toml"

@test "no hooks-registry entry has both on_error=block and async=true" {
    # Extract all [[hooks]] blocks and check for the violating combination.
    # A violation is: a block that contains BOTH on_error = "block" AND async = true.
    #
    # Implementation: split the file into per-entry blocks (on [[hooks]] boundaries),
    # then for each block assert the pair does not co-occur.

    local violations
    violations=$(python3 - <<'EOF'
import re, sys

with open("plugins/vsdd-factory/hooks-registry.toml") as f:
    content = f.read()

# Split on [[hooks]] boundaries
entries = re.split(r'\[\[hooks\]\]', content)[1:]  # skip preamble

bad = []
for i, entry in enumerate(entries, 1):
    has_block = bool(re.search(r'on_error\s*=\s*"block"', entry))
    has_async = bool(re.search(r'async\s*=\s*true', entry))
    if has_block and has_async:
        # Extract name for diagnostics
        m = re.search(r'name\s*=\s*"([^"]+)"', entry)
        name = m.group(1) if m else f"entry #{i}"
        bad.append(name)

if bad:
    print("VIOLATION: on_error=block AND async=true in: " + ", ".join(bad))
    sys.exit(1)
EOF
    )

    [ -z "$violations" ] || {
        echo "$violations"
        return 1
    }
}

@test "registry.rs validate() rejects entry with on_error=block and async=true" {
    # Integration-level: dispatcher hard-errors on violating registry at startup.
    # The dispatcher reads registry from ${CLAUDE_PLUGIN_ROOT}/hooks-registry.toml and
    # reads the hook payload envelope from stdin (JSON). No --registry or --event CLI
    # flags exist; invocation uses env vars + stdin per the actual CLI surface.
    local plugin_root
    plugin_root=$(mktemp -d)
    cat > "$plugin_root/hooks-registry.toml" <<'TOML'
schema_version = 2

[[hooks]]
name = "violating-plugin"
plugin = "hook-plugins/legacy-bash-adapter.wasm"
on_error = "block"
async = true
event = "PostToolUse"
priority = 400

[hooks.config]
script_path = "test-fixtures/exit0.sh"
TOML

    # Minimal valid stdin envelope — dispatcher reads event_name + session_id from JSON stdin.
    local envelope
    envelope='{"hook_event_name":"PostToolUse","tool_name":"Write","session_id":"test-session-001","tool_input":{}}'

    # Dispatcher must exit non-zero and emit a registry_invalid diagnostic to its
    # internal log or stderr when it encounters the block+async invariant violation.
    run sh -c "CLAUDE_PLUGIN_ROOT=\"$plugin_root\" RUST_LOG=error \
        echo '$envelope' | factory-dispatcher 2>&1"
    [ "$status" -ne 0 ]
    [[ "$output" == *"registry_invalid"* ]] || \
        [[ "$output" == *"on_error=block"* ]] || \
        [[ "$output" == *"async"* ]]

    rm -rf "$plugin_root"
}
```

> **Test-infrastructure note:** The `plugin = "hook-plugins/legacy-bash-adapter.wasm"` + `script_path = "test-fixtures/<X>.sh"` pattern in this fixture is test infrastructure used to inject controlled exit codes into the dispatcher under test. The bash stubs are stand-ins for OTHER plugins in the registry, not the production plugin under test (which is itself native WASM per BC-7.06.001 + S-15.01 AC-007). Per project WASM-migration directive, NEW shipped plugins are native WASM; legacy-bash-adapter usage in test fixtures is a transitional convenience that avoids unnecessary per-fixture WASM compilation overhead.

### Harness 3: Positive telemetry-plugin classification (integration test)

Verifies that every plugin named in BC-7.06.001 Invariant 6 is present in the
registry AND carries `async = true`. This is the positive complement to Harness 1
(which catches the negative case: no entry may combine `on_error=block` with
`async=true`). Harness 3 catches a future engineer silently flipping a required
telemetry plugin back to `async = false`, degrading latency without a lint failure.

            "capture-pr-activity",
            "session-start-telemetry",
            "session-end-telemetry",
            "worktree-hooks",
            "tool-failure-hooks",
            "track-agent-start",
            "track-agent-stop",
            "session-learning",

                "BC-7.06.001 Invariant 6 violation: '{}' must be async=true \

## False-Positive Scenarios

| Scenario | Expected behavior |
|----------|------------------|
| Plugin with `on_error = "block"` and `async` field absent | Accepted — absent defaults to `false`, invariant satisfied |
| Plugin with `on_error = "block"` and `async = false` explicit | Accepted — invariant satisfied |
| Plugin with `on_error = "continue"` and `async = true` | Accepted — invariant only constrains the block/async pairing |
| Plugin with `on_error = "block"` and `async = true` | REJECTED at parse time, pre-commit, and CI |
| Registry with no `[[hooks]]` entries | Accepted — empty registry trivially satisfies the invariant |

The lint must not flag entries where only one of the two conditions holds. Producing
false positives would block legitimate telemetry plugin classification
(`on_error = "continue"` AND `async = true` is correct and required).

## Trigger Points

This VP is exercised at three trigger points:

1. **Claude Code PostToolUse Edit|Write hook** — runs before every commit that touches `hooks-registry.toml`
   or any `hooks/*.sh` file. Fails the commit on violation (Property 1 + Property 2).
2. **CI (every PR)** — Harness 1 (negative lint) and Harness 3 (positive classification)
   both run in the PR check suite. Fails the build on any violation.
3. **Dispatcher startup (runtime)** — `registry.rs` `validate()` checks the Property 1
   invariant at load time. Hard error with diagnostic message if violated.

Harness 4 (serde-default unit tests) runs as part of the standard `cargo test` suite
and is exercised on every CI run that builds `crates/factory-dispatcher`.
