# VP-074: Resolver-Error Isolation

## Property Statement

A resolver **panic, trap, timeout, or ABI violation** MUST NOT propagate to the
dispatcher process or abort the hook dispatch flow. Specifically:

1. **Process isolation:** A WASM trap (stack overflow, unreachable, memory out-of-
   bounds, divide-by-zero, etc.) raised inside a resolver's `resolve()` invocation
   is caught by the dispatcher's wasmtime trap-handling boundary. The dispatcher
   process does NOT unwind, panic, or exit due to a resolver trap.

2. **Result safety:** The dispatcher's `invoke_resolver` function returns
   `Result<ResolverOutput, ResolverError>` in ALL cases. The `Err` variant is
   handled without propagating to the dispatch stack. The `Ok` variant is only
   returned when the resolver's WASM execution completed normally and the output
   parsed successfully.

3. **Error classification:** The pure function `classify_resolver_trap(trap_code:
   TrapCode) -> ResolverError` maps every `TrapCode` variant to a `ResolverError`
   variant without panicking and without returning an `Err`. Every `TrapCode`
   variant is handled; no variant reaches an `unreachable!()` or `unwrap()`.

4. **Dispatch continuation:** After a resolver error, the dispatcher:
   (a) emits a `resolver.error` event with resolver name and error detail;
   (b) does NOT merge the failed resolver's output into `plugin_config`;
   (c) proceeds to hook dispatch WITHOUT the missing context key.
   The hook is dispatched in all cases, even if every declared resolver failed.

5. **No silent degrade:** The hook receives an absent key when a resolver fails.
   The dispatcher does not substitute a default value, a null, or a previous
   cached value. The hook is responsible for deciding how to handle the absent key.

## Rationale for this VP

This VP is the resolver analog of DI-002 (plugin crash isolation). The dispatcher
has always been designed so that a WASM plugin crash does not take down the
dispatcher process (VP-002). Resolvers are WASM artifacts and must receive the same
isolation guarantee. Without this VP, a resolver with a bug could abort the
dispatcher process on every hook dispatch that declares `needs_context` for that
resolver — effectively bricking all hooks that use it.

The kani harness covers the pure `classify_resolver_trap` logic (exhaustive
verification of all `TrapCode` variants). The integration test covers the real
wasmtime trap boundary (which is outside kani's pure-function scope).

## Formal Property Statement (kani harness)

```rust
// VP-074 kani harness — lives inline in
// crates/factory-dispatcher/src/resolver_classify_trap.rs

#[cfg(kani)]
mod kani_proofs {
    use super::*;

    /// VP-074-A: classify_resolver_trap is total — every TrapCode maps to a
    /// ResolverError without panic, unwrap, or unreachable!.
    /// Kani verifies this exhaustively over all TrapCode variants.
    #[kani::proof]
    fn proof_classify_trap_is_total() {
        // kani::any() generates a symbolic TrapCode covering all variants.
        let trap_code: TrapCode = kani::any();
        // This call must not panic for any TrapCode value.
        let result = classify_resolver_trap(trap_code);
        // result is a ResolverError value — the function returned normally.
        // No assertion needed beyond "did not panic" — that's what kani verifies.
        let _ = result;
    }

    /// VP-074-B: classify_resolver_trap never returns the NotFound variant.
    /// NotFound is reserved for resolvers not in the registry; a trap classification
    /// must produce Trap, Timeout, or AbiViolation — not NotFound.
    #[kani::proof]
    fn proof_classify_trap_never_returns_not_found() {
        let trap_code: TrapCode = kani::any();
        let result = classify_resolver_trap(trap_code);
        kani::assert(
            !matches!(result, ResolverError::NotFound { .. }),
            "trap classification must not return ResolverError::NotFound"
        );
    }

    /// VP-074-C: invoke_resolver returns Ok or Err — never panics.
    /// Test the pure error-path branch: if trap classification returns an error,
    /// invoke_resolver propagates it as Err, not as a panic.
    ///
    /// Note: This harness tests the pure error-path logic in invoke_resolver
    /// (the branch that handles a classified trap). The full WASM execution
    /// is tested by the integration test below — that's outside kani scope.
    #[kani::proof]
    fn proof_invoke_resolver_error_path_does_not_panic() {
        // Simulate the error classification and result-wrapping logic.
        let trap_code: TrapCode = kani::any();
        let resolver_error = classify_resolver_trap(trap_code);
        // The wrapping path: Err(resolver_error) must be constructible without panic.
        let result: Result<ResolverOutput, ResolverError> = Err(resolver_error);
        kani::assert(result.is_err(), "error path must return Err variant");
    }
}
```

## Integration Test (real WASM trap)

```rust
// VP-074 integration test — lives in
// crates/factory-dispatcher/tests/resolver_error_isolation_test.rs
#[cfg(test)]
mod vp_074_resolver_error_isolation {
    use factory_dispatcher::resolver::ResolverRegistry;
    use factory_dispatcher::executor::Executor;

    /// VP-074 integration: A resolver that traps on resolve() does not abort
    /// the dispatcher. The hook dispatch proceeds and the context key is absent.
    #[test]
    fn test_trapping_resolver_does_not_abort_dispatch() {
        // Load a deliberately-trapping resolver WASM (test fixture).
        // The resolver's resolve() immediately executes `unreachable`.
        let registry = ResolverRegistry::from_toml_str(r#"
            [[resolvers]]
            name = "trap-resolver"
            plugin = "tests/fixtures/trapping_resolver.wasm"
            path_allow = []
        "#).expect("registry parse must succeed");

        let executor = Executor::new_with_resolvers(registry);
        let entry = mock_registry_entry_with_needs_context(vec!["trap-resolver"]);
        let payload = mock_hook_payload();

        // Dispatch must succeed (not panic, not abort).
        let result = executor.dispatch(&entry, &payload);
        assert!(result.is_ok(), "dispatch must succeed even when resolver traps");

        // plugin_config must not contain the trap-resolver's key.
        let config = result.unwrap().plugin_config;
        assert!(
            config.get("trap-resolver").is_none(),
            "failed resolver key must be absent from plugin_config"
        );
    }

    /// VP-074 integration: A resolver that traps causes resolver.error event emission.
    #[test]
    fn test_trapping_resolver_emits_resolver_error_event() {
        let (registry, event_sink) = build_test_registry_with_sink("trap-resolver");
        let executor = Executor::new_with_resolvers_and_sink(registry, event_sink.clone());
        let entry = mock_registry_entry_with_needs_context(vec!["trap-resolver"]);
        let payload = mock_hook_payload();

        let _ = executor.dispatch(&entry, &payload);

        let events = event_sink.drain();
        let resolver_error_events: Vec<_> = events
            .iter()
            .filter(|e| e.event_type == "resolver.error")
            .collect();
        assert!(
            !resolver_error_events.is_empty(),
            "resolver.error event must be emitted when resolver traps"
        );
    }
}
```

## Expected Outcome

**Kani:** All three proof harnesses complete without counterexamples.
`classify_resolver_trap` is proven total: every `TrapCode` variant produces a
`ResolverError` without panic. The `NotFound` variant is never produced by trap
classification. The error-path result-wrapping is panic-free.

**Integration:** The trapping-resolver integration test confirms that a real WASM
trap is caught at the wasmtime boundary and does not propagate. The hook dispatch
proceeds. The context key is absent from `plugin_config`. A `resolver.error` event
is emitted.
