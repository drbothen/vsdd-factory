# VP-068: Tool-Failure Hook Plugin Surface Invariant — All BC-4.08.* Postconditions Hold in Integration Test

## Property Statement

For a fully wired tool-failure-hooks plugin under integration test conditions:

1. Loading `tool-failure-hooks.wasm` and sending a `PostToolUseFailure` envelope through the dispatcher MUST result in exactly one `tool.error` event captured in the file sink. The event contains 10 fields on the wire from three sources (PostToolUseFailure analog of the Tier F 10-field pattern):

   **2 plugin-set fields** (verified as present and non-null): `tool_name`, `error_message`. Both MUST be strings on the wire (per `emit_event.rs:49` string coercion — `.is_string()` not raw value matchers).
   - When the envelope's `tool_name` is absent, `tool_name` MUST equal `"unknown"` (fallback sentinel per BC-4.08.001 EC-002).
   - When the envelope's `error_message` exceeds 1000 characters, `error_message` on the wire MUST be exactly 1000 characters (truncation per BC-4.08.001 EC-001).
   - When the envelope's `error_message` is absent, `error_message` MUST equal `""`.

   **4 host-enriched fields** (auto-injected by `emit_event` host fn from `HostContext`; plugin does NOT set these): `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`. Each MUST be present as a non-empty string (per BC-1.05.012 unconditional enrichment for presence). The plugin MUST NOT set `session_id` — it is RESERVED_FIELDS host-enriched (per BC-1.05.012; BC-4.08.001 Invariant 8).

   **4 construction-time fields** (set by the dispatcher between the plugin's `emit_event` call and the final wire format; the plugin must NOT set them — implementation provenance is opaque from the spec layer): `ts`, `ts_epoch`, `schema_version`, `type`. These MUST be present; `type` MUST equal `"tool.error"`.

   **Wire format note:** Plugin-set field values arrive as JSON strings on the wire (`emit_event.rs:49` coerces all plugin-supplied values to `Value::String`). Harness assertions for `tool_name` and `error_message` MUST use `.is_string()`.

   **PostToolUseFailure field count:** 10 fields (2+4+4). Compare: SessionStart=14 (6+4+4), SessionEnd=11 (3+4+4), WorktreeCreate=10 (2+4+4), PostToolUseFailure=10 (2+4+4), WorktreeRemove=9 (1+4+4). (BC-4.08.001)

2. The `exec_subprocess` CountingMock invocation count MUST equal exactly 0 for `PostToolUseFailure` dispatch. The tool-failure-hooks plugin MUST NOT call `exec_subprocess` at any point during event handling. (BC-4.08.001 Invariant 2)

3. The `read_file` mock (if present) invocation count MUST equal exactly 0 for `PostToolUseFailure` dispatch. The tool-failure-hooks plugin MUST NOT call `read_file` at any point during event handling. (BC-4.08.001 Invariant 1)

4. `hooks.json.template` MUST contain a `PostToolUseFailure` key under `hooks`. For the entry:
   - The entry's `command` field MUST contain `"factory-dispatcher"` (dispatcher binary, NOT a `.wasm` filename).
   - The entry MUST NOT carry `"once": true` — and more specifically, the `once` key MUST BE COMPLETELY ABSENT (not `once: false`, not `once: true` — key must not exist at all).
   - The entry MUST have `"async": true`.
   - The harness assertion is:
   ```rust
   let arr = template["hooks"]["PostToolUseFailure"].as_array()
       .unwrap_or_else(|| panic!("PostToolUseFailure must be an array in hooks.json.template"));
   assert!(!arr.is_empty(), "PostToolUseFailure array must have at least one entry");
   let entry = &arr[0]["hooks"][0];
   let command = entry["command"].as_str()
       .unwrap_or_else(|| panic!("PostToolUseFailure command must be a string"));
   assert!(command.contains("factory-dispatcher"),
       "PostToolUseFailure command must reference dispatcher binary, not a WASM plugin");
   assert!(!command.contains(".wasm"),
       "PostToolUseFailure command must NOT reference a WASM plugin filename (ADR-011 layer separation)");
   // once key MUST BE ABSENT (not once:false, not once:true, not any value — key must not exist)
   // PostToolUseFailure fires per-failure; once:true would suppress subsequent failures.
   // Using is_none() rather than assert_ne!(once, Bool(true)) because Bool(false) and absent
   // have different semantics risk; defensive omission is the canonical pattern.
   assert!(entry.get("once").is_none(),
       "PostToolUseFailure must omit `once` key entirely (not `false`, not `true`); \
        fires per-failure; defensive omission protects against future Claude Code parser changes");
   assert_eq!(entry["async"], true, "PostToolUseFailure async must be true");
   ```
   (BC-4.08.002; per ADR-011 layer separation, once key ABSENT for per-failure semantics.)

5. `hooks-registry.toml` MUST contain exactly one `PostToolUseFailure` entry. For the entry:
   - `name` = `"tool-failure-hooks"`
   - `event` = `"PostToolUseFailure"`
   - `plugin` = `"hook-plugins/tool-failure-hooks.wasm"` (with `hook-plugins/` directory prefix)
   - `timeout_ms` = `5000`
   - NO capability tables (`entry.get("capabilities")` is `None`)
   - Exactly one entry (`post_tool_use_failure_count == 1`)
   - Entry does NOT carry a `once` field — `RegistryEntry` has no such field; `deny_unknown_fields` rejects it.
   (BC-4.08.003)
