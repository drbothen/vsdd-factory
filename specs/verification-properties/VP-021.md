# VP-021: Capability Deny-by-Default — Each Capability Requires Explicit Allow

## Property Statement

All capability-gated host functions (exec_subprocess binary allow, env var allow, read_file path allow) default to deny when the corresponding capability block is absent. An empty capability block also denies. Only explicitly declared entries in the allow list are permitted.

## Source Contract

- **BC:** BC-1.05.002 — exec_subprocess denies binaries not on allow-list
- **Postcondition/Invariant:** DI-004 — Capability denial produces return code and audit event

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| unit-test | Rust unit tests across host/*.rs | yes — absent and empty capability blocks | all three cap-gated host fns |

## Proof Harness Skeleton

```rust
#[test]
fn verify_vp_021_deny_by_default() {
    let bare_ctx = context_with_no_capabilities();

    // exec_subprocess denies unlisted binary
    assert_eq!(exec_subprocess_check(&bare_ctx, "curl"), Err(CAPABILITY_DENIED));

    // env denies unlisted var
    assert_eq!(env_check(&bare_ctx, "SECRET_KEY"), Err(CAPABILITY_DENIED));

    // read_file denies any path
    assert_eq!(read_file_check(&bare_ctx, "/etc/passwd"), Err(CAPABILITY_DENIED));
}
```

## Notes

The deny-by-default property emerges from Capabilities::default() having all fields None or empty Vec. Any code path that uses capabilities.exec_subprocess.as_ref()? will short-circuit to None (deny) if the block is absent. This is a Rust type-level guarantee.
