# VP-065: Session-Start Plugin Surface Invariant — All BC-4.04.* Postconditions Hold in Integration Test

## Property Statement

For a fully wired session-start plugin under integration test conditions:

1. Loading `session-start.wasm` and sending a `SessionStart` envelope through the dispatcher MUST result in exactly one `session.started` event captured in the file sink, containing all required fields: `session_id`, `factory_version`, `plugin_count`, `activated_platform`, `factory_health`, `tool_deps`, `timestamp` (BC-4.04.001).
2. The `factory_health` field in the emitted `session.started` event MUST be one of `"healthy"`, `"warnings"`, `"errors"`, or `"unknown"` — never absent or null. When `factory-health` is mocked as unavailable, `factory_health` MUST equal `"unknown"` (BC-4.04.002).
3. Sending a second `SessionStart` envelope with the same `session_id` to the same plugin instance MUST NOT produce a second `session.started` event — the file sink contains exactly one entry total (BC-4.04.003).
4. `hooks.json.template` MUST contain a `SessionStart` entry with `once: true` referencing `session-start.wasm` that passes schema validation per BC-1.01.001 (BC-4.04.004).

## Source Contract

- **Primary: BC-4.04.001** — session.started event shape + emit_event postcondition
- **BC-4.04.002** — factory-health subprocess invocation + fail-open semantics
- **BC-4.04.003** — idempotency on duplicate SessionStart within same process lifetime
- **BC-4.04.004** — hooks.json.template SessionStart registration with once:true
