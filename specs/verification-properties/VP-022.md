# VP-022: Dispatcher Exit Code Semantics — 0 for Non-Block, 2 for Block

## Property Statement

The dispatcher exits 0 on all conditions except when at least one plugin recorded block_intent=true. It exits 2 if and only if block_intent is true in the execution summary. Registry/payload/engine errors are non-blocking and exit 0 (with an internal.dispatcher_error event emitted).

## Source Contract

- **BC:** BC-1.08.001 — dispatcher exits 0 on registry/payload/engine errors (non-blocking)
- **Postcondition/Invariant:** BC-1.08.002 — dispatcher exit code is 2 iff at least one block_intent recorded

| Method | Tool | Bounded? | Coverage |

## Proof Harness Skeleton

```rust
#[test]
fn verify_vp_022_exit_code_semantics() {
    // No block_intent → exit 0
    assert_eq!(exit_code_from_summary(&summary_no_block), 0);

    // block_intent=true → exit 2
    assert_eq!(exit_code_from_summary(&summary_with_block), 2);

    // Registry error → exit 0 (non-blocking)
    assert_eq!(exit_code_on_registry_error(), 0);

    // Engine error → exit 0 (non-blocking)
    assert_eq!(exit_code_on_engine_error(), 0);
}
```

## Notes

The exit code semantics are the primary signaling mechanism to Claude Code: exit 2 blocks the tool call; exit 0 allows it. This is the most safety-critical exit path in the system. The integration tests (executor_integration.rs) verify summary.exit_code but not the final process exit code. A dedicated test for the main::run → process::exit path is needed.
