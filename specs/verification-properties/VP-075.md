# VP-075: Context-Injection Determinism

## Property Statement

Given **identical inputs**, a resolver's `resolve()` function returns **identical
output**. Formally:

For all `input: ResolverInput`:
```
resolve(input) == resolve(input)
```

Where `==` means structural equality of `ResolverOutput { key, value }`.

This property applies to the **pure computation portion** of the resolver — the
function that processes already-read data and produces a `ResolverOutput`. File I/O
is excluded from the determinism assertion (filesystem state is an external input;
mocking it as in-memory fixtures makes the determinism property testable).

Additionally, the dispatcher's **merging algorithm** is deterministic:

For all `plugin_config: Value` and `resolver_output: ResolverOutput`:
```
merge(plugin_config, resolver_output) == merge(plugin_config, resolver_output)
```

The merge result is a pure function of its two inputs. Registry declaration order
is the tie-breaker for key collisions (first-declared resolver wins; a
`resolver.merge_collision` event is emitted on collision but the result is
deterministic given the same registry order).

## Rationale for this VP

Determinism is foundational to the correctness of the resolver platform.
Non-deterministic resolution means that the same dispatch context may produce
different hook behaviors on different invocations — violating the reproducibility
requirement for convergence gate decisions. The convergence hook
(`validate-per-story-adversary-convergence`) must produce the same block/continue
decision for the same wave state. If the `WaveContextResolver` returns different
story lists on different calls with the same input, the convergence gate is
non-reproducible.

The proptest approach is well-suited here: the determinism property is a
self-consistency check (call with same input twice, assert equal output) that is
trivially constructable as a proptest property with a broad input strategy. 200 trials
with bounded string lengths provides confidence across a wide range of input shapes.

## Formal Property Statement (proptest)

```rust
// VP-075 proptest — lives in
// crates/factory-dispatcher/tests/resolver_determinism_proptest.rs
#[cfg(test)]
mod vp_075_context_injection_determinism {
    use proptest::prelude::*;
    use hook_sdk::resolver::{ResolverInput, ResolverOutput};
    use factory_dispatcher::resolver::{merge_resolver_output};

    /// Strategy for generating arbitrary ResolverInput instances.
    /// Bounded string lengths to avoid pathological inputs.
    fn arb_resolver_input() -> impl Strategy<Value = ResolverInput> {
        (
            "[a-z_]{1,32}",                      // event_type
            "[a-z_]{1,48}",                      // hook_event_name
            proptest::option::of("[a-z]{1,16}"), // agent_type: Option<String>
            "/[a-z0-9/]{1,64}",                  // project_dir
            proptest::collection::hash_map(
                "[a-z_]{1,16}",                  // plugin_config key
                proptest::arbitrary::any::<serde_json::Value>().prop_filter(
                    "non-null JSON values only",
                    |v| !v.is_null()
                ),
                0..8
            ).prop_map(|m| serde_json::Value::Object(m.into_iter().collect()))
        ).prop_map(|(event_type, hook_event_name, agent_type, project_dir, plugin_config)| {
            ResolverInput {
                event_type,
                hook_event_name,
                agent_type,
                project_dir,
                plugin_config,
            }
        })
    }

    /// VP-075-A: The pure computation portion of resolve_wave_context is deterministic.
    /// Given the same ResolverInput (with file I/O mocked), two calls produce equal output.
    proptest! {
        #![proptest_config(proptest::test_runner::Config {
            cases: 200,
            timeout: 5000,  // 5s timeout per trial
            ..Default::default()
        })]

        #[test]
        fn prop_resolve_is_deterministic(input in arb_resolver_input()) {
            use crates::vsdd_context_resolvers::resolve_wave_context_pure;
            // resolve_wave_context_pure is the pure computation function with
            // mocked file I/O (accepts pre-parsed wave-state as an argument).
            let wave_state = test_fixtures::sample_wave_state();

            let output_a = resolve_wave_context_pure(&input, &wave_state);
            let output_b = resolve_wave_context_pure(&input, &wave_state);

            prop_assert_eq!(
                output_a, output_b,
                "resolve_wave_context_pure must return identical output for identical inputs"
            );
        }
    }

    /// VP-075-B: The dispatcher merging algorithm is deterministic.
    /// Given the same plugin_config and ResolverOutput, merge always produces the same result.
    proptest! {
        #![proptest_config(proptest::test_runner::Config {
            cases: 200,
            timeout: 5000,
            ..Default::default()
        })]

        #[test]
        fn prop_merge_is_deterministic(
            base_config in proptest::arbitrary::any::<serde_json::Value>()
                .prop_filter("must be an object", |v| v.is_object()),
            key in "[a-z_]{1,32}",
            value in proptest::arbitrary::any::<serde_json::Value>()
                .prop_filter("non-null", |v| !v.is_null())
        ) {
            let resolver_output = ResolverOutput {
                key: key.clone(),
                value: Some(value),
            };

            let merged_a = merge_resolver_output(base_config.clone(), &resolver_output);
            let merged_b = merge_resolver_output(base_config.clone(), &resolver_output);

            prop_assert_eq!(
                merged_a, merged_b,
                "merge_resolver_output must return identical output for identical inputs"
            );
        }
    }

    /// VP-075-C: Merge is additive — base_config fields are preserved.
    /// Fields in base_config that are NOT the resolver's key must be present in the merged result.
    proptest! {
        #![proptest_config(proptest::test_runner::Config {
            cases: 100,
            timeout: 5000,
            ..Default::default()
        })]

        #[test]
        fn prop_merge_preserves_base_config_fields(
            base_config in proptest::collection::hash_map(
                "[a-z]{1,16}",
                proptest::arbitrary::any::<serde_json::Value>()
                    .prop_filter("non-null", |v| !v.is_null()),
                1..4
            ).prop_map(|m| serde_json::Value::Object(m.into_iter().collect())),
            resolver_key in "resolver_[a-z]{1,16}",  // distinct prefix avoids collision
            resolver_value in proptest::arbitrary::any::<serde_json::Value>()
                .prop_filter("non-null", |v| !v.is_null())
        ) {
            let resolver_output = ResolverOutput {
                key: resolver_key.clone(),
                value: Some(resolver_value),
            };

            let merged = merge_resolver_output(base_config.clone(), &resolver_output);
            let merged_obj = merged.as_object().expect("merged result must be a JSON object");

            // All base_config keys must still be present.
            if let serde_json::Value::Object(base_obj) = &base_config {
                for (k, _v) in base_obj {
                    prop_assert!(
                        merged_obj.contains_key(k.as_str()),
                        "base_config key {} must be preserved in merged output",
                        k
                    );
                }
            }

            // The resolver's key must also be present.
            prop_assert!(
                merged_obj.contains_key(resolver_key.as_str()),
                "resolver key must be present in merged output"
            );
        }
    }
}
```

## Expected Outcome

All proptest property tests pass across 200 trials (VP-075-A and VP-075-B) and 100
trials (VP-075-C). The determinism property is upheld for the full range of
`ResolverInput` shapes generated by the strategy. The merging algorithm preserves
base config fields and injects the resolver's key.

## Limitations

This VP covers the **pure computation portion** of the resolver. It does NOT verify:
- That the resolver reads the filesystem correctly (integration test concern)
- That two calls with different filesystem state produce different results (that
  would violate determinism given the definition — filesystem state IS part of the input)

The proptest uses in-memory fixture wave-state, not a real filesystem read. Real
filesystem integration is covered by the end-to-end integration test in S-12.08.
