# VP-067: Worktree Hook Plugin Surface Invariant — All BC-4.07.* Postconditions Hold in Integration Test

## Property Statement

For a fully wired worktree-hooks plugin under integration test conditions:

1. Loading `worktree-hooks.wasm` and sending a `WorktreeCreate` envelope through the dispatcher MUST result in exactly one `worktree.created` event captured in the file sink. The event contains 10 fields on the wire from three sources (WorktreeCreate analog of F-P7-02):

   **2 plugin-set fields** (verified as present and non-null): `worktree_path`, `worktree_name`. Both MUST be strings on the wire (per `emit_event.rs:49` string coercion — `.is_string()` not raw value matchers). When the envelope's `worktree_path` is absent, `worktree_path` MUST equal `""`. When the envelope's `worktree_name` is absent, `worktree_name` MUST equal `""`.

   **4 host-enriched fields** (auto-injected by `emit_event` host fn from `HostContext`; plugin does NOT set these): `dispatcher_trace_id`, `session_id`, `plugin_name`, `plugin_version`. Each MUST be present as a non-empty string (per BC-1.05.012 unconditional enrichment for presence).

   **4 construction-time fields** (set by the dispatcher between the plugin's `emit_event` call and the final wire format; the plugin must NOT set them — implementation provenance is opaque from the spec layer): `ts`, `ts_epoch`, `schema_version`, `type`. These MUST be present; `type` MUST equal `"worktree.created"`.

   **Wire format note:** Plugin-set field values arrive as JSON strings on the wire (`emit_event.rs:49` coerces all plugin-supplied values to `Value::String`). Harness assertions for `worktree_path` and `worktree_name` MUST use `.is_string()`.

   **WorktreeCreate field count:** 10 fields (2+4+4). Compare: SessionStart=14 (6+4+4), SessionEnd=11 (3+4+4), WorktreeCreate=10 (2+4+4), WorktreeRemove=9 (1+4+4). (BC-4.07.001)

2. Loading `worktree-hooks.wasm` and sending a `WorktreeRemove` envelope through the dispatcher MUST result in exactly one `worktree.removed` event captured in the file sink. The event contains 9 fields on the wire:

   **1 plugin-set field** (verified as present and non-null): `worktree_path`. MUST be a string on the wire. When the envelope's `worktree_path` is absent, `worktree_path` MUST equal `""`. When the envelope's `worktree_path` identifies a worktree not previously registered (unknown worktree — EC-002 in BC-4.07.002), the plugin MUST still emit `worktree.removed` normally — no error, no abort.

   **4 host-enriched fields** and **4 construction-time fields**: identical to WorktreeCreate above, except `type` MUST equal `"worktree.removed"`. (BC-4.07.002)

3. The `exec_subprocess` CountingMock invocation count MUST equal exactly 0 for BOTH `WorktreeCreate` and `WorktreeRemove` dispatches. The worktree-hooks plugin MUST NOT call `exec_subprocess` at any point during either event's handling. (BC-4.07.001 Invariant 2, BC-4.07.002 Invariant 2)

4. `hooks.json.template` MUST contain both a `WorktreeCreate` key and a `WorktreeRemove` key under `hooks`. For EACH event:
   - The entry's `command` field MUST contain `"factory-dispatcher"` (dispatcher binary, NOT a `.wasm` filename).
   - The entry MUST NOT carry `"once": true` (worktree events can re-fire — this is the critical difference from session events).
   - The entry MUST have `"async": true`.
   - The harness assertion is:
   ```rust
   for event_key in &["WorktreeCreate", "WorktreeRemove"] {
       let arr = template["hooks"][event_key].as_array()
           .unwrap_or_else(|| panic!("{event_key} must be an array in hooks.json.template"));
       assert!(!arr.is_empty(), "{event_key} array must have at least one entry");
       let entry = &arr[0]["hooks"][0];
       let command = entry["command"].as_str()
           .unwrap_or_else(|| panic!("{event_key} command must be a string"));
       assert!(command.contains("factory-dispatcher"),
           "{event_key} command must reference dispatcher binary, not a WASM plugin");
       assert!(!command.contains(".wasm"),
           "{event_key} command must NOT reference a WASM plugin filename (ADR-011 layer separation)");
       // once key MUST BE ABSENT (not once:false, not once:true, not any value — key must not exist)
       // Claude Code default semantics: re-fire on reconnect when once key is absent.
       // Using is_none() rather than assert_ne!(once, Bool(true)) because Bool(false) and absent
       // have different semantics risk; defensive omission is the canonical pattern.
       assert!(entry.get("once").is_none(),
           "{event_key} must omit `once` key entirely (not `false`, not `true`); \
            Claude Code default semantics: re-fire on reconnect; explicit `false` semantics \
            are unspecified — defensive omission protects against future Claude Code parser changes");
       assert_eq!(entry["async"], true, "{event_key} async must be true");
   }
   ```
   (BC-4.07.003; per ADR-011 layer separation, once key ABSENT for re-fire semantics.)

5. `hooks-registry.toml` MUST contain BOTH a `WorktreeCreate` entry and a `WorktreeRemove` entry. For EACH entry:
   - `name` = `"worktree-hooks"`
   - `plugin` = `"hook-plugins/worktree-hooks.wasm"` (with `hook-plugins/` directory prefix)
   - `timeout_ms` = `5000`
   - NO capability tables (`entry.get("capabilities")` is `None` for both entries)
   - Exactly one entry per event type (duplicate-detection: `worktree_create_count == 1`, `worktree_remove_count == 1`)
   - Neither entry carries a `once` field — `RegistryEntry` has no such field; `deny_unknown_fields` rejects it.
   (BC-4.07.004)

## Notes

**Feasibility status: `feasible-pending-harness`**

VP-067 is an integration-level verification property that exercises the full plugin invocation path for both worktree events: WASM load → dispatcher routing → host function calls (emit_event only; NO exec_subprocess) → file sink capture → payload shape validation. The integration test harness structure mirrors VP-066.

**S-5.01 + S-5.02 lessons applied from the start (2026-04-28):**

- **Zero-capability profile (simplest possible):** Unlike VP-065 (which needs FactoryHealthMock for subprocess) and VP-066 (which needs CountingMock only for assertion), VP-067's CountingMock is present only to assert `invocation_count == 0` for both events. No subprocess path exists. No read_file mock needed.
- **Two events, one WASM binary:** Both WorktreeCreate and WorktreeRemove dispatch to `worktree-hooks.wasm`. The harness runs two separate dispatch calls (one per event) and verifies independent event emission.
- **`once` key ABSENT assertion on hooks.json.template:** Unlike VP-066 which asserts `once: true`, VP-067 asserts `entry.get("once").is_none()` — the once key must be completely absent. This is the critical behavioral inversion for worktree events. Using `is_none()` rather than `assert_ne!(once, Bool(true))` because `Bool(false)` and absent have different semantics risk under future Claude Code versions; defensive omission is the canonical pattern (HIGH-002 fix, ADV-S5.03-P01).
- **No `capabilities` table assertion (positive absence):** Same as VP-066 — asserts `entry.get("capabilities").is_none()` for BOTH registry entries.
- **No `once` field assertion on registry entries:** Same as VP-066 — `RegistryEntry` has no such field; `deny_unknown_fields` rejects it. Test does not assert `once` on either registry entry.
- **9-field vs. 10-field assertion:** WorktreeRemove emits 9 fields (1+4+4); WorktreeCreate emits 10 (2+4+4). Total field count assertions are event-specific.

**Remaining delivery-time blockers:**

1. **hooks.json.template WorktreeCreate + WorktreeRemove keys:** Must confirm both keys exist in the template. If absent at delivery time, Task 4 of S-5.03 covers adding them.
2. **hooks-registry.toml two new entries:** Task 5 of S-5.03 covers adding the two `[[hooks]]` entries. Test confirms both are present after edit.
3. **workspace_root resolver:** Same decision as VP-065/VP-066 (runtime walker vs. build.rs compile-time env var); document choice in test file.
4. **Internal dispatch on event_name:** The single `worktree-hooks.wasm` crate must dispatch internally based on the `event_name` field in the HookPayload envelope (per HOST_ABI.md and BC-1.02.001/002): `WorktreeCreate` → `worktree_path + worktree_name`; `WorktreeRemove` → `worktree_path` only. Harness verifies the output, not the internal dispatch logic.
